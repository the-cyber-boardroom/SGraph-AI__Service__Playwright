# Elastic + Kibana on EC2 — Runbook
**Version:** v0.1.72  
**Date:** 2026-04-23  
**Status:** PROPOSED — nothing described here exists yet  
**Audience:** DevOps, SRE

---

## Purpose

Run Elasticsearch + Kibana on a dedicated EC2 instance as a side-by-side
alternative (and comparison target) to the OpenSearch / OSD stack managed
by `ob create`.  Both stacks receive the same log stream; dashboards and
query UX can be compared directly.

This document covers provisioning only.  Dashboard authoring and comparison
is a separate workstream.

---

## Licence note

Elasticsearch 7.x and below (up to 7.10.2) was Apache 2.0.  
Elasticsearch 8.x is dual-licensed: **SSPL** (non-OSI) or **Elastic Licence 2.0**.  
Kibana follows the same split.

For internal tooling use, Elastic Licence 2.0 is acceptable (no SaaS
redistribution).  Use the official `docker.elastic.co` images.  Do not
redistribute or offer as a managed service.

## Architecture

```
┌────────────────────────────────────────────────────────────────┐
│  EC2 t3.medium  (elastic-kb-{stack_name})                      │
│                                                                │
│  Docker Compose                                                │
│  ┌──────────────────┐  ┌──────────────────┐                   │
│  │  elasticsearch   │  │     kibana        │                   │
│  │  :9200           │  │  :5601            │                   │
│  └──────────────────┘  └──────────────────┘                   │
│  ┌──────────────────┐                                          │
│  │   fluent-bit     │  ← same log stream as OpenSearch         │
│  │  → ES :9200      │                                          │
│  └──────────────────┘                                          │
│                                                                │
│  nginx → Kibana :5601 (HTTPS via ACM / self-signed)           │
└────────────────────────────────────────────────────────────────┘
         ↑
  SSH / SSM  (no public SSH key required)
```

Kibana is exposed via an nginx reverse proxy on port 443.  The EC2
instance has no inbound SSH; access is via SSM Session Manager.

## EC2 sizing

| Component | Minimum | Recommended |
|---|---|---|
| Instance | `t3.medium` (2 vCPU, 4 GB) | `t3.large` (2 vCPU, 8 GB) |
| Root EBS | 20 GB gp3 | 30 GB gp3 |
| Data EBS | 20 GB gp3 | 50 GB gp3 |
| OS | Amazon Linux 2023 | same |

Elasticsearch needs at minimum 1 GB JVM heap.  The default Docker image
uses 50 % of container memory limit.  Set `ES_JAVA_OPTS=-Xms1g -Xmx1g`
and give the container a 2 GB memory limit.  This fits on a `t3.medium`
alongside Kibana (~512 MB) and fluent-bit (~64 MB).

---

## IAM role

Reuse the `playwright-ec2` role (already has SSM access).  No additional
permissions needed — Elasticsearch is not an AWS service.

If you want Kibana to pull metrics from CloudWatch, attach
`CloudWatchReadOnlyAccess` additionally.

## Security Group

| Port | Source | Purpose |
|---|---|---|
| 443 | `0.0.0.0/0` | Kibana UI (nginx) |
| 9200 | SG of playwright EC2 | Elasticsearch API (fluent-bit ingest) |
| 9300 | self | ES cluster transport (single-node, still needed) |

No port 22.  SSM only.

## Docker Compose

```yaml
# /opt/elastic/docker-compose.yml
# Rendered by provision script; do not edit by hand.

services:

  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.13.4
    environment:
      - discovery.type=single-node
      - ES_JAVA_OPTS=-Xms1g -Xmx1g
      - ELASTIC_PASSWORD=${ELASTIC_PASSWORD}
      - xpack.security.enabled=true
      - xpack.security.http.ssl.enabled=false   # nginx terminates TLS
    ulimits:
      memlock: {soft: -1, hard: -1}
      nofile:  {soft: 65536, hard: 65536}
    volumes:
      - es_data:/usr/share/elasticsearch/data
    ports:
      - "9200:9200"
    restart: unless-stopped
    mem_limit: 2g

  kibana:
    image: docker.elastic.co/kibana/kibana:8.13.4
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
      - ELASTICSEARCH_USERNAME=kibana_system
      - ELASTICSEARCH_PASSWORD=${KIBANA_SYSTEM_PASSWORD}
      - SERVER_BASEPATH=
      - XPACK_SECURITY_ENABLED=true
    ports:
      - "5601:5601"
    depends_on: [elasticsearch]
    restart: unless-stopped

  fluent-bit:
    image: fluent/fluent-bit:3.2
    volumes:
      - /opt/elastic/fluent-bit.conf:/fluent-bit/etc/fluent-bit.conf:ro
      - /var/lib/docker/containers:/var/lib/docker/containers:ro
    depends_on: [elasticsearch]
    restart: unless-stopped

  nginx:
    image: nginx:1.27-alpine
    ports:
      - "443:443"
      - "80:80"
    volumes:
      - /opt/elastic/nginx.conf:/etc/nginx/conf.d/default.conf:ro
      - /opt/elastic/certs:/etc/nginx/certs:ro
    depends_on: [kibana]
    restart: unless-stopped

volumes:
  es_data:
```

## fluent-bit dual-output pattern

To ship the same logs to **both** OpenSearch and Elasticsearch
simultaneously, add a second `[OUTPUT]` block in the fluent-bit config
on the Playwright EC2:

```ini
# Existing output — OpenSearch
[OUTPUT]
    Name            opensearch
    Match           *
    Host            ${OPENSEARCH_ENDPOINT}
    Port            443
    TLS             On
    HTTP_User       ${OB_OS_ADMIN_USER}
    HTTP_Passwd     ${OB_OS_ADMIN_PASS}
    Index           sg-playwright-logs
    Suppress_Type_Name On

# New output — Elasticsearch (same index name for apples-to-apples)
[OUTPUT]
    Name            es
    Match           *
    Host            ${ELASTIC_EC2_IP}
    Port            9200
    HTTP_User       elastic
    HTTP_Passwd     ${ELASTIC_PASSWORD}
    Index           sg-playwright-logs
    Suppress_Type_Name On
```

No changes to log producers.  Both stacks receive identical data.

## Kibana saved objects

Kibana uses a different saved-objects API from OpenSearch Dashboards but
the concept is the same.  Import via:

```
POST /api/saved_objects/_import?overwrite=true
Content-Type: multipart/form-data
```

The `opensearch-instance-lifecycle.ndjson` file is **not** directly
compatible — Kibana 8.x uses `lens` visualisations instead of the legacy
`visualization` type.  A separate `kibana-instance-lifecycle.ndjson` will
need to be authored once the stack is running.

## Provisioning script — new command `elastic-ec2 create`

The simplest path is a new `scripts/elastic_ec2.py` that mirrors
`provision_ec2.py`:

```
sp elastic-ec2 create {name}    # provision + start containers
sp elastic-ec2 delete {name}    # terminate instance
sp elastic-ec2 info   {name}    # show endpoint + credentials
```

Key differences from `provision_ec2.py`:

| Concern | playwright EC2 | elastic EC2 |
|---|---|---|
| Docker Compose | Playwright + mitmproxy + prometheus | Elasticsearch + Kibana + fluent-bit + nginx |
| Instance type | `t3.medium` (existing) | `t3.medium` or `t3.large` |
| IAM role | `playwright-ec2` | same role (reuse) |
| Ports inbound | 80, 443 | 80, 443, 9200 |
| Data volume | none (ephemeral) | `/dev/xvdf` → `/var/lib/docker` volumes |
| Credential output | `OPENSEARCH_ENDPOINT` | `ELASTIC_ENDPOINT`, `ELASTIC_PASSWORD` |

Password generation reuses `_generate_os_password()` from `observability.py`.

## Ephemeral operation model

Like the OpenSearch managed stack, this EC2 should be treated as
**ephemeral** — created in the morning, deleted in the evening, data
preserved via snapshot or backup between runs.

Elasticsearch snapshot API (`PUT /_snapshot/...`) is the native backup
path.  Target: S3 bucket `sg-playwright-elastic-snapshots`.  Register the
repository once; snapshot daily before deletion.

```bash
# Register repo (one-time)
curl -X PUT "http://localhost:9200/_snapshot/s3_backup" \
  -H 'Content-Type: application/json' \
  -d '{"type":"s3","settings":{"bucket":"sg-playwright-elastic-snapshots","region":"eu-west-2"}}'

# Snapshot (before deletion)
curl -X PUT "http://localhost:9200/_snapshot/s3_backup/$(date +%Y-%m-%d)"

# Restore (after creation next morning)
curl -X POST "http://localhost:9200/_snapshot/s3_backup/$(date +%Y-%m-%d)/_restore"
```

The EC2 provisioning script should run snapshot + restore automatically as
part of `elastic-ec2 create` (restore latest) and as a pre-terminate hook
in `elastic-ec2 delete`.

## Comparison matrix (OpenSearch vs Elastic — for reference)

| Feature | OpenSearch 3.5 (AWS managed) | Elasticsearch 8.x (self-hosted EC2) |
|---|---|---|
| Licence | Apache 2.0 | Elastic Licence 2.0 |
| Provisioning time | 15-20 min | ~2 min (Docker) |
| Cost model | Per-hour AWS instance + storage | EC2 + EBS only |
| TLS | AWS-managed | nginx termination |
| Auth | FGAC + basic | xpack security + basic |
| Dashboards | OpenSearch Dashboards 3.x | Kibana 8.x |
| Snapshot target | S3 (native) | S3 (native, needs plugin 8.x) |
| Fluent-bit output plugin | `opensearch` | `es` |
| API compatibility | ES 7.x compatible | ES 8.x |
| Long-term managed | Yes (AWS handles upgrades) | Manual (we own the instance) |
