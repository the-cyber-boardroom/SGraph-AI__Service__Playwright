## Section 1 — The Shipped Foundation (SG_Send__QA)

### 1.1 Same-image-multiple-runtimes (the headline unlock)

**File:** `sg_send_qa/api/Fast_API__QA__Server.py`

A single class extends `Serverless__Fast_API` and exposes both entry points from module-level singletons:

```python
_fast_api = Fast_API__QA__Server().setup()

handler = _fast_api.handler()   # Mangum-wrapped ASGI — Lambda direct invoke
app     = _fast_api.app()       # raw ASGI — uvicorn / Lambda Web Adapter

def run(event, context=None):   # Lambda entry point
    return handler(event, context)
```

This is **exactly** the pattern the 16 April arch brief asks for. Deploys to:

- Local: `uvicorn sg_send_qa.api.Fast_API__QA__Server:app --reload`
- Lambda direct invoke: `HANDLER=sg_send_qa.api.Fast_API__QA__Server.run`
- Lambda Web Adapter: uses `app` — **adapter auto-wraps when `AWS_LAMBDA_RUNTIME_API` env var is present**; no code branch needed
- Starlette TestClient for unit tests: `fast_api.client()`

**What the new service needs on top:** nothing architecturally. This class gets renamed and its routes are replaced.

### 1.2 Stateless-per-request browser session

**File:** `sg_send_qa/api/QA_API__Session.py`

Context manager. `__enter__` boots a fresh browser (local harness or live-URL Playwright), `__exit__` tears down. Comment from the source:

> *"Each request gets a fresh session: browser + servers (local) or browser only (live). Nothing survives between requests. Lambda-compatible by design."*

Handles:
- Browser mode dispatch (`local` vs live URL)
- Access token injection via localStorage in the page context
- Trace ID propagation to all browser HTTP requests via `X-QA-Trace-ID`
- Recording UI state transitions during the workflow (`transitions_observed` list)

**What the new service needs on top:** replace the local-vs-live dispatch with vault-reference dispatch. Swap access token injection for vault-loaded cookies/profile.

### 1.3 Runner orchestration

**File:** `sg_send_qa/api/QA_API__Runner.py`

Single entry point for all workflows. Handles:
- Trace ID auto-generation (`uuid.uuid4().hex[:8]`)
- Capture config defaults populated based on mode (`smoke` vs `qa`)
- Session lifecycle wrapping (`with QA_API__Session(request=...) as session:`)
- Timing (`duration_ms`)
- Error-to-response conversion (`status: 'fail'`, `error: str(exc)`)

Route handlers are thin wrappers that pass a `workflow_fn` closure to `runner.run(request, workflow_fn)`. Clean separation.

**What the new service needs on top:** add a `run_sequence(request, steps)` method that dispatches each step to a registered handler — this is where the 16 April brief's declarative meta-language plugs in.

### 1.4 Capture config — the superset of what 16 April asks for

**File:** `sg_send_qa/api/Schema__Capture_Config.py`

Type_Safe schema with **12 toggleable fields**:

| Flag | Default | What it captures |
|---|---|---|
| `execution_result` | True | pass/fail, step timing, error messages |
| `page_model` | True | `Schema__*Page` snapshot of final page state |
| `screenshots` | False | screenshot at each step (base64 PNG array) |
| `screenshot_on_fail` | True | screenshot only on step failure |
| `video` | False | full browser session recording (base64 webm) |
| `pdf` | False | PDF export of final page state (base64) |
| `browser_console` | False | console.log/warn/error messages |
| `browser_trace` | False | Playwright trace file (base64 zip) |
| `python_console` | False | Python stdout/stderr during execution |
| `python_trace` | False | Python traceback on failure |
| `performance_data` | True | per-step duration, total, browser metrics |
| `prometheus` | True | Prometheus exposition format metrics |
| `network_log` | False | all HTTP requests/responses |

The 16 April arch brief only lists screenshots and video explicitly. **Inherit this schema wholesale**; it already encodes the right separation of concerns.

**Change for the new service:** each of these artefacts goes to a vault reference (`output_vault_path`) rather than base64 in the response. `base64_inline` becomes a separate flag for small payloads where round-trip-via-vault is overkill.

### 1.5 4-layer routes organisation

**Folder:** `sg_send_qa/api/routes/`

| Layer | File | Endpoint pattern | Use |
|---|---|---|---|
| **Layer 0 — Raw** | `routes__browser.py` | `POST /browser/open`, `POST /browser/screenshot` | Building blocks. Maximum granularity. |
| **Layer 2 — Features** | `routes__upload.py`, `routes__download.py` | `POST /upload/combined`, `POST /upload/friendly_token`, `POST /upload/separate_key` | Feature-scoped workflows. |
| **Layer 3 — Personas** | `routes__workflow.py` | `POST /workflow/persona_ab__send_and_receive` | Full user-journey replays. |

Layer 1 (page-level operations) is designed but not yet populated.

**What the new service needs on top:** Layer 0 extends naturally to the arch brief's Direct Action Endpoints (`/browser/navigate`, `/browser/click`, `/browser/fill`, `/browser/evaluate`, `/browser/video/start|stop`). The arch brief's `POST /sequence/execute` is a **new Layer 3 endpoint** that takes declarative JSON steps instead of invoking a specific persona workflow.

### 1.6 Page objects per site

**Folder:** `sg_send_qa/apis_for_sites/send_sgraph_ai/pages/`

Namespaced page object wrappers per target site. Currently only SG/Send. The namespacing hints at the pattern the new service should use: **per-site page-object packages loaded from vault**, so the same container can drive any site (OpenRouter, GitHub, Cyber Boardroom, etc.) without an image rebuild.

### 1.7 State machines

**Folder:** `sg_send_qa/state_machines/`

`State_Machine__Upload`, `State_Machine__Download`, `State_Machine__Snapshot`, `State_Machine__Snapshot__Diff`. The diff machinery is relevant: for the arch brief's sequence mode, we can validate "did this step produce the expected state transition?" rather than only "did this step throw?"

### 1.8 Proven at runtime inside Claude sessions

Reality doc line 89 (main repo): *"Playwright headless Chromium confirmed working in Claude.ai web sessions. Full SG/Send QA suite runs (25 tests in ~44 seconds). Screenshot capture works (18 PNGs)."*

**Takeaway:** the exact stack the new service will package (Python 3.12 + Playwright 1.56 + Chromium + FastAPI) already runs green end-to-end. Containerisation is a packaging task, not a feasibility one.

### Phase 2 — Lambda Container (ready to lift)

The debrief contains the **actual boot loader code**:

```python
# boot_loader.py (in container)
import os, sys, tempfile, zipfile, boto3, importlib

bucket    = os.environ['CODE_ZIP_S3_BUCKET']
key       = os.environ['CODE_ZIP_S3_KEY']
handler_  = os.environ.get('HANDLER', 'main.run')

# Download + extract
tmp = tempfile.mkdtemp()
s3  = boto3.client('s3')
s3.download_file(bucket, key, f'{tmp}/code.zip')
with zipfile.ZipFile(f'{tmp}/code.zip') as z:
    z.extractall(f'{tmp}/code')
sys.path.insert(0, f'{tmp}/code')

# Import and expose handler
module_name, fn_name = handler_.rsplit('.', 1)
module  = importlib.import_module(module_name)
handler = getattr(module, fn_name)
```

| Component | Source | Notes |
|---|---|---|

## Section 3 — The Gap: From QA Service to General Playwright Service

### What MUST change

| QA service (today) | New Playwright service |
|---|---|
| Request `target: 'local'\|<live-url>` dispatch | Request `vault_config: vault://<id>/config.json` dispatch |
| Access token injected via `session.sg_send.storage__set_token()` | Cookies/profile loaded from `vault://<id>/session/*` |
| SG/Send-specific page objects hardcoded | Page objects loaded from vault (per-site) |
| Artefacts returned as base64 in response body | Artefacts committed to output vault; response contains vault refs |
| Workflows as Python closures (`workflow_fn`) | Workflows **also** as declarative JSON (`steps: [...]`) |
| Hardcoded route handlers per feature | Step-handler registry (navigate, click, fill, screenshot, wait_for, evaluate, ...) dispatched from `/sequence/execute` |
| QA-specific auth model | Auth model per architect constraint (04/09): scripts from trusted sources only |

- `Fast_API__QA__Server` → rename to `Fast_API__Playwright__Server`, keep the pattern
- `QA_API__Session` context manager → rename, keep the lifecycle
- `QA_API__Runner` → rename, add `run_sequence()` alongside `run()`
- `Schema__Capture_Config` → use wholesale

- 4-layer routes pattern → use wholesale, extend Layer 0 to full Playwright action surface, add `/sequence/execute` at Layer 3
- State machine validation → optional, but available for sequence step validation
- 26 March boot loader code → lift verbatim

## Section 4 — Hard-Won Knowledge (must survive into the new service)

### 4.1 Playwright does NOT auto-pick-up `HTTPS_PROXY` env vars

**Source:** `SG_Send__QA/team/humans/dinis_cruz/debriefs/03/29/v0.2.40__scheduled-task__qa-daily-smoke.md`

> *"Playwright does not pick up `https_proxy` env vars automatically; the proxy must be passed explicitly to `p.chromium.launch(proxy=...)`. The proxy also performs TLS interception, so `ignore_https_errors=True` is required on the browser context."*

This bit the team in the Claude cloud environment (proxy at `21.0.0.25:15004`). `curl` works because it reads env vars natively. Playwright does not.

**Implication for the new service:** when the vault config contains proxy settings, they must be **explicitly** mapped to the `chromium.launch(proxy=...)` call and `ignore_https_errors=True` must be conditionally set on the browser context for TLS-intercepting proxies.

### 4.2 Python 3.12 pinning is non-negotiable

The main repo stack is Python 3.12 (`.claude/CLAUDE.md` stack table). The QA repo proves the Playwright + osbot stack is tested against 3.12. Default Claude cloud task Python is 3.11, which doesn't work — the 03/29 debrief creates a 3.12 venv at `/tmp/venv312` explicitly.

**Implication for the new service:** Dockerfile base MUST be `python:3.12-slim` (as specified in the 26 March debrief). Not 3.11, not 3.13.

### 4.3 Lambda Web Adapter auto-wrap

**Source:** v0.2.37 debrief, 26 March.

> *"Lambda Web Adapter wraps `app` automatically when `AWS_LAMBDA_RUNTIME_API` is present and the extension is configured. No code branch needed."*

**Implication:** the arch brief's `USE_WEB_ADAPTER` toggle from the 25 March brief is obsolete. One image, zero code branches. Presence of the Lambda extension does the switch.

### 4.4 Cold start budget

5–15s for a headless Chromium Lambda cold start. This is the cost of the ~1 GB image. If the caller needs <5s response guarantee, use Fargate (warm) or keep Lambda provisioned concurrency. **Document this in the service's README so no one is surprised.**

### 4.5 Access to `dev.tools.sgraph.ai` and `openrouter.ai` from Playwright

**Source:** 04/08 QA browser automation debrief.

Network egress in the Claude cloud environment is allowlisted. The allowlist **must include** any domain the service needs to navigate to from inside the container. Config changes only apply to new sessions. This is infrastructure, not code.

### 4.6 Ports must bind to `localhost` not `127.0.0.1` for local dev

**Source:** 04/08 QA debrief.

> *"UI static file server — `python3 -m http.server` on localhost:10062 (must bind to `localhost` not `127.0.0.1` for Web Crypto secure context)"*

Only relevant for local dev mode, but a landmine if repeated.

### 4.7 osbot-fast-api-serverless as base class

Both repos use `Serverless__Fast_API` from `osbot-fast-api-serverless`. This library provides:
- `.handler()` — Mangum-wrapped Lambda handler
- `.app()` — raw ASGI app for uvicorn / Lambda Web Adapter
- `.client()` — Starlette TestClient for unit tests
- `add_routes(RouteClass, **injected_deps)` — dependency injection without `Depends()`
- Config via Type_Safe

**Implication for the new service:** do NOT use vanilla FastAPI. Use `Serverless__Fast_API` for consistency and because it already solves the multi-runtime problem.

### 4.8 No mocks policy

Main repo stack rule: *"No mocks, no patches — full stack starts in-memory in ~100ms."* Enforced across both repos. New service must follow same rule — tests hit a real browser against a real in-memory stack.

## Section 6 — Open Questions the Dev Brief Must Resolve

### Architecture

1. **Service code source of truth.** S3 zip (25 March pattern) vs vault reference (16 April pattern) vs both? The boot loader code is S3-shaped today; vault adds bootstrap complexity (container needs a vault key to start).
2. **Single image with `ENTRYPOINT` switching** vs two images built from the same Dockerfile? 26 March debrief implies single image using Lambda extension detection. 16 April brief says single image. Confirm.
3. **Stateful-within-container session model.** `POST /session/create` returns `session_id`; subsequent `POST /session/{id}/...` calls need an in-memory session map. Works within a single warm Lambda / long-lived Fargate task. Breaks across Lambda cold starts. Decision: accept this, or push sessions to vault after every action (expensive)?
4. **Repo layout.** New package inside `sgraph_ai_app_send/lambda__playwright/`, new top-level package, or new repo? QA repo precedent says new repo is fine.

### Security (architect constraint from 04/09 applies)

5. **Script trust model.** Confirm: the `script` field of `POST /sequence/execute` or `POST /session/{id}/execute` is a **list of declarative steps** (safe) not arbitrary JavaScript (unsafe). `POST /browser/evaluate` that runs JS in the page context must be restricted to a whitelist of evaluation patterns, or disabled in production modes.
6. **Service auth.** Mirror `HEADER__SGRAPH_SEND__ACCESS_TOKEN` from main repo, or new scheme?
7. **Vault capability model.** The request contains a vault reference; does the container verify the caller owns that vault, or does it trust and read? If trust-and-read, the vault reference is itself the auth token.

### Runtime

8. **Concurrent sessions per container.** 16 April brief leans toward one. Confirm. Implication: Fargate scales on session count; Lambda scales via invocation model naturally.
9. **Timeout routing Lambda vs Fargate.** 16 April brief leaves this open. Proposals: caller specifies `deployment_hint: fast|long`; service rejects `long` on Lambda deploys; or service routes based on `timeout_seconds` in request.
10. **Chrome profile persistence.** 16 April brief leans fresh-per-session with vault-stored profile as optional input. Confirm and specify the vault path convention.
11. **Video encoding.** 16 April brief leans WebM only (Playwright default, supported by the video playback component). Confirm.

### CI/CD

12. **Version alignment.** SG_Send__QA has its own version (v0.2.49) separate from main repo (v0.20.55). New service version: own line, or pinned to main repo, or pinned to QA?
13. **Release cadence.** Lambda base image rebuild vs service code zip update — different cadences, different CI workflows.
