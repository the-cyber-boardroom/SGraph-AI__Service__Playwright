## Headline

**Dinis already built this, roughly two years ago.** The OSBot-Playwright repo is a working, production-deployed implementation of **exactly** the architecture the 16 April arch brief specifies: Microsoft Playwright base image + AWS Lambda Web Adapter + FastAPI service + Lambda deployment orchestration + ECR publish + local dev container. It is version-drifted (Playwright 1.40 vs current 1.58; LWA 0.7.1 vs current 1.0.0+; Ubuntu jammy vs noble) but the architectural bones are solid and production-proven.

**Implication for the dev brief:** the task is *refresh and extend*, not *design and build*. The Dockerfile, ECR push logic, Lambda deployment orchestration, local dev workflow, browser process management, and FastAPI scaffolding are already code. What's missing is the vault integration, the declarative sequence meta-language endpoint, the 12-flag capture config (from SG_Send__QA), and the S3-zip code decoupling layer.

**Status:** deployed, tested (integration tests pass Lambda invocations), in use as of repo's most recent commits

```dockerfile

#CMD exec uvicorn --host='0.0.0.0' --reload  --port=$PORT main:app

```

**What this confirms from the research:**
1. ✅ Microsoft Playwright base image — chosen over AWS base + custom install
2. ✅ Lambda Web Adapter as the integration layer — chosen over `awslambdaric`/Mangum
3. ✅ Single image, multiple runtimes (Lambda + local `docker run`)
4. ✅ uvicorn as the server
5. ⚠️ **Version drift** — needs bump to `v1.58.2-noble` + LWA `1.0.0` in the refresh

**`handler.py`** — the entrypoint (boot script):

```python
import sys
sys.path.append('/opt/python')          # LWA loses the path to the layers

def run():
    from osbot_playwright.playwright.fastapi.Fast_API_Playwright \
        import Fast_API_Playwright
    fast_api_playwright = Fast_API_Playwright()
    app = fast_api_playwright.app()

    import uvicorn
    uvicorn.run(app, host="0.0.0.0", port=8000)

if __name__ == '__main__':
    run()
```

**Gotcha captured here:** `sys.path.append('/opt/python')` with comment *"this needs to be done since LWA loses the path to the layers"*. Lambda Web Adapter does not inherit the layer paths by default. Noted — must survive the refresh.

## Section 2 — The Deployment Orchestration (Already Code, Not Spec)

**Four classes, all in `osbot_playwright/docker/`:**

### `Build__Docker_Playwright`
- `build_docker_image()` — builds via `Create_Image_ECR.build_image()`
- `create_container() / start_container()` — local test container (ports 8000→8888)
- `dockerfile()` — reads Dockerfile contents for introspection

### `Local__Docker_Playwright`
- `create_or_reuse_container()` — idempotent container lifecycle using Docker labels (`source=local__docker_playwright`) for identification
- `wait_for_uvicorn_server_running()` — polls container logs for `"Uvicorn running on "` string; returns after max 40 attempts × 0.5s = 20s budget
- `GET(path)` / `POST(path, data)` — HTTP helpers against `http://localhost:8888{path}`
- `delete_container()` — teardown

This is the **full local dev loop**: build image → create container → wait for uvicorn → hit endpoints → inspect logs → teardown. Already working.

### `ECR__Docker_Playwright`
- `ecr_setup()` — creates ECR repo if missing
- `publish_docker_image()` — pushes image to ECR
- `check_for_docker_config_json()` — **a hard-won gotcha**: Docker Desktop's `.docker/config.json` with `{"credsStore": "desktop"}` breaks the Python Docker API's ECR login. The method detects and deletes the file (after logging a warning) before pushing. **Carry this forward.**

### `Lambda__Docker_Playwright`
- `create_lambda(delete_existing, wait_for_active)` — full lifecycle: delete if exists → create from image URI → wait for Active state → create Function URL
- `create_lambda_function_url()` — creates Lambda Function URL with public access; deletes existing first "due to a bug in AWS" (another captured gotcha)
- `update_lambda_function()` — updates existing Lambda to point at new image URI (for rebuild flows)
- `execute_lambda(payload)` — invokes programmatically

**Critical production-tuning signals baked in:**

```python
lambda_function.memory       = 5092    # 5 GB memory — not 2-3 GB I recommended
image_architecture           = 'x86_64' # ARM not supported at build time
lambda_function.set_env_variable(ENV__HTTP_SHELL_AUTH_KEY, ...)
```

The `5092 MB` memory setting is 60% higher than my research doc suggested. **Dinis hit memory issues and settled at 5 GB.** The dev brief should default to this, not the textbook 2-3 GB.

**Shipped endpoints:**

- `GET /playwright/screenshot?url=...` → navigates and returns PNG as `StreamingResponse`
- `GET /shell-server` → remote Python execution (auth-gated)

**Commented-out but historically significant:**

```python
# def code(self, code_data: CodeData):
```

## Section 4 — Browser Factory Abstractions

Not in the arch brief yet, but very relevant for the **deployment target abstraction**.

### `Playwright_Browser` (base class)

Pure Python abstraction over `sync_playwright()`:
- `browser_via_cdp(browser_name, endpoint_url)` — connects to any Chrome DevTools Protocol endpoint
- `context_new / context_close / contexts` — context lifecycle
- `page / pages / new_page / page_close` — page lifecycle
- `playwright()` — lazy singleton starting `sync_playwright()`
- `stop()` — graceful shutdown

### `Playwright_Browser__Chrome`

Uses **local Chromium subprocess** via remote-debugging-port:
- Spawns Chromium with `--remote-debugging-port={port}`, `--user-data-dir=/tmp/...`, `--headless`
- Connects Playwright over CDP to the spawned process
- Persistent process with reuse (`reuse_browser=True`)

### `API_Browserless`

Uses **browserless.io** cloud provider via WebSocket CDP:
```python
def wss_url(self):
    return f'wss://chrome.browserless.io?token={self.auth_key()}'

def browser(self):
    return self.browser_via_cdp(browser_name='chromium', endpoint_url=self.wss_url())
```

**Architectural unlock:** because `Playwright_Browser` is an abstract base, the service can transparently swap between:
- Local Chromium (dev laptop, CI, Fargate)
- Remote Chromium via browserless.io or similar (if Lambda + Chromium ever becomes too painful)
- Local subprocess Chromium with CDP (current Lambda pattern)

**Worth keeping in the dev brief as an option.** The arch brief lists 5 deployment targets — the abstraction for "local browser vs remote browser" is a 6th axis that could be exposed as a config option.

## Section 5 — The `Playwright_Process` Class (Persistent Chromium with Healthcheck)

**Path:** `osbot_playwright/playwright/api/Playwright_Process.py`

This is the single most sophisticated piece in the repo. It solves the **persistent browser session** problem that the 16 April arch brief's `POST /session/create` → `POST /session/{id}/...` flow requires.

### What it does

1. **Spawns Chromium as a long-lived subprocess** with remote-debugging enabled:
```python
params = [
    self.browser_path,
    f'--remote-debugging-port={self.debug_port}',     # e.g. 9910
    f'--user-data-dir={browser_data_folder}',          # /tmp/playwright_chrome_data_folder_in_port__9910
    '--use-mock-keychain',                             # avoid macOS keychain prompts
    '--disable-infobars',
]
if self.proxy:
    params.append(f'--proxy-server={self.proxy}')
    params.append('--ignore-certificate-errors')       # TLS intercepting proxy!
if self.headless:
    params.append('--headless')
```

2. **Persists process metadata** to `/tmp/playwright_chrome_data_folder_in_port__{port}/playwright_process.json`:
```python
{
  'created_at': '2026-04-16 10:30:15',
  'debug_port': 9910,
  'headless': True,
  'process_args': [...],
  'process_id': 12345,
  'reuse_browser': True
}
```

3. **On subsequent requests, reconnects via:**
   - Load JSON file
   - `psutil.Process(pid)` — confirm process still alive
   - Confirm debug port is open
   - If all good, reuse; otherwise restart

4. **`healthcheck()` method** returns 9-field dict:
   ```python
   {'chromium_debug_port': 9910,
    'chromium_debug_port_match': True,
    'chromium_debug_port_open': True,
    'chromium_process_id': 12345,
    'chromium_process_exists': True,
    'chromium_process_status': 'running',      # or 'sleeping'
    'data_folder_exists': True,
    'healthy': True,
    'playwright_process_file_exists': True}
   ```

### Why this matters for the new service

The 16 April arch brief's **session management** (`POST /session/create` returns `session_id`; subsequent `POST /session/{id}/execute` uses that session) depends on a persistent-within-container browser. This class is exactly that pattern, already proven.

**The gotcha captured in code, pre-dating the QA repo's 29 March debrief:**

```python
if self.proxy:
    params.append(f'--proxy-server={self.proxy}')
    params.append('--ignore-certificate-errors')    # TLS intercepting proxy
```

This is the **same finding** the SG_Send__QA 03/29 debrief surfaced — Playwright doesn't auto-pick-up `HTTPS_PROXY` and TLS-intercepting proxies need `--ignore-certificate-errors`. **Dinis solved this here first**, probably around 2024, and SG_Send__QA rediscovered it independently. The dev brief's config-loading path must carry this forward.

### Env var convention

```python
ENV_VAR__PLAYWRIGHT__PROXY_URL = 'PLAYWRIGHT__PROXY_URL'
```

The service already has an env-var convention for proxy config. For the vault-based config path, the startup flow should read vault → set `PLAYWRIGHT__PROXY_URL` → `Playwright_Process` picks it up automatically. No code change needed to `Playwright_Process`.

## Section 6 — The Shell Server (Primordial Sequence Execution)

Mounted via `self.add_shell_server()` in `Fast_API_Playwright`. Uses `osbot_fast_api.utils.http_shell.Http_Shell__Server`.

**Shape (inferred from `test_Lambda__Docker_Playwright.py`):**

```python
POST /shell-server
body: {
  'auth_key': '<HTTP_SHELL__AUTH_KEY>',
  'method_name': 'some_method',
  'method_kwargs': {...}
}
```

Authenticates against an env-var-stored key, executes the method by name, returns result. The test file uses `Http_Shell__Client` + `Lambda_Shell` helpers to exercise it end-to-end against a real Lambda.

**This is prior art for the arch brief's `POST /sequence/execute`.** The evolution:

| Generation | Shape | Security model |
|---|---|---|
| `POST /playwright/code` (commented out) | Accept raw Python, `exec()` | None — auth key only |
| `POST /shell-server` (current) | Accept method name + kwargs, dispatch to registered method | Auth-key gated, method allowlist |
| `POST /sequence/execute` (16 Apr brief) | Accept JSON step list, dispatch each step to step handler | Auth-key gated, step-type allowlist, no eval |

Each generation is strictly safer than the last. The new service should end the lineage at JSON steps with typed dispatch.

## Section 7 — Integration Test Proves End-to-End Lambda Invocation Works

**Path:** `tests/docker/test_Lambda__Docker_Playwright.py`

```python
def test_create_lambda(self):
    delete_existing = True
    wait_for_active = True
    lambda_function = self.lambda_docker.lambda_function()

    if in_github_actions():
        with Duration(prefix='create lambda:'):
            create_result = self.lambda_docker.create_lambda(...)
            assert lambda_info.get('Configuration').get('State') == 'Active'

    payload = {'path': '/config/status', 'httpMethod': 'GET'}
    http_status = '{"status":"ok"}'
    with Duration(prefix='invoke lambda 1st:'):
        invoke_result = lambda_function.invoke(payload)
        assert invoke_result.get('body') == http_status
    # ... 2nd and 3rd invocations to measure warm start
```

The test runs in GitHub Actions, creates a real Lambda from a real image, invokes it with an API-Gateway-shaped event (`{'path': ..., 'httpMethod': ...}`), and asserts the body response. Three sequential invocations demonstrate cold → warm timings.

**The `{'path': ..., 'httpMethod': ...}` payload shape** confirms Lambda Web Adapter is translating Lambda events → HTTP requests → FastAPI routes correctly. This is the working pattern reference.

## Section 10 — Librarian Flags on OSBot-Playwright

1. **Not referenced anywhere in SGraph-AI__App__Send or SG_Send__QA main-repo docs.** The cross-repo graph has no edge from SG/Send to OSBot-Playwright even though this repo is a direct predecessor. The Librarian master index should add it.
2. **This repo is the **original** home of the proxy + TLS gotcha.** `Playwright_Process.start_process()` sets `--ignore-certificate-errors` alongside `--proxy-server=`. The SG_Send__QA 03/29 debrief rediscovered this in the Claude cloud environment. Both repos should cross-reference.
3. **`tests/unit/playwright/fastapi/test_Fast_API_Playwright.py` is currently skipping**: `pytest.skip("Playwright tests started to fails with recent update")`. The refresh is overdue; the tests were flagged as failing at some point and not fixed. The new service should bring these back to green.
4. **The `code` endpoint removal is an important historical decision.** It's commented out because arbitrary `exec()` of caller-supplied Python is a security hole. The 04/09 architect constraint in the main repo (scripts from trusted sources only) is the formalisation of a call Dinis already made in the code here.
5. **`osbot-fast-api` shell server** is a general-purpose primitive that existed before the QA repo's `QA_API__Runner`. Worth documenting as a family: `Http_Shell__Server` (general) → `QA_API__Runner` (QA-scoped) → (new) Sequence dispatcher (vault-and-step-scoped).
