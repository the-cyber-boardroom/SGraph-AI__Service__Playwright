# Health / Diagnostics Patterns in sg_compute_specs — Research

*Version v0.2.6 | 2026-05-10*

---

## Why This Document Exists

The `local-claude` stack revealed a class of bug that is hard to diagnose with a simple pass/fail `health` probe: the boot sequence has **nine distinct steps** that must all succeed in order. When step 4 (Docker CE installed) works but step 6 (ssm-user added to docker group before user existed) is broken, a binary healthy/unhealthy answer tells you nothing useful. You need to see exactly which check in the sequence failed.

This document surveys all existing health/diagnostics patterns in `sg_compute_specs`, identifies the best model, and records the design decision to adopt it for `local-claude` (and eventually all specs).

---

## Patterns Found

### 1. Podman — SSM-reachable → service version (binary)

**Files:**
- `sg_compute_specs/podman/service/Podman__Health__Checker.py`
- `sg_compute_specs/podman/schemas/Schema__Podman__Health__Response.py`

**Approach:** polls in a loop:
1. Get EC2 state; if not `running`, loop.
2. Check SSM reachable via `instance_helper.is_ssm_reachable()`.
3. If reachable, get `podman version` via SSM exec → healthy if non-empty.

**Schema:**
```python
Schema__Podman__Health__Response(
    healthy       : bool,
    ssm_reachable : bool,
    podman_ok     : bool,
    podman_version: str,
    state         : Enum__Podman__Stack__State,
    elapsed_ms    : int,
)
```

**Verdict:** Binary gate (healthy/not-healthy). No named checks. Works well for a one-service stack (Podman version present = healthy), but you cannot see *which* of the three checks failed when unhealthy.

### 2. Elastic — Named check list (best pattern)

**Files:**
- `sg_compute_specs/elastic/service/Elastic__Service.py` — `health()` method
- `sg_compute_specs/elastic/schemas/Schema__Elastic__Health__Check.py`
- `sg_compute_specs/elastic/schemas/Schema__Elastic__Health__Response.py`
- `sg_compute_specs/elastic/enums/Enum__Health__Status.py`

**Approach:** runs 4 ordered named checks:
1. `ec2-state` — EC2 state == running?
2. `public-ip` — instance has public IP?
3. `elastic` — `/_cluster/health` HTTP probe returns green/yellow?
4. `kibana` — `/api/status` probe returns `ready`?

Each check is a `Schema__Elastic__Health__Check(name, status, detail)` where `status ∈ {OK, FAIL, WARN, SKIP}`.

**Schema:**
```python
class Schema__Elastic__Health__Check(Type_Safe):
    name   : Safe_Str__Text
    status : Enum__Health__Status = Enum__Health__Status.SKIP
    detail : Safe_Str__Diagnostic

class Schema__Elastic__Health__Response(Type_Safe):
    stack_name : Safe_Str__Elastic__Stack__Name
    all_ok     : bool = False
    checks     : List__Schema__Elastic__Health__Check
```

**Verdict:** **Best pattern in the codebase.** Named checks with structured status make it immediately obvious which step failed. The check list is also machine-readable (API route returns JSON). *Gap: no Rich CLI renderer — health checks are exposed via FastAPI only, not printed to the terminal.*

### 3. Open Design — Two-phase HTTP wait

**File:** `sg_compute_specs/open_design/cli/__init__.py` — `_wait_healthy()`

**Approach:**
1. Phase 1: `EC2__Instance__Helper.wait_for_running()` — polls until EC2 state == running.
2. Phase 2: `Health__Poller.wait_healthy()` with `Health__HTTP__Probe` against public IP.

Prints freeform status lines (`c.print(...)`) but no structured check list.

**Verdict:** Good for public-facing HTTP services. Does not work for local-claude (vLLM binds to `127.0.0.1` only — not reachable from outside).

---

### 4. Spec__Service__Base — Default HTTP probe (base class)

**File:** `sg_compute/core/spec/Spec__Service__Base.py` — `health()`

**Approach:** HTTP probe against the instance's public IP using `spec.health_port` / `health_path` / `health_scheme`. Returns `Schema__CLI__Health__Probe(healthy, state, last_error, elapsed_ms)`.

**Verdict:** Works for any public-facing service. Not suitable for local-claude. The `render_health_probe()` in `Spec__CLI__Renderers__Base.py` renders it as `✓ healthy` or `✗ not healthy` — binary, no detail.

---

### 5. Local Claude (current) — SSM exec boot-marker + curl

**File:** `sg_compute_specs/local_claude/service/Local_Claude__Service.py` — `health()`

**Approach:**
1. SSM exec: `test -f /var/lib/sg-compute-boot-failed && tail -n 30 /var/log/ephemeral-ec2-boot.log || true`
2. SSM exec: `curl -sf http://127.0.0.1:8000/v1/models`

Returns `Schema__CLI__Health__Probe` (shared schema, not local-claude-specific). No named checks; if unhealthy, the `last_error` field contains raw log text that is hard to parse.

**Verdict:** Better than the HTTP-probe default because it uses SSM and checks the boot-failure marker. But the result is still binary with opaque error text. Does not identify *which* boot step failed.

## The Gap: No CLI Renderer for Elastic-Style Checks

The Elastic pattern (named checks list) is the right model but currently:
- Used in `Elastic__Service` for a FastAPI route only.
- No `render_health_checks()` in `Spec__CLI__Renderers__Base.py`.
- The checks are never printed to the terminal.

## Adopted Design: Sequential Boot Checklist for local-claude

### Boot sequence — 9 steps in order

| # | Check name       | What is tested                                       |
|---|-----------------|------------------------------------------------------|
| 1 | `ec2-state`     | EC2 instance is in `running` state (AWS API)         |
| 2 | `ssm-reachable` | SSM exec `echo ok` succeeds within 10 s              |
| 3 | `boot-failed`   | `/var/lib/sg-compute-boot-failed` is **absent**      |
| 4 | `boot-ok`       | `/var/lib/sg-compute-boot-ok` is **present**         |
| 5 | `docker`        | `systemctl is-active docker` == `active`             |
| 6 | `docker-access` | `sudo -u ssm-user docker info` exits 0               |
| 7 | `gpu`           | `nvidia-smi --query-gpu=name` returns a GPU name     |
| 8 | `vllm-container`| `docker ps --filter name=vllm-claude-code` is non-empty |
| 9 | `vllm-api`      | `curl -sf http://127.0.0.1:8000/v1/models` returns JSON |

Steps 2–9 are skipped (status=SKIP) if any earlier check fails.

### CLI output format

```
  ✓  ec2-state          running
  ✓  ssm-reachable      responsive
  ✓  boot-failed        absent
  ✓  boot-ok            /var/lib/sg-compute-boot-ok present
  ✓  docker             active
  ⚠  docker-access      ssm-user lacks docker group — new SSM session will fix it
  ✓  gpu                NVIDIA A10G (1)
  ✗  vllm-container     no container matching vllm-claude-code
  ⊘  vllm-api           (skipped — container not running)
```

Icons: `✓` = OK (green), `✗` = FAIL (red), `⚠` = WARN (yellow), `⊘` = SKIP (dim).

### Implementation location

- `sp local-claude diag [name]` — new CLI command in `Cli__Local_Claude.py`
- `Local_Claude__Service.diagnose(region, name)` — returns `List[Tuple[str, str, str]]` (name, status, detail)

## Shared Layer Candidates (Audit)

The following patterns were pioneered in local-claude but belong in shared code so every spec gets them for free:

| Pattern | Current location | Should move to |
|---------|-----------------|----------------|
| Boot-failure ERR trap + sentinel file | `Section__Base.TEMPLATE` | already in shared `Section__Base` ✓ |
| Boot-success EXIT marker | `Local_Claude__User_Data__Builder` FOOTER | `Section__Base` EXIT trap |
| SSM-exec health probe | `Local_Claude__Service.health()` | `Spec__Service__Base.ssm_health()` |
| Boot-failure marker check | `Local_Claude__Service.health()` | `Spec__Service__Base.check_boot_markers()` |
| Multi-source logs command | `Cli__Local_Claude._LOG_SOURCES` | `Spec__CLI__Builder._register_logs()` |
| Named-check health response | `Elastic__Service.health()` | `sg_compute/core/health/Schema__Health__Check.py` (shared) |
| Rich CLI renderer for named checks | (does not exist) | `Spec__CLI__Renderers__Base.render_health_checks()` |

Priority order for shared promotion:
1. **Named-check schema** — needed by every spec to return structured health data
2. **Rich renderer for named checks** — needed for usable CLI output  
3. **SSM-exec health base** — eliminate duplicated SSM health logic in each spec
4. **Multi-source logs** — Ollama and elastic would both benefit

## Key Finding: docker-group Race Condition

The ssm-user docker group issue that appeared in the `clever-lovelace` instance illustrates why named checks matter:

- `Section__Docker` calls `usermod -aG docker ssm-user || true`
- ssm-user is created *asynchronously* by `amazon-ssm-agent`
- If `usermod` runs before ssm-user exists, the `|| true` silently swallows the failure
- Result: ssm-user is never in the docker group; `docker ps` → permission denied

**Fix applied in this session:** add `until id ssm-user >/dev/null 2>&1; do sleep 2; done` before the `usermod` call in `Section__Docker`.

The named-check `diag` command makes this visible immediately as a `⚠ docker-access` warning instead of requiring the user to SSH in and manually run `id` / `docker ps`.
