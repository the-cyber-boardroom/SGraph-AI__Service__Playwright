# Network Diagnostic Audit: `fast-hopper.sg-compute.sgraph.ai`

> Captured verbatim from a Perplexity research run on 2026-05-15. **Vantage = the open internet**, not the Claude sandbox. This is the external-world baseline against which the Claude-sandbox failure is contrasted in
> [`team/claude/debriefs/2026-05-15__sandbox-egress-https-block-investigation.md`](../../../team/claude/debriefs/2026-05-15__sandbox-egress-https-block-investigation.md).

## Executive summary

`fast-hopper.sg-compute.sgraph.ai` resolves to `18.169.240.189`, with reverse DNS `ec2-18-169-240-189.eu-west-2.compute.amazonaws.com`, indicating an AWS EC2-hosted endpoint in `eu-west-2`. The tested host is reachable over both HTTP and HTTPS, but the application returns `401 Unauthorized` with the JSON message `Client API key is missing, you need to set it on a header or cookie`. This makes the endpoint technically reachable but not publicly readable by default, which is the most likely reason Claude-like web-fetch agents report that they cannot "open" it.

The certificate is not the likely cause. Strict `curl` validation succeeded, OpenSSL validation returned `Verify return code: 0 (ok)`, the hostname matches the SAN, and the certificate chain terminates at ISRG Root X1 through Let's Encrypt R12.

The highest-priority infrastructure issue is that all 65,535 TCP ports completed a TCP connect scan from this vantage point. Direct probes confirmed successful TCP connections to ports such as `1`, `22`, `25`, `445`, `5432`, `6379`, `27017`, `40000`, and `65535`, but most non-HTTP ports accepted the connection and then timed out or reset during protocol negotiation. This looks like a broad security-group/listener/tarpit/proxy configuration rather than legitimate services on every port.

## HTTP and application behavior

| URL | Result | Body |
| --- | --- | --- |
| `https://fast-hopper.sg-compute.sgraph.ai/` | `401 Unauthorized` | `Client API key is missing, you need to set it on a header or cookie` |
| `http://fast-hopper.sg-compute.sgraph.ai/` | `401 Unauthorized` | Same |
| `OPTIONS https://fast-hopper.sg-compute.sgraph.ai/` | `204 No Content` | CORS preflight headers present |

**From the open internet, both HTTP and HTTPS work.** The 401 is application-layer (no API key), not network-layer.

| Chain | Leaf → Let's Encrypt R12 → ISRG Root X1 |
| Valid from | 2026-05-15 16:39:49 GMT |
| Valid until | 2026-08-13 16:39:48 GMT |
| Strict `curl` verification | `SSL certificate verify ok` |
| OpenSSL verification | `Verify return code: 0 (ok)` |

The cert is a real, valid, browser-trusted Let's Encrypt RSA-2048 cert. **From the open internet there is nothing wrong with it.** Any tool that fails to validate it is presenting a *different* cert (i.e. has been MITM'd by something between the client and the origin).

## TCP and UDP port audit

- Common TCP ports → many reported `open` with `syn-ack`
- `-p-` full scan → **65,535 TCP ports reported `open`**
- Service detection on `80,443`: `80/tcp open http Uvicorn`; `443/tcp open ssl/https`
- Direct TCP probes to `1`, `22`, `25`, `80`, `443`, `444`, `445`, `1024`, `2049`, `2375`, `3000`, `5432`, `6379`, `8080`, `9200`, `27017`, `40000`, `65535` all accepted TCP connections
- HTTP probes on the non-HTTP ports timed out with 0 bytes — accept-and-hang behaviour

Most likely cause: a "hopper" / tarpit / catch-all socket listener accepting on every port and then doing nothing. **Real services are only on `:80` and `:443`.** The other 65,533 are noise that confuses scanners and looks alarming on an external audit but does not affect the working ports.

## Why Claude-like agents may not open this endpoint

| Rank | Cause | Likelihood |
| --- | --- | --- |
| 1 | Application returns `401 Unauthorized` without an API key | High |
| 2 | Agent fetch tool treats HTTP errors as inaccessible content | High |
| 3 | Domain/network allowlists in agent runtimes | Medium |
| 4 | Non-browser/API response shape | Medium |
| 5 | TLS certificate CA problem | **Low** |
| 6 | JavaScript rendering limitation | Low |

> **Important:** the Perplexity vantage point ruled out the certificate as a cause. But Perplexity was **not running inside a Claude sandbox** — it was on the open internet. The actual failure mode for Claude (TLS MITM by the sandbox-egress proxy + upstream 503 for non-pre-warmed FQDNs) is **invisible from the open internet** because the MITM only happens inside the sandbox. See the Claude-vantage debrief for that evidence.

## Recommendations (carried forward)

1. Lock down the SG to only the ports actually serving traffic (close the all-TCP-open tarpit, or document it).
2. Redirect `:80` → `:443` once HTTPS works in all consumer environments. **Status: deliberately not doing this yet** — plain HTTP on `:80` is the only path that works from Claude's sandbox today, and forcing the redirect would break the very integration we are debugging.
3. Enable ALPN, prefer AEAD TLS 1.2 ciphers, add HSTS once the redirect lands.
4. Hide `server: uvicorn` at the reverse proxy if any future fronting layer is added.

## Bottom line (Perplexity)

> "The evidence does not support 'Claude cannot open it because it is a Let's Encrypt certificate.' The certificate is valid, trusted, and correctly matched. The more likely blockers are application-level authorization (`401` without API key), agent web-fetch policies that treat HTTP errors as inaccessible content, and possibly domain allowlist or agent egress rules."

The Claude-sandbox debrief confirms that the **third** of those — "agent egress rules" — is the actual root cause, not the first two.
