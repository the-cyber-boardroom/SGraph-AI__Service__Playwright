# Research: Playwright Base Images, Lambda Integration, and Code-Decoupling Patterns

**Version:** v0.20.55
**Date:** 16 April 2026
**Author:** Librarian session (research pass)
**Scope:** Responds to acceptance criterion #12 of `v0.24.56__arch-brief__playwright-service-decoupled-image.md` — *"Base image research completed before custom build; documented comparison of Microsoft/community images vs custom."*
**Key requirement:** decouple the stable Playwright+Chromium install from the FastAPI service code so code updates do not require ECR redeploys.

---

## TL;DR — Recommended Combination

**Base image:** `mcr.microsoft.com/playwright/python:v1.58.2-noble` (Microsoft official, Ubuntu 24.04). Do NOT build Chromium from scratch. Do NOT use `@sparticuz/chromium` (Node-only, Puppeteer-origin, history of breakage per sources).

**Lambda integration:** **AWS Lambda Web Adapter** (`public.ecr.aws/awsguru/aws-lambda-adapter`), not the Runtime Interface Client. Two reasons: (1) the same image deploys unchanged to Lambda, ECS/Fargate, EC2, and local Docker — exactly what the arch brief requires; (2) a FastAPI/uvicorn app runs natively, no Mangum/handler wrapping needed.

**Code decoupling:** **S3-zip boot loader pattern** (pre-specified in the 26 March debrief). `awslambdaric`-free. Works with Lambda Web Adapter by running the boot step in the uvicorn startup hook. Alternatives (EFS mount, S3 Files, container-bundled) are documented below with trade-offs.

**Architecture:** stable image = `Ubuntu + Python 3.12 + Playwright 1.58 + Chromium + AWS Lambda Web Adapter + boot loader`. Hot-swappable code = the FastAPI service + route handlers + vault client, pulled from S3 zip at container boot.

**Estimated image size:** ~1.8–2.0 GB uncompressed. Well within Lambda's 10 GB image limit. Cold start budget: 5–15s (Chromium load is the dominant cost).

## Section 1 — Base Image Comparison

### Option A: Microsoft official Playwright image (RECOMMENDED)

**Image:** `mcr.microsoft.com/playwright/python:v1.58.2-noble`
**Base OS:** Ubuntu 24.04 LTS (Noble Numbat)
**Also available:** `v1.58.2-jammy` (Ubuntu 22.04) and unversioned for dev
**Size:** ~1.8–2.0 GB uncompressed

**Contents:**
- Python 3 runtime (Ubuntu default 3.12 in Noble)
- Chromium + Firefox + WebKit browsers
- All system dependencies for browser execution
- Does NOT include the `playwright` pip package — must `pip install playwright==1.58.2` (version match is required)

**Pros:**
- Maintained by Microsoft alongside every Playwright release — version tags match Playwright releases exactly
- Ubuntu base matches Playwright's tested environment; no "it works locally but not in Lambda" mismatches
- Sematext case study (2024): they moved from `@sparticuz/chromium` to Microsoft's image precisely because third-party packages break on AWS security patches; Microsoft's image is "a stable runtime environment, immune from changes in third-party packages"
- Published to Microsoft Container Registry (MCR) — pull-through to AWS ECR works
- Both `linux/amd64` and `linux/arm64` tags published (confirmed on Docker Hub)

**Cons:**
- Size: ~2 GB. Much of it is Firefox + WebKit, which we don't need. Single-browser slim variant does not exist officially.
- Runs as `root` by default — OK for trusted code, not OK for web scraping (needs separate user + seccomp). Trusted code in our case.
- Focal tag is being deprecated in favour of Noble.

### Option B: AWS Lambda Python base image + install everything (NOT RECOMMENDED)

**Image:** `public.ecr.aws/lambda/python:3.12`
**Base OS:** Amazon Linux 2023

**Why sources unanimously recommend against this:** Amazon Linux 2023 does not match the library set Playwright targets. You end up rebuilding browser system dependencies by hand; one Amazon Linux security patch breaks your Chromium. Every article consulted (`cloudtechsimplified.com`, `stasdeep.com`, `steele.blue`, `browsercat.com`, `frontendwithhasan.com`) converges on "use Microsoft's image, add Lambda integration on top."

### Option C: Community Lambda-optimised images

**Candidates:**
- `sjw7444/lambda-playwright-python` (also on public ECR: `public.ecr.aws/j9p6g6v0/lambda-playwright-python`) — Lambda-specific, claims ARM support, `awslambdaric` pre-installed. **Status:** community maintained, no corporate backing.
- `VikashLoomba/AWS-Lambda-Docker-Playwright` — GitHub repo with Dockerfile, Node.js only.

**Trade-off:** saves a few lines of Dockerfile vs introduces a dependency on a single maintainer. The sematext reasoning applies here too: prefer Microsoft's image as the single upstream, add ~10 lines of Lambda glue ourselves.

### Option D: `@sparticuz/chromium` layer (NOT APPLICABLE)

Node.js only. No Python bindings. Discounted.

**Decision: Option A (Microsoft official).**

## Section 2 — Lambda Integration: RIC vs Web Adapter

Two ways to make an image work as a Lambda function. They are mutually exclusive for a single deployment target, but the same base image supports both.

### Option 1: AWS Lambda Runtime Interface Client (`awslambdaric`)

**What it is:** Python package that implements the Lambda Runtime API so a non-AWS base image can function as a Lambda. You install it via pip, set `ENTRYPOINT ["/usr/bin/python", "-m", "awslambdaric"]`, and `CMD ["module.handler"]`.

**Canonical Dockerfile pattern** (from `stasdeep.com`, `cloudtechsimplified.com`):

```dockerfile
FROM mcr.microsoft.com/playwright/python:v1.58.2-noble AS build-image
ARG LAMBDA_TASK_ROOT="/var/task"
WORKDIR ${LAMBDA_TASK_ROOT}

RUN apt-get update && apt-get install -y g++ make cmake unzip libcurl4-openssl-dev
RUN pip install --target ${LAMBDA_TASK_ROOT} awslambdaric

FROM mcr.microsoft.com/playwright/python:v1.58.2-noble
ARG LAMBDA_TASK_ROOT="/var/task"
WORKDIR ${LAMBDA_TASK_ROOT}
COPY --from=build-image ${LAMBDA_TASK_ROOT} ${LAMBDA_TASK_ROOT}
COPY app/* ${LAMBDA_TASK_ROOT}

ENTRYPOINT ["/usr/bin/python", "-m", "awslambdaric"]
CMD ["app.handler"]
```

**Handler signature:** traditional Lambda `def handler(event, context)`.

**For a FastAPI app:** requires Mangum wrapper (`from mangum import Mangum; handler = Mangum(app)`).

**Pros:** native Lambda invocation model; works with API Gateway + Lambda Function URLs.

**Cons:** image only runs under Lambda. Cannot `docker run` locally the same image for ECS/Fargate. Requires Mangum for FastAPI, which has subtle limitations (streaming, lifespan events).

### Option 2: AWS Lambda Web Adapter (RECOMMENDED)

**What it is:** AWS-published sidecar extension that sits in `/opt/extensions/lambda-adapter` and translates Lambda invocations into HTTP calls against the container's own web server on a local port. The container runs uvicorn as normal; the adapter bridges.

**Canonical Dockerfile pattern** (from AWS's own `aws-lambda-web-adapter` examples):

```dockerfile
FROM mcr.microsoft.com/playwright/python:v1.58.2-noble

COPY --from=public.ecr.aws/awsguru/aws-lambda-adapter:1.0.0 \
     /lambda-adapter /opt/extensions/lambda-adapter

ENV PORT=8000
ENV AWS_LWA_READINESS_CHECK_PROTOCOL=tcp

WORKDIR /var/task
COPY requirements.txt ./
RUN pip install -r requirements.txt

# Boot loader (decouples service code from image — see Section 3)
COPY bootloader/ /opt/bootloader/

CMD ["/bin/bash", "-c", "python /opt/bootloader/boot.py && exec uvicorn --port=$PORT sgraph_ai_service_playwright.main:app"]
```

**Pros:**
- **Same image, four deployment targets:** local Docker, Lambda (via extension), ECS/Fargate, EC2 — **exactly** the arch brief's five-target matrix minus Docker Hub distribution (which is orthogonal, just a push target)
- **Native FastAPI/uvicorn** — no Mangum, no handler wrapping, standard ASGI
- Adapter is only ~5 MB — negligible image size impact
- Auto-detection: when `AWS_LAMBDA_RUNTIME_API` env var is set (Lambda sets this automatically), the adapter activates; otherwise the container runs as a vanilla uvicorn server
- Supports response streaming for long-running operations (relevant for video capture / long automation flows)
- AWS-maintained (under `aws/aws-lambda-web-adapter` on GitHub)

**Cons:**
- Adds a startup dependency on the extension loading correctly
- Debugging two runtime layers (uvicorn + adapter) can be harder than pure Mangum

**Decision: Option 2 (Lambda Web Adapter).** It uniquely satisfies the "one image, multiple runtimes" requirement. Mangum/RIC forces a Lambda-specific image that cannot run identically on Fargate.

## Section 3 — Code Decoupling Pattern (the critical one)

Goal: the Dockerfile's base image changes ~monthly at most (Playwright version bumps). The FastAPI service code + route handlers change daily. We must NOT rebuild the image for service code changes.

Five patterns considered:

### Pattern A: Container-bundled code (THE PROBLEM)

`COPY app/* /var/task/` in the Dockerfile. Service code ships inside the image. **Every code change requires image rebuild + ECR push + Lambda update.** This is what we're avoiding. Ruled out.

### Pattern B: S3-zip boot loader (RECOMMENDED — matches 26 March pre-spec)

**How it works:**
1. Container builds once with Playwright + FastAPI scaffold + boot loader script
2. Service code is zipped and uploaded to S3 (versioned path)
3. Container boots: boot loader reads `CODE_ZIP_S3_BUCKET` + `CODE_ZIP_S3_KEY` env vars, downloads, extracts to `/tmp/code`, prepends to `sys.path`
4. uvicorn starts, imports the service module from the hot-swapped path
5. To update code: `aws s3 cp code.zip s3://bucket/code/latest.zip` — next cold start picks it up

**Boot loader code** (from the 26 March debrief, verbatim):

```python
# boot.py (baked into container at /opt/bootloader/boot.py)
import os, sys, tempfile, zipfile, boto3

bucket   = os.environ['CODE_ZIP_S3_BUCKET']
key      = os.environ['CODE_ZIP_S3_KEY']

tmp = tempfile.mkdtemp(prefix='sg_code_')
s3  = boto3.client('s3')
s3.download_file(bucket, key, f'{tmp}/code.zip')
with zipfile.ZipFile(f'{tmp}/code.zip') as z:
    z.extractall(f'{tmp}/code')
sys.path.insert(0, f'{tmp}/code')
# uvicorn now imports normally; service module is found on sys.path
```

**Pros:**
- S3 upload takes seconds, not minutes like an ECR push
- Versioned zips support instant rollback (`CODE_ZIP_S3_KEY=code/v0.24.55.zip`)
- Works identically on Lambda, Fargate, EC2, local Docker (pass `CODE_ZIP_S3_*` env vars)
- No VPC, no EFS, no additional AWS infra beyond S3 + IAM
- Pattern is already proven on the existing osbot-fast-api-serverless deployments (per 25 March brief)
- Natural deploy CI: `zip -r code.zip sgraph_ai_service_playwright/ && aws s3 cp code.zip s3://...`

**Cons:**
- Every cold start downloads the zip (~1-5 MB typical, adds ~100-500ms to cold start)
- If zip is corrupt or missing, container fails to boot — need graceful error handling
- Secrets in the zip would be a problem — use env vars or vault for secrets, not the zip

**Mitigation for cold-start cost:** cache the extracted zip in `/tmp` using a content hash; skip re-download if hash matches. Lambda's `/tmp` is scoped to the execution environment so this works across warm invocations.

### Pattern C: EFS-mounted code

**How it works:** Lambda mounts an EFS volume at `/mnt/code`. Service code lives on EFS. `sys.path.append('/mnt/code')` at container start.

**Pros:**
- Zero cold-start download cost — code is already there
- Filesystem semantics (append, lock, multi-writer) if needed
- Same EFS can serve Fargate tasks identically

**Cons:**
- **Requires Lambda to be in a VPC.** Complicates IAM, networking, egress to S3/vault (needs VPC endpoint or NAT gateway). This is the big one.
- NAT gateway costs (~$30/month baseline) or VPC endpoint setup
- EFS throughput limits can bite under concurrent cold starts
- Adds operational complexity (NFS mount health, EFS burst credits)

**Verdict:** viable but heavier than S3 zip. Only worth it if code size becomes large (>50 MB) or cold-start latency becomes intolerable.

### Pattern D: Lambda Amazon S3 Files (NEW — needs verification)

AWS Lambda documentation now mentions: *"Amazon S3 Files – Serverless file system for mounting your Amazon S3 bucket. Amazon S3 Files provides access to your Amazon S3 objects as files using standard file system operations such as read and write on the local mount path."*

**Would work as:** mount a code bucket as `/mnt/code`, put code there, `sys.path.append('/mnt/code')`.

**Pros:** EFS-like semantics without the VPC requirement.

**Cons:** newer feature, limited public documentation, performance characteristics unclear. Worth a spike but do not bet the dev brief on this as primary pattern.

**Verdict:** research spike at most. Not primary.

### Pattern E: LocalStack hot-reload

Development-only. Not applicable to production Lambda. Mentioned for completeness.

### Pattern F: Vault-based code delivery (the 16 April brief's variant of B)

The 16 April brief mentions: *"Pulled from S3 or a vault at container startup."* This is Pattern B with vault replacing S3 as the code source.

**Pros:** code delivery uses the same encryption/versioning/commit-history mechanism as everything else in SG/Send.

**Cons:**
- Container needs a vault key to boot — extra bootstrap secret
- Vault read is a FastAPI round-trip to SG/Send, which introduces a chicken-and-egg (Playwright service depends on SG/Send service to start)
- For a simple zip of service code, the S3 path is dramatically simpler

**Verdict:** defer until there's a clear need. Config (proxy settings, credentials) goes to vault as the arch brief specifies — but the service **code itself** is better off in S3 for Phase 1.

**Decision: Pattern B (S3-zip boot loader)** as primary. Pattern C (EFS) as fallback if cold-start cost exceeds budget. Pattern D (S3 Files) as a spike if we want to simplify further.

## Section 4 — The Recommended Dockerfile

Single image, four deployment targets, decoupled code, per all decisions above:

```dockerfile
# ── Stage 1: base image with Lambda Web Adapter and Python deps baked in ──
FROM mcr.microsoft.com/playwright/python:v1.58.2-noble

# Lambda Web Adapter — enables Lambda deployment without code changes
COPY --from=public.ecr.aws/awsguru/aws-lambda-adapter:1.0.0 \
     /lambda-adapter /opt/extensions/lambda-adapter

# Port for uvicorn; adapter bridges to Lambda invocations on this port
ENV PORT=8000
ENV AWS_LWA_READINESS_CHECK_PROTOCOL=tcp

# Lambda-specific: Chromium writes to $HOME which is read-only in /var/task
# Force it to /tmp which is writable in both Lambda and Fargate
ENV HOME=/tmp

# Install Python-level deps that are STABLE (change rarely)
# These live in the image; service-code deps come via the S3 zip if needed
WORKDIR /opt/service
COPY requirements.base.txt .
RUN pip install --no-cache-dir -r requirements.base.txt

# Boot loader (thin — only job is to download + sys.path the service code)
COPY bootloader/boot.py /opt/bootloader/boot.py

# Fallback service code: if no S3 zip is configured, use a minimal health-check
# FastAPI app baked into the image. Makes the image runnable standalone.
COPY fallback_service/ /opt/fallback_service/

# Start: boot (optional S3 code load) + uvicorn
# uvicorn imports sgraph_ai_service_playwright.main:app which resolves via sys.path
#   - if boot loader ran, comes from /tmp/code/sgraph_ai_service_playwright/
#   - otherwise, comes from /opt/fallback_service/sgraph_ai_service_playwright/
CMD ["/bin/bash", "-c", \
     "python /opt/bootloader/boot.py && \
      exec uvicorn --host 0.0.0.0 --port=$PORT sgraph_ai_service_playwright.main:app"]
```

**`requirements.base.txt` (in image, changes rarely):**

```
playwright==1.58.2
fastapi==0.115.*
uvicorn[standard]==0.32.*
boto3==1.35.*
osbot-utils
osbot-fast-api
osbot-fast-api-serverless
```

**`bootloader/boot.py` (in image, changes almost never):**

```python
# Pulls optional S3-hosted service code and prepends to sys.path.
# If env vars absent, falls back to /opt/fallback_service (baked-in minimal app).
import os, sys, tempfile, zipfile, hashlib, pathlib

CODE_BUCKET = os.environ.get('SG_CODE_S3_BUCKET')
CODE_KEY    = os.environ.get('SG_CODE_S3_KEY')

if CODE_BUCKET and CODE_KEY:
    import boto3
    cache_dir = pathlib.Path('/tmp/sg_code')
    cache_dir.mkdir(exist_ok=True)
    zip_path  = cache_dir / 'code.zip'
    code_dir  = cache_dir / 'extracted'

    s3 = boto3.client('s3')

    # Check etag to skip redundant download on warm starts
    head    = s3.head_object(Bucket=CODE_BUCKET, Key=CODE_KEY)
    etag    = head['ETag'].strip('"')
    marker  = cache_dir / f'.etag-{etag}'

    if not marker.exists():
        s3.download_file(CODE_BUCKET, CODE_KEY, str(zip_path))
        if code_dir.exists():
            import shutil; shutil.rmtree(code_dir)
        code_dir.mkdir()
        with zipfile.ZipFile(zip_path) as z:
            z.extractall(code_dir)
        # clear old etag markers
        for old in cache_dir.glob('.etag-*'):
            old.unlink()
        marker.touch()

    sys.path.insert(0, str(code_dir))
    print(f'[bootloader] loaded code from s3://{CODE_BUCKET}/{CODE_KEY} (etag {etag})')
else:
    sys.path.insert(0, '/opt/fallback_service')
    print('[bootloader] using baked-in fallback service (no S3 code configured)')
```

**Deploy flow:**
- **Base image rebuild** (monthly / Playwright version bump): `docker build → ECR push → Lambda update image`
- **Service code update** (daily): `zip -r code.zip sgraph_ai_service_playwright/ && aws s3 cp code.zip s3://bucket/code/latest.zip` — next cold start picks it up. Warm instances keep the old code until recycled.
- **Force warm-instance refresh:** `aws lambda update-function-configuration --function-name sg-playwright --environment Variables={...}` — Lambda treats env var change as a new version, all warms recycle.

## Section 5 — Hard-Won Knowledge (Consolidated Gotchas)

From every consulted source, merged:

| Gotcha | Source | Fix |
|---|---|---|
| `$HOME` is read-only in Lambda; Chromium tries to write to `$HOME/.cache`, crashes | `steele.blue`, `browsercat.com`, Playwright GitHub issues | `ENV HOME=/tmp` in Dockerfile |
| Playwright does NOT auto-pick-up `HTTPS_PROXY` env var (curl does) | SG_Send__QA 03/29 debrief | Pass proxy explicitly to `chromium.launch(proxy=...)` |
| TLS-intercepting proxies cause Playwright cert errors | SG_Send__QA 03/29 debrief | `browser.new_context(ignore_https_errors=True)` |
| Chromium sandbox requires privileges Lambda doesn't have | `browsercat.com`, `stasdeep.com`, mamezou | Launch with `args=['--no-sandbox', '--disable-gpu', '--single-process', '--disable-dev-shm-usage']` |
| Memory: Chromium needs ≥2 GB to launch reliably | `steele.blue`, `browsercat.com` | Lambda memory: 2048 MB minimum, 3072 MB recommended |
| Default Lambda timeout too short for browser automation | `steele.blue` | Set function timeout to 60–300s; Lambda max 900s (15 min) |
| ARM64 Lambda + Playwright: Chrome For Testing is x86-only per mamezou (July 2024) | mamezou, `sjw7444/lambda-playwright-python` | Use x86_64 architecture. ARM status to re-verify 2026. |
| Image size ~2 GB — pulls slowly on first deploy | microsoft/playwright issues | Acceptable; Lambda caches image layers per region |
| `/dev/shm` is tiny in Lambda containers — Chromium crashes on complex pages | `browsercat.com` troubleshooting | `--disable-dev-shm-usage` in launch args |
| `--ipc=host` Docker flag recommended locally | Playwright docs | Only applies to `docker run` locally, not Lambda |
| Version match required: Docker image Playwright version MUST match pip-installed `playwright` version | Playwright docs | Pin both in sync (`v1.58.2` in FROM and `playwright==1.58.2` in requirements) |
| Noble vs Jammy vs Focal: Focal being deprecated | Playwright Docker docs | Use `-noble` tag (Ubuntu 24.04) for new builds |
| Root user by default in Microsoft image — disables sandbox | Playwright Docker docs | Acceptable for trusted code (our case). For untrusted, add `pwuser` + seccomp profile. |
| Local Lambda testing — use `aws-lambda-rie` (Runtime Interface Emulator) | `stasdeep.com` | For Web Adapter path: not needed; the image runs as normal uvicorn locally. |
| `__enter__`/`__exit__` on Playwright browser may dangle on Lambda timeout | General Playwright | Use `with sync_playwright() as p:` and keep handler lean; ensure cleanup in finally blocks |

## Section 8 — Open Items For the Dev Brief

1. **ARM64 status re-verification.** Mamezou's July 2024 note that "Chrome For Testing is x86-only" on Lambda ARM may be outdated. A 30-min spike: pull `mcr.microsoft.com/playwright/python:v1.58.2-noble` for `linux/arm64`, deploy to a test Lambda with `Architecture: arm64`, run a trivial `chromium.launch() → screenshot()`. If it works, ARM saves ~20% on Lambda compute cost.
2. **Amazon S3 Files research spike.** Confirm the feature exists in our target region, confirm mount semantics match EFS closely enough, benchmark cold-start impact vs S3-zip download. Time-box 2 hours; if unclear, stick with S3-zip.
3. **Fallback service scope.** If the S3 zip is missing or corrupt, the container boots a minimal health-check app. Decide: does this app have a `/health` endpoint only, or does it expose a diagnostics endpoint showing why the real service failed to load? Affects on-call debugging.
4. **Code zip bundle strategy.** Options:
   (a) Pure Python source zip (small, fast, but deps must be in `requirements.base.txt` baked into image — inflexible)
   (b) Source + additional requirements zip (flexible but now the container `pip install`s at boot — adds 10–60s to cold start)
   (c) Source + vendored deps in a `.zip` (large but self-contained)
   Recommend (a) for Phase 1, (b) if specific tools need bespoke deps.
5. **Proxy support in Dockerfile.** The Lambda Web Adapter respects `HTTP_PROXY`/`HTTPS_PROXY`; Playwright does not (see Gotcha #2). The container's startup script must forward proxy env vars into Playwright's `chromium.launch(proxy={...})` call. Logic belongs in the FastAPI app's lifespan hook.
6. **Image registry choice.** Push to private ECR (standard SG/Send deployment target), or also to Docker Hub for public distribution? The arch brief lists Docker Hub as a target — but that implies publishing an opinionated image that others could use, which has different security implications.
7. **Base image rebuild cadence.** Monthly with each Playwright minor release, or pinned indefinitely and only rebuilt on Playwright security patches? Recommend monthly to stay within Microsoft's tested matrix.
8. **Memory / timeout Lambda defaults.** Propose: 3072 MB memory, 300s timeout. Override per-invocation via env vars if needed.
