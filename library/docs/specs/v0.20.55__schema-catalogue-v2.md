## Section 1 — Design Principles

### 1.1 Runs Everywhere

The service must execute identically across five deployment surfaces:

| Surface | Constraints |
|---|---|
| Developer laptop | Full control; can run headful; persistent disk; any duration |
| CI pipeline (GitHub Actions) | Headless; ephemeral; short duration; network-allowlisted |
| Claude Web / Claude Code sessions | Headless; ephemeral; `/tmp` only; TLS-intercepting proxy on egress |
| Container / K8s / Fargate | Headless; long-lived; persistent disk via volume mounts |
| AWS Lambda (container) | Headless; 15-min max per invocation; `/tmp` only; 5 GB memory |

**Schema implication:** the caller declares *intent* (e.g. "I want a persistent session that lives across requests") and the service validates against *capability* on the current deployment target. A persistent-session request against Lambda gets rejected with a helpful error suggesting Fargate. The schemas never assume any single environment.

### 1.2 Sink-Flexible Artefacts

Every artefact (screenshot, video, HAR, trace) must be routable to one of four sinks:

- **vault** — committed to an SG/Send vault (the 16 April arch brief's primary path)
- **inline** — base64-encoded in the HTTP response (small artefacts only)
- **s3** — written directly to an S3 key (large artefacts, no vault roundtrip)
- **local_file** — written to a local path (laptop / container dev only; rejected on Lambda)

Each capture flag has an independent sink choice. Rationale: a laptop dev wants inline screenshots for debugging + no vault; a production QA run wants vault-only; a Fargate content pipeline wants S3 video + vault metadata.

### 1.3 Client Package Model

The HTTP/API surface documented here is the **transport layer**, not the primary caller interface. A separate client package (`sg-playwright-client`) will provide domain-specific methods that wrap these endpoints. Callers of the client package do not need to know about HTTP request details, JSON schemas, or API versioning. The client package resolves version compatibility using the `/info` endpoint's version fields (`service_version`, `playwright_version`, etc.) before dispatching requests.

This means:
- **Wire-format backwards compatibility** is managed by adding new fields with defaults (never removing fields)
- **Breaking changes** are handled by the client package detecting the service version and adapting
- **Step action vocabulary** can grow without breaking old clients — unknown actions are rejected by the dispatcher, not by the schema

### 1.4 Schema Boundaries

- Schemas are **pure data containers** — no methods, no logic. Logic lives in `Playwright__Service`, `Sequence__Dispatcher`, `Session__Manager`, `Artefact__Writer__*` helper classes (routes doc).
- Collection subclasses are **pure type definitions** — no methods.
- All identifiers inherit from the appropriate `Safe_*` primitive — no raw `str`.
- All durations are `Safe_UInt__Milliseconds` — no ambiguity about units.
- All sizes are `Safe_UInt__FileSize` — bytes, with `.to_kb()` / `.to_mb()` helpers built in.
- **No `Literal` types.** All fixed-value sets use proper `Enum__*` classes for runtime validation, serialisation, and IDE discoverability.

### 1.5 Forward References

Type_Safe forward references must point to the same class. Cross-schema references resolve through actual class imports — no `'Schema__Other'` string references.

## Section 2 — Naming Conventions

| Prefix / suffix | Applied to |
|---|---|
| `Schema__` | Type_Safe data container classes |
| `Enum__` | Python Enum subclasses (`(str, Enum)` for JSON serialisation) |
| `Safe_Str__`, `Safe_UInt__`, `Safe_Int__`, `Safe_Float__` | Custom primitive subclasses |
| `Dict__*__By_*`, `List__*`, `Set__*`, `Tuple__*` | Type_Safe collection subclasses |
| `*__Id` | `Safe_Id`-derived identifier types |
| `ENV_VAR__*` | Module-level string constants for environment variable names |
| `TAG__ROUTES_*`, `ROUTES_PATHS__*` | Fast-API routing constants (routes doc) |
