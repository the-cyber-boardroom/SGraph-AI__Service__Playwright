# Vault Sync — Contract Spec

> **PROPOSED — does not exist yet.** The reality doc does not list any `vault-sync` package, route, or controller node. Until code lands and the reality doc is updated, every claim below is aspiration.

## §0. What this is

`vault-sync` is a thin orchestrator that **moves vault data between SG/API instances** using `sgit` as its only transport. It owns three concerns:

1. **Sync** — `push`, `pull`, and `sync pair` of named vaults between named SG/API targets.
2. **Registry** — a self-hosted "vault-sync vault" that stores the list of known vaults, the list of known SG/API targets, the mapping of which vault lives at which target, and the last-known commit per location.
3. **Visibility** — `status` queries (per-vault, per-target, full graph) that read from the registry and report drift.

It is **not** responsible for: provisioning EC2, DNS/cert issuance, vault rendering, vault viewers, allowlist gating, or merging diverging vaults. Those concerns belong to other tools.

## §1. Scope correction vs. the source brief

The 13 May 2026 brief listed `deploy ephemeral/persistent` as first-class verbs and made "vault → fresh certed instance → live URL" one of the two "what good looks like" scenarios. **This spec drops that scope.** Confirmed in thread on 2026-05-16: vault-sync syncs only — it never spins up instances or deploys viewers.

| Brief item | Spec decision |
|---|---|
| `deploy ephemeral` / `deploy persistent` verbs | **Removed.** Use `sg-compute` directly. |
| Scenario 2 ("vault → live URL in one command") | **Removed.** Out of scope. |
| Acceptance criteria #2, #7, #9 | **Removed.** |
| GitHub webhook hook | **Deferred.** v1 is CLI-driven only. See §7. |
| Visualisation dashboard | **Deferred.** v1 emits JSON via `status graph`; UI consumer is a separate spec. |
| Allowlist of vault-ids for privileged ops | **Removed.** Possession of `SG_VAULT_SYNC__KEY` is the authorization. |

## §2. Authorization model

A single env var **`SG_VAULT_SYNC__KEY`** holds the `sgit` key for the vault-sync vault itself. Possession of that key grants full admin capability — list/add targets, register mappings, trigger any sync between any registered vaults.

This is a deliberately concentrated trust boundary. The vault-sync vault is a sensitive asset by construction. Hardening (split-key, audit log, per-target capability scoping) is out of scope for v1 and tracked separately.

Per-target `sgit` credentials are referenced by name from `targets.json` (see §4) and stored **inside** the vault-sync vault for v1. This is the most expedient option and a known concentration risk; isolating per-target credentials into separate vaults is a v2 hardening.

## §3. Transport

`sgit` is the **only** transport. Every `push`/`pull`/`sync` operation is a sub-process invocation of the `sgit` CLI against an SG/API endpoint. No alternative wire protocol is invented; no direct HTTP calls to SG/API are made from vault-sync.

The SGit CLI agentic team is the authority on the `sgit` surface. Any divergence between this spec's invocation patterns and the actual `sgit` CLI is resolved in their favour.

## §4. The vault-sync vault — registry schema

The vault-sync vault holds three families of files. All paths are relative to vault root.

### 4.1 `targets/<target-id>.json`

One file per known SG/API endpoint.

| Field | Type | Notes |
|---|---|---|
| `target_id` | `Safe_Str__Target__Id` | snake_case, matches filename stem |
| `display_name` | `Safe_Str__Display__Name` | human label |
| `sg_api_url` | `Safe_Str__Url` | base URL of the SG/API |
| `sgit_credential_ref` | `Safe_Str__Credential__Ref` | name of credential file in `credentials/` |
| `created_at` | `Timestamp_Now` | |
| `notes` | `Safe_Str__Note` | free text |

### 4.2 `vaults/<vault-id>.json`

One file per known vault.

| Field | Type | Notes |
|---|---|---|
| `vault_id` | `Safe_Str__Vault__Id` | snake_case, matches filename stem |
| `display_name` | `Safe_Str__Display__Name` | |
| `mirrors` | `List__Schema__Vault__Mirror` | one entry per `(target_id, remote_path)` pair |
| `last_seen_commits` | `Dict__Target__Commit` | `target_id → sgit commit sha` |
| `last_sync` | `Dict__Target__Timestamp` | `target_id → last successful sync timestamp` |
| `last_error` | `Dict__Target__Error` | `target_id → last error message` (empty when healthy) |
| `created_at` | `Timestamp_Now` | |

### 4.3 `credentials/<credential-ref>.json`

Per-target `sgit` credentials. v1 stores them inside the vault-sync vault. Schema deliberately opaque here — defined by the SGit CLI team.

### 4.4 No `hooks/` folder in v1

Webhook receiver is deferred (§7). When introduced, it will live at `hooks/<hook-id>.json`.

## §5. CLI surface

`vault-sync` ships as a new top-level spec under `sg_compute_specs/vault_sync/`, conforming to [`v0.2.6__authoring-a-new-top-level-spec.md`](v0.2.6__authoring-a-new-top-level-spec.md). The CLI is generated by `Spec__CLI__Builder` and inherits the 8 standard verbs from [`v0.2.6__spec-cli-contract.md`](v0.2.6__spec-cli-contract.md).

The 8 standard verbs (`list/info/create/delete/wait/health/connect/exec`) operate on **vault-sync controller nodes** — the EC2 instance running `Fast_API__Vault__Sync` (§6). Bringing up the controller is no different from bringing up any other SG/Compute spec.

Vault-sync's domain verbs are **spec extras**, registered on the builder's returned `typer.Typer` per §2 of the CLI contract:

```
sg-compute spec vault_sync ...        # the 8 standard verbs (controller node mgmt)

sg-compute spec vault_sync push <vault-id> --to <target-id>
sg-compute spec vault_sync push <vault-id> --all
sg-compute spec vault_sync pull <vault-id> --from <target-id>

sg-compute spec vault_sync sync pair <vault-id> <target-a> <target-b>
sg-compute spec vault_sync sync all                                   # fan-out fast-forward

sg-compute spec vault_sync status vault <vault-id>
sg-compute spec vault_sync status target <target-id>
sg-compute spec vault_sync status graph                               # JSON for downstream UIs

sg-compute spec vault_sync targets list
sg-compute spec vault_sync targets add <target-id> --url <sg-api-url> --credential-ref <ref>
sg-compute spec vault_sync targets remove <target-id>

sg-compute spec vault_sync vaults list
sg-compute spec vault_sync vaults register <vault-id> --display-name <name>
sg-compute spec vault_sync vaults forget <vault-id>

sg-compute spec vault_sync mirrors add <vault-id> --target <target-id> --remote-path <path>
sg-compute spec vault_sync mirrors remove <vault-id> --target <target-id>

sg-compute spec vault_sync meta version
```

Every domain verb is JSON-in / JSON-out, error-wrapped via `@spec_cli_errors`. All verbs operate against the controller node — the CLI is a thin client that calls `Fast_API__Vault__Sync` (§6).

## §6. `Fast_API__Vault__Sync` — the controller surface

The controller is a long-lived FastAPI service running on a dedicated EC2 instance with its own DNS and cert. It uses the same patterns as the Playwright service: `Serverless__Fast_API` base, `Type_Safe` schemas, routes-no-logic. It is protected by an `X-API-Key` header — same shape as `Fast_API__Compute`.

It is the **only** process that loads `SG_VAULT_SYNC__KEY` and the **only** process that invokes `sgit`. The CLI is a thin client over its HTTP surface.

### 6.1 Route inventory (v1)

| Method | Path | Verb |
|---|---|---|
| `GET`  | `/api/health/info` | health (standard) |
| `GET`  | `/api/health/ping` | ping (standard) |
| `POST` | `/api/sync/push` | push one vault to one or all targets |
| `POST` | `/api/sync/pull` | pull one vault from a source target |
| `POST` | `/api/sync/pair` | sync pair of targets for one vault |
| `POST` | `/api/sync/all` | fan-out fast-forward across all mappings |
| `GET`  | `/api/status/vault/{vault_id}` | per-vault status + drift |
| `GET`  | `/api/status/target/{target_id}` | per-target status |
| `GET`  | `/api/status/graph` | full graph as JSON |
| `GET`  | `/api/targets` | list |
| `POST` | `/api/targets` | add |
| `DELETE` | `/api/targets/{target_id}` | remove |
| `GET`  | `/api/vaults` | list |
| `POST` | `/api/vaults` | register |
| `DELETE` | `/api/vaults/{vault_id}` | forget |
| `POST` | `/api/vaults/{vault_id}/mirrors` | add mirror |
| `DELETE` | `/api/vaults/{vault_id}/mirrors/{target_id}` | remove mirror |

### 6.2 Sync vs. async

Per the 2026-05-16 thread: **default sync.** A request returns when the work is done.

The two exceptions, allowed because they are idempotent and re-firing converges:

| Endpoint | Mode | Why |
|---|---|---|
| `POST /api/sync/push` (with `--all`) | async ack | fan-out to N targets can be slow; each push is idempotent |
| `POST /api/sync/pull` triggered by a future webhook | async ack | webhook needs sub-10s ack |

All other endpoints — including single-target `push`, single-source `pull`, all `status` and `targets`/`vaults`/`mirrors` admin verbs — are synchronous.

The async mechanism is an **in-process worker** on the controller. No external queue, no folder-as-queue inside the vault. Requests that would block return `202 Accepted` with a job id; status is queryable via `/api/status/vault/{vault_id}`.

### 6.3 Conflict policy

`sync pair` is **fast-forward-or-fail**. If neither side is an ancestor of the other, the operation aborts and leaves both vaults exactly as they were. The error response includes both commit shas. The operator resolves divergence using `sgit` directly — vault-sync does not auto-merge.

## §7. Webhooks — DEFERRED

The original brief made hooks a first-class concern. v1 omits them entirely. v2 will add:

- `Routes__Vault__Sync__Webhook` on the controller
- `hooks/<hook-id>.json` registry files in the vault-sync vault
- GitHub webhook signature verification (`X-Hub-Signature-256`) with secret stored in the hook record
- Async ack model per §6.2

Until then, all triggers are CLI-driven (manual, scripted from cron, or invoked by other agents over SSH/API).

## §8. Repository layout decision

vault-sync lives **as a new spec under `sg_compute_specs/vault_sync/`** — same repo as the `sg_compute_specs` package. Justifications:

1. It's a third agent-CLI in the fleet (`sp-cli`, `sg-compute`, now `vault_sync`); reusing `Spec__CLI__Builder` keeps UX consistent.
2. It's an EC2-resident workload; SG/Compute already owns that lifecycle.
3. The webhook receiver (when added) is a FastAPI service; the spec contract already covers Routes registration.

It is **not** added as code in this Playwright repo — only the contract document lives here, because the contract has cross-cutting implications for the routes/schema catalogues this repo owns.

The Playwright repo's responsibility:

- Hold this contract spec.
- Update the reality doc when a vault-sync controller becomes a real consumer of vaults this service produces.
- Nothing else.

## §10. Acceptance criteria (revised)

| # | Criterion | Verification |
|---|---|---|
| 1 | `vault-sync push <vault-id> --to <target>` moves the vault between two SG/API targets via `sgit` | manual end-to-end |
| 2 | `vault-sync pull <vault-id> --from <source>` works symmetrically | manual end-to-end |
| 3 | `vault-sync sync pair` succeeds on fast-forward, **fails cleanly** on divergence, leaves both vaults untouched on failure | integration test |
| 4 | `vault-sync status vault <id>` lists every mapped target with last-seen commit and drift indicator | integration test |
| 5 | `vault-sync status graph` emits JSON consumable by a graph-rendering UI | schema test |
| 6 | Registry schema (§4) is `Type_Safe` end-to-end — no Pydantic, no Literals, no raw primitives | schema test |
| 7 | `Fast_API__Vault__Sync` runs on a long-lived controller node with `Serverless__Fast_API` and `X-API-Key` protection | deploy-via-pytest |
| 8 | `SG_VAULT_SYNC__KEY` is the only credential the controller process loads at boot | boot test |
| 9 | Spec lives under `sg_compute_specs/vault_sync/` and conforms to [`v0.2.6__authoring-a-new-top-level-spec.md`](v0.2.6__authoring-a-new-top-level-spec.md) | manifest contract test passes |
| 10 | Reality doc updated under `team/roles/librarian/reality/sg-compute/` once code lands | librarian sign-off |

## §11. Out of scope (explicit non-goals for v1)

- EC2 / DNS / cert provisioning (use SG/Compute directly)
- Vault rendering / viewing (separate viewer spec)
- Webhook receivers (v2)
- Visualisation UI (separate `sp-cli-vault-sync-graph` web component, v2)
- Auto-merge of diverging vaults (operator's job, via `sgit`)
- Per-target capability scoping / split-key authorization (v2 hardening)
- Persistent audit log of sync operations (v2)
- Scheduled / cron-triggered sync (v2 — relies on hooks)
