# Authoring a New Top-Level Spec — Developer Guide

Status: **Definitive contract.** A new top-level Spec is conformant only if every
mandatory item in this document is present and the manifest contract test
passes. Optional items are clearly labelled.

Audience: developers adding a new workload type (e.g. `redis`, `jupyter`,
`grafana`, `headless-chrome`, …) to SG/Compute.

Pinned to: `sg_compute v0.2.6` / `sg_compute_specs v0.1.0`.

---

## 1. What "Spec" Means

A **Spec** is a self-contained workload definition that the SG/Compute control
plane can launch as an ephemeral EC2 node, expose over HTTP, and surface in the
CLI and dashboard. Each Spec lives in its own Python package under
[`sg_compute_specs/`](../../../sg_compute_specs/) and is discovered by
convention — no central registration table.

Three pilots already ship and serve as canonical references:

| Spec          | Location                                        | Use as template for…                  |
|---------------|-------------------------------------------------|---------------------------------------|
| `docker`      | `sg_compute_specs/docker/`                      | **Full feature set** — schemas, routes, CLI, UI, helpers, tests. Use this as your scaffold. |
| `ollama`      | `sg_compute_specs/ollama/`                      | A simpler service with no UI / routes layer yet — useful when starting "manifest-only". |
| `open_design` | `sg_compute_specs/open_design/`                 | Vault-writing capability example.     |

Any code path that branches on `spec_id` is a smell — Spec discovery is
manifest-driven and convention-routed.

## 2. Architecture in One Picture

```
┌────────────────────────────────────────────────────────────────────────────┐
│  sg_compute_specs/{spec_id}/                                               │
│  ──────────────────────────                                                │
│   manifest.py             ← MANIFEST: Schema__Spec__Manifest__Entry        │
│   schemas/                ← Schema__{Pascal}__Create__Request, …, Info     │
│   service/                ← {Pascal}__Service (Tier-1 orchestrator)        │
│   api/routes/             ← Routes__{Pascal}__Stack (Fast_API__Routes)     │
│   cli/                    ← Cli__{Pascal} (typer.Typer `app`) + Renderers  │
│   primitives/  enums/  collections/   ← per-spec Type_Safe additions       │
│   tests/                  ← test_manifest.py + service/route tests         │
│   ui/card/v0/v0.1/v0.1.0/   ← optional dashboard card web component       │
│   ui/detail/v0/v0.1/v0.1.0/ ← optional dashboard detail web component     │
│   README.md               ← optional, served at /api/specs/{id}/readme     │
│   version                 ← semver, read by manifest.py                    │
└────────────────────────────────────────────────────────────────────────────┘
                  │
                  │  Discovered by three convention-driven loaders:
                  ▼
┌────────────────────────────────────────────────────────────────────────────┐
│  sg_compute/core/spec/Spec__Loader            → walks  *manifest.py        │
│  sg_compute/cli/Spec__CLI__Loader             → imports cli.Cli__{Pascal}  │
│  sg_compute/control_plane/Spec__Routes__Loader → imports api.routes.       │
│                                                  Routes__{Pascal}__Stack   │
└────────────────────────────────────────────────────────────────────────────┘
                  │
                  ▼
┌──────────────────────────────────┐    ┌──────────────────────────────────┐
│  sg-compute  (typer)             │    │  Fast_API__Compute               │
│   spec list                      │    │   GET  /api/specs                │
│   spec info  <id>                │    │   GET  /api/specs/{id}           │
│   spec <id> {list|info|create|…} │    │   GET  /api/specs/{id}/readme    │
│   node {list|info|create|delete} │    │   *    /api/specs/{id}/...       │
│                                  │    │   GET  /api/specs/{id}/ui/...    │
└──────────────────────────────────┘    └──────────────────────────────────┘
```

**Key invariant:** if you add a new spec correctly, the platform learns about
it without anyone editing `Cli__Compute`, `Fast_API__Compute`, or any registry
table. That is the *only* test of "did I name things right".

## 3. Mandatory Pieces

The table below is the contract. Anything marked Mandatory must exist for the
spec to count as "shipped".

### 3.1  Naming Convention

For a spec_id `my_spec` (snake_case, lowercase, regex
`^[a-z][a-z0-9_]*$`), the **PascalCase form is `MySpec`** — `_` is removed and
each word capitalised. Examples:

| spec_id        | PascalCase   | Class name examples                                          |
|----------------|--------------|--------------------------------------------------------------|
| `docker`       | `Docker`     | `Routes__Docker__Stack`, `Cli__Docker`, `Docker__Service`    |
| `open_design`  | `OpenDesign` | `Routes__OpenDesign__Stack`, `Cli__OpenDesign`               |
| `headless_chrome` | `HeadlessChrome` | `Routes__HeadlessChrome__Stack`, `Cli__HeadlessChrome`  |

The PascalCase rule is enforced by
[`sg_compute/cli/Spec__CLI__Loader.py`](../../../sg_compute/cli/Spec__CLI__Loader.py)
and
[`sg_compute/control_plane/Spec__Routes__Loader.py`](../../../sg_compute/control_plane/Spec__Routes__Loader.py).
A misnamed class is silently skipped — discovery just doesn't find your CLI or
routes. If `sg-compute spec` doesn't list your sub-app, check the name first.

CLAUDE.md rules apply throughout: `Type_Safe` only, no Pydantic, no Literals,
zero raw primitives in schema attributes, one class per file, empty
`__init__.py`, 80-char `═══` headers, inline comments only.

### 3.2  Package Skeleton

```
sg_compute_specs/{spec_id}/
├── __init__.py                  # empty
├── version                      # semver, e.g. 0.1.0
├── manifest.py                  # MANDATORY  — exports MANIFEST
├── schemas/
│   ├── __init__.py              # empty
│   ├── Schema__{Pascal}__Create__Request.py     # MANDATORY
│   ├── Schema__{Pascal}__Create__Response.py    # MANDATORY (when routes exist)
│   ├── Schema__{Pascal}__Info.py                # MANDATORY (when routes exist)
│   ├── Schema__{Pascal}__List.py                # MANDATORY (when routes exist)
│   ├── Schema__{Pascal}__Delete__Response.py    # MANDATORY (when routes exist)
│   └── Schema__{Pascal}__Health__Response.py    # optional
├── service/
│   ├── __init__.py              # empty
│   └── {Pascal}__Service.py     # MANDATORY  — Tier-1 orchestrator
├── api/
│   ├── __init__.py              # empty
│   └── routes/
│       ├── __init__.py          # empty
│       └── Routes__{Pascal}__Stack.py           # MANDATORY (when HTTP surface)
├── cli/
│   ├── __init__.py              # empty
│   ├── Cli__{Pascal}.py         # MANDATORY (when CLI surface)
│   └── Renderers.py             # MANDATORY (when CLI surface)
├── primitives/  enums/  collections/            # optional, per-spec types
├── tests/
│   ├── __init__.py              # empty
│   └── test_manifest.py         # MANDATORY contract test
├── ui/                          # optional dashboard components
│   ├── card/v0/v0.1/v0.1.0/sg-compute-{spec_id}-card.{js,html,css}

└── README.md                    # optional, served at /api/specs/{id}/readme
```

`{spec_id}` and `{Pascal}` placeholders follow §3.1.

### 3.3  `manifest.py` — the Single Source of Truth

Every spec **must** export a module-level `MANIFEST` of type
[`Schema__Spec__Manifest__Entry`](../../../sg_compute/core/spec/schemas/Schema__Spec__Manifest__Entry.py).
`Spec__Loader` walks `sg_compute_specs/*/manifest.py` and asserts the type. A
missing or wrong-typed `MANIFEST` raises at boot.

Required field reference:

| Field                    | Type                          | Notes                                                                            |
|--------------------------|-------------------------------|----------------------------------------------------------------------------------|
| `spec_id`                | `str`                         | Lowercase, snake_case, matches the package directory name. Globally unique.      |
| `display_name`           | `str`                         | Human-readable label for CLI/dashboard.                                          |
| `icon`                   | `str`                         | Single-character emoji (or empty). Optional but recommended.                     |
| `version`                | `str`                         | Semver `MAJOR.MINOR.PATCH`. Convention: read from the `version` file (see ollama). |
| `stability`              | `Enum__Spec__Stability`       | `STABLE` / `EXPERIMENTAL` / `DEPRECATED`.                                        |
| `boot_seconds_typical`   | `int`                         | Used by the dashboard to set a launch progress baseline.                         |
| `capabilities`           | `List[Enum__Spec__Capability]` | One or more from the 12-entry enum (`CONTAINER_RUNTIME`, `LLM_INFERENCE`, …).   |
| `nav_group`              | `Enum__Spec__Nav_Group`       | Dashboard navigation tab — `BROWSERS / DATA / OBSERVABILITY / STORAGE / AI / DEV / OTHER`. |
| `extends`                | `List[str]`                   | spec_ids this spec composes on top of. Cycles are rejected by `Spec__Resolver`. |
| `soon`                   | `bool`                        | When `True`, dashboard renders the spec as a "coming soon" card.                |
| `create_endpoint_path`   | `str`                         | Must start with `/api/specs/`. Tells the dashboard where to POST a create request. |

Reference (the docker manifest is the canonical minimal example —
[`sg_compute_specs/docker/manifest.py`](../../../sg_compute_specs/docker/manifest.py)):

```python
# ═══════════════════════════════════════════════════════════════════════════════
# SG/Compute Specs — {Pascal}: manifest
# ═══════════════════════════════════════════════════════════════════════════════

from pathlib                                                                  import Path

from sg_compute.core.spec.schemas.Schema__Spec__Manifest__Entry              import Schema__Spec__Manifest__Entry
from sg_compute.primitives.enums.Enum__Spec__Capability                      import Enum__Spec__Capability
from sg_compute.primitives.enums.Enum__Spec__Nav_Group                       import Enum__Spec__Nav_Group
from sg_compute.primitives.enums.Enum__Spec__Stability                       import Enum__Spec__Stability


def _read_version() -> str:
    return (Path(__file__).parent / 'version').read_text().strip()


MANIFEST = Schema__Spec__Manifest__Entry(
    spec_id              = '{spec_id}'                                ,
    display_name         = '{Display Name}'                           ,
    icon                 = '🟢'                                        ,
    version              = _read_version()                            ,
    stability            = Enum__Spec__Stability.EXPERIMENTAL         ,
    boot_seconds_typical = 120                                        ,
    capabilities         = [Enum__Spec__Capability.CONTAINER_RUNTIME] ,
    nav_group            = Enum__Spec__Nav_Group.OTHER                ,
    extends              = []                                         ,
    soon                 = False                                      ,
    create_endpoint_path = '/api/specs/{spec_id}/stack'               ,
)
```

**If your `Enum__Spec__Capability` value doesn't already exist**, add it to
[`sg_compute/primitives/enums/Enum__Spec__Capability.py`](../../../sg_compute/primitives/enums/Enum__Spec__Capability.py)
before referencing it. Same for `Enum__Spec__Nav_Group`.

### 3.4  Schemas

All payload schemas extend `Type_Safe`. Schemas are pure data — no methods.
**Zero raw primitives**: every attribute must be a `Safe_*` primitive, an
`Enum__*`, another schema, or a typed collection subclass
(`List__*` / `Dict__*` from
[`osbot_utils.type_safe.Type_Safe__List`](https://pypi.org/project/osbot-utils/)
or per-spec collections).

Mandatory schemas when you have an HTTP surface:

| Schema                                  | Used by                                       |
|-----------------------------------------|-----------------------------------------------|
| `Schema__{Pascal}__Create__Request`     | `POST /api/specs/{spec_id}/stack`             |
| `Schema__{Pascal}__Create__Response`    | `POST` response — wraps the `Info` plus message/timing. |
| `Schema__{Pascal}__Info`                | `GET  /api/specs/{spec_id}/stack/{name}` and embedded in responses. |
| `Schema__{Pascal}__List`                | `GET  /api/specs/{spec_id}/stacks`             |
| `Schema__{Pascal}__Delete__Response`    | `DELETE /api/specs/{spec_id}/stack/{name}`     |
| `Schema__{Pascal}__Health__Response`    | optional — `GET /api/specs/{spec_id}/stack/{name}/health`. |

Reference: [`sg_compute_specs/docker/schemas/`](../../../sg_compute_specs/docker/schemas/).

A `Create__Request` for an EC2-backed workload typically reuses these shared
primitives from `sg_compute/`:

- `Safe_Str__AWS__Region`
- `Safe_Str__SSM__Path`
- `Safe_Str__Stack__Name` (or your own per-spec stack-name primitive with a
  tighter regex — see
  [`Safe_Str__Docker__Stack__Name`](../../../sg_compute_specs/docker/primitives/Safe_Str__Docker__Stack__Name.py))
- `Safe_Int__Hours`, `Safe_Int__Max__Hours`
- `Safe_Str__AMI__Id` (under `sg_compute/platforms/ec2/primitives/`)

Anything your spec adds that isn't in `sg_compute/primitives/` lives under
`sg_compute_specs/{spec_id}/primitives/`, one class per file.

### 3.5  Service — the Tier-1 Orchestrator

`{Pascal}__Service` is the single class the CLI and routes call. It holds
references to per-spec helpers (AWS client, mapper, name generator, user-data
builder, health checker), built lazily in `setup()`.

Mandatory public surface (used by routes / CLI):

```python
class {Pascal}__Service(Type_Safe):
    aws_client        : Optional[{Pascal}__AWS__Client]        = None
    user_data_builder : Optional[{Pascal}__User_Data__Builder] = None
    mapper            : Optional[{Pascal}__Stack__Mapper]      = None
    ip_detector       : Optional[Caller__IP__Detector]         = None
    name_gen          : Optional[Stack__Name__Generator]       = None
    health_checker    : Optional[{Pascal}__Health__Checker]    = None

    def setup(self) -> '{Pascal}__Service': ...

    def create_stack    (self, request: Schema__{Pascal}__Create__Request,
                         creator: str = '') -> Schema__{Pascal}__Create__Response: ...
    def list_stacks     (self, region: Safe_Str__AWS__Region) -> Schema__{Pascal}__List: ...
    def get_stack_info  (self, region: Safe_Str__AWS__Region,
                         stack_name: Safe_Str__Stack__Name) -> Optional[Schema__{Pascal}__Info]: ...
    def delete_stack    (self, region: Safe_Str__AWS__Region,
                         stack_name: Safe_Str__Stack__Name) -> Schema__{Pascal}__Delete__Response: ...
    def health          (self, region: Safe_Str__AWS__Region,
                         stack_name: Safe_Str__Stack__Name,
                         timeout_sec: int = 300, poll_sec: int = 10
                         ) -> Schema__{Pascal}__Health__Response: ...      # optional
```

To participate in cross-spec node creation (`POST /api/nodes` and
`sg-compute node create {spec_id}`), also implement:

```python
    def create_node(self, base_request,
                    api_key_ssm_path: Safe_Str__SSM__Path = Safe_Str__SSM__Path()
                    ) -> 'Schema__Node__Info':
        ...                                                                    # see Docker__Service.create_node
```

`base_request` is a `Schema__Node__Create__Request__Base`. Map its fields to
your spec's `Create__Request`, call `create_stack`, then return a
`Schema__Node__Info`. `EC2__Platform.create_node` dispatches on `spec_id` and
will call this method on your service.

Reference: [`Docker__Service`](../../../sg_compute_specs/docker/service/Docker__Service.py).

**Side effects:** emit a `Schema__Stack__Event` on the singleton `event_bus`
for `created` and `deleted` so consumers (LETS, dashboard streams) can
observe the lifecycle.

### 3.6  Routes — `Routes__{Pascal}__Stack`

Routes are zero-logic delegators. They MUST extend `Fast_API__Routes` and
declare a `service` attribute typed to your `{Pascal}__Service` so
`Fast_API__Compute._make_service()` can auto-construct it.

Skeleton (copied from
[`Routes__Docker__Stack`](../../../sg_compute_specs/docker/api/routes/Routes__Docker__Stack.py)):

```python
# ═══════════════════════════════════════════════════════════════════════════════
# SG/Compute Specs — {Pascal}: Routes__{Pascal}__Stack
# ═══════════════════════════════════════════════════════════════════════════════

from fastapi                                                                  import HTTPException
from osbot_fast_api.api.routes.Fast_API__Routes                               import Fast_API__Routes

from sg_compute_specs.{spec_id}.schemas.Schema__{Pascal}__Create__Request    import Schema__{Pascal}__Create__Request
from sg_compute_specs.{spec_id}.service.{Pascal}__Service                    import DEFAULT_REGION, {Pascal}__Service


TAG__ROUTES_{UPPER} = '{spec_id}'


class Routes__{Pascal}__Stack(Fast_API__Routes):
    tag     : str             = TAG__ROUTES_{UPPER}
    service : {Pascal}__Service

    def list_stacks(self, region: str = '') -> dict:
        return self.service.list_stacks(region or DEFAULT_REGION).json()
    list_stacks.__route_path__ = '/stacks'

    def info(self, name: str, region: str = '') -> dict:
        result = self.service.get_stack_info(region or DEFAULT_REGION, name)
        if result is None:
            raise HTTPException(status_code=404, detail=f'no {spec_id} stack matched {name!r}')
        return result.json()
    info.__route_path__ = '/stack/{name}'

    def create(self, body: Schema__{Pascal}__Create__Request) -> dict:
        return self.service.create_stack(body).json()
    create.__route_path__ = '/stack'

    def delete(self, name: str, region: str = '') -> dict:
        response = self.service.delete_stack(region or DEFAULT_REGION, name)
        if not response.deleted:
            raise HTTPException(status_code=404, detail=f'no {spec_id} stack matched {name!r}')
        return response.json()
    delete.__route_path__ = '/stack/{name}'

    def setup_routes(self):
        self.add_route_get   (self.list_stacks)
        self.add_route_get   (self.info       )
        self.add_route_post  (self.create     )
        self.add_route_delete(self.delete     )
```

You get these endpoints for free (auto-mounted under
`/api/specs/{spec_id}/`):

| Method   | Path                                                |
|----------|-----------------------------------------------------|
| `GET`    | `/api/specs/{spec_id}/stacks`                       |
| `GET`    | `/api/specs/{spec_id}/stack/{name}`                 |
| `POST`   | `/api/specs/{spec_id}/stack`                        |
| `DELETE` | `/api/specs/{spec_id}/stack/{name}`                 |
| `GET`    | `/api/specs/{spec_id}/stack/{name}/health`  (optional) |

Plus, regardless of whether you wrote routes:

| Method | Path                            | Source                       |
|--------|---------------------------------|------------------------------|
| `GET`  | `/api/specs`                    | `Routes__Compute__Specs`     |
| `GET`  | `/api/specs/{spec_id}`          | `Routes__Compute__Specs`     |
| `GET`  | `/api/specs/{spec_id}/readme`   | `Routes__Compute__Specs` (404 if no `README.md`) |
| `GET`  | `/api/specs/{spec_id}/ui/...`   | `StaticFiles` mount (only if `ui/` exists) |

### 3.7  CLI — `Cli__{Pascal}` and Renderers

The CLI module **must** export a module-level `app: typer.Typer`. Discovery is
done by [`Spec__CLI__Loader.load`](../../../sg_compute/cli/Spec__CLI__Loader.py),
which imports `sg_compute_specs.{spec_id}.cli.Cli__{Pascal}` and reads `app`.
A missing class or wrong file name silently means "no CLI for this spec" —
your `sg-compute spec` group will not include the sub-app.

Conventional verbs (mirror docker exactly to keep UX consistent):

```
sg-compute spec {spec_id} list   [--region X]
sg-compute spec {spec_id} info   <stack-name> [--region X]
sg-compute spec {spec_id} create [--region X] [--instance-type T] [--max-hours N]
                                  [--name X] [--registry R] [--api-key K]
sg-compute spec {spec_id} delete <stack-name> [--region X] [--yes]
```

Renderer functions live in `cli/Renderers.py`. They take a Rich `Console` and
a schema, and emit pretty output. **No business logic, no AWS calls in the
renderer module.**

Reference:
- [`Cli__Docker`](../../../sg_compute_specs/docker/cli/Cli__Docker.py)
- [`Renderers (docker)`](../../../sg_compute_specs/docker/cli/Renderers.py)

Common error guard you should keep in every command (typer-friendly AWS
credential reporting):

```python
except Exception as e:
    if 'credential' in str(e).lower() or 'NoCredential' in type(e).__name__:
        typer.echo(f'AWS credentials not configured: {e}', err=True)
        raise typer.Exit(1)
    raise
```

### 3.8  Manifest Contract Test (Mandatory)

Every spec ships a `tests/test_manifest.py` that proves the manifest is
loadable and minimally well-formed. Use the docker file as a template
([`sg_compute_specs/docker/tests/test_manifest.py`](../../../sg_compute_specs/docker/tests/test_manifest.py)):

```python
# ═══════════════════════════════════════════════════════════════════════════════
# SG/Compute Specs — {Pascal}: manifest contract test
# ═══════════════════════════════════════════════════════════════════════════════

from unittest                                                              import TestCase

from sg_compute.core.spec.schemas.Schema__Spec__Manifest__Entry            import Schema__Spec__Manifest__Entry
from sg_compute_specs.{spec_id}.manifest                                   import MANIFEST


class test_manifest(TestCase):

    def test_manifest_is_typed(self):
        assert isinstance(MANIFEST, Schema__Spec__Manifest__Entry)

    def test_spec_id(self):
        assert MANIFEST.spec_id == '{spec_id}'

    def test_version_is_semver(self):
        parts = MANIFEST.version.split('.')
        assert len(parts) == 3
        assert all(p.isdigit() for p in parts)

    def test_capabilities_non_empty(self):
        assert len(MANIFEST.capabilities) >= 1

    def test_create_endpoint_path(self):
        assert MANIFEST.create_endpoint_path.startswith('/api/specs/')
```

You should additionally write integration tests for `Service`, route, and
schema round-trips, following the `in_memory_stack` patterns in
`sg_compute__tests/`. **No mocks, no patches** — see
[`library/guides/v3.1.1__testing_guidance.md`](../../guides/v3.1.1__testing_guidance.md).

### 3.9  PEP 621 Entry Point

Add a line to
[`sg_compute_specs/pyproject.toml`](../../../sg_compute_specs/pyproject.toml)
under `[project.entry-points."sg_compute.specs"]`:

```toml
{spec_id} = "sg_compute_specs.{spec_id}.manifest"
```

This is what makes installed wheels of `sg-compute-specs` discoverable by
[`Spec__Loader._load_from_entry_points`](../../../sg_compute/core/spec/Spec__Loader.py).
For in-tree development the filesystem walk picks it up too — but never ship
without the entry point.

## 4. Optional Pieces

### 4.1  Per-Spec Primitives, Enums, Collections

Anything narrower than the cross-spec primitives in `sg_compute/primitives/`
goes under `sg_compute_specs/{spec_id}/primitives/`,
`…/enums/`, `…/collections/`. Examples in docker:

| Class                              | Purpose                                            |
|------------------------------------|----------------------------------------------------|
| `Safe_Str__Docker__Stack__Name`    | regex `^[a-z][a-z0-9\-]{1,62}$`                    |
| `Safe_Str__IP__Address`            | dotted-quad IPv4                                   |
| `Enum__Docker__Stack__State`       | docker-specific lifecycle states                   |
| `List__Schema__Docker__Info`       | typed collection used in `Schema__Docker__List`    |

Naming: one class per file, file named exactly after the class, empty
`__init__.py` in the package. Never re-export through `__init__.py`.

### 4.2  EC2 Helpers

If your spec launches an EC2 node, mirror docker's per-spec helpers under
`service/`:

| Helper                              | Responsibility                                              |
|-------------------------------------|-------------------------------------------------------------|
| `{Pascal}__AWS__Client`             | Composes ec2/ssm boto facades from `osbot_aws`. Never use boto3 directly. |
| `{Pascal}__SG__Helper`              | Per-stack security group. **GroupName MUST NOT start with `sg-`** (CLAUDE.md rule 14). |
| `{Pascal}__AMI__Helper`             | AMI lookup (e.g. AL2023 SSM parameter).                     |
| `{Pascal}__Instance__Helper`        | List/find/terminate, plus SSM probes.                       |
| `{Pascal}__Launch__Helper`          | `run_instances` wrapper.                                    |
| `{Pascal}__Tags__Builder`           | The 6-tag set; `Name` tag prefixed via `Stack__Naming(section_prefix='{spec_id}')`. **Never double-prefix** (CLAUDE.md rule 15). |
| `{Pascal}__Stack__Mapper`           | boto3 dict → `Schema__{Pascal}__Info`.                      |
| `{Pascal}__User_Data__Builder`      | Cloud-init / user-data script. Reuse `Section__*` composables from `sg_compute/platforms/ec2/user_data/`. |
| `{Pascal}__Health__Checker`         | Two-stage poll — EC2 state → service-level probe.           |

Compose with the shared classes under
[`sg_compute/platforms/ec2/`](../../../sg_compute/platforms/ec2/) wherever
possible — `Section__Docker`, `Section__Sidecar`, `Caller__IP__Detector`,
`Stack__Name__Generator`, `Stack__Naming`, etc.

### 4.3  UI (Dashboard)

Add web components under `ui/card/` and `ui/detail/` using the IFD versioning
convention `v0/v0.1/v0.1.0/sg-compute-{spec_id}-{card,detail}.{js,html,css}`
(see
[`sg_compute_specs/docker/ui/`](../../../sg_compute_specs/docker/ui/)). The
control plane mounts the directory as `StaticFiles` at
`/api/specs/{spec_id}/ui` automatically.

If you ship UI, also bump the package-data globs in
`sg_compute_specs/pyproject.toml` (already covers `*/ui/**/*`).

Component-import rule: detail components use **absolute** paths
(`/ui/shared/...`) so they work both in dev and packaged builds.

### 4.4  README

Drop a `README.md` at `sg_compute_specs/{spec_id}/README.md` and the route
`GET /api/specs/{spec_id}/readme` will serve it as `text/markdown`.

### 4.5  Cross-Spec Node CLI Dispatcher

`sg-compute node create {spec_id}` lives in
[`sg_compute/cli/Cli__Compute__Node.py`](../../../sg_compute/cli/Cli__Compute__Node.py).
This is the **one place** outside the spec package where adding a new spec
requires editing core code today.

Add an entry in the `_DISPATCHERS` dict and a `_create_{spec_id}` helper that
mirrors `_create_docker`. (This is a known wart — it duplicates the per-spec
CLI's `create` verb. A future refactor will collapse it into a generic
delegator that calls `EC2__Platform.create_node`. Until then, mirror the
existing pattern.)

`POST /api/nodes` already dispatches on `spec_id` via
`EC2__Platform.create_node` — no edit required there once your `Service`
implements the `create_node(base_request, …)` method (§3.5).

## 5. Wiring Cheatsheet — What Discovers What

| Discoverer                        | What it looks for                                           | Failure mode                          |
|-----------------------------------|-------------------------------------------------------------|---------------------------------------|
| `Spec__Loader._load_from_package` | `sg_compute_specs/*/manifest.py` exporting `MANIFEST`       | `RuntimeError` on wrong type          |
| `Spec__Loader._load_from_entry_points` | `[project.entry-points."sg_compute.specs"]` rows        | Silent skip if module raises          |
| `Spec__Resolver.validate`         | `extends` graph: every parent registered, no cycles         | `ValueError` at boot                  |
| `Spec__CLI__Loader.load`          | `sg_compute_specs.{spec_id}.cli.Cli__{Pascal}.app`          | Silent skip — sub-app just absent     |
| `Spec__Routes__Loader._find_routes_class` | `sg_compute_specs.{spec_id}.api.routes.Routes__{Pascal}__Stack` | Silent skip — no HTTP surface |
| `Fast_API__Compute._make_service` | The `service` annotation on the routes class — must be a `Type_Safe` subclass | Routes mounted with `service=None`    |
| `Spec__UI__Resolver.ui_path_for_spec` | `sg_compute_specs/{spec_id}/ui/` directory              | No `/ui/...` mount                    |
| `Spec__Readme__Resolver.readme_path_for_spec` | `sg_compute_specs/{spec_id}/README.md` file         | `404` on `GET /api/specs/{id}/readme` |

If your spec is "missing" from any surface, walk this table top-down — one of
these conventions has been violated.

## 6. End-to-End Walkthrough — "Adding `redis`"

A minimum-viable Redis spec, scaffolded from docker:

1. **Create the package skeleton.**
   ```
   sg_compute_specs/redis/
     __init__.py          (empty)
     version              (0.1.0)
     manifest.py
     schemas/  service/  api/routes/  cli/  tests/
   ```

2. **Write `manifest.py`.** spec_id `redis`, capability list including a new
   `Enum__Spec__Capability.KEY_VALUE_STORE` (add it to the enum first), nav
   group `DATA`, `create_endpoint_path='/api/specs/redis/stack'`.

3. **Write the five schemas** under `schemas/`:
   `Schema__Redis__Create__Request`, `…__Response`, `…__Info`, `…__List`,
   `…__Delete__Response`. Reuse `Safe_Str__AWS__Region`,
   `Safe_Str__Stack__Name`, `Safe_Int__Max__Hours`. Add a per-spec
   `Safe_Str__Redis__Stack__Name` if you want a tighter regex than
   `Safe_Str__Stack__Name`.

4. **Build the helpers** under `service/`: `Redis__AWS__Client`,
   `Redis__SG__Helper`, `Redis__Tags__Builder`, `Redis__Stack__Mapper`,
   `Redis__User_Data__Builder` (cloud-init that installs and starts
   `redis-server`), `Redis__Service`. Mirror docker's structure.

5. **Write `Routes__Redis__Stack`** in `api/routes/`. Annotate
   `service: Redis__Service` — that's all that `Fast_API__Compute` needs to
   construct it.

6. **Write `Cli__Redis`** in `cli/` with `list / info / create / delete`
   verbs and a `Renderers.py`. Mirror docker.

7. **Write `tests/test_manifest.py`** (the §3.8 template), plus tests for
   `Redis__Stack__Mapper`, `Redis__Tags__Builder`,
   `Redis__User_Data__Builder` — same pattern as
   [`sg_compute_specs/docker/tests/`](../../../sg_compute_specs/docker/tests/).

8. **Add the entry point** in `sg_compute_specs/pyproject.toml`:
   ```toml
   redis = "sg_compute_specs.redis.manifest"
   ```

9. **(Optional, for `sg-compute node create redis`)** add a dispatcher entry
   in `sg_compute/cli/Cli__Compute__Node.py`. **(Optional, for
   `POST /api/nodes`)** implement `Redis__Service.create_node`.

10. **Update the reality doc** —
    [`team/roles/librarian/reality/sg-compute/index.md`](../../../team/roles/librarian/reality/sg-compute/index.md):
    add a row under "Spec catalogue" and describe the Redis sub-tree under a
    new `### sg_compute_specs/redis/ structure — EXISTS` section. Same commit
    as the code.

11. **Run the tests:**
    ```bash
    pytest sg_compute_specs/redis/tests/ -v
    pytest sg_compute__tests/core/spec/test_Spec__Loader.py -v
    sg-compute spec list                 # redis must appear
    sg-compute spec info redis           # manifest renders
    sg-compute spec redis --help         # sub-app present
    ```

## 9. Quality Bar — Pre-Merge Checklist

Before opening a PR for a new spec:

- [ ] `sg_compute_specs/{spec_id}/manifest.py` exports `MANIFEST` of the right type.
- [ ] `tests/test_manifest.py` covers the §3.8 cases and **passes**.
- [ ] `pytest sg_compute__tests/core/spec/` passes (Spec__Loader picks up the spec).
- [ ] `sg-compute spec list` shows the new entry.
- [ ] `sg-compute spec {spec_id} --help` shows the sub-app.
- [ ] `GET /api/specs` includes the spec; `GET /api/specs/{spec_id}` returns the manifest.
- [ ] One class per file; empty `__init__.py`; 80-char `═══` headers; inline comments only.
- [ ] Zero raw primitives in any schema attribute (no bare `str/int/list/dict`).
- [ ] Type_Safe everywhere — no Pydantic, no Literals.
- [ ] No boto3 imports in spec code (use `osbot_aws`).
- [ ] Security group `GroupName` does not start with `sg-`.
- [ ] AWS `Name` tag is not double-prefixed.
- [ ] PEP 621 entry point added in `sg_compute_specs/pyproject.toml`.
- [ ] Reality doc updated:
      `team/roles/librarian/reality/sg-compute/index.md` — same commit.
- [ ] Branch follows `claude/{description}-{session-id}` and PR opens against `dev`.

## 10. Where to Look When Something Goes Wrong

| Symptom                                            | Most likely cause                                                              |
|----------------------------------------------------|--------------------------------------------------------------------------------|
| `RuntimeError: failed to load manifest for spec '…'` | `MANIFEST` missing or wrong type. Check §3.3.                                |
| `ValueError: cycle detected in spec composition graph` | `extends` graph has a loop. Drop or refactor the parent reference.         |
| Spec appears in `/api/specs` but not in CLI        | `Cli__{Pascal}.app` not exported, or PascalCase wrong (§3.1).                  |
| Spec appears in `/api/specs` but no per-spec routes | `Routes__{Pascal}__Stack` class name wrong, or wrong path (§3.6).             |
| `404` on `/api/specs/{id}/readme`                  | No `README.md` at the spec root, **or** the spec_id isn't registered.          |
| `/api/specs/{id}/ui/...` returns 404               | `ui/` folder absent at boot — only mounted if it exists at startup.            |
| `POST /api/nodes` returns 501 / `NotImplementedError` | `{Pascal}__Service.create_node` not implemented. See §3.5.                  |
| Routes mount but `body` deserialisation fails      | Schema attribute uses a raw primitive — replace with `Safe_*` (CLAUDE.md rule 2). |
| AWS calls work locally, fail in Lambda             | Likely missing IAM on `playwright-ec2` profile. Out of scope for this guide.   |
