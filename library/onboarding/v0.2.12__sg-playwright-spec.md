# sg playwright — end-to-end onboarding brief

> **TL;DR.** `sg playwright create --wait` launches an ephemeral EC2 instance running a 2-container compose stack (`host-plane` + `sg-playwright`) — or 3-container with `--with-mitmproxy` adding `agent-mitmproxy`. The Playwright FastAPI lands on `http://<public-ip>:8000`, gated by an `X-API-Key` header (or cookie). Verified end-to-end in ~2m 6s from `create` to the first PNG screenshot returning HTTP 200.

The spec sits under `sg_compute_specs/playwright/` and mirrors the `sg_compute_specs/vault_app/` and `sg_compute_specs/local_claude/` conventions: `Spec__Service__Base` for create/list/info/wait/health/connect/exec/delete, `Spec__CLI__Builder` for the typer wiring, shared `sg_compute` EC2 helpers (no per-spec clones). The legacy `sgraph_ai_service_playwright__cli/playwright/` cli-plugin and the inline `pw_app` typer in `scripts/provision_ec2.py` are both retired — `sg playwright` now routes through `sg_compute/cli/Cli__SG.py` to the new spec.

## 1. What you get

| Command                                | Notes                                                                                          |
|----------------------------------------|------------------------------------------------------------------------------------------------|
| `sg playwright create --wait`          | Launch + block until `/health/status` returns. Surfaces the api_key once and the full URL set. |
| `sg playwright list`                   | All live `playwright`-tagged EC2 instances in the region.                                       |
| `sg playwright info <name>`            | Rich panel — URLs, mode, api_key, set-cookie-form link, JS bookmarklet, terminate-at, hints.   |
| `sg playwright wait <name>`            | Staged-progress probe — `[t+Ns]` lines, dedupe, stage-timings table, boot-log tail on timeout. |
| `sg playwright health <name>`          | One-shot probe (no progress UI).                                                                |
| `sg playwright connect <name>`         | Interactive SSM shell on the EC2 host.                                                          |
| `sg playwright exec <name> -- <cmd>`   | One-shot SSM command.                                                                           |
| `sg playwright delete <name>`          | Terminate + delete the security group.                                                          |
| `sg playwright logs <name> -s boot -f` | Stream `/var/log/ephemeral-ec2-boot.log` (or `cloud-init` / `journal`); `-f` polls every 4s.    |
| `sg playwright extend <name> --ah 1`   | Push the auto-terminate timer out by N hours.                                                   |
| `sg playwright ami …`                  | Bake / list / delete an AMI from a warm stack (Spec__CLI__Builder sub-typer).                   |
| `sg playwright cert …`                 | Cert sub-typer (Spec__CLI__Builder).                                                            |

**Create-time flags worth knowing**:
- `--with-mitmproxy` — adds the `agent-mitmproxy` container (3-container stack). sg-playwright is wired to route browser traffic through it.
- `--intercept-script <path-or-source>` — only meaningful with `--with-mitmproxy`. Reads a local file if the string is a path, otherwise treats it as inline source. Baked into `/opt/sg-playwright/interceptors/active.py` and mounted into agent-mitmproxy at `/app/interceptors/active.py:ro`.
- `--ami <id>` — boot from a pre-baked AMI (skips Docker install + image pulls).
- `--image-tag <tag>` — `diniscruz/sg-playwright:<tag>` instead of `:latest`.
- `--no-use-spot` — on-demand instead of spot (default is spot, ~70% cheaper).
- `--api-key <value>` — pre-supply the FAST_API__AUTH__API_KEY__VALUE; auto-generated and surfaced once on create if blank.

Then the info panel renders with the auth-helper block — the api_key reused from the EC2 tag, the `/auth/set-cookie-form` link, and a one-click JavaScript bookmarklet pre-filled with the actual key. Pasting the bookmarklet on the playwright tab sets the `X-API-Key` cookie + reloads, and `/screenshot/simple` returns a valid PNG in ~1.7 s.

The HTTP 401 on the wait probe is **expected** — `/health/status` is API-key-gated, so any 4xx response (specifically 401/403) is treated as "service up". A 5xx or connection refusal counts as not-yet-ready and keeps polling.

## 3. How it works

### 3.1 Per-EC2 container layout

| Container          | Image (source)                                                | Published port | Why it's there                                                |
|--------------------|---------------------------------------------------------------|----------------|---------------------------------------------------------------|
| `host-plane`       | `<ecr>/sgraph_ai_service_playwright_host:<tag>` (**ECR**)     | none           | Internal docker-control sidecar; reachable via SSM port-forward only. |
| `sg-playwright`    | `diniscruz/sg-playwright:<tag>` (**Docker Hub**)              | `8000:8000`    | The Playwright FastAPI — the externally-visible surface.      |
| `agent-mitmproxy`  | `<ecr>/agent_mitmproxy:<tag>` (**ECR**, only `--with-mitmproxy`) | `8001:8000` admin | Intercepting proxy on `:8080` (sg-net only) + admin API on `:8001`. |

The mixed-registry pull is deliberate. `diniscruz/sg-playwright` is a public Docker Hub image (built + pushed by `.github/workflows/ci-pipeline.yml`'s `build-and-push-playwright-image` job, multi-arch). The other two are private ECR images. The user-data still does `aws ecr get-login-password | docker login <ecr>` for the ECR pulls; Docker Hub pulls are anonymous. `docker compose up` handles the mix natively.

> **Earlier bug shipped in 46da36d**: the compose template was prefixing every image with the ECR registry, including `diniscruz/sg-playwright`. The generated compose pointed at a non-existent ECR path; `docker compose up` failed the pull and the boot died at line 137 (exit=15) with no useful symptom. **Fixed in 6e46361d** by dropping the ECR prefix on the sg-playwright line specifically. Tests assert the regression form (`<ecr>/diniscruz/sg-playwright`) is never re-introduced.

### 3.2 The user-data lifecycle

`Playwright__User_Data__Builder.render()` (`sg_compute_specs/playwright/service/Playwright__User_Data__Builder.py`) emits a bash script that runs on first boot:

1. **Section__Base preamble** (`sg_compute/platforms/ec2/user_data/Section__Base.py`)
   - Strict mode + log redirection to `/var/log/ephemeral-ec2-boot.log`
   - **Auto-terminate timer (set BEFORE any failable work)** — `systemd-run --on-active=<seconds>s /sbin/shutdown -h now`. Even if `dnf install` crashes, the instance still self-terminates. The dev observed this line as the first stage marker (`t+21s`).
   - SSM agent + `ssm-user` provisioning so `sg playwright connect` works without an interactive SSM session ever having opened.

2. **`[playwright] installing Docker CE`** — `dnf install -y docker` + curl Docker's compose plugin from GitHub releases + `systemctl enable --now docker`. The dev observed this stage at `t+44s` (`+23s` after the timer).

3. **`[playwright] Docker ready`** — `t+79s` (`+35s` after Docker install starts).

4. **`[playwright] interceptor script written`** — only when `--intercept-script` was provided. Writes the contents to `/opt/sg-playwright/interceptors/active.py`.

5. **Stack files** — writes `/opt/sg-playwright/.env` (chmod 600; `ECR_REGISTRY`, `IMAGE_TAG`, `FAST_API__AUTH__API_KEY__NAME=X-API-Key`, `FAST_API__AUTH__API_KEY__VALUE=<api_key>`) and `/opt/sg-playwright/docker-compose.yml` (the rendered compose template).

6. **ECR login + compose up** — `aws ecr get-login-password | docker login <ecr>` then `docker compose --env-file /opt/sg-playwright/.env up -d` then `docker logout <ecr>`.

7. **`[playwright] stack started ({mode})`** — marker for "compose up returned".

8. **`[playwright] boot complete`** — footer touches `/var/lib/sg-compute-boot-ok`.

The boot-log markers `[playwright] *` and `[ephemeral-ec2] *` are what the staged-wait probe parses (see §3.4).

### 3.3 The EC2 tag set

`Playwright__Service.create_stack()` writes the following tags on the instance:

| Tag                 | Source                            | Used by                                          |
|---------------------|-----------------------------------|--------------------------------------------------|
| `Name`              | stack_name                        | Console UX                                       |
| `Purpose`           | `'ephemeral-ec2'`                 | Generic filter                                   |
| `StackName`         | stack_name                        | `sg playwright info <name>` lookup               |
| `StackType`         | `'playwright'`                    | `sg playwright list` filter (`list_by_stack_type`) |
| `CallerIP`          | auto-detected or `--caller-ip`    | SG ingress audit                                  |
| `CreatedBy`         | git email / `$USER`               | Audit                                            |
| `StackWithMitmproxy`| `'true'` / `'false'`              | `Schema__Playwright__Info.with_mitmproxy`        |
| `StackApiKey`       | the api_key                       | `Schema__Playwright__Info.api_key` → info panel  |
| `TerminateAt`       | `now + max_hours` (ISO-8601 Z)    | `Schema__Playwright__Info.terminate_at` + `time_remaining_sec` |

`StackApiKey` is the vault-app pattern (`AccessToken` there). Trade-off: anyone with `ec2:DescribeInstances` can read the api_key. Acceptable for ephemeral caller-IP-restricted stacks; rotate by re-creating the stack. The api_key is **only** written at create time — pre-existing stacks won't have the tag and `info` falls back to a `YOUR_API_KEY` placeholder in the bookmarklet.

### 3.4 The staged-wait probe

`Playwright__Service.health()` overrides the inherited `Spec__Service__Base.health()` with a probe that mirrors `Vault_App__Service.health()` exactly (same helper names: `_http_status`, `_probe_endpoint`, `_probe_progress`, `_print_stage_timings`, `_print_boot_log_tail`). Every `poll_sec` (default 10s) during a `wait`:

1. `get_stack_info(region, name)` — re-fetch EC2 state from `DescribeInstances`.
2. If `public_ip` is known, `urllib` HTTP probe of `http://<ip>:8000/health/status`. Any 2xx/3xx/4xx response = "up" (4xx because auth is gated). Only connection-refused / 5xx counts as "still booting".
3. If still booting, one SSM round-trip combining two greps with a `|||CONTAINERS|||` separator:
   - `tail -n 1` of the latest `^\[(playwright|ephemeral-ec2)\]` line from the boot log → current stage
   - `docker ps --filter label=com.docker.compose.project=sg-playwright` → `N/M Up` summary
4. Build a `(state, ip, stage, containers)` signature. Print a `[t+Ns]` line **only when the signature changes** — drops the 5-identical-lines-per-stage noise the inherited probe would produce.
5. Record `(elapsed_s, stage_label)` for the final stage-timings table.
6. On timeout, print the last 20 lines of `/var/log/ephemeral-ec2-boot.log` via one more SSM call.

The dev's stage-timings table from the verified run captured the full boot rhythm:

```
t+ 21s   auto-terminate timer set (Section__Base, before any failable work)
t+ 44s   installing Docker CE      (+23s — dnf install + repo work)
t+ 79s   Docker ready              (+35s — dnf finish + compose-plugin curl)
t+126s   HTTP up — HTTP 401        (+47s — image pull + container start + first OK probe)
```

`HTTP 401` is the success signal — see §2.

### 3.5 The info auth-helper block

`render_playwright_info` (`sg_compute_specs/playwright/cli/Renderers.py`) emits a Rich panel + table. The auth-helper rows are the operator's one-stop shop for browser access:

| Row              | Renders when…                            | Value                                                                                  |
|------------------|------------------------------------------|----------------------------------------------------------------------------------------|
| `api-key`        | `info.api_key` is set (the `StackApiKey` tag was found) | `<value>  (X-API-Key header or cookie)` — both work because the same env-var name configures both. |
| `set-cookie-form`| `info.playwright_url` is set             | `<url>/auth/set-cookie-form` — the browser-friendly UI from `osbot_fast_api.api.routes.Routes__Set_Cookie`. |
| `browser-auth`   | `info.playwright_url` is set             | One-click JS bookmarklet, pre-filled with the actual key (or `YOUR_API_KEY` placeholder if the tag wasn't set). Drag to bookmark bar; click on the playwright tab; cookie set + page reloaded. |

The bookmarklet is the fastest browser-side path:

```javascript
javascript: document.cookie = "X-API-Key=<value>; path=/"; location.reload();
```

After that, the `/screenshot/simple` UI works without pasting the key into the API-key textbox.

## 6. What's still pending

- **Stage 2 of the `provision_ec2.py` retirement.** The file is dead code as the `sg` entry point but `sgraph_ai_service_playwright__cli/ec2/service/Ec2__Service.py` still lazy-imports `provision()` + 5 tag constants for the legacy `/api/ec2/playwright/*` FastAPI routes. Full deletion needs to retire that whole coupling cluster (see `library/reference/v0.2.11__provision_ec2-legacy-snapshot.md`).
- **`--with-aws-dns` on playwright.** The vault-app slice added parallel Route 53 work + auto-derived FQDN + LE HTTPS during create (`sp vault-app create --with-aws-dns --wait` → `https://<stack>.sg-compute.sgraph.ai`). The playwright spec doesn't have this yet — pattern is `Spec__CLI__Builder`'s `post_launch_fn` hook + a `Playwright__Auto_DNS` helper mirroring `Vault_App__Auto_DNS`.
- **`diag` command.** vault-app + local-claude both implement `Spec__Service__Base.diagnose()` returning a generator of `(check_name, status, detail)` tuples — the foundation for a structured boot checklist. Playwright doesn't have one yet.
- **The `degraded` health badge** the web UI shows is probably `Capability__Detector` reporting a non-fatal capability as unavailable (mitmproxy / artefact sink / vault). Worth tracing `Capability__Detector` → `Routes__Health` to silence it for the default 2-container shape.
- **`Cli__SG__Repl` per-section dispatch.** The REPL currently has a `SECTIONS` registry with only a `nodes` entry. The `sg/playwright>` prompt the dev's session shows comes from a newer commit (the section name is whatever the user types — see `Cli__SG__Repl.py:138`). A `playwright` entry in `SECTIONS` mapping verbs to the typer commands would close the loop cleanly; today, anything not in `SECTIONS[section]` falls through to "Unknown command".

## 7. Quick-start cheatsheet

```bash
# Launch + wait
sg playwright create --wait

# Get the full panel (URLs, api-key, bookmarklet, hints)
sg playwright info <name>

# Stream the boot log live
sg playwright logs <name> --source boot -f

# Open an interactive SSM shell on the EC2 host
sg playwright connect <name>

# Push the auto-terminate timer out by 1h
sg playwright extend <name> --add-hours 1

# Tear down
sg playwright delete <name>
```

To run the image locally (no AWS):

```bash
docker run --rm -p 8000:8000 \
  -e FAST_API__AUTH__API_KEY__VALUE=local-dev-key \
  diniscruz/sg-playwright:latest
# UI:           http://localhost:8000
# OpenAPI:      http://localhost:8000/docs
# Cookie form:  http://localhost:8000/auth/set-cookie-form
```

`docker compose up`-style mixing of ECR + Docker Hub images works on the EC2 host because the user-data does ECR login before compose-up and the Docker Hub image is public. The build-time guard in the Dockerfile guarantees `playwright==1.58.0` is installed for the same `python3` interpreter the CMD invokes — the original `ModuleNotFoundError` can't ship silently again.
