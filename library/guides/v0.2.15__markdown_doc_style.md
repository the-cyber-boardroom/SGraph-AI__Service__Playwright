# Markdown doc style — briefs, plans, debriefs, reviews

Applies to every Markdown deliverable under `team/` and `library/` — Architect briefs, plans, debriefs, role reviews, comms changelogs, anything that renders on GitHub.

---

## Frontmatter, not heading-block metadata

**Do** open the file with a YAML frontmatter block, then a single H1 title:

```markdown
---
title: "Architect Briefing — sg aws dns: Route 53 DNS Management Center"
file: architect__sg-aws-dns__plan.md
author: Architect (Claude)
date: 2026-05-15 (UTC hour 08)
repo: SGraph-AI__Service__Playwright @ dev (v0.2.15 line)
status: PLAN — no code, no commits. For human ratification before Dev picks up.
parent: team/humans/dinis_cruz/claude-code-web/05/15/03/architect__vault-app__cf-route53__plan.md
---

# Architect Briefing — sg aws dns: Route 53 DNS Management Center

> **PROPOSED — does not exist yet.** ...
```

**Don't** open with the `═══` header block used in Python files:

```markdown
# ═══════════════════════════════════════════════════════════════════════════════
# Architect Briefing — sg aws dns: Route 53 DNS Management Center
# File   : architect__sg-aws-dns__plan.md
# Author : Architect (Claude)
# ═══════════════════════════════════════════════════════════════════════════════
```

GitHub renders every `# Key : Value` line as its own H1 with a divider rule, turning the first screen of the doc into chrome. Frontmatter renders as a tidy two-column table at the top instead.

Multi-value fields use YAML list syntax:

```yaml
parent:
  - team/roles/architect/reviews/05/14/v0.2.6__vault-app-tls-options.md
  - team/roles/architect/reviews/05/14/v0.2.6__vault-app-tls-poc-fastapi-sidecar.md
```

---

## Why `═══` is for code, not Markdown

CLAUDE.md rule #7 — **`═══` 80-char headers — every file, every section** — applies to Python source files. The reason it works there is that `#` is a comment in Python, so `# ═══...` is invisible to the interpreter and structural to the reader.

In Markdown, `#` is heading syntax. The same line that's a comment in Python becomes an H1 on GitHub. The two conventions look identical in a raw editor and behave completely differently when rendered. Always use frontmatter for Markdown.

---

## Section dividers inside the body

Inside the body, separate top-level sections with `---` (an HR rule) — not `═══`. Numbered section headers (`## 1. Context`, `## 2. Goals`) are fine.
