# Playwright API — Guide for Agents

**Version:** v0.2.6
**Date:** 2026-05-15
**Audience:** Claude (or any agent) driving the SG/Playwright browser-automation
REST API. **This document is intentionally self-contained.** Paste it into the
agent's prompt together with the stack's URL + access token; nothing else is needed.

The user supplies two values per session:

```
BASE_URL     = http://<public-ip>:11024     # plain HTTP — see "TLS" below
ACCESS_TOKEN = <the access-token from `sp vault-app info`>
```

Every request below assumes those two values are substituted in.

---

## 1. Authentication

Every request **must** carry the access token as the `X-API-Key` header:

```
X-API-Key: <ACCESS_TOKEN>
```

Missing or wrong key → `401 {"detail": "..."}`. The token is opaque; treat it
as a single secret. It also doubles as the vault's `x-sgraph-access-token` (the
same value gates `https://<public-ip>/...` for vault calls), but for *this*
API only `X-API-Key` matters.

---

## 2. Quick start — three commands that cover 80% of use cases

### 2.1 Is it up?

```bash
curl -s "$BASE_URL/health/status" -H "X-API-Key: $ACCESS_TOKEN" | jq
```

Returns `{ "status": "ok", "checks": [...], ... }`. Use this as a readiness gate
before issuing browser-driving requests. Browser launches cold-start in ~2–4s
on the first call, then warm.

### 2.2 Screenshot any URL (PNG → raw bytes)

```bash
curl -s "$BASE_URL/browser/screenshot" \
  -H "X-API-Key: $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"url": "https://example.com", "full_page": true}' \
  > screenshot.png
```

`Content-Type: image/png` — pipe it straight to a file or stream it.

### 2.3 Run a multi-step sequence (the orchestration endpoint)

```bash
curl -s "$BASE_URL/sequence/execute" \
  -H "X-API-Key: $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "steps": [
      {"action": "navigate",   "url": "https://example.com"},
      {"action": "screenshot", "full_page": true},
      {"action": "get_url"}
    ]
  }' | jq
```

`/sequence/execute` is the **right endpoint for anything beyond a single action** —
it runs the whole step list in one Chromium process, returns timing + per-step
results, and tears down. See §5 for the step vocabulary.

## 3. Endpoint reference

All paths are relative to `BASE_URL`. All POSTs accept JSON; the body schema
is enumerated below.

| Method | Path                        | Body shape (key fields)                                | Returns                  |
|--------|-----------------------------|--------------------------------------------------------|--------------------------|
| GET    | `/health/info`              | —                                                      | service name/version JSON|
| GET    | `/health/status`            | —                                                      | health JSON              |
| GET    | `/health/capabilities`      | —                                                      | declared capabilities    |
| GET    | `/metrics`                  | —                                                      | Prometheus text          |
| POST   | `/browser/navigate`         | `{url, wait_until?, timeout_ms?, viewport?, browser_config?}`            | navigation result JSON   |
| POST   | `/browser/click`            | `{url?, selector, timeout_ms?, ...}`                                     | click result JSON        |
| POST   | `/browser/fill`             | `{url?, selector, value, ...}`                                           | fill result JSON         |
| POST   | `/browser/get-content`      | `{url?, selector?, ...}`                                                 | `{content: "<html>..."}` |
| POST   | `/browser/get-url`          | `{url?, ...}`                                                            | `{url: "https://..."}`   |
| POST   | `/browser/screenshot`       | `{url, full_page?, click?, javascript?, format?}`                        | **raw PNG bytes**        |
| POST   | `/screenshot/screenshot`    | same as above                                                            | `{image_b64, ...}` (JSON wrapping) |
| POST   | `/screenshot/batch`         | `{requests: [Schema__Screenshot__Request, ...]}`                          | batched JSON             |
| POST   | `/sequence/execute`         | `{steps: [...], browser_config?, capture?, sequence_id?, credentials?}`  | per-step results + timings|

**Single-shot vs sequence:** `/browser/*` are stateless one-shot helpers that
each spin up a fresh Chromium for the one action. `/sequence/execute` is the
right call when you need ≥2 actions on the same page — same browser, shared
DOM state, lower overhead.

**`/browser/screenshot` vs `/screenshot/screenshot`:** identical inputs, two
output shapes. The former returns raw `image/png` bytes (faster, file-friendly);
the latter wraps it in a JSON envelope with metadata + base64 (easier to embed
in a JSON response chain). Pick by what you do with the bytes.

### 3.1 Common body fields

| Field             | Type                  | Default                | Notes |
|-------------------|-----------------------|------------------------|-------|
| `url`             | string (URL)          | required for one-shots | full URL incl. scheme; on `/sequence/execute` use a `navigate` step instead |
| `wait_until`      | enum                  | `"load"`               | `"load"` / `"domcontentloaded"` / `"networkidle"` / `"commit"` |
| `timeout_ms`      | int                   | service default        | per-action timeout |
| `selector`        | string (CSS / text)   | required for click/fill | standard Playwright selector syntax |
| `viewport`        | `{width, height}`     | `{1280, 720}`          | shorthand; full control via `browser_config.viewport` |
| `browser_config`  | object                | service defaults       | see §4 |
| `full_page`       | bool                  | `false`                | screenshot only |
| `format`          | enum                  | `"png"`                | `"png"` / `"jpeg"` |
| `click`           | string (selector)     | none                   | screenshot one-shot helper — click before snapping (e.g. dismiss a banner) |
| `javascript`      | string                | none                   | screenshot one-shot helper — run JS after load, before snapping. **Allowlist-gated** — see §6 |

## 4. `browser_config` — when defaults aren't enough

`browser_config` is optional on every request. Common overrides:

```json
{
  "browser_config": {
    "engine"     : "chromium",
    "headless"   : true,
    "viewport"   : {"width": 1920, "height": 1080},
    "user_agent" : "Mozilla/5.0 (custom)",
    "locale"     : "en-GB",
    "timezone"   : "Europe/London",
    "java_script_enabled": true
  }
}
```

Defaults: `chromium`, `headless`, `1280x720`, system locale, JS on. Omit the
whole object to use them.

**Proxy routing is fixed at boot, not per-request.** Every browser launch in
this stack already routes through `agent-mitmproxy:8080` so traffic is
inspectable via mitmweb. Don't try to set a proxy in `browser_config` — it
won't apply, and the existing wiring works.

## 5. The step language — `/sequence/execute`

Each entry in `steps` is `{"action": "<verb>", ...action-specific fields...}`.
The dispatcher routes on `action`. 16 verbs are available:

| Action            | Required fields                | Optional                              | What it does |
|-------------------|--------------------------------|----------------------------------------|--------------|
| `navigate`        | `url`                          | `wait_until`, `timeout_ms`             | Go to URL |
| `click`           | `selector`                     | `timeout_ms`, `button`                 | Click element |
| `fill`            | `selector`, `value`            | `timeout_ms`                           | Fill form field |
| `press`           | `key`                          | `selector`                             | Keyboard press |
| `select`          | `selector`, `value`            | —                                      | Select dropdown option |
| `hover`           | `selector`                     | `timeout_ms`                           | Mouse hover |
| `scroll`          | `direction` or `selector`      | `pixels`                               | Scroll viewport or element |
| `wait_for`        | one of `selector` / `url` / `state` | `timeout_ms`                      | Wait for condition |
| `screenshot`      | —                              | `full_page`, `selector`, `format`      | Capture screenshot (added to sequence artefacts) |
| `video_start`     | —                              | `dir`                                  | Begin recording |
| `video_stop`      | —                              | —                                      | End recording |
| `evaluate`        | `expression`                   | —                                      | Run JS — **deny-by-default allowlist**, see §6 |
| `dispatch_event`  | `selector`, `event_type`       | `event_init`                           | Synthetic DOM event |
| `set_viewport`    | `width`, `height`              | —                                      | Change viewport mid-sequence |
| `get_content`     | —                              | `selector`                             | Capture HTML/text into the per-step result |
| `get_url`         | —                              | —                                      | Capture current URL into the per-step result |

### 5.1 Worked sequence — visit, accept cookies, screenshot, extract title

```json
{
  "steps": [
    {"action": "navigate",  "url": "https://example.com"},
    {"action": "click",     "selector": "button:has-text('Accept')"},
    {"action": "wait_for",  "state": "networkidle"},
    {"action": "screenshot","full_page": true},
    {"action": "evaluate",  "expression": "document.title"},
    {"action": "get_url"}
  ]
}
```

Response (abridged):
```json
{
  "sequence_id": "seq-...",
  "status": "ok",
  "steps": [
    {"action": "navigate",  "status": "ok", "elapsed_ms": 1834, "result": {...}},
    {"action": "click",     "status": "ok", "elapsed_ms":  127, "result": {...}},
    {"action": "wait_for",  "status": "ok", "elapsed_ms":  642, "result": {...}},
    {"action": "screenshot","status": "ok", "elapsed_ms":  311, "artefact": {"path": "...", "size_bytes": 84210}},
    {"action": "evaluate",  "status": "ok", "elapsed_ms":   18, "result": "Example Domain"},
    {"action": "get_url",   "status": "ok", "elapsed_ms":    5, "result": "https://example.com/"}
  ],
  "timings": {"total_ms": 2937, ...}
}
```

Each step has a `status` of `ok` / `failed` / `skipped`. On the first failure
the sequence stops by default; remaining steps surface as `skipped`. The
response is always a valid JSON object — never an HTTP error, even on
per-step failures.

## 6. The JS expression allowlist — `evaluate` is deny-by-default

`evaluate` steps and the screenshot `javascript` shortcut go through a
deny-by-default allowlist (`JS__Expression__Allowlist`). Constant, side-effect-
free expressions are accepted (e.g. `document.title`, `window.isSecureContext`,
`document.querySelectorAll('a').length`); arbitrary scripts are rejected at
schema-validation time with a clear error.

If you need an expression that isn't on the allowlist:
- **Check first** by submitting it — the error message tells you whether the
  block is at the allowlist or at runtime.
- Common reads that are typically allowed: `document.title`, `document.URL`,
  `window.location.href`, `window.innerWidth`, `document.readyState`.
- **Don't try to bypass** with `eval` / `Function` constructors / template
  strings — those patterns trigger an explicit rejection.

For data extraction at scale, prefer `get_content` (HTML) over `evaluate` (JS
read) — `get_content` has no allowlist and is the right tool when you want a
chunk of DOM, not a derived value.

## 7. Errors & status codes

| Status | Meaning                                              | Common cause |
|--------|------------------------------------------------------|--------------|
| 200    | success                                              | — |
| 400    | request schema validation failed                     | typo in field; wrong type; missing required field |
| 401    | missing / wrong `X-API-Key`                          | — |
| 403    | `evaluate` expression not in the allowlist           | use a read-only expression, or `get_content` |
| 408    | timeout (selector not found, page didn't load, etc.) | check selector / `timeout_ms` / network |
| 500    | service-internal (rare)                              | report with response body — usually a Playwright crash |

Per-step failures inside `/sequence/execute` return **200** with `status:"failed"`
on the offending step — that's by design, the sequence reports its own outcome
in the body.

## 8. TLS / certificates / browser warnings

Playwright API itself is served over **plain HTTP** on `:11024` by default — the
token is the gate, the link to the vault is the X-API-Key header. Don't expect
HTTPS at this layer unless your operator explicitly enabled it.

When the *browser* inside Playwright fetches HTTPS pages, the agent-mitmproxy
sidecar intercepts and presents forged certs. The service is already configured
with `SG_PLAYWRIGHT__IGNORE_HTTPS_ERRORS=true` so target HTTPS endpoints work
transparently — you don't need to do anything special.

## 9. Performance & limits

- **Cold start:** first request ~2–4s while Chromium spawns. Subsequent calls
  are sub-second per action.
- **One process per request** for `/browser/*` one-shots — costly if chained.
  Use `/sequence/execute` for any multi-step flow.
- **Default timeout** per action is ~30s. Long-running pages: override
  `timeout_ms` per step.
- **No session handles** on the wire — every call is stateless from the
  client's POV. Continuity within a sequence is internal to that one
  `/sequence/execute` call.
- **Artefacts** (screenshots, videos) are written by the server-side
  `Artefact__Writer` and surfaced as `result.artefact.path` (for vault-stored
  artefacts) or returned inline (raw screenshots). The path is server-internal;
  the response has the bytes you need.

## 10. Discovery — when this doc doesn't cover what you need

The live service auto-serves its OpenAPI spec at:

```
GET $BASE_URL/openapi.json     # machine-readable
GET $BASE_URL/docs             # Swagger UI (for humans)
```

If a field, value, or error reply doesn't match this guide, `/openapi.json` is
the ground truth — this doc may be one version behind the running service.
Fetch it, find the relevant `paths`/`components` entry, and try the request.

## 11. End-to-end example — Python

```python
import os
import requests

BASE = os.environ['BASE_URL']        # e.g. http://13.40.193.154:11024
TOKEN = os.environ['ACCESS_TOKEN']   # from `sp vault-app info`
HDRS  = {'X-API-Key': TOKEN}

# 1. wait for service to be up
assert requests.get(f'{BASE}/health/status', headers=HDRS, timeout=10).status_code == 200

# 2. one-shot screenshot
r = requests.post(f'{BASE}/browser/screenshot',
                  json={'url': 'https://example.com', 'full_page': True},
                  headers=HDRS, timeout=60)
r.raise_for_status()
open('out.png', 'wb').write(r.content)

# 3. multi-step sequence — visit, accept cookies, snap, extract title
seq = requests.post(f'{BASE}/sequence/execute', json={'steps': [
    {'action': 'navigate',   'url': 'https://example.com'},
    {'action': 'click',      'selector': "button:has-text('Accept')"},
    {'action': 'wait_for',   'state': 'networkidle'},
    {'action': 'screenshot', 'full_page': True},
    {'action': 'evaluate',   'expression': 'document.title'},
], 'browser_config': {'viewport': {'width': 1920, 'height': 1080}}},
    headers=HDRS, timeout=120).json()

for step in seq['steps']:
    print(f"{step['action']:<12} {step['status']:<8} {step.get('elapsed_ms', 0)}ms")
print('Title:', next(s['result'] for s in seq['steps'] if s['action'] == 'evaluate'))
```
