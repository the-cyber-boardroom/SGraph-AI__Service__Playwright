## Why this document exists

`provision_ec2.py` accumulated three responsibilities that didn't belong together:

1. **The `sg` typer entry point** ‚Äî `app = typer.Typer(name='sg', ...)` and the
   ~15 `app.add_typer(...)` registrations for every spec's sub-CLI.
2. **An inline `pw_app` with 19 playwright commands** (`create`/`list`/`info`/
   `delete`/`connect`/`shell`/`env`/`run`/`exec`/`exec-c`/`logs`/`diagnose`/
   `forward`/`wait`/`clean`/`open`/`screenshot`/`smoke`/`health` +
   `create-from-ami`) plus an inline `ami_app` with 4 verbs.
3. **A `provision()` function + 5 tag constants** still imported by the legacy
   FastAPI ec2 routes (`sgraph_ai_service_playwright__cli/ec2/service/Ec2__Service.py`).

Responsibilities (1) and (2) are now in `sg_compute/cli/Cli__SG.py` and the
playwright spec. Responsibility (3) is the reason the file is still on disk ‚Äî
see "Stage 2" below.

### "‚ùå Not replaced" ‚Äî recovery plan if a user actually misses one

These are the candidates to fold into the spec as extras (alongside `logs` and
`extend`) when someone needs them:

- `diagnose` ‚Äî biggest gap. Implement `Playwright__Service.diagnose()` returning
  a generator of `(check_name, status, detail)` tuples; mirror
  `Vault_App__Service.diagnose` and `Local_Claude__Service.diagnose`.
- `forward` ‚Äî useful enough that it should probably go into
  `Spec__CLI__Builder` for every spec, not just playwright. (`sg <spec>
  forward <port>:<port>`.)
- `clean` ‚Äî should be a sub-step of `ami bake`, not a user-facing verb.
- `env` ‚Äî one-line `os.environ`-emitting helper; trivial to add as a spec extra.
- `screenshot` / `smoke` ‚Äî these are **runtime API calls** against the deployed
  Playwright stack, not infra ops. They belong in a client library that talks
  HTTP to `playwright_url`, not in `sg playwright`. (Today they essentially
  curl the stack's own API.)
- `shell` / `run` / `exec-c` ‚Äî overlap with builder `exec`. Probably fine to
  drop entirely; if someone needs container-targeted exec, add `--container
  <name>` to the builder's `exec` flag.

go away. They are an alternate path to the same provisioning that
`sg playwright create` does ‚Äî there is no functionality lost that isn't
already covered by the CLI. If something *outside this repo* calls those HTTP
routes (a notebook, a script, a CI job), it must move to either the CLI or
the new spec's programmatic surface (`Playwright__Service().setup().create_stack(...)`).
