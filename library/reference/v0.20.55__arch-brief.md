## Context

This is a follow-up to the 06 Apr serverless Playwright API brief (`v0.20.37__dev-brief__serverless-playwright-api.md`). We now have the pieces needed to implement it properly: the vault can be used as a communication medium (confirmed via 04/11 Phase 2 + Phase 3), the tree-only loading pattern means vault access is cheap, and the JS API primitive pattern is established.

Time to build it.

## The Goal

A Docker container running Playwright with Chrome, deployable as:

- Local container (developer laptop)
- Docker Hub image
- ECR image
- AWS Lambda function (via container deployment, since Lambda does not natively support Playwright/Chrome)

The container exposes a FastAPI service that maps Playwright actions to HTTP endpoints, with maximum granularity. Callers can execute individual commands (navigate, click, screenshot, capture video) or sequences of commands described in an agentic meta-language.

## The Core Architectural Decision: Decouple the Image from the Code

The single most important decision in this brief: the Playwright Docker image must be stable. The code that actually gets executed (the FastAPI service, the route handlers, the action sequencers) must be loaded separately at runtime.

### Why

Playwright + Chrome images are heavy. Rebuilding and redeploying the entire image every time we update a route handler or add a new endpoint is wasteful and slow. The Playwright version, the Chrome version, and the base dependencies change rarely. Our service code changes often.

### How

The Docker image contains:

- Base OS (Ubuntu or similar)
- Python runtime
- Playwright library + browser binaries (Chrome, maybe Firefox)
- A thin bootstrap script that loads the service code at startup

The service code (FastAPI app, route handlers, action sequencers) is:

- Pulled from S3 or a vault at container startup
- Or mounted as a volume in local development
- Or included via Lambda layer for Lambda deployment

When we deploy a new version of the service, we update the code source (S3, vault, or layer) but not the base image. The next container start picks up the new code. The image stays stable for weeks or months between Playwright/Chrome upgrades.

### Research Starting Point

Before building from scratch, check if a suitable base image already exists:

- Official Playwright Docker images (e.g. `mcr.microsoft.com/playwright`)
- AWS Lambda Playwright layers (community or official)
- Existing community images for serverless Playwright

If something close exists, start there and add the thin bootstrap layer on top.

## The API Design: Two Modes

### Mode 1: Direct Action Endpoints

Each Playwright action is an HTTP endpoint. Maximum granularity. Callers control the session explicitly.

```
POST /session/start          -> returns session_id
POST /session/{id}/navigate  -> {url}
POST /session/{id}/click     -> {selector}
POST /session/{id}/fill      -> {selector, value}
POST /session/{id}/screenshot -> returns image (or vault path)
POST /session/{id}/video/start
POST /session/{id}/video/stop -> returns video (or vault path)
POST /session/{id}/evaluate  -> {script} (execute JS in page)
POST /session/{id}/close
```

This is the building-block mode. Agents, QA scripts, and tools compose these primitives into workflows.

### Mode 2: Sequence Execution (Meta-Language)

A single endpoint accepts a sequence of commands in a JSON structure and executes them in order. This is the agentic mode.

```json
POST /sequence/execute
{
  "config": {
    "proxy": { "source": "vault://<vault-id>/proxy-config.json" },
    "output_vault": "vault://<vault-id>/outputs/"
  },
  "steps": [
    { "action": "navigate", "url": "https://example.com" },
    { "action": "wait_for", "selector": ".content-loaded" },
    { "action": "screenshot", "save_as": "step-1.png" },
    { "action": "click", "selector": "#menu" },
    { "action": "screenshot", "save_as": "step-2.png" },
    { "action": "evaluate", "script": "return document.title" }
  ]
}
```

The response contains the result of each step and pointers to any artefacts written to the output vault.

This is how agents and LLMs describe browser workflows: a declarative sequence, not imperative code.

## Vault as Communication Medium

This is the architectural unlock that was not available when the 06 Apr brief was written.

### Configuration Input

Secrets, proxy settings, credentials, session state, and other configuration are stored in a vault. The Playwright container receives a vault reference (vault ID + key) and fetches only the files it needs using the tree-only loading pattern.

```
Configuration flow:
  Caller -> vault://<id>/config/proxy.json
  Container starts -> reads proxy.json from vault
  Container executes -> uses proxy for all requests
```

No environment variables with secrets. No config files baked into the image. No API keys in command line arguments. Everything sensitive lives in the encrypted vault, accessed on demand.

### Output Destination

Screenshots, videos, HAR files, console logs, and any other artefacts produced during execution are written directly to a vault. The caller provides an output vault reference. The container commits artefacts to that vault as it produces them.

```
Output flow:
  Container executes sequence
  Step produces screenshot -> vault commit
  Step produces video -> vault commit (may be large, uses presigned S3)
  Caller reads outputs from vault (on-demand, tree-only access)
```

This means the caller does not need to poll, stream, or maintain a connection during long-running sessions. The container works. The vault fills with results. The caller reads when ready.

### Why This Works Now

The vault's dual branch model means the container can write to a clone branch safely. The tree-only loading means the container only pulls the config files it needs (not the whole vault). The immutable caching means repeat runs can reuse cached Playwright scripts or test data without re-fetching.

## Deployment Targets

| Target | Use Case | Notes |
|--------|----------|-------|
| Local Docker | Development, testing, CI | `docker run -p 8000:8000 ...` |
| Docker Hub | Public distribution | Easy for users to pull and run themselves |
| ECR | Customer-hosted deployments | Private image registry per customer |
| AWS Lambda (container) | Serverless, auto-scaling | Up to 15-minute execution time, cold start trade-off |
| Fargate | Long-running sessions | For workflows that exceed Lambda limits |

The same image deploys to all of these. The only difference is the invocation layer (HTTP server vs Lambda handler).

## Open Questions

1. **Chrome profile persistence:** Should sessions persist Chrome profiles across invocations (useful for maintaining logins) or start fresh each time? Leaning toward fresh-per-session with vault-stored profile as optional input.

2. **Timeout handling:** Lambda has a 15-minute hard limit. For longer sessions, we need Fargate. Does the caller specify the deployment target, or does the service route automatically based on estimated duration?

3. **Concurrent sessions per container:** One browser per container, or multiple browsers per container? One-per-container is simpler and safer. Multiple adds throughput but complicates resource management.

4. **Video encoding:** Playwright records WebM by default. Do we transcode to MP4 for broader playback compatibility? Leaning toward WebM only initially (it plays in all modern browsers and the video playback component from 04/08 supports it).
