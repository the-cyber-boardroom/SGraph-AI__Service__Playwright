# Vault-publish bootstrap — operator test guide

This document walks through testing the v0.2.23 vault-publish cold path end-to-end against your real AWS account.

**Why no Lambda was created by CI:** the CI pipeline (`ci-pipeline__dev.yml`) runs unit tests + builds Docker images. It does NOT call `sg vp bootstrap` — that's an operator-driven command that creates real AWS resources, so it requires explicit human consent (no CI account ever runs it).

---

## 0. Prerequisites

Before you run anything, confirm these in your shell session.

### 0.1 AWS credentials

```bash
aws sts get-caller-identity
# Expected: Account 745506449035 (the account that owns the *.aws.sg-labs.app cert)
```

If this fails, set `AWS_PROFILE` or `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` first.

### 0.2 Region

CloudFront is always `us-east-1` (global). Lambda + EC2 should be your normal region (`eu-west-2` per repo defaults).

```bash
echo "${AWS_DEFAULT_REGION:-eu-west-2}"
```

### 0.3 The wildcard cert (already issued)

```bash
aws acm describe-certificate \
  --region us-east-1 \
  --certificate-arn arn:aws:acm:us-east-1:745506449035:certificate/99346343-dc1e-4a62-a6d3-0f22ab7bfffa \
  --query 'Certificate.[DomainName,Status]' --output text
# Expected: *.aws.sg-labs.app  ISSUED
```

### 0.4 The Route53 wildcard ALIAS record (you already created this manually)

```bash
aws route53 list-resource-record-sets \
  --hosted-zone-id $(aws route53 list-hosted-zones-by-name --dns-name aws.sg-labs.app --query 'HostedZones[0].Id' --output text | sed 's:.*/::') \
  --query "ResourceRecordSets[?Type=='A' && starts_with(Name, '\*.')]"
# Expected: at least one ALIAS A record for *.aws.sg-labs.app pointing to a CloudFront distribution
```

If this comes back empty, you'll need to create the ALIAS after step 3 below, pointing at the CF distribution `sg vp bootstrap` returns.

### 0.5 Lambda execution role — CREATE THIS FIRST

`sg vp bootstrap --role-arn` is a required input. The role needs **two** permission sets:

1. **Basic Lambda execution** (CloudWatch logs)
2. **EC2 describe + start** (so the Waker can wake stopped vault-app instances)

One-time setup — save this as `/tmp/waker-trust-policy.json`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    { "Effect": "Allow",
      "Principal": { "Service": "lambda.amazonaws.com" },
      "Action": "sts:AssumeRole" }
  ]
}
```

And `/tmp/waker-permissions.json`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    { "Effect": "Allow",
      "Action": ["logs:CreateLogGroup", "logs:CreateLogStream", "logs:PutLogEvents"],
      "Resource": "arn:aws:logs:*:*:*" },
    { "Effect": "Allow",
      "Action": ["ec2:DescribeInstances"],
      "Resource": "*" },
    { "Effect": "Allow",
      "Action": ["ec2:StartInstances"],
      "Resource": "arn:aws:ec2:*:*:instance/*",
      "Condition": { "StringEquals": { "aws:ResourceTag/StackType": "vault-app" } } }
  ]
}
```

Then create the role:

```bash
aws iam create-role \
  --role-name sg-compute-vault-publish-waker-role \
  --assume-role-policy-document file:///tmp/waker-trust-policy.json

aws iam put-role-policy \
  --role-name sg-compute-vault-publish-waker-role \
  --policy-name waker-permissions \
  --policy-document file:///tmp/waker-permissions.json

# Capture the ARN — you'll pass it to bootstrap
WAKER_ROLE_ARN=$(aws iam get-role --role-name sg-compute-vault-publish-waker-role --query 'Role.Arn' --output text)
echo "$WAKER_ROLE_ARN"
# Expected: arn:aws:iam::745506449035:role/sg-compute-vault-publish-waker-role
```

## 3. Run bootstrap

This is the headline command. It deploys the Waker Lambda, creates a Function URL, and creates a CloudFront distribution.

```bash
sg vp bootstrap \
  --cert-arn arn:aws:acm:us-east-1:745506449035:certificate/99346343-dc1e-4a62-a6d3-0f22ab7bfffa \
  --zone aws.sg-labs.app \
  --role-arn "$WAKER_ROLE_ARN"
```

You'll get a `[y/N]` confirmation prompt. Type `y`.

**Expected output (after ~30s):**

```
  ✓  Bootstrapped
      Lambda      : sg-compute-vault-publish-waker
      Waker URL   : https://<random>.lambda-url.eu-west-2.on.aws/
      CF dist     : E<13-digit-id>
      CF domain   : d<random>.cloudfront.net
      Zone        : aws.sg-labs.app
      elapsed     : ~30000ms
```

**Capture the four IDs.** You'll need them for the next steps and for cleanup:

```bash
# Edit these from the output above
WAKER_NAME='sg-compute-vault-publish-waker'
WAKER_URL='https://...lambda-url.eu-west-2.on.aws/'
CF_DIST_ID='E...'
CF_DOMAIN='d...cloudfront.net'
```

### 3.1 Common failures at this step

| Symptom | Cause | Fix |
|---------|-------|-----|
| `lambda deploy failed: User is not authorized to perform: iam:PassRole` | Your IAM user can't pass the waker role | Add `iam:PassRole` on the waker role ARN to your user policy |
| `lambda deploy failed: The role defined for the function cannot be assumed by Lambda` | Trust policy missing or wrong | Re-check `/tmp/waker-trust-policy.json` from §0.5 |
| `lambda deploy failed: Unzipped size must be smaller than ...` | Waker folder includes heavyweight deps | The deployer only zips `sg_compute_specs/vault_publish/waker/` — confirm with `ls -la` |
| Hangs after "Bootstrapping…" | CloudFront create call (~20-30s normal); if >2 min check `aws cloudfront list-distributions` |

## 4. Verify each piece

### 4.1 Lambda exists

```bash
sg aws lambda deployment list | grep "$WAKER_NAME"
# Expected: one row showing the function with runtime python3.12, state ACTIVE
```

### 4.2 Function URL is reachable

```bash
sg aws lambda url show "$WAKER_NAME"
# Expected: function_url matches $WAKER_URL, auth_type NONE

# Hit the health endpoint directly
curl -sS "${WAKER_URL}health"
# Expected: {"status":"ok","service":"vault-waker"}
```

### 4.3 Waker handles an unknown slug

```bash
# Spoof the Host header so the Waker routes through Slug__From_Host
curl -sS -o /dev/null -w "%{http_code}\n" \
  -H 'Host: nonexistent-slug.aws.sg-labs.app' \
  "$WAKER_URL"
# Expected: 404 (slug not registered)
```

### 4.4 CF distribution exists

```bash
sg aws cf distribution show "$CF_DIST_ID"
# Expected: status InProgress (or Deployed if >15 min has passed), aliases includes *.aws.sg-labs.app
```

### 4.5 CF wait-for-Deployed (optional, slow — 10-15 min)

```bash
sg aws cf distribution wait "$CF_DIST_ID"
# Polls every 30s until status == Deployed
```

You can do the rest of the test without waiting — CF returns 503s until Deployed but the direct-to-Lambda path (§4.3) works immediately.

### 4.6 Route53 ALIAS points at the new distribution (you already created it manually)

```bash
HOSTED_ZONE_ID=$(aws route53 list-hosted-zones-by-name --dns-name aws.sg-labs.app --query 'HostedZones[0].Id' --output text | sed 's:.*/::')

aws route53 list-resource-record-sets \
  --hosted-zone-id "$HOSTED_ZONE_ID" \
  --query "ResourceRecordSets[?Name=='\\052.aws.sg-labs.app.']"
# Expected: AliasTarget.DNSName matches $CF_DOMAIN (or your existing dist)
```

If the ALIAS still points at an old distribution, update it now via the Route53 console or:

```bash
aws route53 change-resource-record-sets --hosted-zone-id "$HOSTED_ZONE_ID" --change-batch "{
  \"Changes\": [{
    \"Action\": \"UPSERT\",
    \"ResourceRecordSet\": {
      \"Name\": \"*.aws.sg-labs.app.\",
      \"Type\": \"A\",
      \"AliasTarget\": {
        \"HostedZoneId\": \"Z2FDTNDATAQYW2\",
        \"DNSName\": \"${CF_DOMAIN}\",
        \"EvaluateTargetHealth\": false
      }
    }
  }]
}"
```

`Z2FDTNDATAQYW2` is CloudFront's global hosted zone ID — same for every account.

## 5. End-to-end cold-path test

This is the headline test from the original brief (P2d success criteria).

### 5.1 Register a slug (this creates a vault-app stack — costs money)

```bash
# Pick a unique slug
SLUG=hello-vault-publish

sg vp register "$SLUG" --vault-key dev-pack
```

This takes **~2 minutes** — it provisions a real EC2 vault-app stack at `${SLUG}.aws.sg-labs.app`.

### 5.2 Confirm the warm path works

```bash
sg vp status "$SLUG"
# Expected: state RUNNING, fqdn ${SLUG}.aws.sg-labs.app, vault_url https://...

# CF needs to be Deployed for this to work — see §4.5
curl -sSI "https://${SLUG}.aws.sg-labs.app/"
# Expected: HTTP/2 200 (proxied through CF → Waker → vault-app)
```

### 5.3 Stop the vault-app instance (cold)

```bash
sg vault-app stop "$SLUG"
# Expected: state stopped; per-slug A record deleted (which is fine — wildcard ALIAS still routes)
```

### 5.4 First cold request → warming page

```bash
curl -sS "https://${SLUG}.aws.sg-labs.app/"
# Expected: HTML body containing "warming" and the slug name
# Side effect: Waker called ec2:StartInstances on the stopped instance
```

Watch the EC2 console — the instance state should flip `stopped` → `pending` → `running` over ~30 seconds.

### 5.5 Subsequent requests during boot → still warming

```bash
# Run a few times over the next minute
for i in 1 2 3; do
  curl -sSI "https://${SLUG}.aws.sg-labs.app/" | head -1
  sleep 10
done
# Expected: HTTP/2 202 (warming) until the instance health check passes
```

### 5.6 Eventually → proxied to vault-app

```bash
sleep 60
curl -sSI "https://${SLUG}.aws.sg-labs.app/"
# Expected: HTTP/2 200 (vault UI loaded directly via the Waker proxy)
```

## 6. Inspect Waker logs

The Lambda's CloudWatch log group is `/aws/lambda/sg-compute-vault-publish-waker`.

```bash
# Latest 50 lines
aws logs tail /aws/lambda/sg-compute-vault-publish-waker --since 10m --max-items 50

# Follow live
aws logs tail /aws/lambda/sg-compute-vault-publish-waker --follow
```

You should see one invocation per cold request. Each invocation logs the host header, the resolved slug, the EC2 state, and the action taken (start / 202 / proxy).

> **Note:** `sg vp waker logs/info/invoke` sub-commands are on the v0.2.24 follow-up plan — for now, use `aws logs tail` directly.

## 7. Cleanup (when you're done testing)

**Order matters.** CloudFront needs to be disabled before delete, and the disable → deleted transition takes ~15 minutes.

### 7.1 Unpublish the test slug

```bash
sg vp unpublish "$SLUG" --yes
# Deletes the vault-app stack + the SSM slug entry
```

### 7.2 Disable the CF distribution

```bash
sg aws cf distribution disable "$CF_DIST_ID"
# Returns immediately — CF starts the disable async
```

### 7.3 Wait for it to fully disable (~15 min)

```bash
sg aws cf distribution wait "$CF_DIST_ID"
# Polls until Status=Deployed AND Enabled=false
```

### 7.4 Delete the CF distribution

```bash
sg aws cf distribution delete "$CF_DIST_ID"
```

### 7.5 Delete the Lambda + Function URL

```bash
sg aws lambda url delete "$WAKER_NAME" --yes
sg aws lambda deployment delete "$WAKER_NAME" --yes
```

### 7.6 (Optional) Delete the IAM role

```bash
aws iam delete-role-policy --role-name sg-compute-vault-publish-waker-role --policy-name waker-permissions
aws iam delete-role          --role-name sg-compute-vault-publish-waker-role
```

### 7.7 (Optional) Update the Route53 ALIAS

If you keep the ALIAS pointed at the now-deleted CF distribution, lookups will fail. Either re-point it at a new distribution or delete it.

## 8. Cost notes

| Resource | Idle cost | Per-use cost |
|----------|-----------|--------------|
| Lambda function (waker) | $0 | ~$0.0000002 per cold request |
| Lambda Function URL | $0 | included above |
| CloudFront distribution | $0 | $0.085/GB egress; first 1 TB/mo free |
| Route53 ALIAS | $0 | included in zone hosting ($0.50/zone/mo) |
| Vault-app EC2 (stopped) | EBS only (~$0.02/GB-month for the root volume) | $0.0416/hr for `t3.medium` when running |

A stopped vault-app + a deployed Waker costs <$1/month of standing capacity. The big variable is how long instances run when woken.
