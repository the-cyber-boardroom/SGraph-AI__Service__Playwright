1. The reframe in one sentence
═══════════════════════════════════════════════════════════════════════════════

The stack is not a QA stack. It is a generic execution substrate for any
vault-resident app (`vault_app`) needing persistent vault storage,
browser automation, and a transparent capturing proxy. QA is the first
vault-app; other use cases (traffic simulation, scraping, agent workbench,
regression harnesses) follow the same shape and reuse the same stack image.

```
┌──────────────────────────────────────────────────────────────────────┐
│  EC2 instance (or laptop, or Fargate task)                           │
│                                                                      │
│  ┌──────────────────────────┐                                        │
│  │ host control plane        │  (provisioning, log shipping,         │
│  │                           │   SSM agent, compose lifecycle)       │

│  │ sg-send-vault             │    │ sg-playwright             │         │
│  │ (UI + Send API + vault   │←───│ (browser automation)      │         │
│  │  storage)                 │    │ HTTP_PROXY → mitmproxy    │         │
│  │ :8080 EXTERNAL            │    │ INTERNAL ONLY             │         │
│  └──────────────────────────┘    └──────────────────────────┘         │
│           │                                  │                       │
│           └──────────────┬───────────────────┘                       │
│                          ▼                                            │
│  ┌──────────────────────────────────────────────────────┐            │
│  │ mitmproxy                                             │            │
│  │ - passive capture only                                │            │
│  │ - intercept script baked at image-build time          │            │
│  │ - NO admin API exposed                                │            │
│  │ - NO external ports                                   │            │
│  └──────────────────────────────────────────────────────┘            │
└──────────────────────────────────────────────────────────────────────┘
```

External surface: ONE port — sg-send-vault :8080 (TLS-fronted in prod via
caddy / traefik / ALB). Operators wanting to inspect playwright or
mitmproxy internally use SSM port-forward; no host-published ports.

6. Vault preload contract
═══════════════════════════════════════════════════════════════════════════════

Two preload modes — both supported on day one — orthogonal to where the
sg-send-vault's `/data` lives physically.

6.1 Preload modes

(a) Bind-mount. Vaults already on disk at startup.
    `-v /host/path:/data` on the sg-send-vault container. Vault picks them
    up via its default `disk` backend.

(b) Seed env var. Clone vaults at first boot.
    `SG_VAULT_APP__SEED_VAULT_KEYS=key1,key2,key3` on the stack. The
    host control plane (or an init hook) shells out to `sgit clone` once
    per key into the running vault. Idempotent: skip keys whose vault
    already exists.

These compose. Bind-mount sets up the substrate; seed env var fills in
missing vaults on first boot.

6.2 `/data` backing storage (orthogonal to preload)

| Backing                  | When                                                | How                                              |
|--------------------------|-----------------------------------------------------|--------------------------------------------------|
| Local disk (ephemeral)   | Laptop dev, CI smoke                                | No mount — writable layer only                   |
| EBS volume               | Single-EC2 persistent; snapshot-able                | EBS attached to EC2, mounted on host, bind into `/data` |
| Native S3 backend        | Multi-replica, durable, shared                       | `SEND__STORAGE_MODE=s3` + AWS creds — vault talks to S3 directly, no filesystem |
| Mountpoint-for-S3        | Filesystem-view over S3 is specifically needed       | Mount bucket at `/mnt/s3data`, bind into `/data` |

Recommendation: native S3 first (no POSIX emulation overhead, no append-
only constraints, no eventual-list quirks); Mountpoint only if a concrete
reason emerges. EBS for single-EC2 persistent deployments where snapshot
backups are wanted. Local disk for dev / CI.

7. Auth — single stack-wide secret
═══════════════════════════════════════════════════════════════════════════════

Reuse `SGRAPH_SEND__ACCESS_TOKEN` across the stack. Already wired to
`FAST_API__AUTH__API_KEY__VALUE` in the playwright service. CLI already
sets it from a user-provided value.

- Vault UI (in the browser) sends `x-sgraph-access-token` on every API
  call it issues to either sg-send-vault or sg-playwright from inside a
  vault-app — same header, same value, same middleware.
- Playwright service → sg-send-vault calls (when staging artefacts via
  `VAULT` sink) carry the same token in the same header.
- No new auth code in this repo. Existing middleware covers it.

Per-container keys are a future enhancement, out of scope here.

8. Mitmproxy — passive only
═══════════════════════════════════════════════════════════════════════════════

Reuse what already exists in the agent_mitmproxy spec — same intercept
script pattern, same CLI knobs:

- `SG_PLAYWRIGHT__DEFAULT_PROXY_URL` (already in reality doc) — points
  playwright at the mitmproxy container's proxy port on the docker
  network.
- `SG_PLAYWRIGHT__IGNORE_HTTPS_ERRORS` — preserved.
- Intercept script and addon configuration knobs from the existing
  mitmproxy spec.

Constraints for THIS brief:
- No admin API exposed. The 6 admin endpoints documented for
  agent_mitmproxy stay disabled in this stack's compose.
- No external ports. Mitmproxy listens on the docker network only.
- Intercept script provided at image-build time. Default is a no-op
  passthrough that logs to stdout in the existing NDJSON shape.
- Per-purpose stack images can FROM the base and bake in their own
  intercept script. Runtime bind-mount of a script over the baked-in
  default is an explicit override, not the default path.

No new code in this repo — the existing mitmproxy spec already covers
everything needed.

    separate, so playwright service releases don't force stack rebuilds
    and vice versa.
