2. Scope of "QA Service" inside this repo
═══════════════════════════════════════════════════════════════════════════════

The brief spans five concerns. The Architect's call on where each lives:

| Brief concern                              | Lives in                                                  | Why |
|-------------------------------------------|-----------------------------------------------------------|-----|
| Vault layout (`qa-vault/`)                 | Vault server (sibling repo) + a tiny client in sg_compute_specs/qa/ | Vault is already a separate stack; agents read/write via existing vault primitives. |
| QA UI (App Mode, iframe mode)              | sgraph_ai_service_playwright__api_site/ (UI lives here)   | UI is already on a sibling tree; the iframe-mode JS is UI-side and does not touch this service. |
| Real-browser execution                     | THIS repo (Playwright service) — new `/qa/scenario/run`   | Browser automation is the contract this service owns. |
| MITM capture per run                       | agent_mitmproxy sidecar — add a per-run-id log sink       | Mitmproxy already exists; only its persistence layer needs a per-run key. |
| Scheduler (5-min loop)                     | SG/Compute scheduler (new `sg_compute_specs/qa/` spec)    | Scheduling is not browser automation; lives in the SG/Compute stack alongside other specs. |
| Daily summary / static reports             | SG/Compute reports generator (same `sg_compute_specs/qa/`) | Consumes the vault `runs/` tree; reads-only against this service. |

The boundary this repo owns: receive a scenario, run it in a real browser,
return a typed result envelope, optionally write artefacts to caller-chosen
sinks. Everything else is upstream (vault) or downstream (scheduler, UI,
reports).

3. New surface area in this repo
═══════════════════════════════════════════════════════════════════════════════

3.1 New endpoint (one)

`POST /qa/scenario/run` — accepts `Schema__QA__Scenario__Run__Request`,
returns `Schema__QA__Scenario__Run__Response`. Owned by a new
`Routes__QA` under sg_compute_specs/playwright/core/fast_api/routes/.

This is a THIN wrapper. The route does no logic; it calls
`Playwright__Service.qa_scenario_run(...)` which:

  a) builds a `Schema__Sequence__Request` from the scenario,
  b) delegates to the existing `Sequence__Runner.execute()`,
  c) hands the `Schema__Sequence__Response` to a new
     `Assertion__Evaluator` to compute per-assertion verdicts,
  d) returns the union (sequence response + assertion results) as the QA
     response envelope.

Why a separate endpoint rather than overloading `/sequence/execute`:

- Assertions are a different concern from steps. Steps mutate page state;
  assertions inspect it. Mixing the two muddies `Schema__Step__*`.
- The QA envelope carries scenario-level metadata (`scenario_id`,
  `environment`, `mode`) that does not belong in
  `Schema__Sequence__Request`.
- Layering: `/qa/scenario/run` is a Layer-4 wrapper on top of the existing
  Layer-3 `/sequence/execute`. Same pattern as Layer-3 wraps Layer-0.

3.2 New schemas (all `Type_Safe`, one class per file, no Pydantic, no Literals)

Under sg_compute_specs/playwright/core/schemas/qa/:

  `Schema__QA__Scenario`
      scenario_id        : Safe_Str__Id
      target_url         : Safe_Str__Url
      environment        : Safe_Str__Key                                       # "dev" / "main" / "prod" — value-only; not enum (env list grows)
      mode               : Enum__QA__Execution__Mode

      capture_config     : Schema__Capture__Config
      steps              : List[dict]                                         # heterogeneous; parsed by STEP_SCHEMAS
      assertions         : List[dict]                                         # heterogeneous; parsed by ASSERTION_SCHEMAS
      tags               : List[Safe_Str__Key]
      notes              : Safe_Str__Text                                     # short; full notes live in vault notes.md

  `Schema__QA__Scenario__Run__Request`
      scenario           : Schema__QA__Scenario

      trace_id           : Safe_Str__Trace_Id = None

  `Schema__QA__Scenario__Run__Response`

      scenario_id        : Safe_Str__Id

3.4 New service classes

Two new classes under sg_compute_specs/playwright/core/service/qa/:

  `QA__Scenario__Runner`
      - Public entrypoint called by `Playwright__Service.qa_scenario_run`.
      - Composes a `Schema__Sequence__Request` from a `Schema__QA__Scenario`
        (steps, browser_config, capture_config flow through unchanged).
      - Delegates to existing `Sequence__Runner.execute()` (NO new code
        touches `page.*`; the rule that `Step__Executor` is the sole `page.*`
        caller is preserved).
      - After the sequence completes, hands the response + the live session
        (or a fresh page if the session was torn down) to
        `Assertion__Evaluator`.

  `Assertion__Evaluator`
      - Reads each `Schema__QA__Assertion__*` and runs it.
      - URL / status_code / header assertions read from `Schema__Sequence__
        Response` and the MITM-supplied network log — NO `page.*` needed.
      - `SELECTOR_VISIBLE` / `SELECTOR_TEXT_*` assertions need DOM access.
        These are implemented as synthetic `Schema__Step__Wait_For` and
        `Schema__Step__Get_Content` invocations appended to the sequence
        BEFORE handoff to `Sequence__Runner`, so `Step__Executor` is still
        the only `page.*` caller. The evaluator interprets the resulting
        `Schema__Step__Result__*` to produce
        `Schema__QA__Assertion__Result`.
      - This keeps the boundary rule (Architect quality gate) intact:
        Step__Executor remains the sole `page.*` site.

Cross-schema validation goes into the existing `Request__Validator`:

  - `validate__qa_scenario_request(...)` ensures every assertion type has a
    registered schema, every selector-based assertion has a selector, the
    requested capture sinks are supported by current capabilities, and the
    JS allowlist gate is honoured if any step is `EVALUATE`.

|------------------|--------------------------------|--------------------------------------|
| wait             | `Schema__Step__Wait_For`       | Maps `selector` / `state` / `ms`.    |
| fill             | `Schema__Step__Fill`           | Direct.                              |
| click            | `Schema__Step__Click`          | Direct.                              |

3.6 MITM integration touchpoints (changes outside this service, listed for
completeness)

This service exposes ONE new contract toward the sidecar: every QA run
attaches a `X-SG-Run-Id` request header (added in `Browser__Launcher` /
`Step__Executor` via the existing `extra_http_headers` channel on
`Schema__Session__Credentials`). The sidecar groups its NDJSON audit log
by `X-SG-Run-Id`. After the run, the QA service calls a small sidecar
endpoint (`GET /capture/network-log/{run_id}`, NEW on the sidecar — not
this repo) to fetch the JSON log and stage it as an artefact via
`Artefact__Writer`, producing the `network_log_ref` in the response.

The Architect notes that the sidecar endpoint is a sibling-repo change.
This repo only adds the header injection + the typed `network_log_ref`
field; the sidecar work is logged as a dependency, not part of any slice
in this plan.

4. Contracts with the outside (vault, UI, scheduler)
═══════════════════════════════════════════════════════════════════════════════

4.1 Request shape (scheduler / UI → this service)

POST /qa/scenario/run
Content-Type: application/json
X-API-Key: <api-key>     (existing API-key middleware)

{
  "scenario": {
    "scenario_id": "login-happy-path",
    "target_url": "https://app.example.com/",
    "environment": "dev",
    "mode": "real_browser",
    "browser_config": {...},      # optional; defaults apply
    "capture_config": {           # required (Type_Safe forces a value)
      "screenshot_on_fail": {"enabled": true, "sink": "vault", ...},
      "network_log":        {"enabled": true, "sink": "vault", ...}
    },
    "steps":      [{...}, {...}],
    "assertions": [{"type": "url_contains", "expected_substring": "/home"}],
    "tags":  ["smoke", "login"],
    "notes": ""
  },
  "run_id":   "2026-05-13T21-00__login-happy-path",   # optional, idempotency
  "trace_id": null
}

4.2 Response shape (this service → scheduler / UI)

A full `Schema__QA__Scenario__Run__Response.json()` — `run_id`,
`scenario_id`, `trace_id`, `status`, the full `sequence_response`
(per-step results + cumulative artefact list), `assertion_results`,
timestamps, and an optional `network_log_ref`.

4.3 Artefact addressing (the decision)

Use `Schema__Artefact__Ref` as-is. The caller picks the sink in
`capture_config`. The Architect's recommendation for the QA UI/scheduler
is `sink=VAULT` keyed by run, written by `Artefact__Writer` into the
vault path `runs/{date}/{run_id}/{filename}` — but this service does not
enforce the path; the vault writer (sibling repo) imposes the layout.

Reasoning for NOT inlining base64: a run can produce multi-MB
screenshots / videos / HAR — Lambda response-size limits (6 MB sync,
1 MB streaming) bite quickly. Vault references stay small.

4.4 Idempotency / run-id strategy

If the caller supplies `run_id`, the service uses it verbatim and the
caller takes responsibility for uniqueness. If omitted, the service
generates `{ISO-timestamp-second}__{scenario_id}__{uuid8}` — same scheme
as the brief's `runs/{date}/{run-id}/` layout suggests. The service does
NOT dedupe re-runs of the same `run_id`; idempotency is the vault's
concern (a put with an existing key is the caller's mistake to
detect).

6. Risks, open questions, decisions to escalate
═══════════════════════════════════════════════════════════════════════════════

Q1 (HIGH). Brief talks about "iframe injection" as a first-class execution
mode. This service can ONLY do real-browser; it has no mechanism (and
shouldn't) to ship JS into a UI iframe. Architect proposes:
`Enum__QA__Execution__Mode.IFRAME_INJECTION` is REJECTED by
`Request__Validator` with a 422 "mode_not_supported_by_service" — the UI
must not route iframe-mode runs here. Confirm?

Q2 (MED). The current MITM sidecar writes NDJSON to stdout. Capturing
per-run logs needs the sidecar to add a `GET /capture/network-log/{run_id}`
endpoint backed by an in-memory ring buffer keyed by `X-SG-Run-Id`. Should
this brief spawn that sidecar slice now, or block until a separate brief
is filed?

Q3 (MED). The 11 acceptance criteria reference "daily summary" and "static
projections". The Architect places these in the SG/Compute scheduler spec,
NOT in this service. Confirm — or do you want a `/qa/report/...` surface
exposed here too?

Q4 (LOW). "environment" (dev / main / prod) is modelled as
`Safe_Str__Key` (free-form value), not `Enum__QA__Environment`, because
the brief uses it as a string and the environment list is owned by the
vault. Confirm this is correct (vs forcing an enum that has to grow each
time a tenant adds an environment).

Q5 (LOW). Run-id strategy: free-form `Safe_Str__Id` or `Safe_Str__Run_Id`
new primitive enforcing the `YYYY-MM-DDTHH-mm-ss__{slug}__{uuid8}` shape?
Architect leans toward Safe_Str__Id (looser; callers can use timestamps if
they want). Confirm.

Q6 (LOW). The brief implies `assertions[]` can include arbitrary JSON.
This plan forces every assertion to match a registered
