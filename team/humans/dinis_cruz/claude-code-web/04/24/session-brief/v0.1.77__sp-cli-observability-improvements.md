## Background you need to know

The `sp` CLI (`scripts/provision_ec2.py`) provisions an EC2 instance running a
Docker Compose stack. The stack includes:

| Service | Purpose |
|---|---|
| `playwright` | FastAPI browser automation service (port 8000) |
| `agent-mitmproxy` | Sidecar proxy, intercepts browser traffic (port 8001 admin) |
| `fluent-bit` | Log shipper → OpenSearch (conditional on `OPENSEARCH_ENDPOINT`) |
| `prometheus` + `cadvisor` + `node-exporter` | Metrics (conditional on `AMP_REMOTE_WRITE_URL`) |
| `dockge` | *(new)* Docker Compose UI, replaces Portainer (port 5001, SSM-forward only) |
| `browser` + `browser-proxy` | KasmVNC Chromium (conditional on upstream proxy) |

The observability stack (OpenSearch, AMP, Grafana) is **ephemeral** — created
each morning, backed up and deleted each evening. fluent-bit ships logs from
all containers on the EC2 host to OpenSearch. The prior state of the fluent-bit
config sent only raw `log` strings with two static fields (`stack`, `environment`),
making every record in OpenSearch Discover look identical.

## Change 1 — OpenSearch log quality (commit `2d032a7`)

**Files:** `scripts/provision_ec2.py`, `tests/unit/scripts/test_provision_ec2.py`

### Problem
Every log record in OpenSearch differed only in the `log` field. No structured
fields. Health-check GETs (`GET /health/ping 200`) made up the majority of
records — pure noise.

### Solution

#### New constant: `FLUENT_BIT_PARSERS_CUSTOM`
Written to `/opt/sg-playwright/config/parsers_custom.conf` at boot. Defines a
`uvicorn_access` parser using regex that splits FastAPI/uvicorn access log lines:

```
INFO:     172.18.0.1:49234 - "POST /v1/browser/action HTTP/1.1" 200 OK
```

into four indexed fields:

| Field | Type | Example |
|---|---|---|
| `http_method` | string | `POST` |
| `http_path` | string | `/v1/browser/action` |
| `http_status` | integer | `200` |
| `client_ip` | string | `172.18.0.1` |

The original `log` field is **preserved** alongside the new fields (`Reserve_Data On`,
`Preserve_Key On`), so no data is lost.

#### Updated `FLUENT_BIT_CONF_TEMPLATE`
Three new filter blocks added (in order — order matters in fluent-bit):

```
[FILTER] parser  → runs uvicorn_access on every record's log field
[FILTER] grep    → Exclude http_path ^/health/   (drops health-check records)
[FILTER] grep    → Regex log \S                  (drops blank lines)
```

Also added `Path_Key container_path` to the INPUT block — each record now
carries the source file path (which contains the container ID).

#### `render_observability_configs_section` updated
Writes three files at boot instead of two:
1. `prometheus.yml`
2. `parsers_custom.conf` ← new
3. `fluent-bit.conf`

#### `COMPOSE_SVC_FLUENT_BIT` updated
Switched from mounting individual files to mounting the whole config directory:
```yaml
command: /fluent-bit/bin/fluent-bit -c /opt/sg-playwright/config/fluent-bit.conf
volumes:
  - /opt/sg-playwright/config:/opt/sg-playwright/config:ro   # ← replaces per-file mounts
```
The built-in parsers file is still loaded from its original path inside the
image (`/fluent-bit/etc/parsers.conf`).

## Change 2 — Container name enrichment (commit `622330e`)

**Files:** `scripts/provision_ec2.py`, `tests/unit/scripts/test_provision_ec2.py`

### Problem
`container_path` contains a 64-char hex container ID, not the human-readable
service name. You couldn't tell in OpenSearch whether a log came from
`playwright` or `agent-mitmproxy`.

### Solution

#### New constant: `FLUENT_BIT_LUA_CONTAINER_NAME`
Written to `/opt/sg-playwright/config/container_name.lua`. A Lua filter that:

1. Extracts the 12-char short container ID from the `container_path` field
2. Looks it up in `/opt/sg-playwright/config/container-names.txt` (a flat
   `<id> <name>` file written by the host after `docker compose up -d`)
3. Caches results — the file is only re-read when an unknown ID is seen
   (handles containers that started after fluent-bit)
4. Adds `container_name` field to the record (e.g. `playwright`,
   `agent-mitmproxy`, `fluent-bit`, `cadvisor`, etc.)

#### `container-names.txt` written at boot
Both `AMI_USER_DATA_TEMPLATE` and `USER_DATA_TEMPLATE` now include this step
after `docker compose up -d`:

```bash
sleep 5
docker ps --format '{{.ID}} {{.Names}}' \
  | sed 's/ sg-playwright-/ /; s/-[0-9]*$//' \
  > /opt/sg-playwright/config/container-names.txt || true
```

The `sed` strips the project prefix (`sg-playwright-`) and replica suffix (`-1`)
so you get clean names like `playwright`, not `sg-playwright-playwright-1`.

#### Lua filter added to `FLUENT_BIT_CONF_TEMPLATE`
```
[FILTER] lua → calls add_container_name on every docker.* record
```
Placed immediately after `record_modifier` (before the parsers), so
`container_name` is available to all downstream filters.

### New OpenSearch fields (full set after both changes)

| Field | Source | Always present? |
|---|---|---|
| `log` | Docker JSON log | Yes |
| `stream` | Docker JSON log (stdout/stderr) | Yes |
| `stack` | record_modifier | Yes |
| `environment` | record_modifier | Yes |
| `container_path` | fluent-bit Path_Key | Yes |
| `container_name` | Lua filter | Yes (fallback: short ID) |
| `http_method` | uvicorn_access parser | FastAPI access logs only |
| `http_path` | uvicorn_access parser | FastAPI access logs only |
| `http_status` | uvicorn_access parser | FastAPI access logs only |
| `client_ip` | uvicorn_access parser | FastAPI access logs only |

## Change 3 — Portainer → Dockge (commit `20edea9`)

**File:** `scripts/provision_ec2.py`

### Why
Portainer is a general container manager. We only use it to view compose
stacks and logs. Dockge is compose-first (created by the same author as
Uptime Kuma), lighter, and has a better UI for our use case. It does **not**
require Podman — it runs on Docker like everything else in the stack.

### What changed

| Before | After |
|---|---|
| `EC2__PORTAINER_PORT = 9000` | `EC2__DOCKGE_PORT = 5001` |
| `EC2__PORTAINER_IMAGE = portainer/portainer-ce:latest` | `EC2__DOCKGE_IMAGE = louislam/dockge:1` |
| Named Docker volume `portainer_data:` | Host bind mount `/opt/dockge/data:/app/data` |
| `COMPOSE_SVC_PORTAINER` | `COMPOSE_SVC_DOCKGE` |
| `sp forward-portainer` | `sp forward-dockge` |

**Key config trick:** `DOCKGE_STACKS_DIR=/opt` — Dockge scans `/opt/<name>/docker-compose.yml`
for stacks. Our compose is at `/opt/sg-playwright/docker-compose.yml`, so Dockge
discovers and displays it as the `sg-playwright` stack with no path changes needed.

**Volumes section:** The compose footer now omits `volumes:` entirely when no
named volumes are needed (previously `portainer_data:` was always there).
`prometheus_data:` still appears when AMP is configured.

**Access:** `sp forward-dockge` → `http://localhost:5001/` — first visit sets
admin password.

## Change 4 — CI test fix (commit `622330e`, same commit as container name)

**File:** `tests/unit/scripts/test_provision_ec2.py`

### Problem
4 tests in `test_preflight_check` were failing in CI with
`botocore.exceptions.NoCredentialsError: Unable to locate credentials`:

```
test__generates_random_key_when_not_set
test__prints_account_region_registry_and_image_uris
test__returns_account_region_registry
test__warns_when_api_key_value_not_set
```

### Root cause
`preflight_check` calls `ensure_caller_passrole(account)` which immediately
calls `boto3.client('sts').get_caller_identity()` directly — bypassing the
`aws_account_id` wrapper that `_stub_aws` was stubbing. CI has no AWS
credentials, so this failed.

### Fix
`_stub_aws` now also stubs `ensure_caller_passrole`:

```python
provision_ec2.ensure_caller_passrole = lambda account: {
    'ok': True, 'action': 'already_exists', 'detail': 'stubbed'
}
```

All 4 tests now pass. Full suite: **39/39 green**.

## Current state of `render_observability_configs_section`

The function writes 4 files to `/opt/sg-playwright/config/` at boot:

```
prometheus.yml          — Prometheus scrape config + optional AMP remote write
parsers_custom.conf     — fluent-bit custom parser definitions (uvicorn_access)
container_name.lua      — Lua filter for container_name field enrichment
fluent-bit.conf         — main fluent-bit config (references the three above)
```

fluent-bit also reads the **5th file** at runtime (not written at boot):
```
container-names.txt     — written by docker ps after compose up, refreshed by Lua on cache miss
```

## What's NOT done yet (natural next steps)

### Mitmproxy structured logs
The `agent-mitmproxy` sidecar logs proxy flows to stdout. These are currently
shipped as raw `log` strings. Adding a mitmproxy parser would give structured
fields like `proxy_url`, `proxy_method`, `proxy_status`, `proxy_duration`.
The exact log format depends on the mitmproxy addon configuration — check what
the container actually emits before writing a parser.

### Drop infrastructure noise
`cadvisor`, `prometheus`, `node-exporter` emit verbose Go-style logs with low
signal value. These can be filtered out by adding a grep filter that excludes
logs from those `container_name` values:
```ini
[FILTER]
    Name    grep
    Match   docker.*
    Exclude container_name  ^(cadvisor|node-exporter|prometheus|fluent-bit)$
```
This wasn't done yet because it requires the container_name field to be
populated first (which it now is).

### OpenSearch saved objects import (unresolved from prior session)
The `ob dashboard-import` command was seeing 400 errors on some saved objects.
A three-layer import strategy was added (bulk_create → per-object → system-index
fallback) in a prior session. Status of that fix should be confirmed by looking
at recent `ob create` run logs.
