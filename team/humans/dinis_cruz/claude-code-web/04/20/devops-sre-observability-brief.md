## Context

The SG Playwright service runs as two Docker containers on a t3.large EC2 instance:

| Container | Port (host) | Role |
|-----------|-------------|------|
| `playwright` | 8000 | FastAPI browser-automation service |
| `agent-mitmproxy` | 8001 | mitmproxy sidecar (all browser traffic routed through it) |

EC2 instances are provisioned and managed via `sg-ec2` CLI (`scripts/provision_ec2.py`). AMI baking is automated in `.github/workflows/bake-ami.yml`.

**Current visibility is almost zero.** The only observability today is:

- `sg-ec2 health` — 3 health endpoints polled manually
- `sg-ec2 smoke` — 4-URL smoke test with cold/warm timing
- AWS CloudWatch basic EC2 metrics (CPU, network — no process-level detail)
- mitmproxy web UI at port 8001 (no persistence, resets on container restart)

### Observed performance issues (smoke test results)

```
url                        1st_ms    warm_ms    notes
https://www.google.com     31566     17681      Both cold and warm very slow
https://sgraph.ai          26231     21072      
https://send.sgraph.ai     31185      1133      Warm suspiciously fast — noise?
https://news.bbc.co.uk      3549     33419      Warm SLOWER than cold — high variance
```

Root cause hypotheses:
1. **Every request = fresh Chromium** (confirmed stateless) — 3–8s overhead minimum per request
2. **mitmproxy HTTPS interception** — MITM on all HTTPS adds TLS overhead × number of sub-resources
3. **t3 burstable CPU credits** — CPU credit balance observed at 27.44 and recovering; spikes to 50.3% during smoke test
4. The high variance between cold/warm suggests the "warm" label is misleading — all three requests are cold starts (fresh Chromium per request)

## Objective

Add **maximum visibility** into what is happening on the EC2 stack using open-source tools. Specifically:

1. Per-request Chromium lifecycle timing (launch → navigate → screenshot → teardown)
2. Container resource usage (CPU, memory, network) over time
3. mitmproxy traffic analytics (flows, hosts, latency, error rates)
4. System-level metrics (disk I/O, process count, memory pressure)
5. Alerting / anomaly detection for slow requests

---

## Proposed stack (open-source, self-hosted on same EC2)

| Tool | Role | Port |
|------|------|------|
| **Prometheus** | Metrics scraping + time-series store | 9090 |
| **Grafana** | Dashboards + alerting UI | 3000 |
| **cAdvisor** | Container-level CPU/mem/net metrics | 8080 (internal) |
| **Node Exporter** | Host-level metrics (CPU, disk, mem) | 9100 (internal) |
| **Loki** | Log aggregation | 3100 (internal) |
| **Promtail** | Log shipping (Docker → Loki) | — |

All added to the existing `docker-compose.yml` generated by `render_compose_yaml()` in `scripts/provision_ec2.py`.

The observability containers should:
- NOT be exposed publicly (no new security group rules)
- Be accessible via `sg-ec2 forward 3000` (Grafana) and `sg-ec2 forward 9090` (Prometheus)

## Specific tasks

### 1. Add observability containers to `render_compose_yaml()`

In `scripts/provision_ec2.py`, extend the `COMPOSE_YAML_TEMPLATE` to include:

```yaml
  prometheus:
    image: prom/prometheus:latest
    volumes:
      - /opt/sg-playwright/prometheus.yml:/etc/prometheus/prometheus.yml:ro
    restart: always
    networks: [sg-net]

  grafana:
    image: grafana/grafana:latest
    environment:
      GF_SECURITY_ADMIN_PASSWORD: '{api_key_value}'
      GF_AUTH_ANONYMOUS_ENABLED: 'false'
    volumes:
      - grafana_data:/var/lib/grafana
    restart: always
    networks: [sg-net]

  cadvisor:
    image: gcr.io/cadvisor/cadvisor:latest
    volumes:
      - /:/rootfs:ro
      - /var/run:/var/run:ro
      - /sys:/sys:ro
      - /var/lib/docker/:/var/lib/docker:ro
    restart: always
    networks: [sg-net]

  node-exporter:
    image: prom/node-exporter:latest
    restart: always
    networks: [sg-net]
```

Prometheus config (`prometheus.yml`) should be embedded in the user-data script alongside `docker-compose.yml`.

### 2. Add a `/metrics` endpoint to the Playwright FastAPI service

The Playwright service (`sgraph_ai_service_playwright/`) should expose Prometheus-compatible metrics:

- `sg_playwright_request_total` — counter, labels: `endpoint`, `status`
- `sg_playwright_request_duration_seconds` — histogram, labels: `endpoint`
- `sg_playwright_chromium_launch_seconds` — histogram (from `timings.browser_launch_ms`)
- `sg_playwright_navigate_seconds` — histogram (from `timings.steps_ms`)
- `sg_playwright_chromium_teardown_seconds` — histogram (from `timings.browser_close_ms`)

Use the `prometheus_client` Python library. Add a `Routes__Metrics` class following the existing pattern in `sgraph_ai_service_playwright/fast_api/routes/`.

### 3. Add Grafana dashboards

Pre-provision dashboards via `grafana/provisioning/` volume mounts (embedded in user-data):

**Dashboard 1 — Request Performance**
- Chromium launch time percentiles (p50, p90, p99) over time
- Total request duration histogram
- Requests per minute by endpoint

**Dashboard 2 — Container Resources**
- CPU % per container (playwright vs mitmproxy vs observability)
- Memory usage per container
- Network bytes in/out per container

**Dashboard 3 — mitmproxy Traffic**
- Total flows per minute
- Top 10 hosts by request count
- Error rate (4xx/5xx from mitmproxy perspective)
- Note: requires mitmproxy to expose a Prometheus endpoint or custom scraper

### 4. Update `sg-ec2` CLI

New commands:
- `sg-ec2 forward grafana` → forwards port 3000 via SSM (convenience alias)
- `sg-ec2 forward prometheus` → forwards port 9090
- `sg-ec2 metrics` → prints current Prometheus metric values (raw text, no port-forward needed — goes through the API key-protected port 8000)

### 5. Security group — no new open ports

All observability ports (3000, 9090, 9100, 8080) must remain on the `sg-net` Docker bridge only. Access only via `sg-ec2 forward`. The FastAPI `/metrics` endpoint goes through port 8000 (API-key protected).
