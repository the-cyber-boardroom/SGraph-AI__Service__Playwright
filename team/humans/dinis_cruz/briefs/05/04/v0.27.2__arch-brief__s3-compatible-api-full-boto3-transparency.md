## The Non-Negotiable Requirement

A boto3 client pointed at our endpoint must have **zero idea** it is not talking to the real AWS S3. Not "mostly compatible." Not "works for the common cases." Identical. A normal HTTP client, the AWS CLI, boto3, any S3 SDK in any language: all must work without modification.

```python
import boto3

# This must work identically whether endpoint_url points to
# AWS S3 or to our server
s3 = boto3.client('s3', endpoint_url='https://our-server:9000')

s3.put_object(Bucket='my-bucket', Key='file.txt', Body=b'hello')
obj = s3.get_object(Bucket='my-bucket', Key='file.txt')
s3.list_objects_v2(Bucket='my-bucket')
s3.create_bucket(Bucket='new-bucket')
```

This means we must handle not just the S3 data operations but also the AWS request signing (SigV4), the XML response format, the error response format, the headers, and the edge cases that boto3 relies on.

## The Call Log (Discovery-First Approach)

We will not guess which S3 APIs to implement. We will discover them.

### How

Every request that hits our API is logged in full:

```json
{
  "timestamp": "2026-04-30T10:15:32.456Z",
  "method": "PUT",
  "path": "/my-bucket/file.txt",
  "headers": {
    "Authorization": "AWS4-HMAC-SHA256 ...",
    "Content-Type": "application/octet-stream",
    "x-amz-content-sha256": "...",
    "x-amz-date": "20260430T101532Z"
  },
  "query_params": {},
  "body_size": 1234,
  "aws_action": "PutObject",
  "response_status": 200,
  "response_time_ms": 45,
  "handled_by": "local",
  "proxied": false
}
```

### Why

When we start using our S3 server with real workloads (SG/Send's Memory-FS, osbot-aws, SGit, the Cyber Boardroom), the call log tells us exactly which APIs are actually called. We implement those. We do not implement APIs nobody uses.

The call log also reveals:

- Which APIs are called most frequently (optimise these first)
- Which APIs we have not implemented yet (they will fail, and the log shows what to build next)
- Which APIs are IAM-related (STS, identity calls that boto3 makes before any S3 operation)
- Request patterns (bulk uploads, sequential reads, presigned URL generation)

## Four Modes of Operation

### Mode 1: Full Local

All requests handled locally. No connection to AWS. Data stored on disk, in a vault, or in memory.

```
Client -> Our API -> Local Storage (disk/vault/memory)
```

Use case: air-gapped environments, development, testing, offline operation. The S3 server is completely self-contained.

### Mode 2: Full Proxy (Pass-Through to AWS)

All requests forwarded to real AWS S3. Our server acts as a transparent proxy. The client talks to us; we talk to AWS on their behalf.

```
Client -> Our API -> AWS S3
```

Use case: traffic logging and analysis. The client's S3 operations work exactly as before, but every request passes through our call log. We see everything. We can audit, monitor, and analyse without the client knowing.

### Mode 3: Hybrid (Local + Proxy Fallback)

Implemented APIs are handled locally. Unimplemented APIs are proxied to AWS.

```
Client -> Our API -> Do we handle this? 
                         |
                         +--> YES: handle locally
                         +--> NO: proxy to AWS S3
```

Use case: development of the S3 server itself. We implement APIs one at a time. Everything we have not built yet still works because it falls through to AWS. The call log tells us what to build next.

### Mode 4: Selective (Local Storage with AWS Sync)

Requests handled locally, but data is also synced to AWS S3 (or vice versa). Our server is both a cache and a proxy.

```
Client -> Our API -> Local Storage
                         |
                         +--> Background sync to/from AWS S3
```

Use case: performance (local reads are fast), reliability (data survives if AWS is unavailable), migration (gradually move data from AWS to local or from local to AWS).

## Two Credential Modes

### Platform Credentials (Single AWS Identity)

Our server uses one set of AWS credentials for all S3 operations. The client does not provide AWS credentials. The client authenticates to our API (via API key, per the existing pattern), and our server uses its own AWS credentials for any operations that need real S3.

```
Client -> API key auth -> Our API -> (uses platform AWS credentials) -> AWS S3
```

Use case: managed service. The client does not need an AWS account. We handle the AWS side. We control costs, quotas, and access.

### Client Credentials (Relay)

The client provides their own AWS credentials (via the standard Authorization header with SigV4 signing). Our server relays those credentials to AWS. The client's AWS IAM permissions determine what they can do.

```
Client -> SigV4 signed request -> Our API -> (relays client credentials) -> AWS S3
```

Use case: enterprise deployment where the customer wants to use their own AWS account and their own IAM policies. Our server is a proxy/enhancement layer, not a credential holder.

### How SigV4 Signing Works in Both Modes

boto3 always signs requests with SigV4. Our server must handle this:

**Platform credentials mode:** Our server validates the signature using its own access key/secret key pair (which it issued to the client). The signature proves the client has the correct API key. We then make a new request to AWS (if proxying) with our own credentials.

**Client credentials mode:** Our server does NOT validate the signature. It relays the request (including the Authorization header) directly to AWS. AWS validates the signature. If the client's credentials are valid, the request succeeds.

## IAM Call Interception

boto3 does not just make S3 calls. Before any S3 operation, it may call:

- **STS GetCallerIdentity** (to verify credentials are valid)
- **STS AssumeRole** (for cross-account access)
- **S3 GetBucketLocation** (to determine the correct regional endpoint)

Our server must handle these. In platform credentials mode, we can return synthetic responses:

```python
# STS GetCallerIdentity
{
    "UserId": "sg-compute-user",
    "Account": "000000000000",
    "Arn": "arn:aws:iam::000000000000:user/sg-compute"
}
```

In client credentials mode, we proxy these to the real AWS STS endpoint.

The call log will reveal exactly which IAM calls boto3 makes. We implement them as we discover them.

## Implementation Strategy

### Phase 1: Call Log + Full Proxy

Start as a pure proxy. Every request is logged and forwarded to AWS. Nothing is handled locally yet. This gives us:

- A complete call log of real S3 usage patterns
- Confidence that the proxy is transparent (boto3 works without knowing)
- A baseline to measure local implementation against

### Phase 2: Core Operations Local

Implement the most-called APIs locally (based on call log data). Proxy everything else.

Expected order based on typical boto3 usage:
1. PutObject, GetObject, DeleteObject, HeadObject
2. ListObjectsV2, CreateBucket, ListBuckets
3. CopyObject, DeleteObjects (batch)
4. Presigned URL generation
5. Multipart upload

### Phase 3: Full Local

All commonly-used APIs handled locally. Proxy only for rare or advanced operations. The call log shows when we have reached coverage: if 99% of requests are handled locally, we are done.

### Phase 4: Storage Backends

With the API layer complete, plug in different storage backends:

| Backend | Implementation |
|---------|---------------|
| Disk | Files on local filesystem. Simplest. |
| Memory | In-memory store. Fastest. Ephemeral. |
| Vault | Objects stored as vault files. Encrypted, versioned. |
| Real S3 | Proxy/cache mode. Local reads, S3 writes. |

The API layer does not know or care which backend is active. Same pattern as Memory-FS.

## What This Enables

With a fully boto3-compatible S3 server:

- **SG/Send's Memory-FS** can point at our server instead of AWS S3. Zero code changes.
- **osbot-aws** S3 operations work against our server. Zero code changes.
- **SGit** can use our server for vault storage. Zero code changes.
- **Any AWS SDK client** in any language works without modification.
- **Air-gapped deployment** of the entire platform without AWS.
- **Cost visibility:** the call log shows exactly how much S3 usage each component generates.
- **Testing:** run integration tests against local S3 without AWS credentials or costs.
