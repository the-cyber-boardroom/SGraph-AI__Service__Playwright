# Serverless SG Workflows: Moving Execution Beyond The Laptop

**version** v0.27.43
**date** 15 May 2026
**from** Human (project lead)
**to** Strategy, Architect, Developer (lead), @Dev
**type** Strategy brief

---

## What This Is

A strategic framing for the next phase of SG/Compute: **applying the serverless principle not just via Lambda, but across our full substrate of EC2, Fargate, and our own container hosts.** The Lambda-first model has served us well; the next iteration says many of the workflows we currently run on developer laptops or in Lambda should run on managed-but-not-Lambda compute, and the CLI primitives we have been building make this newly easy.

The trigger is operational: Lambda is good for a class of workloads, but the costs are creeping (a recent £20 spent on a SnapStart misconfiguration was a useful signal), the performance is not always what we need, and most importantly **the deployment friction for Lambda is higher than we now need it to be**. With Fargate and our own EC2 container hosts coming online (today's Fargate experiment brief), we have substrates that we can push to **without packaging a Lambda function**, which changes the calculus.

## The Bigger Theme

The principle: **when an action could run somewhere other than my laptop, it probably should**. Not for performance reasons in every case, but for **logging, capture, reproducibility, and parallelism**. An action run on my laptop leaves no trace. An action run on a managed substrate leaves a vault commit, a log, a timestamp, and a measurable cost.

Today's example use case from the voice memo: "if I want to do a particular action and I want it to be logged and captured, let's implement it so that we can run that action not on my laptop but inside one of our EC2 containers or Fargate." This is the pattern. The laptop is the development environment; the substrate is where production work happens, even when "production" means a single ad-hoc invocation.

## Why Now

Three changes in the last two weeks make this possible:

1. **CLI-native AWS support**. We can now create EC2 instances, Lambda functions, Route 53 records, and (today) Fargate tasks from the CLI in a couple of seconds. The deployment friction has dropped substantially.
2. **The container path is real**. Yesterday's vault-collections-and-containerised-hosts brief, plus today's Fargate experiment, mean we have a credible container substrate, not just Lambda.
3. **The agent-CLI architecture (13 May briefs)**. Every workflow is already structured as CLI commands. A CLI command can run anywhere with the right environment. The hard work of designing the command structure is already done.

## The Substrate Spectrum

Different workloads want different substrates. The mature view treats this as a **spectrum**, not a binary:

| Substrate | Best For | Trade-offs |
|-----------|----------|-----------|
| **Laptop (current default for ad-hoc)** | Interactive development, exploration | No logging, no reproducibility, no parallelism |
| **Lambda** | Fast, short, stateless workloads with very low frequency | Cold-start tax, deployment friction, hidden costs (SnapStart, etc.) |
| **Fargate** | Container-shaped workloads with moderate duration | One-off startup cost; after that, fast and cheap |
| **EC2 with container hosts** | Many small workloads that can share an instance | Routing layer needed; warm pool gives near-zero idle |
| **Dedicated EC2** | Long-running, regulated, or isolation-sensitive workloads | Higher per-task cost; strongest isolation |

The hypothesis the voice memo opens: **for many workloads currently on the laptop or in Lambda, an EC2 container host or Fargate is the right substrate**. We have a hunch that the cost difference will be small or negative once the SnapStart-class hidden costs of Lambda are accounted for.

This is testable. The Fargate experiment brief (today) starts the measurement. This brief is the broader frame: Fargate is one of several substrates we want measured, not just an isolated experiment.

## What Workflows Migrate

A non-exhaustive list of workflows worth moving off the laptop and into a managed substrate:

| Workflow | Current Substrate | Proposed Substrate | Why |
|----------|-------------------|---------------------|-----|
| Routine vault operations (sync, snapshot, publish) | Laptop / Lambda | EC2 container host or Fargate | Capture, logging, reproducibility |
| Playwright-based automated tests | Laptop | EC2 container host (with Playwright base image cached) | Parallelism, multi-region, scheduling |
| Multi-region user simulation tests | Cannot do on laptop | EC2 container hosts per region | Genuine multi-region behaviour |
| Daily-band experiment production steps | Laptop | EC2 container host or Fargate | Captured in vault; no manual intervention |
| Infographic generation batches | Laptop | Fargate | Cost capture; queueing |
| Vault collection packaging | Laptop | Fargate or container host | Reproducibility; trigger from any source |
| Periodic vault health checks | Lambda (currently) | Possibly EC2 container host | Cost; richer environment |
| Agent CLI invocations from orchestration | Laptop / Lambda | EC2 container host | Long-running; rich filesystem; can hold the vault open |

The pattern is consistent: anything that benefits from being captured, parallelised, or reproducible is a candidate. Ad-hoc exploration stays on the laptop.

## The Operational Reality To Measure

The voice memo flags four dimensions worth understanding empirically before committing:

| Dimension | Question |
|-----------|----------|
| **Cost** | What does each substrate actually cost for a typical workload? Including hidden costs like SnapStart, data egress, instance idle overhead |
| **Usability** | How easy is it to push work to each substrate? What does the developer experience look like? |
| **Timings** | How long does each workload take on each substrate? Including the startup, the work itself, and the teardown |
| **Operational behaviour** | What happens when things go wrong? How easy is it to debug, retry, observe? |

These are the same dimensions from today's Fargate experiment, generalised to all substrates. The Fargate experiment is the first instance; this brief positions it as part of a broader measurement programme.

## The CLI Compounds

A key strategic point: **as we add CLI primitives, the workflows above become trivial to migrate**. Once `sg-compute container-host create` and `sg-compute fargate run-task` exist as solid primitives, every workflow that wants to use them becomes one extra CLI call away.

This is the agent-CLI architecture (13 May) paying off. Workflows are not big programs that need to be rewritten; they are compositions of CLI calls. Changing the substrate is a flag, not a rewrite. The investment in clean primitives compounds.

## Hypothesis: Lambda Is Not Where The Cost Sits

The voice memo names a specific hypothesis worth testing publicly: **for our actual workload, Lambda is not giving us a huge advantage**. The £20 SnapStart misconfiguration is the kind of hidden cost that adds up. The cost of running these workloads on Fargate or an EC2 container host might be similar or lower, and the developer experience clearly simpler (no Lambda packaging, no SnapStart cliff, no cold-start anxieties).

If the hypothesis holds, **Lambda becomes a more specialised tool** in our substrate kit: the wildcard DNS spinup function, occasional very-short triggers, glue between AWS services. The bulk of our workflow execution shifts to container substrates.

If the hypothesis does not hold, we learn something useful: Lambda is the right answer for our shape. Either result is actionable.

## What This Does Not Yet Commit To

Several deliberate non-commitments:

- **Not committing to abandon Lambda.** Lambda continues to serve specific cases (DNS spinup function, very-short triggers, low-volume glue work). The shift is in default substrate for new workflows.
- **Not committing to Kubernetes.** The container host primitive is "Kubernetes-shaped" in some ways but we are not building a full scheduler. The companion primitive brief explains the lighter-weight approach.
- **Not committing to a specific migration order.** That depends on measurements. The Fargate experiment is the first data point.
- **Not committing to a fixed substrate-per-workload mapping.** Different workloads might use different substrates. The point is to give workflows the choice.
