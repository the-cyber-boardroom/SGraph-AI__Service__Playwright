# Unified Observability Session: Logs And Stats From One Place

**version** v0.27.43
**date** 15 May 2026
**from** Human (project lead)
**to** Architect, Developer (lead), @Dev, DevOps
**type** Arch brief

## What This Is

A consolidation effort: **bring all our logging and observability sources into a single interactive session** where a developer can pull logs, query stats, and reason across sources without switching tools or contexts.

Right now we have logs landing in S3 (our own captures), logs in CloudWatch (Lambda invocations, ECS tasks, anything that uses CloudWatch Logs natively), audit trails in CloudTrail (IAM events, API calls), and an assortment of bespoke captures from our agents and CLIs. Each one has its own access pattern. Each one requires its own mental model. Each one creates friction when the question being asked spans multiple sources, which is increasingly often.

This brief proposes a **single observability surface** that knows about all of these sources and lets you query them as if they were one logical log stream, with appropriate routing of each query to the right backend.

This is companion to today's S3 native CLI support brief: the S3 primitive gives us fluent access to S3-stored logs; this brief is about combining that with the other sources into something more powerful than the sum.

## The Sources

Concretely, the sources we want unified:

| Source | What It Contains | Current Access |
|--------|------------------|----------------|
| **Our S3-backed logs** | SG/Compute task outputs, agent CLI invocations, vault operations, infographic generations, daily-band production steps | Manual S3 browsing; some bespoke scripts |
| **CloudWatch Logs** | Lambda function logs, ECS task logs, anything emitting to CloudWatch | AWS Console or `aws logs` CLI |
| **CloudWatch Metrics** | CPU, memory, network, custom metrics per service | AWS Console dashboards |
| **CloudTrail** | All AWS API calls, IAM events, who-did-what audit trail | AWS Console; `aws cloudtrail` CLI |
| **AWS Cost Explorer / Billing** | Per-service cost breakdowns, daily and per-tag spend | AWS Console |
| **Vault commit histories** | Per-vault audit logs (already the audit trail for vault operations, per the vaults-of-vaults brief) | Vault inspection tools |
| **Application-emitted logs** | Anything our applications log directly (currently mixed across stdout, files, S3) | Source-by-source |

Each one is useful. Together they tell the story of what happened across our platform. Currently the story is fragmented; this brief makes it whole.

## The Interactive Session Concept

The proposed UX is an **interactive session** the developer enters, with state and context that persist across queries. Sketch:

```
$ sg-compute logs

SG/Compute Observability Session
Connected sources: s3, cloudwatch, cloudtrail, billing, vaults

> sources
  s3          : 47 known prefixes
  cloudwatch  : 23 log groups
  cloudtrail  : connected
  billing     : connected (last sync: 2h ago)
  vaults      : 142 vaults indexed

> tail --source cloudwatch --group /aws/lambda/vault-spinup --since 1h
  [stream of recent Lambda invocations]

> query "errors" --since 24h
  [unified view across all sources, filtered for entries matching "errors"]

> stats --source billing --by-service --since 7d
  [cost breakdown by AWS service over the last week]

> agent-trace researcher --session abc123
  [pulls together the full trace of one agent invocation: agent log,
   underlying Bedrock calls, S3 reads, vault commits, costs]

> exit
```

The session keeps state: which sources are connected, recent queries, current time window, default filters. Queries are short because the session has context. Cross-source queries are first-class.

This is **a log interface, not a logging system**. The data stays where it is; the session reads from each source on demand. We are not building yet-another-logging-backend; we are building the tool that knows how to talk to the backends we already have.

## Why "Interactive Session"

The alternative would be a set of one-shot CLI commands (`sg-compute logs query "..."`). That is fine for scripting, but for the typical investigation flow it is wrong. Investigations are exploratory:

- Start broad, narrow down based on what you find
- Pivot to a different source when something interesting appears
- Adjust the time window mid-investigation
- Refer back to recent results without re-running queries

An interactive session matches that workflow naturally. The session is the conversation between the developer and the data; one-shot commands are how you script a known-answer query after the investigation is done.

We support both: the session for investigation, individual commands for scripting. Same code paths underneath.

## Cross-Source Queries

The most useful capability is queries that span sources. Examples:

| Question | Sources Touched |
|----------|----------------|
| What was happening when this vault deploy failed? | CloudWatch (Lambda spinup log), S3 (deploy task output), CloudTrail (EC2 RunInstances call), Vault (deploy command history) |
| How much did the infographic batch cost last week? | Billing (AWS spend), S3 (infographic generation outputs with token counts), Vault (credential manager usage logs) |
| Why is this agent running slow? | Vault (agent commit history), CloudWatch (any Lambda or ECS components), S3 (agent stdout / stderr captures), Bedrock observability (if a Bedrock agent) |
| Who modified this IAM role and when? | CloudTrail (IAM events), Vault (if changes went through our IAM lockdown workflow) |

Each one is currently 3-5 separate investigations across 3-5 separate tools. The unified session makes it one investigation.

## The Source Abstraction

Each source has the same interface, even though backends differ:

```
Source contract:
    - connect()         : establish connection / verify credentials
    - list()            : enumerate available log streams / metric streams
    - tail(stream)      : stream recent entries
    - query(stream, q)  : run a query (varies per source's capability)
    - stats(stream, agg): compute aggregations
    - schema(stream)    : describe what fields are available
```

For each source we implement this contract using the underlying API. The session orchestrates queries across sources by knowing each one's contract.

This is similar to the agent-CLI architecture from 13 May: a uniform contract over heterogeneous backends. Same pattern, applied to observability.

## The Storage Question

A natural follow-on: if we are reading from many sources, would it be useful to **also write a consolidated copy** to a single destination (probably S3) so historical queries are fast and cheap?

The answer is "probably yes, eventually, but not in v1." V1 reads from the sources directly. V2 considers consolidation if and only if the read patterns justify it. Most of the time, the source has indexes we can use; consolidation buys us little except for cross-source queries that we run repeatedly.

This is the same principle from the strategy brief today: do not build infrastructure to solve problems we have not yet measured.

## What "Done" Looks Like

A scenario: I notice that the daily-band production agent ran much slower last night than usual. The investigation flow:

1. `sg-compute logs` → enter session
2. `agent-trace band-producer --date 2026-05-14` → pulls the full trace
3. The trace shows the Bedrock call took 3x longer than typical
4. `bedrock-stats --model claude-opus-4.7 --since 24h` → shows Bedrock had elevated latency yesterday
5. `cost --service bedrock --since 7d` → confirms cost was within the normal range, just slower
6. Conclusion: external Bedrock issue, not our problem. Move on.

Currently that investigation is 5+ tool switches and probably 20+ minutes of clicking. With the session, it should be 5 commands and 2 minutes. The session pays off when investigations like this are routine, which they are.

## What This Does Not Try To Be

- **Not a real-time alerting system.** Alerts come from CloudWatch alarms, Bedrock observability, our own watchers. The session is for investigation, not detection.
- **Not a metrics visualisation tool.** We can summarise metrics, but for full-fidelity dashboards CloudWatch and Grafana exist.
- **Not a log retention system.** Each source has its own retention; we read from the sources as they are.
- **Not a SIEM.** Security-specific aggregation and correlation lives in a different (later) brief if we decide we need one.

These are deliberate scope limits to keep v1 focused.

## Open Questions

| Question | Notes |
|----------|-------|
| What is the session UX precisely? | TUI? REPL? Notebook-style? Probably REPL for v1, possibly with rich output rendering |
| How are queries expressed across sources? | SQL-like? Natural language? Our own DSL? Open; needs prototyping |
| Authentication model? | Reuses the credential manager / dynamic credential delivery patterns |
| How do we handle very large result sets? | Pagination; streaming; explicit `--limit` defaults |
| Should the session itself be loggable? | Yes; queries the user runs go into a vault location so investigations are reproducible |
| Multi-region support? | First-class; CloudWatch logs and CloudTrail events are region-scoped |
| Performance? | Source-dependent; we should surface query latency clearly |
| Plugin model? | Yes eventually, so new sources can be added without modifying core; v1 hardcodes the known sources |

## What This Asks For

1. **Validate the source list.** Are there sources we are missing? Are there sources on this list we should not bother with in v1?
2. **Pick the session UX.** Probably REPL-style for v1; needs a clear decision.
3. **Implement the source contract.** Six methods per source; consistent across all of them.
4. **Implement the session loop.** Stateful, persistent across queries, with sensible defaults.
5. **Implement the cross-source query model.** This is the value-add over individual tools; needs careful design.
6. **Add the first concrete cross-source feature.** Probably `agent-trace` because it spans many sources and is a frequent need.
7. **Iterate based on actual use.** This is the kind of tool that gets better the more it gets used; ship early and adjust.

Estimated effort: 1-2 weeks for the core session and three sources (S3, CloudWatch, CloudTrail). Add sources incrementally after that.
