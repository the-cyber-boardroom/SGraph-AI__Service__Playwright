# SG/Compute Container Hosts: Containers On Our Own EC2 Nodes

**date** 15 May 2026
**from** Human (project lead)

## What This Is

A new primitive in SG/Compute: **container hosts**, our own EC2 instances configured to run containers with node-pool-style behaviour. Workflows submit container tasks; the system places them on existing hosts where capacity exists, spins up new hosts when needed, and tears them down after a configurable idle window.

This is **Kubernetes-shaped without being Kubernetes**. The node-pool concept is borrowed; the implementation is much lighter (a Lambda placement function, a small per-host agent or REST endpoint, the existing wildcard DNS + spinup pattern). The goal is to get most of the value of cluster-style scheduling with a fraction of the operational surface.

The primitive serves two distinct workload classes that have been pulling us in the same direction: **vault hosting** (per the density brief from yesterday) and **ad-hoc workload execution** (per the serverless-SG-workflows strategy brief today). Both want containers; both want fast placement on existing capacity when available; both want auto-teardown when idle.

## Why This Primitive

Three forcing functions converge on this:

1. **The vault-hosting density brief** named "multi-container per EC2" as the default density mode. We need the primitive that runs containers on EC2 hosts to make that mode real.
2. **The serverless-SG-workflows strategy brief** (companion today) calls for managed substrates beyond Lambda. EC2 container hosts are the cheapest path to that capability for many workloads.
3. **The multi-region testing use case** (Playwright tests, simulated users across regions) needs a substrate we can fire up cheaply per region. Fargate is one option; our own container hosts are another, and likely cheaper for repeated use.

## The Workflow

The pattern in concrete terms:

```
User: sg-compute container run \
        --image sg-playwright:latest \
        --regions us-east-1,eu-west-1,ap-southeast-2 \
        --count-per-region 10 \
        --command "run-tests --suite checkout-flow"

CLI:  for each region:
        Is there a container host available with capacity?
          YES → REST-call the host: please run N containers with this image and command
          NO  → spin up a new container host (10-20 sec with warm pool)
                → wait until ready
                → REST-call the host

Host: pulls image if not cached locally
      runs N container instances in parallel
      reports back: stdout, stderr, exit codes, logs, durations

CLI:  collects results across regions
      writes them to a vault location
      reports back to user
```

Each container host is a single EC2 instance running Docker (or Podman). The host has a small REST API the CLI talks to. Containers within the host are isolated as usual; the host enforces a max-containers-per-host limit so performance stays predictable.

## The Primitive's CLI Surface

Following the established `sg-compute aws X` pattern:

```
sg-compute container-host create --region <r>          (provision a new host)
sg-compute container-host list                         (list active hosts)
sg-compute container-host describe <host-id>           (host capacity, running containers, idle timer)
sg-compute container-host stop <host-id>               (graceful shutdown)
sg-compute container-host metrics <host-id>            (resource utilisation)

sg-compute container run --image <img> --command <cmd>    (run one container, placement automatic)
sg-compute container run --image <img> --regions <list> --count-per-region <n>   (multi-region)
sg-compute container list                                  (list running containers across all hosts)
sg-compute container logs <container-id>                  (stream / fetch logs)
sg-compute container stop <container-id>                  (kill a running container)
```

The `container run` is the high-level primitive most workflows will use. It handles placement, host provisioning if needed, image pull, execution, log collection. The `container-host` commands are the lower-level controls for inspecting and managing the host fleet directly.

## The Idle Timeout Mechanic (Reset-On-Activity)

A small but important detail from the voice memo, worth being explicit about: **the idle timeout should reset on each new container start, not run as a fixed window from host creation**.

The intuition: if a host has just served a workload, it is likely to serve another shortly. Tearing it down 20 minutes after creation regardless of recent activity wastes capacity. Resetting the timer on each new container start keeps active hosts alive, lets idle hosts time out, and matches the burst pattern most workloads have.

Concretely:

| Configuration | Behaviour |
|---------------|-----------|
| `--idle-timeout 15m` | Host shuts down 15 minutes after the **last** container was started (not after creation) |
| Container start | Resets the timer |
| Container running | Does not affect the timer (a long-running container plus a fast idle timeout is intentional) |
| Manual `stop` | Immediate teardown |

This is also a useful pattern for **Fargate tasks and EC2 instances generally**. The "reset on activity" timeout is a feature worth adding across our ephemeral substrates, not just container hosts. Captured as a small cross-cutting feature in this brief; should be implemented consistently.

## The Multi-Region Use Case

The container host primitive makes a previously-painful workflow trivial: **multi-region parallel test execution**.

```
sg-compute container run \
    --image sg-playwright:latest \
    --regions us-east-1,eu-west-1,ap-southeast-2 \
    --count-per-region 10 \
    --command "run-tests --suite checkout-flow --user-locale auto"
```

Each region gets 10 Playwright containers running the test suite. Each container reports back. Results aggregate into a vault. The test runs from three real-world regions with real network conditions.

This is something that does not work cleanly on Lambda (size limits, startup characteristics) and is expensive to set up per-region in Fargate (less so once the Fargate experiment validates that path). Container hosts make it cheap and repeatable.

## The Capacity Model

A simple, explicit capacity model:

| Setting | Default | Notes |
|---------|---------|-------|
| **Max containers per host** | 10 | Per the voice memo's framing; configurable |
| **Container memory request** | Image-default | Optional override |
| **Container CPU request** | Image-default | Optional override |
| **Host instance type** | Configurable; default to a small burstable instance | Sized for the workload class |
| **Idle timeout** | 15 minutes from last activity | Per the reset-on-activity rule above |
| **Max hosts per region** | Configurable | Safety cap |

When a workload requests N containers and existing hosts have capacity for less than N, the placement logic spins up additional hosts to cover the gap. When demand drops, hosts time out and tear themselves down.

This is genuinely the Kubernetes node-pool pattern, but implemented in a Lambda placement function plus a small per-host REST endpoint. No control plane to operate. No scheduler. No CNI. Just a placement decision and a REST call.

## The REST Surface On Each Host

Each container host runs a small REST server with endpoints like:

```
GET  /capacity           → { running: N, max: M, idle_since: timestamp }
POST /run                → { image, command, env } → { container_id }
GET  /containers         → list of running containers
GET  /containers/{id}    → status of one container
GET  /containers/{id}/logs → stream logs
DELETE /containers/{id}  → stop a container
```

This is intentionally minimal. Authentication is via mutual TLS (the host has a cert from our internal CA; the CLI has a cert from the same CA). No fancy authorization model; the CLI has the cert, the CLI can do anything. We are not running a multi-tenant scheduler; the hosts belong to us.

The voice memo flagged that **we might not even need the AWS API for this** once a host is up. Just the REST surface. The AWS API is for the initial spinup (and to find IPs); after that, everything is REST. That is a clean separation.

## What This Replaces (Partially)

Several existing patterns get cleaner:

| Current Pattern | New Pattern |
|----------------|-------------|
| Run Playwright tests on the laptop | Run on container hosts in three regions in parallel |
| Run vault sync as a Lambda function | Run as a container task on an existing host |
| Run daily-band production steps via cron on a single instance | Run as scheduled container tasks across the fleet |
| One-off ad-hoc workloads on the laptop | Submit to the container fleet; get a logged, captured execution |
| Bespoke per-task EC2 spinup scripts | Submit a container task; placement is automatic |

The pattern is consistent: **the CLI submits a task; the system decides where to run it; results come back captured**.

## How This Plugs Into The Density Architecture

Yesterday's vault-hosting-density-and-scaling brief named three modes (dedicated EC2, multi-container per EC2, multi-vault per app). The container host primitive **is** the multi-container-per-EC2 mode. With this primitive in place:

- A vault deployment via `vault-sync deploy ephemeral <vault-id> --density shared-container` becomes a container-run on an existing or new container host
- Vault collections (yesterday's brief) mount as the container's data volume
- DNS and TLS plumbing (last week's milestone) routes traffic to the right container on the host

The pieces fit. Today's primitive is the missing connective tissue.

## What This Does Not Do

Deliberate non-features for v1:

- **No genuine multi-tenancy isolation between containers on the same host.** Containers are isolated as containers normally are; we do not run untrusted code from different security domains on the same host. Workloads that need hard isolation get dedicated EC2 (the existing density mode).
- **No cross-region migration of running containers.** A container that needs to move regions gets stopped in one region and restarted in another.
- **No persistent storage on the host.** Containers are ephemeral; persistent data goes to S3 or to a mounted vault collection.
- **No GPU support.** v1 is CPU-only. GPU container hosts can come later if a workload justifies them.
- **No fancy scheduling.** Placement is "first host with capacity, or spin up a new one." Bin-packing optimisation can come later.

## Open Questions

| Question | Notes |
|----------|-------|
| Docker or Podman? | Probably Podman for the rootless story; Docker is more familiar. Decide before implementation. |
| What is the right per-host base image? | Probably a small Linux AMI with Docker/Podman, our agent REST server, and our internal CA cert |
| How do we cache popular container images on hosts? | The voice memo's Playwright example wants this: pre-pull common images at host creation. Probably a config flag |
| How small can the host instance be? | The instance sizing experiments brief (companion) covers this |
| Where does the host agent live? | Probably a small Go or Rust binary baked into the AMI; restart on failure |
| How do we secure the REST surface? | mTLS with our internal CA; CLI fetches certs at startup |
| How do we handle host crashes mid-task? | Container task moves to a different host; partial results may be lost; surface to the user |

## What This Asks For

Implementation order:

1. **Bake a container host AMI** with Docker/Podman, the REST agent, and our internal CA cert
2. **Implement the host REST surface** (the seven endpoints above)
3. **Implement the placement Lambda** (find host with capacity; spin up if needed)
4. **Implement the CLI commands** for `container-host` and `container`
5. **Add the reset-on-activity idle timeout** consistently across substrates
6. **Run a real workload through it**: pick one (Playwright tests across regions is a good first target)
7. **Document the primitive** as part of SG/Compute

Estimated effort: 1-2 weeks for the AMI, REST agent, placement Lambda, and CLI. Real workloads start landing as soon as the basic placement works.
