## What This Is

A planning brief for adding **Bedrock-native CLI support** to SG/Compute, following the established pattern of `sg-compute aws ec2`, `sg-compute aws lambda`, `sg-compute aws dns`, and (today) `sg-compute aws fargate`. Bedrock is the next AWS surface worth absorbing into the CLI.

Bedrock has grown significantly. What used to be "managed access to a few foundation models" is now a broad agentic platform with multiple distinct capabilities (Runtime, Memory, Gateway, Browser, Code Interpreter, Identity, Policy, Observability, Evaluations, Payments). This brief plans how to expose those capabilities through the SG/Compute CLI in a way that hides Bedrock's messiness behind clean abstractions and connects naturally to our vault model.

This is a **planning brief**, not an implementation brief: it maps the surface, names the abstractions, and prepares the ground for a subsequent implementation brief that turns the plan into commands.

## Why Bedrock, Why Now

Three reasons:

1. **The pattern is paying off.** EC2, Lambda, Route 53, and Fargate (today) prove that CLI-native AWS support compounds. Each new surface added makes the others more useful. Bedrock fits the pattern.
2. **Bedrock has matured enough to be interesting.** AgentCore (Runtime, Browser, Code Interpreter, etc.) makes Bedrock more than "a model gateway." It is now an agentic compute platform in its own right, parallel to ours. Worth absorbing.
3. **The vault is the right grounding.** Most of Bedrock's capabilities benefit from persistent state: conversation history, agent memory, tool definitions, evaluation results. The vault is naturally suited to hold this. Adding Bedrock to the CLI without vault grounding misses the leverage.

## The Messiness Bedrock Brings

Worth naming what we are abstracting over. Today's Bedrock surface includes:

| Layer | What It Contains |
|-------|------------------|
| **Foundation Models** | Anthropic Claude family (Opus 4.7, Sonnet 4.6, Haiku 4.5), Amazon Nova (Lite, Pro, Micro, Premier), Meta Llama (3.x, 4 Scout/Maverick), Mistral, Cohere, AI21, Titan, DeepSeek, Qwen, and now OpenAI frontier models |
| **AgentCore Runtime** | Serverless agent hosting with session isolation, multi-modal support, MCP and A2A protocols |
| **AgentCore Memory** | Short-term and long-term agent memory |
| **AgentCore Gateway** | Turns APIs and Lambda functions into MCP tools |
| **AgentCore Browser** | Cloud-based browser runtime for agent web interactions |
| **AgentCore Code Interpreter** | Sandboxed code execution in Python, JavaScript, TypeScript |
| **AgentCore Identity** | Auth and permission delegation for agents |
| **AgentCore Policy** | Cedar-based fine-grained policy for agent actions |
| **AgentCore Observability** | Tracing, debugging, monitoring |
| **AgentCore Evaluations** | Batch evaluations, A/B tests, recommendation engine |
| **AgentCore Payments** | Coinbase / Stripe wallet integration for autonomous agent transactions |
| **Knowledge Bases** | Managed RAG over S3-backed content |
| **Guardrails** | Content filters, PII detection, topic restrictions |
| **Prompt Management** | Versioned prompts with rollback |
| **Multi-Agent Collaboration** | Supervisor + specialist agent coordination |
| **Managed Agents (powered by OpenAI)** | Limited preview, OpenAI harness on Bedrock |

This is a lot. A vanilla CLI wrapper for all of it would be unusable. Our value-add is the **abstraction layer** that picks the right defaults, the right groupings, and the right vault integration so that day-to-day work is fast and the underlying complexity is reachable when needed.

## The Command Surface

Following the agent-specific CLI architecture (13 May arch brief), Bedrock support should be a tree organised by purpose, not a flat mirror of AWS service names. Proposed top-level:

```
sg-compute aws bedrock
    ├── chat        (talk to a model: pure inference)
    │     ├── claude
    │     ├── nova
    │     ├── llama
    │     ├── openai
    │     ├── list-models
    │     └── ...
    ├── agent       (build, run, manage agents)
    │     ├── create
    │     ├── invoke
    │     ├── list
    │     ├── memory
    │     ├── tools
    │     └── ...
    ├── tool        (built-in agent tools as standalone CLI commands)
    │     ├── browser           (AgentCore Browser sessions)
    │     ├── code-interpreter  (AgentCore Code Interpreter sessions)
    │     └── ...
    ├── kb          (Knowledge Bases: RAG)
    │     ├── create
    │     ├── ingest
    │     ├── query
    │     └── ...
    ├── guardrail   (Guardrails: content filtering, PII, topic restrictions)
    │     ├── create
    │     ├── apply
    │     └── ...
    ├── eval        (AgentCore Evaluations: batch tests, A/B)
    │     ├── run
    │     ├── recommendations
    │     └── ...
    ├── observe     (Observability: tracing, debugging)
    │     ├── traces
    │     ├── debug
    │     └── ...
    └── meta
          ├── status
          ├── models
          └── pricing
```

Each leaf is a CLI command following the JSON-in / JSON-out contract from the 13 May agent-CLI architecture briefs. Each branch corresponds to one of Bedrock's distinct capabilities.

## The Chat Sub-Command (First Priority)

The simplest and most immediately useful entry point. Pattern:

```
sg-compute aws bedrock chat claude --prompt "What is X?" --model opus-4.7
sg-compute aws bedrock chat nova --prompt "Summarise this" --input file.md
sg-compute aws bedrock chat llama --prompt "Generate code for ..." --model llama4-maverick
sg-compute aws bedrock chat openai --prompt "..." --model gpt-something
sg-compute aws bedrock chat list-models                    # show what is available
sg-compute aws bedrock chat any --prompt "..." --provider anthropic
```

Each provider gets a sub-command. Each sub-command takes a `--model` flag with reasonable defaults if omitted. The `list-models` shows what is currently enabled in the account. A catch-all `chat any` accepts a provider name for cases where the user wants to script across providers without thinking about the sub-command structure.

Critical detail: the **model IDs change**. Anthropic's Bedrock IDs in particular have inference profile complications (Opus 4.7 only available via cross-region inference profiles in some regions; some models require ARN-versioned IDs; others need application inference profiles). The CLI hides this. The user says `claude`; the CLI picks the right ID for their region, account, and inference path.

## The Agent Sub-Command (Second Priority)

Agents are where Bedrock's value adds up. The agent sub-command exposes AgentCore Runtime plus the related services:

```
sg-compute aws bedrock agent create --name researcher --model claude-opus-4.7 \
    --tools "browser,code-interpreter" --memory short-and-long --vault /vault/agents/researcher
sg-compute aws bedrock agent invoke researcher --prompt "..." --session <id>
sg-compute aws bedrock agent list
sg-compute aws bedrock agent stop <session>
sg-compute aws bedrock agent memory list --agent researcher
sg-compute aws bedrock agent memory clear --agent researcher --scope short
sg-compute aws bedrock agent tools add --agent researcher --gateway-tool <tool-arn>
```

The agent is identified by name, stored locally in a vault location, and the CLI maintains the mapping between names and the underlying Bedrock ARNs. This is where the vault grounding pays off: the agent definition (name, model, tools, memory config, custom prompts) lives in our vault, not in AWS console state. Reproducibility, version control, sharing all come free.

## The Tool Sub-Command (Third Priority)

AgentCore Browser and Code Interpreter are useful as **standalone tools**, not just as agent capabilities. The CLI exposes them directly:

```
sg-compute aws bedrock tool browser session start
sg-compute aws bedrock tool browser session navigate <url>
sg-compute aws bedrock tool browser session click <selector>
sg-compute aws bedrock tool browser session screenshot

sg-compute aws bedrock tool code-interpreter session start --language python
sg-compute aws bedrock tool code-interpreter run --code "..." --files file.csv
sg-compute aws bedrock tool code-interpreter session stop
```

This means a script can invoke a browser session or code execution without spinning up an entire agent. Useful for one-off automation, testing, or as building blocks in our own CLI-first agent workflows.

## The Vault Grounding

Every Bedrock interaction should be vault-grounded by default. Concretely:

| Bedrock Output | Vault Destination |
|----------------|-------------------|
| `chat` response | Written to a session vault with prompt, response, tokens, cost |
| `agent` definition | Stored in a vault location; vault is the source of truth |
| `agent memory` | Persisted to the vault, mirrored to AgentCore Memory; vault is the audit layer |
| Tool session traces | Captured in the vault, organised by session ID |
| `kb` ingest source list | Tracked in the vault so we know what was ingested when |
| `eval` results | Stored in the vault with timestamp, model, parameters |
| Token usage and cost | Recorded in the vault for the cost-tracking workflow |

This means the vault becomes the **system of record for AI work**, with Bedrock as the execution layer. Two consequences:

1. **Reproducibility.** A vault can be replayed: rerun the same prompts, same agent definitions, same evaluations, get comparable results.
2. **Auditability.** Every interaction is captured with its provenance. The newsroom-style "how this was made" pattern (12 May) applies to AI work.

This is also how Bedrock plugs into the **pre-auth payments model** (yesterday's strategy brief). Cost tracking per Bedrock call goes through the same metering layer as the rest of the platform.

## Where The Credential Manager Fits

Yesterday's credential manager vault brief proposed Simple Tokens that resolve to API keys server-side. Bedrock fits this naturally:

- A user has a Simple Token (or pre-auth balance)
- The CLI uses the Simple Token to authenticate against the credential manager
- The credential manager has scoped Bedrock access (AWS IAM credentials or assumed roles)
- The Bedrock call happens server-side; the user never sees AWS credentials
- Cost is metered against the user's pre-auth balance

This is the multi-provider abstraction the credential manager brief mentioned as a future capability. Bedrock is the natural place to start because Bedrock itself is multi-model; adding multi-provider routing on top is incremental.

## The MCP Connection

Bedrock now ships an MCP Server (`AWS MCP Server`) and Agent Toolkit for AWS. AgentCore Gateway turns APIs and Lambda functions into MCP tools. Multi-Agent Collaboration uses A2A protocols.

Our agent-CLI architecture (13 May briefs) is parallel to this work. The two should interoperate cleanly. Specifically:

- Our CLIs can be exposed as MCP tools so Bedrock agents can invoke them
- Bedrock-hosted agents can be invoked as our CLI sub-commands (so an orchestration script can mix our agents and Bedrock agents)
- The vault can be the shared workspace both sets of agents operate on

This is worth a separate dev brief later. For now, the planning brief notes the interoperability is a design constraint, not an afterthought.

## What This Replaces

For our current AI work:

| Current Pattern | New Pattern |
|----------------|-------------|
| Call OpenAI directly with hand-managed API keys | Call Bedrock with vault-grounded outputs and credential-manager-mediated auth |
| Manage prompts in text files scattered across projects | Bedrock Prompt Management mirrored to vault; versioned |
| Hand-roll RAG with pgvector / similar | Bedrock Knowledge Bases for the cases where managed RAG is enough |
| Custom guardrail logic per integration | Bedrock Guardrails applied uniformly |
| Run agent loops in our own orchestration code | Optionally delegate to AgentCore Runtime where session isolation matters |

This is not a "rip and replace": our existing patterns continue working. Bedrock becomes another substrate in the toolbox. The CLI lets us reach for it cleanly when it is the right answer.

## What This Does Not Replace

Worth being explicit:

- **Our CLI-first agent architecture** stays. Our agents are CLI commands; Bedrock agents are an additional resource our CLI commands can use.
- **Our vault model** stays. Bedrock interactions write to vaults; vaults remain the source of truth.
- **Our pre-auth payments model** stays. Bedrock costs flow through the same metering.
- **Our hosting density modes** stay. Bedrock is a managed service we call; not a substrate we host.

Bedrock is a **capability we absorb**, not a replacement for what we have built.

## What This Asks For

Concrete next steps:

1. Validate the command tree above with the team (this brief is the starting point, not the final tree)
2. Pick the implementation order: chat → agent → tool → kb → guardrail → eval → observe (or adjust)
3. Decide on the vault grounding schema (folder structure, naming, what gets captured per call type)
4. Plan the credential-manager integration so Bedrock auth flows through Simple Tokens / pre-auth
5. Plan the MCP interop (separate dev brief later; design constraint for now)
6. Write the implementation dev brief once the plan is settled
7. Implement the chat sub-command first as the smallest useful unit
8. Iterate from there

This is roughly a 1-2 week effort for the chat + agent + tool surfaces. The rest of the tree follows as we use it.

## Open Questions For The Plan

| Question | Notes |
|----------|-------|
| Default region for Bedrock calls? | Probably us-east-1 or us-west-2; configurable per call |
| How to handle Bedrock inference profile complexity? | CLI hides it; library handles regional/profile selection |
| How to manage model availability checks? | `list-models` should reflect what is actually enabled, not the full Bedrock catalogue |
| Stream responses by default or buffer? | Stream for interactive; buffer for scripted/JSON-out |
| How to format JSON-out for streaming? | Probably NDJSON; consistent with our other CLIs |
| Where do tool sessions persist on long-running operations? | Vault, with session ID; aligned with AgentCore session model |
| Should the CLI use the AgentCore SDK or boto3 directly? | Probably boto3 for chat, AgentCore SDK for agent/tool work |
| How does our `agent-{role}` CLI relate to a Bedrock `agent`? | Distinct concepts; Bedrock agents are a tool our agents can use |
| When do we expose multi-agent collaboration? | After single-agent flows are solid; probably v2 |
| When do we expose AgentCore Payments? | After the credential-manager integration; the two should work together |
