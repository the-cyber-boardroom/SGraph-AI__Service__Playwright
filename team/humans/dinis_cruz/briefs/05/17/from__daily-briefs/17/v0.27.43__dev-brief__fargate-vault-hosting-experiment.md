# Fargate For Vault Hosting: Performance Experiment vs Lambda

**version** v0.27.43
**date** 15 May 2026
**from** Human (project lead)
**to** Developer (lead), @Dev, DevOps
**type** Dev brief

## What This Is

An experiment to test a specific hypothesis: **Fargate (containerised) hosting for the SG vault will outperform the current Lambda-based approach** for the workload shapes we care about, even after accounting for Fargate's one-off startup cost.

This brief also continues the workstream that has been quietly compounding for the past several days: adding CLI-native support on SG/Compute for AWS primitives. EC2 instances, Lambda functions, DNS records via Route 53 (yesterday's milestone), and now Fargate. The pattern works because it sits between what the CLI can do and what the web console offers: automation where automation pays off, manual control where it does not.

## The Hypothesis

The Lambda-based path is **slower than expected** at scale. Concretely:

- Each invocation makes a significant number of API calls
- Lambda functions scale up slowly when those API calls compound
- SnapStart helps but has hidden costs and does not eliminate the underlying issue
- Cold start mitigations exist but never feel right

The hypothesis is that **Fargate's one-off container startup cost is amortised quickly** by significantly better per-operation performance after the container is warm. For workloads that involve more than a handful of operations per session (most of ours), the trade-off favours Fargate.

This is testable. The result either confirms the hypothesis (and reshapes the deployment substrate accordingly) or refutes it (and we keep Lambda where it serves us well).

## Why Now

Three reasons this is the right week to run this:

1. **The AWS CLI primitives are in place.** We have already added build-to-test for EC2, then Lambda, then DNS. Adding Fargate fits the existing pattern. The investment compounds.
2. **The vault collections architecture (yesterday) maps naturally onto containers.** A vault collection zip mounted into a Fargate container is the same shape we already designed for. This experiment validates the design decision.
3. **The Kubernetes-comparison conversation is fresh.** We have just publicly articulated the per-tenant VM model. Fargate sits adjacent to that conversation; understanding its performance characteristics matters for the architectural narrative.

## What "Better" Means Here

Concretely, the experiment needs to measure:

| Metric | Lambda Baseline | Fargate Test | Notes |
|--------|----------------|--------------|-------|
| Cold start time | Current measured value | Container provisioning + first response | Lambda wins on simple cold start; question is what comes after |
| Time to clone a specific git repo | Current measured value | Container value | Includes repeated API calls; this is where Lambda is suspected to slow down |
| Time to clone a large repo | Current measured value | Container value | Larger workloads stress the difference |
| Time for a sequence of N vault operations | Current measured value | Container value | The realistic workload; the steady-state matters more than first-byte |
| Cost per operation | Lambda pricing | Fargate pricing | One-off vs amortised |
| Cost per session | Lambda total | Fargate total | The unit that matters for the daily band experiment and partner POCs |
| Concurrency behaviour | Lambda scale-out | Fargate scale-out | What happens when 10 vaults need to run at once |

Each metric gets a measurement on both paths. Each is recorded in the SG/Compute experiment log. The numbers decide, not the priors.

## The Experiment Design

The experiment has three phases:

### Phase 1: Add Fargate To The CLI

Following the same pattern as EC2, Lambda, and Route 53 in recent days:

- `sg-compute fargate create-cluster`
- `sg-compute fargate run-task --image <image> --vault <vault-id>`
- `sg-compute fargate list-tasks`
- `sg-compute fargate stop-task`
- `sg-compute fargate logs <task-id>`

This is the CLI-first agent architecture (13 May arch brief) applied to a new substrate. Each command JSON in, JSON out, debuggable, scriptable.

### Phase 2: Deploy SG Vault As A Container

The vault app already runs as a service. Containerising it requires:

1. A clean Dockerfile (or container image build path)
2. The vault app entry point as the container's CMD
3. An empty volume mount point for vault data (per the vault collections brief)
4. A workflow to load a vault collection zip into the container at startup
5. The same TLS / DNS plumbing pattern we already have for EC2 (adapted for Fargate's networking)

The container image bake aligns with the AMI bake work from yesterday. Same principle: most of the configuration is in the image, very little happens at runtime.

### Phase 3: Run The Comparisons

For each metric:

1. Run the operation N times on Lambda (current baseline)
2. Run the same operation N times on Fargate (new path)
3. Record numbers, variance, anomalies
4. Compute the cost per operation including any infrastructure overhead
5. Document the results in the SG/Compute experiment log
6. Decide: does Fargate win the workload, lose, or split?

The decisions are workload-shape-dependent. Fargate may win for "10 vault operations in one session" and lose for "single 200ms operation per session." Both results are useful.

## What This Validates Or Refutes

The hypothesis being tested is broader than Fargate-vs-Lambda. It is really: **for our actual workload shape, is the right substrate something that pays a one-off startup cost and then runs fast, or something that has very fast startup and slightly slower per-operation cost?**

If Fargate wins, the architecture leans toward containers as the default workload substrate (per the vault-collections + containerised-hosts arch brief from yesterday). Lambda continues to serve specific cases (the wildcard DNS spinup function, for example, where the work is trivial and the latency matters).

If Lambda wins, the architecture stays where it is. We learn that the Lambda overhead we are feeling is fixable through other means (better caching, fewer API calls, SnapStart with the kinks worked out).

Either result is actionable. The point is to **measure rather than assume**.

## The SnapStart Question

A specific sub-hypothesis worth recording: SnapStart on Lambda **looks** like a fix for the cold-start problem but has hidden costs. Worth probing during the experiment:

- The extra cost of maintaining the snapshot
- The behaviour on first invocation after long idle periods
- The compatibility with our specific Lambda dependencies
- The actual measured performance gain on our workloads (not the AWS marketing numbers)

If SnapStart is genuinely good enough, it raises the bar for Fargate to clear. If SnapStart has the hidden costs the hypothesis suggests, Fargate's case strengthens.

## How This Plugs Into The Density Architecture

Yesterday's vault-hosting-density-and-scaling brief named three modes (dedicated EC2, multi-container per EC2, multi-vault per app). Fargate sits naturally between modes:

- It is a **container substrate** (so closer to multi-container per EC2)
- It is **managed** by AWS rather than self-hosted on EC2 (so we do not run the routing layer)
- It can be **per-tenant** (one Fargate task per tenant) or **shared** (multiple tenants in one Fargate task with routing)

Fargate is not a replacement for the three modes; it is a fourth substrate option that may be the right default for ephemeral workloads. The Vault Synchronizer's `deploy` command would gain another flag value:

```
vault-sync deploy fargate <vault-id>
```

If the experiment validates the hypothesis.
