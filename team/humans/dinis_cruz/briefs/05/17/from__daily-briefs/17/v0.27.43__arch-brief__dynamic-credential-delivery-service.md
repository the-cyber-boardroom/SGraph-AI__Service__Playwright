# Dynamic Credential Delivery Service: Per-Action AssumeRole Credentials

**date** 15 May 2026
**from** Human (project lead)

## What This Is

A mechanism for delivering **temporary, narrowly-scoped AWS credentials** to a piece of code (CLI invocation, agent action, scheduled job) at the moment that code needs to perform a specific operation. Rather than running every SG command with broad standing credentials, each operation requests the privileges it needs, receives temporary credentials, performs the action, and the credentials expire.

This is the AWS STS AssumeRole pattern applied as a first-class platform feature. The pattern itself is standard AWS practice; what this brief adds is the **service layer** that makes it usable across all of our agent CLIs, SG commands, and scheduled workflows without each piece reinventing the wheel.

The companion brief (`v0.27.43__dev-brief__iam-graph-visualisation-and-lockdown.md`) addresses **what the scoped roles are**. This brief addresses **how the scoped credentials get into the hands of the right code at the right time.**

## The Pattern

Today, an SG command runs with a privileged account. The credentials are long-lived. The command's actual operation might only need three specific API calls on three specific resources, but the credentials it carries give it the keys to the kingdom.

The proposed pattern:

```
Code wants to perform an action
        ↓
Code requests credentials for that action (with scope)
        ↓
Credential service:
    - Validates the request (the code is authorised to ask for this scope)
    - Calls STS AssumeRole on the matching scoped role
    - Returns temporary credentials (15 min, 1 hour, or task-bounded)
        ↓
Code uses the credentials for the specific action
        ↓
Credentials expire automatically
```

The code never holds long-lived broad credentials. The scope is explicit per-action. Credentials expire whether the code remembers to discard them or not. Every assumption is logged.

This is **just-in-time, just-enough** access. It is what AWS recommends and what most production AWS environments aspire to. The reason it does not get implemented more often is friction: every piece of code has to implement the STS request flow, handle credential refresh, manage expiry, and be configured with the right role ARNs. The service layer eliminates that friction.

## The Service Layer

A local service (initially) that:

1. Holds the master credentials capable of calling STS AssumeRole
2. Knows the catalogue of scoped roles available
3. Answers credential requests from clients via a simple local API
4. Caches active credentials per (caller, scope) tuple while they are valid
5. Refreshes on expiry
6. Logs every assumption with caller, scope, role, timestamp

Clients (SG commands, agent CLIs, scheduled jobs) make a single local API call:

```bash
sg-creds get --scope "lambda:list" --role-hint sg-aws-lambda-list
```

The response includes the temporary credentials in the format the AWS SDK expects. The client uses them; they expire; next call goes through the service again.

This is the same pattern as `aws-vault` or `aws sso login` for developer credentials, but applied to **agent and service** credentials, with scopes that match the agent-CLI architecture.

## Why "Local Initially"

Two reasons to start with a local service:

1. **Validates the pattern with low risk.** A local service can hold the master credentials in well-protected local config. Mistakes are bounded to the local machine.
2. **Avoids a chicken-and-egg problem.** A remote credential service needs its own deployment, its own IAM scoping, its own credential to do its job. Starting local lets us iterate the API and the role catalogue before adding network and deployment complexity.

The pattern is the same whether the service is local or remote. The local version is a stepping stone.

## The Service-Driven Version (Later)

Once the local pattern works, the service moves to a deployed location:

- Runs on its own SG/Compute instance (or Fargate task, if the experiment validates that path)
- Holds its master credentials via instance role (not stored at rest in a way that can be exfiltrated)
- Exposes the same API as the local version
- Accessible only from authorised callers (the SG/Compute network, or via signed requests)
- Logs every assumption to a vault for audit

The clients do not change. The same `sg-creds get --scope X` call now hits the deployed service. The transition is invisible to client code.

This is the standard pattern for moving from "works on my machine" to "works as infrastructure." We have used it before for the vault app (local first, then EC2, then Fargate). Same shape applies here.

## What Defines A Scope

A scope is a named permission profile. Examples:

| Scope | What It Allows |
|-------|----------------|
| `lambda:list` | List Lambda functions in the account |
| `lambda:invoke:specific:fn-name` | Invoke one specific Lambda function |
| `route53:create-record:specific:hostzone` | Create DNS records in one specific hosted zone |
| `ec2:provision:t4g-nano:specific:vpc` | Launch t4g.nano instances in a specific VPC |
| `s3:read:specific:bucket-name` | Read objects from one bucket |
| `iam:read-only` | Read IAM state (for the IAM graph tool) |
| `iam:cleanup:specific:role-prefix` | Delete roles matching a specific prefix |

Each scope corresponds to a real IAM role with the matching policy. The scope catalogue is the authoritative list of "what kinds of permission can be requested."

The catalogue itself lives in a vault (per the vaults-of-vaults principle). It is version-controlled, reviewable, and auditable. Adding a new scope is a vault commit. Removing or tightening a scope is a vault commit. The history shows when each scope was created, by whom, with what justification.

## Authorisation Of The Request

A subtle but critical point: **the credential service decides who can request which scope**. Not every caller can request every scope. The caller's identity matters.

For the local version, the caller's identity is the OS user running the command. The local service config specifies which OS users can request which scopes.

For the deployed version, the caller's identity comes from signed requests (the calling code has its own credentials specifically for asking the credential service for other credentials). This sounds recursive but is not: the calling code's identity-credential is tightly scoped to "I am allowed to ask for these scopes" and nothing else. The downstream scoped credentials are what actually act on AWS.

This authorisation layer is what makes the service safe. Without it, anyone with access to the service could request any scope. With it, the principle of least privilege is enforced both at "what does this action do" and "who is allowed to do this action."

## Why This Matters For Agents

The agent-CLI architecture from 13 May said every operation is a CLI. The CLIs need to do real AWS work. Each CLI today runs with the credentials whatever process invoked it inherited.

With this service, each agent CLI gets its own scope:

- `agent-archivist` has scope `vault:write:specific:band-vault`
- `agent-musician` has scope `bedrock:invoke:specific:music-model`
- `agent-publisher` has scope `cloudfront:invalidate:specific:distribution`

When an agent CLI runs, it requests credentials for its scope, gets them, performs its action, and is done. If the agent CLI is compromised or behaves unexpectedly, the blast radius is exactly its scope. Nothing more.

This is the security argument from the platform-strategy validation brief (yesterday) made concrete. We claim our model has a smaller blast radius than Kubernetes. The credential service is one of the mechanisms that makes that claim true at the AWS-permissions level.

## The Self-Documenting Property

Every assumption logged means the system documents its own usage:

- Which scopes are actually requested
- Which scopes are requested but never used (candidates for removal)
- Which combinations of scopes co-occur (candidates for consolidation)
- Which time-of-day or rate patterns each scope follows
- Anomalies (a scope suddenly requested in much higher volume than usual)

This feeds back into the IAM graph tool from the companion brief. The credential service produces evidence; the IAM graph tool consumes evidence; the recommendations get sharper over time.

## What This Is Not

Worth being explicit about scope boundaries:

- **Not a replacement for AWS Secrets Manager** for application secrets (API keys to third parties, database credentials, etc.). Those have different requirements and lifecycles.
- **Not a replacement for AWS SSO** for human developer credentials. Humans should continue to use SSO; this service is for code.
- **Not a replacement for AWS IAM itself.** This service uses IAM; it does not redefine it.
- **Not yet a customer-facing capability.** This is internal infrastructure. Partner deployments might benefit from the same pattern eventually, but that is a later decision.

## Phasing

| Phase | What | Why First |
|-------|------|-----------|
| Phase 1 | Local credential service with 5-10 scoped roles | Validate the pattern with the highest-value SG commands |
| Phase 2 | Scope catalogue stored in a vault, version-controlled | Make the scopes auditable and reviewable |
| Phase 3 | Migrate SG commands to use scoped credentials | Eat our own dog food; lock down the most-used surface |
| Phase 4 | Migrate agent CLIs to use scoped credentials | Apply the pattern to the broader agent surface |
| Phase 5 | Deploy the credential service remotely | Move from "works on my machine" to "works as infrastructure" |
| Phase 6 | Audit dashboard reading from assumption logs | The evidence layer feeding the IAM graph tool |

Phase 1 ships in days. Phase 2-3 in a week or two. Phase 4-5 mature over the following weeks. Phase 6 follows after enough data has accumulated.
