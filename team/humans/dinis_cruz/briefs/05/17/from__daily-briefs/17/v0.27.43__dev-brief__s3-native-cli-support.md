## What This Is

The next AWS surface to absorb into the SG/Compute CLI: **S3**. Following the same pattern as EC2, Lambda, Route 53, Fargate (today), and the upcoming Bedrock integration. S3 has been an essential substrate for us from day one (vault storage, logs, exports, snapshots, AMI artefacts), but until now we have interacted with it through a mix of AWS CLI, console clicks, and bespoke scripts. Time to consolidate.

The trigger: we recently added a vim-from-CLI edit capability to SG/Compute that works really well. That capability plus S3 native support means I can edit an S3 object as if it were a local file. Combined with the existing vault tooling, this turns S3 from "a storage backend I poke at occasionally" into "a first-class workspace I can navigate from the CLI fluently."

## Why S3 Now

Three forcing functions converge:

1. **The vim-from-CLI edit capability just landed.** Editing a remote S3 file is now the same as editing a local file. The friction is gone; the primitive is begging to be used.
2. **We are already heavy users.** Vault data, logs, snapshots, AMI artefacts, exported reports, the Cyber Boardroom data, infographic outputs: much of our state already lives in S3. Treating it as something we have to leave the CLI to interact with is silly.
3. **The CLI-primitives pattern is paying off.** Each AWS surface added to the CLI makes the others more useful. S3 in particular is something almost every other workflow needs to touch; native support compounds across the existing primitives.

## The Command Surface

Following the established `sg-compute aws X` pattern:

```
sg-compute aws s3
    ├── ls                          (list buckets, or list within a path)
    ├── view <s3-path>              (view file contents; auto-handles common formats)
    ├── edit <s3-path>              (open in vim; save writes back to S3)
    ├── cat <s3-path>               (raw content to stdout)
    ├── tail <s3-path>              (last N lines; useful for logs)
    ├── head <s3-path>              (first N lines)
    ├── cp <src> <dst>              (copy; either side can be S3 or local)
    ├── mv <src> <dst>              (move; same as cp + delete)
    ├── rm <s3-path>                (delete; with confirmation)
    ├── sync <src> <dst>            (sync directory and bucket trees)
    ├── stat <s3-path>              (metadata: size, ETag, modified, encryption, etc.)
    ├── presign <s3-path>           (generate presigned URL with TTL)
    ├── search <bucket> <pattern>   (find keys matching pattern)
    ├── bucket-create <name>        (create bucket with sensible defaults)
    ├── bucket-list                 (list buckets)
    ├── bucket-stat <name>          (size, object count, costs)
    └── bucket-config <name>        (versioning, lifecycle, encryption settings)
```

Each command is a normal SG CLI: JSON in, JSON out for scripting; pretty output for interactive use; consistent flags and error handling.

The path format is `s3://bucket/key/path` or, where a bucket is implied or configured as default, just `key/path`. The CLI should handle both, with sensible defaults so common operations are short.

## The vim Integration

The recently-added vim capability gets first-class treatment. The flow:

```
sg-compute aws s3 edit s3://my-bucket/configs/agent.json
    ↓
    Downloads the file to a temporary local path
    ↓
    Opens it in vim (or $EDITOR)
    ↓
    On save and exit, uploads back to S3
    ↓
    Reports any conflict (if the ETag changed during the edit)
    ↓
    Cleans up the temp file
```

Conflict handling matters: if someone else edited the file while you were editing it, the upload should fail safely and offer a merge view, not silently overwrite. This is the same property git users expect.

A configuration flag controls whether unchanged files are still uploaded (default: skip), whether to use ETag-based conflict detection (default: yes), and whether to keep a local cached copy (default: no).

## Smart Format Handling

`view` and related commands should auto-detect common formats and render them sensibly:

| Format | Behaviour |
|--------|-----------|
| JSON | Pretty-print with syntax highlighting |
| YAML | Pretty-print with syntax highlighting |
| Markdown | Render as plain text with light formatting hints |
| CSV / TSV | Render as a column-aligned table |
| Plain text / logs | Raw display |
| Binary (images, PDFs, etc.) | Show metadata and offer to download or open externally |
| `.tar.gz`, `.zip` | Show contents listing without extracting |
| Encrypted files (`.encrypted`, with our vault encryption header) | Note the encryption state; offer to decrypt if a key is available |

This is the kind of small affordance that turns "I need to use a separate tool to look at this file" into "I just typed `s3 view` and got what I needed."

## Sync Behaviour

`sync` is the workhorse command for moving directories of files. The semantics match what rsync users expect:

- Compare source and destination, transfer only changed files
- Support `--delete` to remove destination files not in source (off by default)
- Support `--dry-run` to preview without changes
- Support `--exclude` patterns
- Support `--include` patterns (more selective than exclude)
- Support `--checksum` mode for content-based comparison (slower but more reliable)
- Support `--reverse` to swap source and destination semantics

This is also how we sync vaults to S3 and back, so the `sync` primitive becomes the foundation for many vault workflows.

## Integration With The Vault Model

S3 is not just storage; it is one of the substrates a vault can live on. The S3 CLI commands should know about our vault format:

| Vault Operation | S3 CLI Equivalent |
|----------------|-------------------|
| Open a vault stored in S3 | `s3 vault-open s3://bucket/vaults/<id>` |
| Sync a local vault to S3 | `s3 vault-sync ./vault-dir s3://bucket/vaults/<id>` |
| Diff a local vault against its S3 copy | `s3 vault-diff ./vault-dir s3://bucket/vaults/<id>` |
| List vaults in an S3 prefix | `s3 vault-ls s3://bucket/vaults/` |

These are thin wrappers around the basic primitives plus vault-format awareness. They make the common vault-to-S3 workflows one command instead of three.

## What This Replaces

A non-exhaustive list of current patterns this brief makes obsolete:

| Current Pattern | New Pattern |
|----------------|-------------|
| `aws s3 cp` with verbose flags | `sg-compute aws s3 cp` with sensible defaults |
| AWS Console clicks to browse buckets | `sg-compute aws s3 ls` |
| Downloading a file to inspect it | `sg-compute aws s3 view` |
| Downloading a file, editing locally, uploading back | `sg-compute aws s3 edit` |
| Bespoke scripts for daily-rotated log files | `sg-compute aws s3 tail` or `view` with format awareness |
| Manual generation of presigned URLs | `sg-compute aws s3 presign` |
| Hand-rolled bucket-stat scripts | `sg-compute aws s3 bucket-stat` |

This is **substrate consolidation**: many small pieces of friction removed at once.

## What This Does Not Try To Be

Deliberate non-features for v1:

- **Not a replacement for the AWS CLI in scripting contexts where someone already has a working `aws s3` command.** We are adding native support, not banning the AWS CLI.
- **Not a graphical browser.** S3 inspection happens at the CLI; if a graphical view is needed, the AWS Console exists.
- **Not a backup tool.** S3 backup workflows exist (lifecycle policies, versioning, cross-region replication); this brief does not duplicate them.
- **Not a multi-cloud abstraction.** This is S3-specific. A future abstraction layer over S3 + other object stores might come later, but starts here.

## Open Questions

| Question | Notes |
|----------|-------|
| Default bucket? | A configured default bucket so common commands do not need the full S3 URI |
| Credentials? | Use the same credential resolution as other SG AWS commands; ties into the credential manager and dynamic credential delivery briefs from today |
| Region defaults? | Use the bucket's region; configurable override per command |
| Concurrency limits? | For sync and bulk operations; sensible defaults with overrides |
| Streaming behaviour? | For very large files, stream rather than buffer; `cat`, `tail`, `cp` all need this |
| Compression handling? | Transparently handle `.gz`, `.zstd` files on view / cat / tail |
| Encryption? | Transparently handle our vault encryption format; surface encryption metadata in `stat` |

## What This Asks For

1. Implement the core command set (ls, view, edit, cat, tail, head, cp, mv, rm, stat)
2. Add the vim integration with ETag-based conflict detection
3. Add format-aware rendering for view / cat
4. Implement sync with the rsync-style flags
5. Add the bucket-level commands (create, list, stat, config)
6. Add the vault-aware wrappers (vault-open, vault-sync, vault-diff, vault-ls)
7. Wire into the credential manager / dynamic credential delivery patterns
8. Document the command tree alongside the other `sg-compute aws X` primitives
9. Identify the next workflow to migrate from "AWS CLI + scripts" to this primitive

Estimated effort: 3-5 days for the core command set, plus a few more for the vault wrappers and bucket-level commands. Most of the work is wrapping boto3 with sensible defaults and good error handling.
