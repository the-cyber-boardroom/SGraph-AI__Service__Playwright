# Instance Sizing And Startup Experiments: Finding The Right Shape For Ephemeral Workloads

## What This Is

A measurement programme to answer a class of questions we keep hand-waving: **which EC2 instance types are optimal for our ephemeral workloads, and what does the startup-time / sustained-cost / burst-behaviour trade-off actually look like?**

The voice memo phrased it well: "it could be that we can have a bigger instance that actually starts faster, which means we don't need to have it running for as long." This is the kind of non-obvious result that only shows up when you measure. We have not measured.

This brief is a focused experiment to produce the measurements, then use them to inform default instance choices for container hosts (today's companion brief), Fargate tasks (today's earlier brief), warm pool members (yesterday's brief), and the dedicated EC2 mode (yesterday's density brief).

## What We Do Not Know (And Should)

Concrete open questions:

| Question | Why It Matters |
|----------|---------------|
| What is the **smallest** instance that can host a container and run our typical workload? | Determines minimum cost per host |
| How does **startup time** vary with instance size? | Larger instances may start faster (less of a trade-off than expected) |
| How do **burstable** vs **non-burstable** instances behave under our patterns? | Our workloads are bursty by nature; burstable might be the natural fit |
| What is the **break-even** point between a smaller-cheaper-slower-startup instance and a larger-faster-startup one? | Determines the right shape per workload duration |
| Do **ARM-based (Graviton)** instances change the economics? | Possibly significantly cheaper for the same throughput |
| Are there **per-region** price differences that matter? | Multi-region workloads might benefit from per-region instance choices |
| What is the **cold-cache** penalty when a fresh instance pulls a container image? | Determines whether image caching strategy is worth it |
| How does **memory pressure** affect performance on small instances? | Affects how many containers we can pack |

These are all directly relevant to the substrate-spectrum decision in the strategy brief today. Without numbers, we are guessing.

## The Hypothesis Worth Testing

The non-obvious one from the voice memo: **a bigger instance that starts faster may be more cost-effective than a smaller instance running for longer**, even though per-hour pricing favours the smaller instance.

Intuition: if a `c7i.xlarge` boots in 8 seconds and a `t3.nano` boots in 30 seconds, and the workload takes 60 seconds, the bigger instance is alive for 68 seconds vs 90 seconds. The xlarge costs about 10x per second, but it lives 0.75x as long. Outcomes depend on the actual numbers, which we have to measure.

This is also relevant to **warm pool sizing decisions**. A warm pool of fast-starting larger instances may be cheaper than a warm pool of slow-starting smaller instances, because the warm pool can be smaller for the same response-time SLO.

## Experiment Design

### Phase 1: Startup Time Matrix

Measure startup time (from AWS API call to "instance ready to accept a connection") across a matrix of instance types:

| Instance Class | Sizes To Test |
|----------------|---------------|
| **Burstable (t-series)** | t3.nano, t3.micro, t3.small, t4g.nano, t4g.micro, t4g.small |
| **General purpose (m-series)** | m7i.large, m7i.xlarge, m8g.large |
| **Compute optimised (c-series)** | c7i.large, c7i.xlarge, c8g.large |
| **ARM (Graviton)** | c8g.medium, c8g.large, m8g.medium, m8g.large |

For each: 10 runs, recording (a) API-call-to-running-state, (b) running-state-to-SSH-ready, (c) SSH-ready-to-container-runtime-ready (via the host agent REST endpoint). Total: about 100-150 measurements; runnable in a single afternoon.

### Phase 2: Workload Performance

For each viable instance type from Phase 1, run a standardised workload:

| Workload | Approximates |
|----------|-------------|
| Spin up a single Playwright container; run a 30-second test | The most common test pattern |
| Spin up 5 Playwright containers; run them in parallel for 30 seconds | Multi-container load |
| Container image pull from ECR (cold cache) | First-time pull latency |
| Container image pull from ECR (warm cache) | Repeat pull latency |
| Sustained CPU load for 5 minutes | Burstable credit consumption pattern |

Measure: wall time, CPU utilisation, memory utilisation, network throughput, cost (computed from per-second pricing).

### Phase 3: Break-Even Analysis

Combine startup time and workload performance into a model:

```
total_cost(instance_type, workload_duration) =
    per_second_price(instance_type) *
    (startup_time(instance_type) + workload_duration)
```

Plot for varying workload durations. Identify the crossover points: "for workloads under N seconds, use instance X; for workloads over N seconds, use instance Y."

This is the actually-useful output of the experiment. The default-instance-per-workload decision falls out of the numbers.

### Phase 4: Burstable Credit Dynamics

A specific sub-experiment for burstable instances. The t-series have credit-based CPU bursting that complicates the simple model.

Measure:

- How long can a t3.nano sustain full CPU before credits run out?
- How does that affect a workload's wall time?
- Is "unlimited mode" worth the extra cost?
- For our pattern of bursty-then-idle workloads, do burstable instances replenish credits during idle?

The answer determines whether burstable instances are the natural fit for our pattern (likely yes for many cases) or whether non-burstable is more predictable.

## Why This Matters

Three reasons this measurement is worth the time:

1. **Default instance choices compound.** Every workflow we migrate to a managed substrate inherits the default. Getting it right early saves money and latency across hundreds of future invocations.
2. **The Graviton question is meaningful.** ARM-based instances may meaningfully change the economics for our typical workloads. Worth quantifying.
3. **The startup-time question is non-obvious.** The hypothesis that a larger faster-starting instance may beat a smaller slower-starting one is the kind of insight that only shows up with data. Worth proving or disproving.

## What This Does Not Do

Deliberate scope limits:

- **Not benchmarking AWS in general.** We are benchmarking specifically the instance types and workload patterns relevant to SG/Compute.
- **Not committing to one cloud provider's instance shapes.** The methodology applies to any provider; the specific numbers are AWS-only for now.
- **Not exploring spot instances.** That is a separate optimisation that compounds on top of any instance choice; worth a follow-up brief once the base measurements are in.
- **Not exploring custom AMI optimisations (e.g., pre-pulled images, init scripts).** Those are improvements we apply after we know the base numbers.

## What This Asks For

Concretely:

1. Set up the experiment infrastructure (a script that spins up instances of each type, measures, records, tears down)
2. Run Phase 1 (startup time matrix); should be a few hours of work and a few hours of measurement
3. Run Phase 2 (workload performance); a day of work
4. Run Phase 3 (break-even analysis); analysis day
5. Run Phase 4 (burstable credit dynamics); follow-up half-day
6. Publish the results in a vault
7. Update default instance choices in the container host primitive, warm pool, and Fargate-experiment briefs based on findings

Estimated effort: 3-4 days end to end, including the analysis. The output informs every substrate decision for months.
