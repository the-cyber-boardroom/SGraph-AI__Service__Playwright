## What This Is

The next step in our AWS-CLI-primitives workstream: bringing IAM into the same type-safe SG/Compute treatment we have applied to EC2, Lambda, Route 53, and (today) Fargate. But IAM is a different kind of problem from the others. The other AWS primitives are about **provisioning things**; IAM is about **understanding and tightening relationships**. A graph view is the right interface, not a CLI list.

Two parallel goals:

1. **Aggressively delete** roles and policies that are not needed, including the many that AWS adds by default and never removes
2. **Aggressively expand** the policies and roles we actually use, so each operation runs with the minimum privilege required

The current state: SG commands run with a privileged account because we have not yet bottomed out exactly which permissions each command needs. That is operationally fragile and a security weakness. The goal is to know, for every SG command, exactly which AWS permissions it needs at the moment of execution, and to grant exactly those, no more.

The companion brief (`v0.27.43__arch-brief__dynamic-credential-delivery-service.md`) addresses the **mechanism** for delivering those scoped credentials. This brief addresses the **understanding and structuring** of the underlying IAM landscape.

## Why Graph, Not List

IAM resources have inherent graph structure that flat lists hide:

- A user belongs to groups
- A group has policies attached
- A role can be assumed by users, services, or other roles
- A policy grants permissions on specific resources (which may match patterns)
- Trust relationships define who can assume what
- Permissions boundaries cap what other policies can grant

Flat AWS console views show one node at a time. The CLI commands show one resource per call. Reasoning about "what can this user actually do, transitively?" requires reconstructing the graph mentally every time.

A graph view collapses this work. Nodes are roles, users, policies, services. Edges are attachments, assumptions, trust relationships. Walking the graph answers questions like:

- Which services and roles can read this S3 bucket?
- What is the full transitive permission set for this role?
- Which roles are unused (no edges leading to them being assumed)?
- Where are there permission overlaps that suggest consolidation?
- Where are permissions broader than necessary for the resource patterns we actually have?

These questions are answerable today via cobbled-together IAM Access Analyzer outputs, CloudTrail queries, and manual SDK calls. None of that is fast, repeatable, or visually intuitive. A purpose-built graph view changes the workflow.

## What The Tool Looks Like

A new section in SG/Compute, working name **SG IAM Graph** (subject to renaming alongside the broader naming work from yesterday):

### Discovery

A discovery pass that pulls the current state of IAM in our AWS account:

- All users, groups, roles, policies (inline and managed)
- All trust relationships
- All permission boundaries
- AWS-default-created roles flagged separately from user-created
- Recent CloudTrail activity per role (which roles have been used recently)

The output is a structured representation written to a vault. The vault is the source of truth for the visualisation; it is also the audit trail.

### Visualisation

The vault renders as a graph view (using the SG/App framework). The view supports:

- **Filter by recency**: hide unused roles, show only active ones
- **Filter by source**: AWS-default vs user-created
- **Permission walking**: pick a node, see everything transitively reachable
- **Trust relationship view**: focus on who can assume what
- **Resource scope view**: for each policy, what specific resources it affects
- **Evidence overlay**: per role, the CloudTrail-derived list of recently-used actions

The visualisation is read-only. Actions (delete, modify, create) happen through the CLI; the graph reflects the current state after each change.

### Evidence-Driven Recommendations

The interesting layer. For each role, the tool computes:

- What permissions the role actually used in the last N days (from CloudTrail)
- What permissions the role has but did not use
- Suggested tighter policy that covers only the observed usage
- Confidence interval (a role active for 30 days with 200 distinct API calls is more confidently tightened than one used twice)

The operator reviews the recommendations. The graph view highlights where tightening is high-confidence and where it requires judgement.

This is the "what we want to do" pattern from the voice memo made concrete: **understand exactly what is needed, then grant exactly that.**

## The Cleanup Workflow

The first pass through the tool is mostly deletion. AWS environments accumulate roles over years:

- Roles created by services you tried once and abandoned (CodeBuild, CodePipeline, Glue, Step Functions, etc.)
- Roles created by CloudFormation templates that have been deleted
- Roles created automatically that nothing actually assumes
- Policies attached "just in case" that no role uses

The graph view surfaces these because they have no in-edges (nothing assumes them) and no recent CloudTrail activity (nothing is using them). The CLI provides bulk-delete commands scoped by the graph filter.

The cleanup commands:

```
sg-iam graph discover --output current.json
sg-iam graph filter --unused --no-cloudtrail-30days --output candidates.json
sg-iam graph delete --from candidates.json --dry-run
sg-iam graph delete --from candidates.json --confirm
```

Dry-run is the default. Confirmation is explicit. Each deletion is recorded in the vault for audit.

## The Expansion Workflow

The mirror of cleanup. Where roles are too privileged, the tool helps narrow them. The flow:

1. Pick a role currently using `*:*` or broad permissions
2. Pull CloudTrail history for that role
3. Generate a tighter policy covering only observed actions and resources
4. Save the tighter policy as a new version
5. Test by applying in a non-production context (or by simulating against historical CloudTrail)
6. Roll out when confident

The tool tracks the lockdown progress per role: how many roles are at `*`, how many are scoped to action prefixes, how many are scoped to specific actions, how many are scoped to specific actions on specific resources. This is a measurable security posture indicator over time.

## What Changes For SG Commands

Currently, SG commands run with a privileged account because they need to do many different things. The lockdown work has implications:

- Each SG command (or each command group) gets its own narrowly-scoped role
- The commands are migrated off the privileged account
- The companion dynamic credential delivery service hands the right scoped credentials to each command at execution time

This means **SG/Compute commands themselves become evidence in the IAM graph**. Each command's role is documented; each command's actual permissions are derived from its observed behaviour; each command's lockdown progress is tracked.

There is recursion here that is worth pointing out: the tool we are building to lock down IAM also locks down its own commands. That is the right shape.

## Why This Matters Now

Three reasons:

1. **The Lambda spinup function (from yesterday's DNS milestone) needs a tight role.** It currently has more permission than it should. The cleanup work makes this concrete.
2. **The Fargate experiment (today's other brief) will add new IAM surface.** Doing the cleanup before adding more is much easier than after.
3. **The platform-strategy validation work (yesterday's strategic analysis) named security as one of our differentiators against Kubernetes.** Internally we should be the strongest example of what tight IAM looks like.

The Komodor 2025 finding from yesterday's research brief was that misconfigurations are the #1 cause of breaches in production Kubernetes environments. The equivalent in AWS is over-privileged IAM. Both reduce when you can see the graph, walk the permissions, and tighten with evidence.

## What This Asks For

The phasing:

| Phase | What | Why First |
|-------|------|-----------|
| Phase 1 | IAM discovery pass + vault representation | Foundation; everything else reads this |
| Phase 2 | Graph visualisation in SG/App | Makes the data legible |
| Phase 3 | Cleanup workflow with dry-run + confirm | The biggest immediate win; safest place to start |
| Phase 4 | Evidence-driven recommendations from CloudTrail | The "what does this role actually use?" layer |
| Phase 5 | Expansion workflow: per-role policy tightening | The expansion mirror of cleanup |
| Phase 6 | SG commands migrated to scoped roles | Eat our own dog food; ties to the credential service brief |

Phase 1 + Phase 3 alone produce a meaningful security-posture improvement within days. Phases 4-6 mature over weeks.
