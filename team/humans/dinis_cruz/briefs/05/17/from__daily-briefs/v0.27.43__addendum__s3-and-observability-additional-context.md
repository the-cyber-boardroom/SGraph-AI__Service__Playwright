## What This Is

The first memo transcript was incomplete; a fuller version surfaces several pieces of context that should now apply to both briefs and that change the framing in meaningful ways.

Rather than re-issue the two briefs, this addendum captures what was missing and notes how it extends what is already in place. Six additions, summarised:

1. The **concrete real-world pipelines** that ground the unified observability work
2. **AWS dashboard screenshots** as a capability worth integrating
3. The **strategic framing**: real users are coming, and we need real visibility before they arrive

5. The connection to **SG billing** (the billing we give customers, not AWS billing)
6. The **methodology reference**: the billing primitive worked really well; follow the same pattern

## Addition 1: The Concrete Pipelines

The unified observability brief named the sources abstractly. The fuller memo names two specific real pipelines that are already running and feeding S3:

| Pipeline | What Lands In S3 |
|----------|------------------|
| **CloudFront → Firehose → S3** | Traffic logs from the CloudFront edge. Every request, every cache hit / miss, every origin pull. |
| **CloudTrail → S3** | All AWS API calls, who-did-what audit trail across the account. |

Both of these are **already producing data**. We have a Firehose set up that pulls CloudFront logs and dumps them into an S3 bucket; we have CloudTrail writing to S3 as standard practice. The observability session is not creating new data sources, it is consuming data sources that exist and currently are not being looked at.

This matters for sequencing: **the data we need is already on the floor.** The bottleneck is the tool to read it, not the pipeline to capture it. The S3 native CLI gives us the read capability; the unified observability session gives us the query capability over it. Together they unlock data that is currently invisible.

## Addition 2: AWS Dashboard Screenshots

The fuller memo flagged something we have not built before: **AWS offers a screenshot / dashboard-export capability** for CloudWatch dashboards and similar surfaces. Pulling those into the observability session as part of an investigation report is a real capability worth integrating.

The use case: when we run an investigation across sources and want to capture "what the dashboards looked like at the moment we noticed the issue," we can fetch the AWS-rendered images and attach them to the investigation record in the vault. This is the newsroom-style "show your work" pattern applied to operations.

Concretely, the observability session should gain:

```
> dashboard fetch <dashboard-name> --time 2026-05-14T03:00:00Z
> dashboard list
> dashboard screenshot <dashboard-name> --output /tmp/dash.png
```

This is a small addition to the source contract but a useful one. Add to the v1 scope if cheap, or v2 if not.

## Addition 3: The Strategic Framing - Real Users Coming

The most important piece missing from the first transcript. Paraphrased: **real users and real customers are coming. Real billing and real tracking come with them. We need to visualise exactly what is going on on the website, and we do not have that today.**

This changes the urgency profile of the observability work. Building this tooling is not a nice-to-have; it is a precondition for safely operating with real customers. Without it:

- We will not know when something is broken until a customer tells us
- We will not be able to answer "is this user doing what they should be?" or "is this user being charged correctly?"
- We will not be able to demonstrate to customers what they are using
- We will not be able to debug customer-reported issues

Each one of these is a blocker for the customer-facing posture. The observability session is on the critical path for the CV/portfolio launch, partner POCs becoming production, and any paying-user workflows.

This means the observability work and the S3 native CLI are not just "nice consolidation," they are **operational readiness work** for the customer-facing chapter of the project. Worth treating accordingly.

## Addition 4: The Product Analytics Layer

Distinct from infrastructure observability is **product analytics**: what users are doing on the platform, in plain English. Concrete questions the fuller memo named:

- How many people are cloning vaults?
- How many vault downloads are happening, and which vaults?
- Who is signing up, who is converting to paid, who is churning?
- What features are being used by whom?
- How does engagement vary across the user base?
- What does the funnel look like?

These are **product questions, not platform questions**. They sit on top of the infrastructure observability rather than alongside it. The infrastructure observability tells us "the platform is healthy"; the product analytics tells us "users are getting value."

The fuller memo treats these as part of the same workstream, and that is broadly right: the **substrate** (S3 native access, source aggregation, unified queries) is the same. The **questions** are different. The right framing:

| Layer | Question Class | Primary Sources |
|-------|----------------|-----------------|
| **Infrastructure observability** | Is the platform healthy? Are workloads behaving correctly? | CloudWatch, CloudTrail, our S3 logs |
| **Product analytics** | What are users doing? What is converting? What is engaging? | CloudFront logs, vault commit histories, application emissions, SG billing |
| **Customer-facing reporting** | What should we show this specific customer about their usage? | Product analytics, filtered to one customer |

V1 of the unified observability session is layer 1. Layer 2 builds on the same substrate but exposes different queries. Layer 3 is a customer-facing surface that uses layer 2 as its data source.

This is worth a separate brief once the substrate is solid. For now, the addendum captures the shape so the substrate work serves all three layers cleanly.

## Addition 5: SG Billing Connection

A specific kind of product analytics: **the billing we give our customers, not AWS billing**. The fuller memo separates these explicitly and rightly so:

- **AWS billing** is what we pay AWS. Already integrated via Cost Explorer.
- **SG billing** is what our customers pay us. Per the pre-auth + micropayments brief from yesterday, this is the substrate for all paid features.

The observability layer needs to handle both. AWS billing answers "what does this customer cost us?"; SG billing answers "what should we charge this customer?" The two together let us compute margin per customer, which matters for pricing decisions.

Concretely:

| Question | AWS Billing | SG Billing |
|----------|-------------|------------|
| What did we spend on EC2 last week? | Yes | No |
| What did Customer X cost us? | Tagged costs in AWS billing | No |
| What did we charge Customer X? | No | Yes |
| What is our margin on Customer X? | Both | Both |
| Per-feature unit economics? | Both, plus product analytics | Both, plus product analytics |

These all feed into the same observability substrate. The pre-auth payments substrate from yesterday emits the SG billing data; the observability session consumes it.

## Addition 6: The Methodology - Billing Primitive As Reference

The fuller memo references the billing primitive as a success pattern: **it worked really well**. The approach was first-pass-of-raw-data, mapping primitives first then combining.

The observability work should follow the same pattern explicitly:

1. **Map out the primitives** (read from each source individually; one source contract per backend)
2. **Map out the combined view** (cross-source queries; the session as the orchestrator)
3. **Connect the dots within SG/Send** (product analytics; how the platform's own state shows up in observability)
4. **Connect the dots with billing** (SG billing emission; AWS billing import)
5. **Add the customer-facing layer** (filtered views per customer)

This is the same shape as the billing primitive's growth path. Worth being explicit so the team can see the pattern and execute against it.

## What This Surfaces As New Work

Beyond the two existing briefs, this addendum surfaces work that should become its own brief once the substrate is in place:

1. **Product Analytics Layer**: a brief specifically about the user-behaviour questions (vault clones, downloads, conversions, funnels). Builds on the observability substrate; distinct queries and possibly distinct UI.
2. **SG Billing Emission**: how the pre-auth payments substrate emits billing events that flow into observability. Probably already implicit in the payments brief, but worth being explicit.
3. **Per-Customer Reporting**: a customer-facing view of their own usage and billing. The first product surface that depends on the observability substrate.

Each of these is a separate piece of work; this addendum names them so they do not get lost.

## What This Does Not Change

The methodology in the two existing briefs stays the same. The substrate spectrum, the source contract, the interactive session UX, the S3 command surface: all unchanged. The additions are contextual and prioritising, not architectural.

Likewise the deferrals stand: V1 is the substrate plus layer 1 (infrastructure observability). Layers 2 and 3 follow. The product analytics work and customer-facing reporting are real work items but downstream of the substrate.
