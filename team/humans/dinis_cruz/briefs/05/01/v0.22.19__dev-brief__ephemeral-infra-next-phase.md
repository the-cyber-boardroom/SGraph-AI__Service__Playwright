## Context

The first round of the ephemeral infrastructure UI is working: instance creation, start/stop, basic frontend with sg-layout, vault integration for state, API and CLI parity. This brief covers the next set of features needed to take it from "working MVP" to "simulated AWS Marketplace."

## Feature 1: Split Creation from Live Instances

The current UI combines instance creation and running instance management in one view. These need to be separate sections because they serve different purposes and need different real estate.

### Creation Section

Focused on provisioning: what can I start, how do I configure it, what AMIs are available?

Three creation modes, each with different speed and flexibility:

| Mode | What It Does | Speed | When to Use |
|------|-------------|-------|-------------|
| **Create instance** | Starts a fresh EC2 instance, downloads and configures everything at boot | ~2 minutes (Elastic example) | First time setup, or when AMI is not available |
| **Bake AMI** | Takes a running, configured instance and creates an AMI from it | ~5-10 minutes | After configuring an instance to your liking, save it for fast re-use |
| **Create from AMI** | Launches a new instance from a pre-baked AMI | ~80 seconds (Elastic example) | Production path, fastest boot |

The difference is significant: Elastic from scratch takes ~2 minutes. Elastic from AMI takes ~80 seconds. The 40-second saving matters when the whole model is "spin up, use, destroy."

The CLI and API already support these three modes. The UI needs to expose them clearly:

```
+-----------------------------------------------------------+
|  Create Instance                                           |
+-----------------------------------------------------------+
|                                                           |
|  Instance Type: [Elastic ▾]                               |
|                                                           |
|  Creation Mode:                                           |
|  ( ) Fresh build (~2 min)                                 |
|  (●) From AMI (~80 sec)     AMI: [ami-xxx (22 Apr) ▾]   |
|                                                           |
|  Size: [t3.small ▾]    Timeout: [10 min ▾]              |
|                                                           |
|  [Launch]                                                 |
|                                                           |
|  AMI Management:                                          |
|  [Bake AMI from running instance ▾]                      |
|                                                           |
+-----------------------------------------------------------+
```

## Feature 2: AMI Management

AMIs need their own management UI within the Creation section:

- List available AMIs per instance type (with creation date, source instance, size)
- Bake a new AMI from a running instance (one click, progress indicator)
- Delete old AMIs (to avoid accumulating stale images)
- Set a default AMI per instance type (used when "Create from AMI" is selected)

This is the beginning of the "simulated AWS Marketplace" concept: the AMI library is the catalogue of available environments. Each AMI is a pre-configured, ready-to-launch product.

## Feature 3: SG/Send Vault Instance

Add a new instance type: the SG/Send vault server, running as an ephemeral EC2 instance.

This is significant because it provides storage within the ephemeral infrastructure. Currently, all instances are compute-only. Adding a vault server instance means:

- Agents and other instances can read/write to vaults on the ephemeral server
- The vault server is independent of the main sgraph.ai production server
- Data isolation: ephemeral workflows use ephemeral storage, not production vaults
- The vault server can be pre-loaded with specific vault snapshots (per the 21 Apr ephemeral vault brief)

The vault instance should be launchable like any other instance type:

```
Instance Type: [SG/Send Vault Server ▾]
Creation Mode: (●) From AMI
[Launch]
```

Once running, other instances in the same network can be pointed at it for vault operations. This is the "storage primitive" completing the platform alongside compute.

## Feature 4: Docker Container Management Inside Instances

Some instance types (Playwright, VNC, Linux) need to start Docker containers inside the EC2 instance. The UI needs to support:

- List running containers on an instance
- Start a new container (from a catalogue of available images)
- Stop/restart containers
- View container logs
- Expose container ports (add to security group or route through the instance's FastAPI)

For Playwright specifically: the ability to start multiple Playwright containers on one instance (for parallel browser automation sessions). Each container gets its own port.

## Feature 5: Remote Shell

Three approaches, in order of priority:

### Option A: API-Based Shell (via Instance FastAPI)

Each instance runs a FastAPI container. Add shell execution endpoints:

```
POST /shell/execute   { "command": "ls -la" }  -> { "stdout": "...", "stderr": "...", "exit_code": 0 }
POST /shell/stream    { "command": "tail -f /var/log/syslog" }  -> WebSocket stream
```

This is the simplest approach. The UI renders a terminal that sends commands to the FastAPI and displays output. No SSH, no SSM, no additional infrastructure.

Limitation: not a true interactive terminal (no tab completion, no curses-based apps). But for most admin tasks (checking logs, running commands, inspecting files), it is sufficient.

### Option B: SSM from Lambda

AWS Systems Manager Session Manager invoked from the Lambda function. This gives a true interactive terminal without opening SSH ports.

Research question: can a Lambda function invoke SSM StartSession and proxy the WebSocket connection to the user's browser? If yes, this is the cleanest solution. If not, Option A is the fallback.

### Option C: SSH Over WebSocket

A WebSocket-to-SSH bridge running on the instance. The UI connects via WebSocket, the bridge proxies to localhost SSH. More complex to set up but gives a full interactive terminal.

For the MVP, start with Option A (API-based shell). It works today with what we have.

## Feature 6: Prometheus Endpoints

Add Prometheus-compatible metrics endpoints to each instance's FastAPI:

```
GET /metrics    -> Prometheus-format metrics
```

Expose: CPU usage, memory usage, disk usage, network I/O, container count, container status, uptime, request count.

This enables:

- The observability pipeline (22 Apr brief) to scrape metrics from running instances
- The Elastic dashboards to show real-time instance health
- Future Grafana integration if needed

Even without a Prometheus server, the metrics endpoint is useful: the UI can poll it and display instance health inline.

## Feature 7: Stacks (Multi-Instance Bundles)

The concept: instead of starting instances one at a time, define a "stack" that bundles multiple instances together.

```json
{
  "name": "full-observability",
  "description": "Elastic + Playwright + Vault Server",
  "instances": [
    { "type": "elastic", "from_ami": true },
    { "type": "playwright", "from_ami": true },
    { "type": "sg-vault", "from_ami": true }
  ],
  "networking": {
    "all_instances_same_vpc": true,
    "vault_server_accessible_from_all": true
  }
}
```

One click: "Launch full-observability stack." Three instances start. They can communicate with each other. The vault server provides storage. Playwright provides browser automation. Elastic provides visualisation.

This is a simplified CloudFormation. No YAML templating. No dependency graphs. Just "these instances should exist together and talk to each other."

For the MVP, stacks are just a JSON config that triggers multiple instance launches. Networking is handled by putting all instances in the same security group. No orchestration complexity.
