## What Changed

Podman now works alongside Docker as a container runtime for our ephemeral EC2 instances. An experiment confirmed it runs cleanly. This means we have two runtimes today (Docker, Podman) and a third on the horizon (Kubernetes/minikube for multi-container orchestration at scale).

The key architectural decision: **plugins and workloads must not know which runtime they are running on.** A Playwright container, a Firefox browser, an Elastic instance, an SGit server: none of them should care whether they are on Docker, Podman, or eventually Kubernetes. They speak "containers." The runtime is an implementation detail underneath.

## The Container Runtime Abstraction

```
Plugins / Workloads
    |
    v
Container Abstraction Layer (unified API)
    |
    +--> Docker runtime
    +--> Podman runtime
    +--> Kubernetes runtime (future)
```

The abstraction layer exposes a common interface:

```python
# The plugin calls this. It does not know what runtime is underneath.
runtime = get_container_runtime()   # returns Docker, Podman, or K8s adapter

container = await runtime.start(
    image="sgraph/playwright-service:latest",
    ports={"8000": 8000},
    env={"API_KEY": "random-key-here"},
    name="playwright-1"
)

status = await runtime.status("playwright-1")
logs = await runtime.logs("playwright-1", tail=100)
await runtime.stop("playwright-1")
await runtime.remove("playwright-1")

containers = await runtime.list()   # equivalent of docker ps / podman ps
```

Each runtime adapter (Docker, Podman, Kubernetes) implements this interface using its native API. The caller never imports Docker or Podman directly.

## The Host FastAPI Control Plane

Every EC2 instance needs a FastAPI service that acts as the control plane for containers running on that instance. This is the "bastion API": the only container that runs with privileged access to the host OS.

### What It Exposes

```
# Container management
GET    /containers                    -> List all containers (ps)
POST   /containers                    -> Start a new container
GET    /containers/{name}             -> Container details (status, ports, uptime)
GET    /containers/{name}/logs        -> Container logs (tail, stream)
POST   /containers/{name}/stop        -> Stop a container
POST   /containers/{name}/restart     -> Restart a container
DELETE /containers/{name}             -> Remove a container

# Host information
GET    /host/status                   -> CPU, memory, disk, network
GET    /host/runtime                  -> Which runtime (docker/podman), version

# Shell access
POST   /host/shell/execute            -> Execute a command on the host
WS     /host/shell/stream             -> WebSocket interactive shell

# Metrics
GET    /metrics                       -> Prometheus-format metrics
```

### Implementation Base

Build this on the existing `serverless_fast_api` module (osbot-fast-api-serverless). This means:

- The same FastAPI service is 100% compatible with Lambda deployment (even though Docker/Podman will not be available in Lambda, the API structure and auth work identically)
- API key authentication out of the box
- Class-based route organisation (consistent with all other services)
- Can be tested locally without an EC2 instance

The API key for each instance is generated at boot and stored in the vault for that session (per the plugin architecture brief).

### Privilege Model

The host FastAPI is the only container that needs privileged access:

| Container | Privileged | Why |
|-----------|-----------|-----|
| Host FastAPI (control plane) | **Yes** | Needs to talk to Docker/Podman socket, access host OS for shell, read system metrics |
| Playwright | No | Runs inside its own container, talks to the web |
| Firefox | No | Runs inside its own container |
| Elastic/Kibana | No | Runs inside its own container |
| Any plugin workload | No | Plugins never need host access |

This is the security boundary: only one container has privilege. Everything else is isolated.

## Two Types of Workloads

### Long-Running Containers

Services that start and stay running for the duration of the session: Playwright, Firefox, Elastic, Kibana, VNC, SGit server. These are started, used repeatedly, and stopped when the session ends.

### Ephemeral / Single-Use Containers

Containers that spin up, execute one task, and are destroyed: run an SGit command, execute a QA test, run a data transformation, execute a script. These are fire-and-forget.

```
# Long-running
POST /containers { image: "sgraph/playwright-service", mode: "long-running" }
# Container stays running, accessible via its port

# Ephemeral
POST /containers/run { image: "sgraph/sgit-runner", command: "sgit clone <key>", mode: "ephemeral" }
# Container starts, runs command, returns output, is destroyed
```

The ephemeral mode is powerful for QA: spin up a container, run a test sequence that sends traffic to the platform, capture the output, destroy the container. No persistent state, no cleanup needed.

## Use Cases to Build First

### 1. Start Nginx (Proof of Pattern)

The simplest possible test: start an Nginx container, verify it is accessible, view logs, stop it. This proves the control plane API works end-to-end without any SG/Send-specific logic.

### 2. Start a FastAPI Service (Plugin Testing)

Start a custom FastAPI container with dynamic code loading (per the 18 Apr layered container brief). This proves the "load code at runtime" pattern works through the control plane.

### 3. Run an SGit Command (Ephemeral Mode)

Spin up a container with sgit-ai installed, execute `sgit clone <key>`, capture the output, destroy the container. This proves the ephemeral workload pattern.

### 4. QA Traffic Generation

Spin up a Playwright container, execute a test sequence against the live platform, capture results, destroy. This is the QA traffic generation from the 19 Apr brief, now executable through the control plane API.

## Sidecar Patterns

Three sidecar types that can be attached to any workload container:

### MITM Proxy Sidecar

Already working with Playwright and Firefox. Route a container's traffic through the proxy for full capture and control.

### Remote Browser Sidecar

Neko or VNC container that provides a streamable desktop view into the workload container. Solves the "I want to see what is happening inside this container" problem.

### Desktop Streaming Sidecar

Full desktop environment (Xfce or similar) with VNC/Neko streaming. For use cases where the workload is a GUI application, not just a browser.

Each sidecar is a plugin. It can be attached to any container through the control plane API:

```
POST /containers/{name}/sidecars
{ type: "mitm-proxy" }

POST /containers/{name}/sidecars
{ type: "remote-browser", provider: "neko" }
```

## Plugin Isolation (Reinforced)

The plugin architecture from the 28 Apr brief applies here too. Container-type plugins (Playwright, Firefox, Elastic, etc.) and sidecar plugins (MITM, Neko, desktop) should:

- Live in their own folders
- Have their own config, routes, CLI commands, tests
- Be loadable from separate repos eventually
- Never touch the core container runtime abstraction or the host FastAPI control plane

Adding a new container type or a new sidecar should be: create a plugin folder, define the config, implement the routes, enable in settings. No changes to core.
