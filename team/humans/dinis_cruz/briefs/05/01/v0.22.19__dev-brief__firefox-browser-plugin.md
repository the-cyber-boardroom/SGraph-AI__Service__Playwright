## What We Have

A well-maintained community Docker container providing a fully configured Firefox browser with:

- Login page (configurable username/password)
- MITM proxy connected and working
- Clean, functional setup out of the box

This is a concrete example of the plugin architecture from yesterday's brief: a self-contained instance type with its own specific needs, its own UI panel, and its own configuration options that do not apply to any other instance type.

## Why Firefox Gets Its Own Plugin (Not Just "Another Browser")

Firefox with MITM proxy is a different product from Playwright with MITM proxy:

| | Playwright | Firefox Browser |
|---|---|---|
| Interaction model | Programmatic (API-driven, no human at the keyboard) | Interactive (human browsing, mouse and keyboard) |
| Primary use | Automation, screenshots, sequences | Browsing, testing, debugging, secure web access |
| MITM proxy role | Traffic capture and audit | Traffic capture AND interception/modification |
| Session type | Stateless (spin up, execute, tear down) | Stateful (user browses, accumulates history, bookmarks) |
| Authentication | API key on the service | Username/password on the Firefox login page |

These differences mean the UI panel for Firefox needs different controls, different settings, and different information displayed than a Playwright panel.

## Configuration Options (Firefox-Specific)

| Setting | What It Does | Stored In |
|---------|-------------|-----------|
| Username / password | Login credentials for the Firefox container's auth page | Vault (per instance session) |
| MITM intercept script | Python script that mitmproxy executes on each request (modify headers, inject content, block domains) | Vault (uploadable via UI) |
| Self-signed certificate settings | Whether to accept self-signed certs, whether to intercept SSL | Container environment variables |
| Start page / homepage | URL that Firefox opens on launch | Container config |
| Firefox profile | Pre-configured bookmarks, extensions, preferences | Loadable from vault or AMI |

## AMI Support (Same as Other Plugins)

The Firefox plugin follows the same three-mode creation pattern from the ephemeral infra next-phase brief:

- **Create fresh:** Start Firefox container from base image. Configure manually. ~60-90 seconds.
- **Bake AMI:** Save current configured instance (with Firefox profile, MITM scripts, settings) as an AMI.
- **Create from AMI:** Launch a pre-configured Firefox environment in ~60 seconds.

The AMI bake is particularly valuable for Firefox because users will want to save configured profiles: specific MITM intercept scripts for testing a particular application, specific bookmarks and extensions for a particular workflow, specific security settings for a particular compliance requirement.

## Vault Storage (What Gets Persisted)

Per-session data stored in the vault for this instance:

| Data | Purpose |
|------|---------|
| Session credentials (username, password) | Restore access on reconnect |
| MITM intercept script(s) | Persist custom proxy scripts across sessions |
| MITM proxy logs | Traffic capture for the session |
| Firefox profile (if saved) | Bookmarks, extensions, preferences |
| Screenshots | If the user takes screenshots during the session |
| Instance metadata | Type, AMI used, start time, configuration applied |

## Health Checks

The Firefox plugin should display health status:

- Container running (Docker health check)
- Firefox process alive (process check inside container)
- MITM proxy connected (proxy health endpoint)
- Network connectivity (can the browser reach the internet through the proxy)
- Login page accessible (HTTP check on the auth endpoint)

Display as a simple status indicator (green/amber/red) in the panel, with drill-down to individual check results.

## Inline Viewing

The browser view in the right side of the panel should render the Firefox session inline. Two approaches:

**If the container supports iframe embedding:** Render directly in an iframe. Simplest, best performance.

**If iframe is blocked:** Use the remote browser component (Neko or noVNC) to stream the Firefox desktop. This adds a layer but works universally.

Given that this is a Docker container we control, we should be able to configure it to allow iframe embedding (remove X-Frame-Options headers). So the iframe approach should work.

|---|-----------|-------------|
