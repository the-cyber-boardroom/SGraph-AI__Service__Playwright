# Ephemeral Observability: Elastic and Kibana on Demand

**date** 21 Apr 2026
**from** Human (project lead)
**to** DevOps (lead), Developer, Architect
**type** Dev brief

## The Shift

We are decoupling data capture from data visualisation. Logs and metrics are captured continuously and stored in vaults or S3. The visualisation infrastructure (Elastic + Kibana) spins up only when you need it, loads the data, lets you analyse it, and shuts down. You do not pay for dashboards nobody is looking at.

This is the ephemeral infrastructure principle applied to observability.

## Why Not Run Elastic 24/7

Three reasons:

**Cost.** Elastic/Kibana on a dedicated instance runs 24/7 whether anyone is looking at dashboards or not. For our usage pattern (periodic analysis, not continuous monitoring), this is paying for 24 hours of compute to use 1-2 hours.

**Security.** A permanent Elastic instance is another attack surface to maintain, patch, and secure. An ephemeral instance that exists for 30 minutes and is destroyed is inherently more secure.

**AWS OpenSearch is too slow.** The managed OpenSearch service takes approximately 15 minutes to spin up. That is 14 minutes too long. A self-managed Elastic + Kibana on an EC2 instance (or better, from a pre-baked AMI) should start in under 2 minutes.

## The Architecture

### Phase 1: Ephemeral EC2 with Elastic + Kibana

```
CLI command: "start observability"
    |
    v
Launch EC2 instance (from pre-baked AMI)
    |
    +--> Elastic + Kibana already installed and configured
    +--> Pre-configured dashboards, index patterns, visualisations
    +--> FastAPI admin service for data management
    |
    v
Instance ready (target: under 2 minutes)
    |
    v
Import data from vault or S3
    |
    v
Analyse, visualise, explore in Kibana
    |
    v
Export: save dashboards, snapshots, screenshots
    |
    v
CLI command: "stop observability"
    |
    v
Instance terminated. No persistent compute.
```

### Two Launch Modes

**From scratch (slower, full flexibility):** Launch a base EC2 instance, install Elastic + Kibana, configure everything at boot. This takes 5-10 minutes but allows maximum customisation.

**From AMI (faster, pre-configured):** Bake an AMI with Elastic + Kibana already installed, dashboards pre-configured, unnecessary features stripped out. Launch from AMI. Target: under 2 minutes to a working dashboard. This is the production path.

The AMI approach also lets us strip Elastic down to only what we need. Remove unused plugins, disable features we do not use, pre-configure security settings. A lighter Elastic starts faster and uses fewer resources.

### Phase 2: Admin API

Extend the existing observability CLI with an API layer on the ephemeral instance:

```
POST /admin/data/import           -> Import data from vault or S3 path
POST /admin/data/export           -> Export current Elastic data to vault or S3
POST /admin/backup/create         -> Create Elastic snapshot
POST /admin/backup/restore        -> Restore from snapshot
GET  /admin/status                -> Instance health, data loaded, uptime
POST /admin/dashboards/screenshot -> Take screenshot of a specific dashboard
```

The import/export capability is critical: it means the data survives the instance. Import data, analyse it, export the results (or just the Elastic snapshot), destroy the instance. Next time, restore from snapshot instead of re-importing. Restore is faster than re-indexing.

### Phase 3: Vault Integration

This is where it gets powerful. Instead of loading data from S3 directly, load from vaults:

```
"Start observability, import data from vault X"
    |
    v
Instance starts
    |
    +--> sgit shallow-clone vault X (only fetch the log files needed)
    +--> Import log files into Elastic
    +--> Analyse in Kibana
    |
    v
"Export results to vault Y"
    |
    +--> Export dashboards, screenshots, analysis as files
    +--> Commit to vault Y
    +--> Push
```

Benefits of vault-backed data:

- **Access control:** The ephemeral instance only gets access to the vault it needs. No broad S3 permissions. The vault key is the credential.
- **Audit trail:** Every data import and export is a vault operation with a commit history.
- **Encryption:** Log data in vaults is encrypted at rest. The ephemeral instance decrypts at import, processes in memory, and the decrypted data is destroyed with the instance.
- **Stress test:** Large vaults with log data test the vault infrastructure at scale. Shallow clones mean we only fetch the files we need.
- **Scoped credentials:** No need to give the Elastic instance AWS credentials for broad S3 access. Give it a vault key. It accesses exactly what it needs.

## Data Flow: Capture vs Visualise

```
Continuous (always running):
    Lambda functions, SG/Send server, Playwright service
        |
        +--> Generate logs
        +--> Store in vault (raw logs vault) or S3
        |
    No visualisation infrastructure running.
    Logs accumulate. Cost: storage only.

On demand (when needed):
    "Start observability"
        |
        +--> Ephemeral Elastic + Kibana
        +--> Import from raw logs vault
        +--> Analyse, visualise, explore
        +--> Export results to analysis vault
        +--> Screenshot dashboards (via Playwright)
        +--> "Stop observability"
        |
    Instance destroyed. Cost: compute for session duration only.
```

The critical insight: **do not use Elastic as a data source.** Elastic is a visualisation and analysis engine. The data lives in vaults or S3. Elastic loads it when needed and the indexed data is destroyed with the instance (or snapshotted for fast restore next time).

## Dashboard Screenshots via Playwright

This closes a loop from the Playwright service briefs. Once Kibana is running, we can use the Playwright container to take automated screenshots of specific dashboards. These screenshots can be:

- Saved to a vault (as part of a daily observability report)
- Included in automated briefings
- Compared over time (visual regression for metrics)
- Shared via SG/Send tokens

The daily agent workflow becomes: start observability, import latest logs, take dashboard screenshots, save to vault, stop observability. The dashboard screenshots persist in the vault even after the Elastic instance is gone.

## Using Our Infrastructure to Monitor Our Infrastructure

This is meta but real: the ephemeral Elastic instance loads SG/Send server logs from a vault, analyses them in Kibana, exports the analysis to another vault, and uses the Playwright service to screenshot the dashboards. Every tool in the pipeline is our own tool.

This is both dogfooding and stress testing. Any problems we find while monitoring ourselves are problems our customers would hit.
