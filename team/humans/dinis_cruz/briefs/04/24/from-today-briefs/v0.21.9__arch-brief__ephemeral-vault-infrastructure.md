# Ephemeral Vault Infrastructure: Isolated, Versioned, Monitored

## The Problem

We are starting to have genuinely confidential data managed through vaults: website assets, agent configurations, deployment credentials, customer review materials. This data is encrypted, but the attack surface is not just the encryption. It is also about who can access the vault server, which network paths exist to it, and how much noise surrounds the access patterns.

Using the main shared SG/Send server for everything means internal sensitive vaults sit alongside public traffic. An attacker (or a misconfiguration) on the shared server has a path to internal vaults. The access patterns for internal vaults are drowned in the noise of public traffic, making anomaly detection nearly impossible.

We need to separate sensitive vault deployments into isolated, ephemeral, monitored infrastructure.

## The Core Principle

**Reduce the attack surface by reducing the infrastructure lifetime and the number of consumers.**

A vault used by the ambassador to manage the website should only be accessible by two consumers: the Claude agent session that edits it, and the CI pipeline that deploys from it. Not the public internet. Not other agents. Not other vaults. Two consumers, one ephemeral server, aggressive timeouts.

If you reduce the consumers to two and the server lifetime to 30 minutes, anomaly detection becomes trivial: any access that is not one of those two consumers is suspicious. Compare that to detecting anomalies on a shared server with hundreds of users and thousands of requests per hour.

## The Architecture: Ephemeral Compute + Vault Snapshots

### The Vault Snapshot

All vault data on the server side is just files in a folder. You can zip the entire folder and that zip IS the vault backup. No database dumps, no export tools. Just a zip of the bare directory.

The zip contains only encrypted ciphertext. It is useless without the vault keys. You could store it on S3, commit it to Git, or even put it on a public server, and the data remains secure. But we still restrict access because defence in depth.

### The Workflow

```
Step 1: Agent session starts (e.g. Ambassador updating the website)
    |
    v
Step 2: Spin up ephemeral compute (EC2 instance or Docker container)
    |
    +--> Boot time: under 1 minute
    +--> Load vault snapshot (zip) from versioned storage (S3 bucket)
    +--> Extract to disk
    +--> Start SG/Send server pointing at the extracted folder
    |
    v
Step 3: Agent works (makes changes to vaults, updates files, etc.)
    |
    +--> All vault operations happen on the ephemeral instance
    +--> Instance has aggressive timeout (5-10 minutes idle = destroy)
    |
    v
Step 4: Agent finishes
    |
    +--> Save: zip the vault folder
    +--> Version it: store as v0.7.17 (or dated snapshot) in S3
    +--> Optionally: add a manifest file with metadata (who, when, what changed)
    +--> Destroy the compute instance
    |
    v
Step 5: CI pipeline picks up the new snapshot
    |
    +--> Same pattern: spin up ephemeral compute
    +--> Load the latest vault snapshot
    +--> Run QA, tests, validation
    +--> Deploy to production
    +--> Destroy the compute instance
```

### Version Control of Vault Snapshots

Each save produces a versioned zip. This is version control of version control: the individual vaults inside have their own commit history (via SGit), but the snapshot captures the state of ALL vaults at a point in time.

This solves the "multiple vaults changed at once" problem. If the ambassador updates five vaults during a session (website content, images, configuration, navigation, metadata), the snapshot captures the combined state. Rolling back means loading a previous snapshot, which restores all five vaults to their prior state atomically.

```
snapshots/
  v0.7.15.zip    # Tuesday morning
  v0.7.16.zip    # Tuesday afternoon (ambassador session)
  v0.7.17.zip    # Wednesday morning (CI pipeline deployed)
  manifest.json  # { "v0.7.17": { "author": "ambassador", "date": "...", "changes": "..." } }
```

### API for Upload/Download

The ephemeral server should expose admin API endpoints for vault snapshot management:

```
POST /admin/vault-snapshot/upload    -> Upload a zip, extract, serve
GET  /admin/vault-snapshot/download  -> Zip current state, return
GET  /admin/vault-snapshot/status    -> Current snapshot version, loaded from, last modified
```

This makes the snapshot lifecycle API-driven, not SSH-driven.

## The Cost Question

Maintaining permanent infrastructure has visible costs (EC2 instances, EBS volumes) and hidden costs (patching, monitoring, configuration drift, security reviews). Ephemeral infrastructure has compute costs (per-minute EC2 billing) but zero maintenance costs between sessions.

For a vault that is accessed for 30 minutes per day by one agent and one CI pipeline, the ephemeral model is almost certainly cheaper. A t3.micro running for 30 minutes costs fractions of a cent. Maintaining that same t3.micro 24/7 costs orders of magnitude more, plus all the maintenance overhead.

The break-even question: at what usage level does permanent infrastructure become cheaper than ephemeral? For internal vaults with low access frequency, ephemeral wins convincingly.

## Security Properties

| Property | Shared Server | Ephemeral Instance |
|----------|--------------|-------------------|
| Attack surface | Large (public traffic + internal) | Minimal (2 consumers) |
| Anomaly detection | Hard (drowned in noise) | Trivial (any unexpected access is suspicious) |
| Data at rest | Encrypted but persistent on server | Encrypted, exists only during session |
| Network exposure | Public internet | Can be VPC-only, no public endpoint |
| Credential exposure | Server credentials persistent | Credentials loaded at boot, destroyed at shutdown |
| Lateral movement | Attacker on server can probe other vaults | Attacker on instance sees only one vault set |
| Patching | Must patch running infrastructure | Fresh image every session (always latest) |

## Future: PGP Signing

Eventually, vault snapshots should be PGP-signed. The ambassador's private key signs the snapshot on save. The CI pipeline's public key verifies before deployment. This ensures that only authorised agents can produce deployable snapshots, and any tampering between save and deploy is detectable.

This is not needed for the MVP but should be the target architecture. The PKI infrastructure for this already exists in the SG/Send design (per-branch PKI, signed commits).

## Where This Applies First

**The website vault.** The ambassador manages website content via vaults. The CI pipeline deploys from those vaults. This is the simplest case: two consumers, low frequency, high sensitivity (the vault controls what goes on the public website).

**Internal collaboration vaults.** Vaults used for agent-to-agent communication during development sessions. These should not live on the public server. Spin up ephemeral compute, collaborate, save the snapshot, destroy.

**Customer review vaults.** When delivering a security review or assessment via vault, the review data should be on isolated infrastructure, not co-located with public file transfers.

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | Vault folder zippable into a single snapshot file | Zip the server's vault directory, verify it contains all vault data |
| 2 | Snapshot loadable on a fresh EC2 instance in under 1 minute | Start instance, upload zip, extract, server running with vault data |
| 3 | Ephemeral instance destroyed after session with no persistent state | Terminate instance, verify no EBS volumes or data remain |
| 4 | Vault snapshots versioned in S3 with manifest metadata | Multiple snapshots in bucket, manifest tracks author, date, version |
| 5 | Rollback works by loading a previous snapshot | Load v0.7.15 instead of v0.7.17, verify vaults match the earlier state |
| 6 | Admin API for snapshot upload/download functional | POST upload, GET download, GET status all return expected results |
| 7 | Aggressive timeout destroys instance after 5-10 minutes idle | Leave instance idle, verify auto-termination |
| 8 | Access restricted to two consumers (agent session + CI pipeline) | Security group or VPC config limits access, verify no other paths |
| 9 | Anomalous access detectable (any request not from known consumers flagged) | Generate unexpected request, verify it is logged and alertable |
| 10 | Cost comparison documented (ephemeral vs permanent for this use case) | Analysis showing per-session cost vs 24/7 cost |
