## Context

The Playwright service has a growing API with full session management, sequence execution, and detailed browser control. That is powerful but heavy for the most common use case: take a screenshot of a URL.

This brief adds a simplified "screenshot" API surface that covers 80% of use cases with minimal parameters. Keep it short and snappy. The full session-based API remains for complex workflows.

## The Screenshot Endpoint

### Single Screenshot

```
POST /screenshot
{
  "url": "https://example.com",
  "click": "#accept-cookies",          // optional: element to click after load
  "javascript": "window.scrollTo(0, 500)", // optional: JS to execute after load
  "full_page": true,                    // optional: full page or viewport only
  "format": "png"                       // optional: png or html
}
```

Returns: the screenshot image (PNG) or the rendered HTML.

That is it. URL in, image out. No session management, no start/close, no sequence definition. One call, one result.

Every parameter except `url` is optional with sensible defaults (no click, no JS, viewport only, PNG format). An empty or missing value means "skip this step."

### Execution Order

When parameters are provided, they execute in this order:

1. Navigate to `url`
2. Wait for page load
3. Execute `javascript` (if provided)
4. Click `click` element (if provided)
5. Wait for any triggered navigation or rendering
6. Take screenshot
7. Return image

### Batch Screenshots

```
POST /screenshot/batch
[
  { "url": "https://example.com/page1" },
  { "url": "https://example.com/page2", "click": ".show-more" },
  { "url": "https://example.com/page3", "javascript": "document.querySelector('.modal').remove()" }
]
```

Returns: array of screenshots, one per entry. Same schema as the single endpoint, just as an array.

### Screenshot at Every Step (Optional)

```
POST /screenshot/batch
{
  "steps": [
    { "url": "https://example.com", "click": "#login" },
    { "url": "https://example.com/dashboard" }
  ],
  "screenshot_per_step": true
}
```

When `screenshot_per_step` is true, a screenshot is taken after each step completes (not just at the end). This gives a visual trail of multi-step workflows without needing the full sequence execution API.

## What This Is NOT

This is not a replacement for the full Playwright API. The session-based endpoints (`/session/start`, `/session/{id}/navigate`, etc.) and the sequence execution endpoint (`/sequence/execute`) remain for complex workflows that need fine-grained control.

The screenshot API is the fast path for the common case. It should stay simple. As the full API grows with more options and more capabilities, the screenshot API should resist feature creep. If a use case needs more than URL + click + JS + full_page, the caller should use the full API.

## Mini Frontend

Add a simple static frontend to the Playwright service itself (served from the same container). This page:

- Explains what the service does (one paragraph)
- Has a URL input field and a "Screenshot" button
- Shows the result inline
- Has a "Batch" tab for multiple URLs
- Links to the full API documentation (FastAPI's built-in Swagger UI at `/docs`)

This makes the service immediately usable without any API client. Open the browser, type a URL, click screenshot, see the result. It also serves as the landing page when someone first accesses the service.

## Naming

The working name "quick" is fine for development but "screenshot" is clearer for the API surface. The endpoints are `/screenshot` and `/screenshot/batch`. The concept is self-describing.

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | `POST /screenshot` with just a URL returns a PNG | Provide URL only, receive screenshot image |
| 2 | `click` parameter clicks the specified element before screenshot | Provide URL + cookie banner selector, screenshot shows banner dismissed |
| 3 | `javascript` parameter executes JS before screenshot | Provide URL + scroll JS, screenshot shows scrolled position |
| 4 | `full_page` captures entire page height | Compare viewport vs full_page screenshots, full_page is taller |
| 5 | All parameters except URL are optional | Call with URL only, no errors, sensible defaults applied |
| 6 | `POST /screenshot/batch` accepts array, returns array of screenshots | Submit 3 URLs, receive 3 images |
| 7 | `screenshot_per_step` returns intermediate screenshots | Submit 3-step batch with flag, receive 3 screenshots |
| 8 | Mini frontend served from the container root | Open browser to container URL, see input field and screenshot button |
| 9 | Mini frontend screenshot button invokes the API and displays result | Type URL, click button, see screenshot inline |
| 10 | Full API endpoints unchanged and still functional | Existing session-based and sequence endpoints work as before |
