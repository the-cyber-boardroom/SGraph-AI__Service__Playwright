# Brief: `sgit` CLI — vault is now on HTTPS + two auth headers

**Version:** v0.2.6
**Date:** 2026-05-15
**Audience:** sgit CLI team (owners of the `sgit` PyPI package `sgit-ai`)
**Status:** PROPOSED — the **vault side** of this is shipped and live in production
(`diniscruz/sg-send-vault:latest` honours the §8.2 TLS contract; `sp vault-app
create --wait` now produces a browser-trusted Let's Encrypt IP cert by default).
The **`sgit` side** — picking up the new URL scheme and the second auth header —

## What changed on the vault side

A `vault-app` stack created via `sp vault-app create` now (by default):

1. Serves the vault UI / API over **HTTPS on `:443`** — not plain HTTP `:8080`.
2. Presents a **real Let's Encrypt** certificate for the EC2 public IP — browser-trusted, no warning, valid ~6 days (the LE `shortlived` profile, since IP certs are issue-only with no renewal — ephemeral stacks outlive no cert).
3. Enforces **two distinct authentication headers** on every request — both required, both with the same value.

The URL the CLI previously hit:
```
http://<public-ip>:8080
```
is now:
```
https://<public-ip>
```
(No port — `:443` is implicit; no `:8080` is published when TLS is on.)

---

## The two auth headers — both required, same value

| Header | What enforces it | Comes from .env var |
|---|---|---|
| `X-API-Key`              | FastAPI middleware (`enable_api_key=True` in the new image) | `FAST_API__AUTH__API_KEY__VALUE` |
| `x-sgraph-access-token`  | The vault app's own auth — the existing session/access-token check | `SGRAPH_SEND__ACCESS_TOKEN` |

**Good news:** both env vars are set to **the same auto-generated value** at create time (see `sg_compute_specs/vault_app/service/Vault_App__User_Data__Builder.py` — the `_STACK_TEMPLATE` `.env` block). So `sgit` only needs to remember **one** token per stack, and send it on **both** headers.

Failure mode if you only send one: a `401` with body
```json
{"data": null, "error": null,
 "message": "Client API key is missing, you need to set it on a header or cookie",
 "status": "error"}
```
That's the FastAPI gate refusing the request before the vault app even sees it. Same response shape as before, but now both headers must be present, not one.

## Discovering the token

Three paths, in order of friction:

1. **At create time** — `sp vault-app create` prints the token once in a boxed
   "access token" panel right after launch. Same as today.
2. **From `sp vault-app info`** *(new)* — the token is now stamped on the EC2
   instance as an `AccessToken` tag, and surfaced by `info`:
   ```
   access-token  KX8xggC8DShJm2HWlFm4HHb7thmX1W4Q  (X-API-Key + x-sgraph-access-token)
   ```
   Available any time, no SSM needed.
3. **From the box (legacy)** — `sp vault-app connect` → `sudo cat /opt/vault-app/.env`. Still works.

Note on the tag visibility: anyone with `ec2:DescribeInstances` on the account can read this tag. The token was already recoverable from the box via SSM, so this just makes the AWS-API path explicit. If that visibility is unacceptable for a deployment, run `sp vault-app create --no-with-tls-check` for the plain-HTTP shape — but most fleet usage should accept the trade for the operational win of "token visible via the same API as the stack itself."

## TLS root — which CA store does `sgit` need?

| Default | Cert issuer | `sgit` needs… |
|---|---|---|
| `sp vault-app create --wait` (production)                       | Real Let's Encrypt — `R3` / equivalent  | Nothing special. Stock system CA store works. |
| `sp vault-app create --wait --no-acme-prod` (debug / staging)   | `(STAGING) Ersatz Emmer YR2`            | Either install LE's staging root, **or** verify=False (recommended for the staging-only debug path). |
| `sp vault-app create --wait --tls-mode self-signed` (offline)   | self-signed, CN = the public IP          | verify=False; the cert is intentionally untrusted. |

A `--verify-tls / --no-verify-tls` switch on the `sgit` CLI is the cleanest shape — default on, off for staging/self-signed development paths.

## Concrete request shape

```http
GET /api/.../files/some/path HTTP/1.1
Host: 13.40.119.90
X-API-Key: KX8xggC8DShJm2HWlFm4HHb7thmX1W4Q
x-sgraph-access-token: KX8xggC8DShJm2HWlFm4HHb7thmX1W4Q
Accept: application/json
```

The full equivalent with `requests`:
```python
import requests

TOKEN = '...the value from `sp vault-app info` access-token row...'
HEADERS = {
    'X-API-Key'             : TOKEN,
    'x-sgraph-access-token' : TOKEN,
}

r = requests.get('https://13.40.119.90/api/some/endpoint', headers=HEADERS, timeout=10)
```

## Handoff checklist for the sgit CLI team

- [ ] **URL scheme** — switch the default base URL from `http://<host>:8080` to `https://<host>` (drop the port). Keep the old shape behind a `--http` / `--port` escape hatch if you have one for backward compat.
- [ ] **Second header** — send `X-API-Key` alongside the existing `x-sgraph-access-token`, both with the same value.
- [ ] **TLS verify switch** — `--verify-tls` default on, `--no-verify-tls` to opt out for staging / self-signed development stacks.
- [ ] **One-line "configure for stack" UX (optional but nice)** — a `sgit configure --stack <name>` that calls `sp vault-app info`, reads the `access-token` + `vault-url` rows, and writes them to the user's sgit config. Removes the copy-paste step entirely.
- [ ] **Error message** — if a request returns `401` with body `"Client API key is missing"`, tell the user "your `sgit` token is set but the `X-API-Key` header is not being sent — upgrade sgit to vX.Y." This is the only error shape that distinguishes the old-one-header world from the new-two-header one.

## Open questions

- **Cookie vs header.** The vault's browser flow uses a cookie (`x-sgraph-access-token=...; path=/`). For sgit / API callers, **headers are the right choice** — same value, easier in non-browser HTTP clients, and the FastAPI gate accepts the header per the existing convention. Confirm this matches sgit's existing pattern.
