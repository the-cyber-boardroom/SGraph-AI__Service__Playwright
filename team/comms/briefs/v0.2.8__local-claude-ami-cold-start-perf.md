# BUG — local-claude AMI cold start is brutally slow

**Status:** open
**Filed:** 2026-05-11
**Affected spec:** `local-claude` (potentially any future GPU spec that bakes large model files into the AMI)
**Branch where surfaced:** `claude/sg-compute-continuation-Ko0RY`

---

## Summary

Launching a `local-claude` instance **from a baked AMI** is dramatically slower than launching from the base AMI and pulling the model fresh. The first checkpoint shard load can sit at `0%` for many minutes, and vLLM startup that should complete in <1 min ends up taking 10+ min on the very first launch from a snapshot.

The slowdown is reproducible across **both AMIs baked so far** in this session:
- `ami-…` (first bake from `kind-volta`, snapshot ~50 GiB)
- `ami-0b540a6e1c7b3297e` (`local-claude-kind-volta-1778460815`, snapshot ~50 GiB on a 200 GiB volume)

Both AMIs hold ~50 GiB of actual data (OS + Docker layers + HF model cache) on a 200 GiB root volume. The vLLM container itself works fine; the slowness is in the file reads under it.

---

## Symptoms

User-visible: `sg lc logs --source vllm` parks for many minutes on:

```
(EngineCore pid=32) INFO 05-11 01:47:04 Using FLASHINFER attention backend …
(EngineCore pid=32) INFO 05-11 01:47:06 Filesystem type for checkpoints: XFS.
                       Checkpoint size: 15.66 GiB. Available RAM: 12.61 GiB.
(EngineCore pid=32) INFO 05-11 01:47:06 Auto-prefetch is disabled because the
                       filesystem (XFS) is not a recognized network FS
                       (NFS/Lustre) and the checkpoint size (15.66 GiB)
                       exceeds 90% of available RAM (12.61 GiB).
Loading safetensors checkpoint shards:   0% Completed | 0/6 [00:00<?, ?it/s]
```

…and stays at `0%` for an extended period.

A second observation from the same session: when reusing the same instance for a follow-up vLLM start, load is **fast** — i.e. the slow first read is a one-time cost.

---

## Root cause

**EBS lazy-load from snapshot.**

When an EBS volume is created from a snapshot, AWS does **not** copy the snapshot data onto the new volume up front. The volume is created empty and blocks are fetched on demand from S3 the first time they are read. From the workload's perspective:

```
process → kernel → EBS → S3 fetch → block now on EBS → returned to process
```

That first-touch S3 fetch is ~50–100x slower than steady-state EBS reads. Every safetensors shard vLLM touches for the first time pays this cost. Because the model is 15.66 GiB and instance RAM is 16 GiB (12.61 GiB usable after kernel/processes), vLLM's `Auto-prefetch is disabled` log line confirms it cannot mmap the model into page cache, so it has to keep going back to disk — and every "first read" hits S3.

This is not a defect in the bake process or the gp3 volume class. It is the documented behaviour of EBS volumes created from snapshots, and it scales with snapshot size.

### Why the AMI workflow makes this worse

- AMIs ship the model **inside the snapshot** (`/root/.cache/huggingface/…`).
- The snapshot is ~50 GiB, of which ~15 GiB is the model.
- A fresh launch from the AMI mounts an unwarmed EBS volume → vLLM is the first reader of those 15 GiB → every shard is paged in from S3.
- Subsequent reads are fast because the block now lives on EBS (and partially in page cache).

In contrast, the **base-AMI workflow** downloads the model from HuggingFace at boot — at ~600–1000 MB/s on g5's 10 Gbps NIC, the 15 GiB pull finishes in 20–30s, and the file is written to a freshly-allocated (non-lazy-loaded) EBS region, so vLLM's read is fast.

---

## Why it slipped through

- The first instance ever used to test the model-from-AMI path was the same instance the AMI was baked from. That instance's EBS volume was already fully populated, so reads were instant. The "AMI works fast" mental model came from this same-instance test, which never actually exercised the snapshot→new-volume path.
- AMI bake validation focuses on "does it boot, does vLLM come up, does Claude Code respond". All three pass eventually on a lazy-loaded volume — just slowly.

---

## Proposed solutions

Ranked by ROI for a dev-time GPU workflow.

### 1. Slim AMI + local NVMe model cache  *(recommended)*

g5.xlarge ships with **1×250 GB NVMe instance store** at ~3 GB/s. It is wiped on stop/start but persists across reboot — perfectly fine for an ephemeral box.

- Bake a **small AMI** containing only OS + Docker + vLLM image (~10–15 GiB snapshot)
- At boot: format and mount the local NVMe at `/mnt/hf-cache`
- Pull the model fresh from HuggingFace into `/mnt/hf-cache` (~20–30s on 10 Gbps with `HF_HUB_ENABLE_HF_TRANSFER=1`)
- vLLM reads checkpoints from NVMe at ~3 GB/s, well above any EBS class

Trade-offs:
- Each new instance re-downloads the model from HF (requires HF_TOKEN for higher rate limits)
- Requires a small boot-script section for NVMe mount + cache redirect
- AMI bake time drops from 15–30 min to ~5 min
- Snapshot storage cost drops ~70%

This is the proper architecture for an ephemeral GPU box. Aligns with what the user already pays AWS for (the NVMe is included in the instance price).

### 2. Boot-time prewarm of the model cache  *(quick win, keeps current architecture)*

Add a section that runs after the EBS is mounted and before vLLM starts:

```bash
# Force lazy-loaded blocks to materialize on EBS now, in background, so vLLM
# does not pay the S3 fetch cost when loading checkpoints.
nohup bash -c '
  find /root/.cache/huggingface -type f \
    -name "*.safetensors" -exec dd if={} of=/dev/null bs=1M status=none \;
' >/dev/null 2>&1 &
```

- Cost: 3–5 min one-time on each fresh launch, runs in background, does not block boot
- Pro: zero architectural change, no HF dependency at runtime, AMI workflow stays the same
- Con: doesn't fix the underlying "model bigger than RAM" issue — vLLM will still have to re-page from disk, but at full EBS speed instead of S3 speed
- Best as a stopgap or as a complement to (1)

### 3. Bump instance to g5.2xlarge (32 GiB RAM)

- Pro: model fits in page cache, vLLM's auto-prefetch re-enables, second+ loads are essentially instant
- Con: ~2x hourly cost
- Doesn't fix the **first** load — still pays lazy-load — but the subsequent reads are fast
- Best combined with prewarm (2)

### 4. AWS Fast Snapshot Restore (FSR)

AWS pre-warms a snapshot in advance so volumes created from it perform at full speed immediately.

- Pro: zero boot-time penalty, no architectural change
- Con: **~$0.75/GiB-month per AZ**. For a 200 GiB AMI in one region/AZ that's ~$150/month standing cost. For a 50 GiB slim AMI it'd be ~$38/month/AZ.
- Verdict: only justifiable for a high-frequency launch pattern, or once we slim the AMI per (1)

### 5. gp3 throughput/IOPS bump or io2 Block Express

- gp3 throughput: 125 MB/s → 1000 MB/s, ~$8/month per 200 GiB
- io2 Block Express: up to 4 GB/s, 256k IOPS, ~3x gp3
- **Neither fixes the bottleneck.** The slow path is the S3→EBS fetch, not the EBS device. Once a block is on EBS, gp3-baseline is already fast enough for vLLM. Upgrading the volume class is the wrong layer.

### 6. Don't bake the model into the AMI at all  *(equivalent to (1) without the NVMe)*

If we don't want to deal with NVMe mounts, just pull the model from HF at boot into the regular EBS volume. The model lands on freshly-allocated blocks (no lazy load) and the first vLLM read is fast.

- Pro: simplest change; small AMI; no NVMe handling
- Con: model lives on EBS, which is slower than NVMe (but still much faster than lazy-loaded reads from S3); also occupies snapshot space if the user later bakes again

---

## Recommended path

1. **Now:** ship (2) — boot-time prewarm — as a small section under `sg_compute/platforms/ec2/user_data/`. Keeps the existing AMIs usable.
2. **Next:** implement (1) — slim AMI + local NVMe — and re-bake. Deregister the two existing fat AMIs once verified.
3. **Later (only if launch frequency justifies it):** evaluate FSR on the slim AMI.

Options (3), (4), and (5) are not recommended as first moves.

---

## Acceptance criteria

- A fresh `sg lc create --ami <id>` reaches vLLM `/v1/models` returning `200` in **< 2 min** total (target; current is 10+ min on a lazy-loaded fat AMI).
- `sg lc diag` shows all 8 checks green within the same window.
- AMI bake time is **< 10 min** (target; current is 15–30 min).
- Snapshot size is **< 25 GiB** (target; current is ~50 GiB).

## Notes

- This bug class will recur for **any** spec that bakes large data into the AMI (ollama models, large container images, dataset caches). Documenting the solution under `sg_compute` rather than under `local_claude` so future specs can adopt it.
- The two existing fat AMIs are still usable — just slow on first launch. No need to deregister urgently; let them age out when the slim AMI lands.
