## Goal — one sentence

**Prove that the sgi capture/replay approach takes local-claude from ~10 minutes cold-start to ≤ 2 minutes, using a build instance + S3 bundle + minimal user-data, without touching any existing `sg_compute/` code.**

If the spike hits the number, we extract sgi to its own repo and execute the full pack. If it misses, we revisit before more investment.

---

## The pain we're solving

Today's `sg lc create` cold-start path, end-to-end:

| Phase | Time | What happens |
|---|---|---|
| EC2 `RunInstances` → SSH ready | ~17–37 s | DLAMI floor (EBS lazy-load on /) |
| user-data installs nvidia-container-toolkit, docker, etc. | ~30–60 s | `dnf install`, daemon restart |
| `docker pull vllm/vllm-openai:latest` | ~60–120 s | ~5 GB image from Docker Hub or ECR |
| `docker run` + vLLM downloads model from HF | ~5–8 min | ~15.66 GB model, sequential safetensors load |
| vLLM serves `/v1/models` 200 | (cumulative) | **~600 s total** |

The dominant cost is the model: 15.66 GB on a 16 GB-RAM instance, lazy-loaded from EBS-from-snapshot at S3 latency, with no auto-prefetch headroom. Full session diagnosis in the cold-start perf brief.

## The sgi bet

Replace the install+download phases with **one parallel multipart S3 GET into NVMe instance store**, decompressed in place, services started against the warm files. Predicted:

| Phase | Predicted time | Mechanism |
|---|---|---|
| EC2 `RunInstances` → SSH ready | ~17 s | AL2023 base, NOT DLAMI; no driver tax for the spike (see §Spike scope) |
| Fetch bundle from S3 to NVMe | ~25–40 s | 20+ GB at ~600 MB/s saturated multipart |
| Extract tar.zst to disk | ~10–15 s | zstd -d \| tar x, NVMe write |
| Start vLLM against extracted files | ~20–30 s | warm files, no model download |
| vLLM serves `/v1/models` 200 | (cumulative) | **~75–100 s total** |

Numbers are predictions, not guarantees. **The spike's only job is to replace them with measurements.**

## Constraints

1. **No mutation of `sg_compute/` or `sg_compute_specs/`.** sgi may read them, shell out to `sg lc *`, and consume the live local-claude instance produced by `sg lc create`, but cannot edit them.
2. **sgi lives at `sg_image_builder/`** at the repo root, with tests under `tests/sg_image_builder/`. Empty `__init__.py` everywhere (CLAUDE.md rule 22). Lift-and-shift to its own repo happens after the spike.
3. **`Type_Safe`, no Pydantic, no boto3 direct, no Literals** — same rules as the rest of this repo (`.claude/CLAUDE.md`). osbot-aws for AWS, osbot-utils SSH for exec.
4. **No mocks / no patches in tests.** `Storage__In_Memory` + `Exec_Provider__Twin` are real implementations that record calls.
5. **The spike is GPU-only.** Don't bother with `ssh_file_server`/`python_server` etc. — go straight to `vllm_docker`-style local-claude. If sgi doesn't help local-claude, the rest of the ladder is irrelevant.
6. **No strip workflow.** Capture the fat bundle, replay it fat. Strip is M7 in the pack and a v0.2 concern.
7. **No CLI polish.** A single entry-point script (e.g. `python -m sg_image_builder.spike capture | publish | load | bench`) is enough for the spike. The full typer CLI lands when the pack proper is implemented.

### Step 2 — Capture local-claude state (~1 day)

Pre-condition: a working local-claude instance exists, launched via the **existing** `sg lc create --spec local_claude`. We do **not** modify that path.

The capture script:

1. Resolves the running instance's SSH endpoint (via `sg lc info <name>` shell-out, parse the existing output — A6 in the architect review).
2. Connects via `Exec_Provider__SSH`.
3. Runs a known-good set of file enumerations on the target:
   - `/etc` (config, especially `/etc/systemd/system/`, `/etc/docker/`, `/etc/nvidia-container-runtime/`)
   - `/opt`
   - `/usr/local`
   - `/var/lib/docker/` (the Docker image's overlay2 + content store + image manifests)
   - `/root/.cache/huggingface/` and any other model cache locations
   - `~/.local`, `~/.config` for the service user
3. Records `sha256sum`, mode, owner, mtime, size for each file (one `find … -printf` + `xargs sha256sum`, streamed).
4. Writes the result as a `Schema__Bundle__Manifest` JSON.
5. Copies the files into a local tar.zst, organised as `rootfs/<absolute-path>` per the pack's payload convention.

**Measurement:**
- Capture completes without errors against the live local-claude instance.
- Resulting tar.zst is somewhere in the 18–22 GB range (we expect ~5 GB vLLM image + ~15.66 GB model + small leftovers).
- Round-trip extraction on a tempdir reproduces byte-identical files (verify via per-file sha256).
- Capture wall-clock time recorded (informational — capture happens once, replay happens always).

### Step 3 — Publish to S3 (~½ day)

Use osbot-aws for multipart upload (8 MiB chunks per A9 in the architect review). Reuse an existing bucket if available; otherwise create `sg-compute-sgi-artifacts-dev` in `eu-west-2` with the existing `playwright-ec2` IAM role getting read access.

IFD path per the pack:

```
s3://sg-compute-sgi-artifacts-dev/bundles/local_claude/local-claude-vllm/v0/v0.1/v0.1.0/
  ├── manifest.json
  ├── payload.tar.zst
  └── publish.json
```

Skip `sidecar.zip` for the spike (A7 in the architect review — bundle-scoped tests aren't relevant yet).
Skip `catalog.json` for the spike — single-writer publishes don't need an index file (A2).

**Measurement:**
- Upload wall-clock time (informational — published once).
- Download wall-clock time from a `g5.xlarge` in the same AZ, single-stream and multi-stream. **This is the load-bearing measurement** — it's the floor on `first_load` time. Run 3× each.

### Step 4 — Load onto a fresh AL2023 instance (~1–2 days)

This is the spike's centrepiece and the most novel work.

`Exec_Provider__SSH` launches an EC2 from base AL2023 with a minimal user-data script:

```bash
#!/bin/bash
set -e

# Mount NVMe instance store (g5.xlarge ships with 250 GB at /dev/nvme1n1)
mkfs.xfs -f /dev/nvme1n1
mkdir -p /mnt/nvme
mount /dev/nvme1n1 /mnt/nvme

# Pull bundle from S3 to NVMe
aws s3 cp --quiet \
  s3://sg-compute-sgi-artifacts-dev/bundles/local_claude/local-claude-vllm/v0/v0.1/v0.1.0/payload.tar.zst \
  /mnt/nvme/payload.tar.zst

# Extract to / (manifest puts everything under rootfs/)
cd /
zstd -d -c /mnt/nvme/payload.tar.zst | tar -x --strip-components=1

# Per-instance fixups (the parts that have to be live)
systemctl daemon-reload
systemctl enable --now docker
systemctl enable --now <vllm-systemd-unit>
```

The user-data is generated by sgi (a `Section__Bundle_Fetch.py`-style class **in sgi's tree**, not in `sg_compute/`). To avoid touching sg-compute, the spike's launch path is:

- sgi calls `osbot-aws` `EC2_Instance.create` directly with sgi's user-data (not `sg lc create`).
- The fresh instance is **not registered with sg-compute's `lc list`** — it's an sgi-managed instance for the spike only.
- This means `sg lc connect <name>` won't work on the spike instance. That's expected. SSH directly using the key pair sgi provisioned, or use SSM via the standard `aws ssm start-session` if the instance profile is `playwright-ec2`.

The Docker daemon will see the captured `/var/lib/docker/` and pick up the vllm image natively. vLLM systemd unit (captured from local-claude's working setup) will start with `--model /path/to/captured/model`. **No `docker pull` or HF download runs.**

**Measurement (the headline number):**

Run the cold-start 10 times. For each:
- `t0` = `ec2:RunInstances` API response timestamp
- `t1` = first SSH connection succeeds
- `t2` = `aws s3 cp` completes (extracted from user-data log via SSM `GetCommandInvocation` or `cloud-init` log)
- `t3` = tar extract completes
- `t4` = `systemctl is-active vllm` → `active`
- `t5` = `curl -s http://localhost:8000/v1/models` returns 200 with the expected model

Compute and report:

```
first_load   = t3 - t1     # bundle download + extract (the pack's P17)
boot         = t4 - t3     # service start
execution    = t5 - t4     # first health check
cold_start   = t5 - t0     # composite
```

Capture each `t*` and the sub-timings into `team/comms/briefs/v0.2.8__sgi-spike-results.md` as a results table.

### Step 5 — Decision gate (~½ day)

Compare to the existing local-claude cold-start (~600 s).

| Outcome | Action |
|---|---|
| `cold_start_p50 ≤ 120 s` | **Strong pass.** Proceed to extract sgi to its own repo per the pack. Pack's M0–M2 → M4 → M9 path lands the full surface. |
| `120 s < cold_start_p50 ≤ 240 s` | **Conditional pass.** Win is real but smaller than predicted. Investigate sub-timing (which step is the surprise cost?), file a follow-up brief before extracting. |
| `cold_start_p50 > 240 s` | **Fail or partial.** Sub-timings determine whether the design is wrong, the spike is misconfigured, or there's a fundamentally different bottleneck. Don't extract; revisit the pack. |

The decision gate produces a debrief at `team/claude/debriefs/<date>__sgi-spike-result.md`.

## What we explicitly defer

- **Strip workflow (pack's M7).** Fat bundle is fine for the spike. We can ship a 20 GB bundle and still validate the load path. Strip is a 30% improvement on top, not the headline.
- **`Exec_Provider__Sg_Compute` provider.** The spike uses SSH for launch and SSH for capture-target. The Sg_Compute provider matters for *running* sgi-the-tool on sg-compute-managed boxes, which is not the validation path.
- **`Storage__Local_Disk` fully wired.** The spike only needs `Storage__S3` for the load step. `Storage__Local_Disk` exists for the round-trip integration test only.
- **Full sgi CLI (`sgi bundle capture-from`, etc.).** A single `python -m sg_image_builder.spike.run` driver script orchestrates the spike. The typer CLI is a packaging concern.
- **Recipes, KPIs, events bus, Kibana dashboards.** All add value but none gate the cold-start measurement.
- **Sidecar.zip, SKILL.md/USAGE.md/SECURITY.md.** Spike has no consumers reading the sidecar.
- **The 8 other specs (`ssh_file_server` through `ollama_docker`).** Out of scope for the spike. They become relevant only after sgi is extracted.

## Risks and how we'd notice

| Risk | Symptom | Pre-mitigation |
|---|---|---|
| **Capture misses a file the workload needs.** | Replay starts but vLLM fails to load model / Docker daemon can't find image. | Diff-based capture is enumerative; capture `/` minus a small exclude list (`/proc`, `/sys`, `/dev`, `/run`, `/tmp`, `/var/cache`, `/var/log`) rather than allowlisted paths. Larger bundle, more confidence. |
| **Captured `/var/lib/docker/` is corrupted on extract.** | Docker daemon errors on start. | Stop the Docker daemon on the build host **before** capture. Capture is taken against a quiesced filesystem. Add as explicit Step 2 sub-task. |
| **NVMe instance store is not large enough.** | `aws s3 cp` fills the disk. | g5.xlarge ships 250 GB NVMe; bundle is ~20 GB; headroom is ~10×. Acknowledge in user-data: fail loudly if `df` shows < 50 GB free before download. |
| **vLLM systemd unit references hard-coded paths that don't exist on the spike instance.** | `systemctl status` shows path-not-found errors. | Capture explicitly includes `/etc/systemd/system/vllm.service` (and equivalents). Spike's verification step asserts `systemctl is-active`. |
| **Cold-start is slow because of EBS lazy-load on `/` even with bundle on NVMe.** | `boot` time is anomalously high; sub-timings show `t4 - t3` is the cost. | The bundle's services should be on NVMe-mounted paths after extract. If they're on `/`, we accidentally inherited the same EBS-lazy-load tax. Verify capture-side paths during Step 2. |
| **Multipart S3 download is single-stream-bound.** | `first_load` time matches single-stream rate (~80 MB/s), not multi-stream. | osbot-aws S3 helpers should default to multi-stream; if not, file an osbot-aws issue and use boto3's `TransferConfig` directly (narrow exception, like the existing Lambda Function URL precedent). |
| **The whole thing works but only by chance — captured state is non-deterministic.** | Second run produces a different bundle sha256 from a re-capture. | Out of scope for the spike. Document, defer. The capture-determinism question is a real one for v0.2 but doesn't gate the cold-start measurement. |

## What this brief does NOT do

- **It does not lock the v1 design.** The pack does. This brief is the path to a yes/no on whether to commit to the pack.
- **It does not produce a production tool.** The spike is a measurement instrument. Polish, error messages, retry logic — all deferred.
- **It does not handle multi-AZ, multi-region, spot, or any non-`g5.xlarge`.** Single AZ, on-demand, one instance type.
- **It does not solve secrets management.** The vLLM unit and the model are public-ish artefacts; the spike's S3 bucket has no PII. Real key management is a v0.2 concern.

## Open questions before starting

1. **Which existing AMI is the spike's base?** AL2023 (slower base boot, no driver tax during boot — drivers come from the captured bundle) or DLAMI AL2023 OSS (faster driver provisioning, slower base boot)? **Recommended: AL2023.** The whole point is to prove the bundle carries everything. If DLAMI is needed, the bundle is incomplete.
2. **Where do we get the local-claude build instance?** Spin up one fresh via `sg lc create --spec local_claude`, capture, terminate? Or use an existing running one (e.g. the user's current dev box)? **Recommended: spin up a fresh one via the existing path** so the capture is reproducible from a known good state.
3. **Which model size is in the bundle?** Qwen3-Coder (~15.66 GB per the cold-start brief) is the existing local-claude default. Stick with it unless there's a reason to vary. **Recommended: match production.**
4. **What's the IAM identity for the spike S3 bucket?** The existing `playwright-ec2` instance profile or a new one? **Recommended: reuse `playwright-ec2`**, add S3 read on the bucket. Avoids new IAM surface during the spike.

These are decisions to make in the first session, not blockers.

End of brief.
