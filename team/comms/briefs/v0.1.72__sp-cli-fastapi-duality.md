# Brief: SP CLI ↔ FastAPI Duality
**Version:** v0.1.72  
**Date:** 2026-04-23  
**Audience:** Architect, Dev, DevOps  
**Status:** PROPOSED — nothing described here exists yet

---

## Design principle: the observability layer is ephemeral

The OpenSearch domain, AMP workspace, and AMG workspace are **not permanent
infrastructure**.  They are created on demand (morning), used, and destroyed
(evening).  Data is preserved between runs via `ob backup` / `ob restore`.
Dashboards and saved objects are imported fresh on every `ob create`.

This is intentional:
- **Cost** — managed OpenSearch on `t3.small` costs ~$0.04/hr.  Running it only
  during business hours (≈10 h/day) cuts the bill by ~60 % vs always-on.
- **Hygiene** — ephemeral stacks cannot accumulate drift.  Every morning is a clean
  slate, guaranteed by the create sequence.
- **Testing** — the create/delete/restore cycle is itself a tested workflow; any
  breakage is caught in CI before it becomes a production incident.

**The API duality refactor exists to make this lifecycle fully automated** — no
human needs to remember to run `sp ob create` in the morning or `sp ob delete`
in the evening.  GitHub Actions does it via HTTP POST/DELETE to the FastAPI layer.

---

## Problem

Every `sp` / `ob` command embeds business logic directly inside a `typer` command
function.  A `Console` object is mixed into the core computation; the only way to
invoke the logic is via the CLI.  That makes:

- **Automation hard** — scheduling `ob create` / `ob delete` requires shell access or
  subprocess calls from GitHub Actions.
- **Integration impossible** — the Playwright service itself cannot trigger stack
  operations programmatically.
- **Testing awkward** — unit tests must either invoke typer's test runner or
  duplicate the logic.

---

## Goal

Every `sp` / `ob` operation should be callable in **two ways** — same code path,
same result, no duplication:

| Invocation | Entry point | Output |
|---|---|---|
| `sp ob create sp-observe-1` | typer CLI | Rich console |
| `POST /v1/observability/stack` | FastAPI route | JSON schema |
| `sp ob delete sp-observe-1` | typer CLI | Rich console |
| `DELETE /v1/observability/stack/sp-observe-1` | FastAPI route | JSON schema |
| `sp ob list` | typer CLI | Rich table |
| `GET /v1/observability/stacks` | FastAPI route | JSON schema |
| `sp ob backup sp-observe-1` | typer CLI | Rich console |
| `POST /v1/observability/stack/sp-observe-1/backup` | FastAPI route | JSON schema |
| `sp ob restore sp-observe-1` | typer CLI | Rich console |
| `POST /v1/observability/stack/sp-observe-1/restore` | FastAPI route | JSON schema |

Backup and restore are **first-class operations** in this design, not
afterthoughts.  The daily automation workflow is:

```
06:00  POST /v1/observability/stack          ← create (morning)
06:20  POST /v1/observability/stack/{}/restore  ← restore yesterday's data
...business hours...
20:00  POST /v1/observability/stack/{}/backup   ← snapshot before teardown
20:05  DELETE /v1/observability/stack/{}        ← delete (evening)
```

## Architecture

### Layer separation (three tiers)

```
┌─────────────────────────────────────────────────────────┐
│  Tier 1 — Pure logic (no I/O, no console, returns data)  │
│  observability/service.py  ·  ec2/service.py             │
│  All functions return Type_Safe schemas                   │
└──────────┬──────────────────────────────────────────────┘
           │  called by both ↓
    ┌──────┴──────┐          ┌──────────────────┐
    │  Tier 2A    │          │  Tier 2B          │
    │  CLI layer  │          │  FastAPI layer    │
    │  typer cmds │          │  route handlers   │
    │  print/Rich │          │  return .json()   │
    └─────────────┘          └──────────────────┘
```

### Tier 1 — service functions

Each operation becomes a pure Python function in a dedicated module.  No `typer`,
no `Console`, no `sys.exit`.  Returns a `Type_Safe` schema.  Raises a typed
exception on error.

```
sgraph_ai_service_playwright/
  observability/
    __init__.py
    service.py          ← Observability__Service  (pure logic)
    schemas/
      Schema__Stack__Create__Request.py
      Schema__Stack__Create__Response.py
      Schema__Stack__Info.py
      Schema__Stack__List.py
      Schema__Stack__Delete__Response.py
      Schema__Import__Result.py
```

`Observability__Service` extends `Type_Safe` and exposes:

```python
def create(self, name: str, region: str = '',
           admin_user: str = '', admin_pass: str = '',
           no_wait: bool = False, no_import: bool = False,
           no_grafana: bool = False) -> Schema__Stack__Create__Response

def delete(self, name: str, region: str = '') -> Schema__Stack__Delete__Response

def info(self, name: str, region: str = '') -> Schema__Stack__Info

def list_stacks(self, region: str = '') -> Schema__Stack__List

def dashboard_import(self, name: str, region: str = '',
                     admin_user: str = '',
                     admin_pass: str = '') -> Schema__Import__Result
```

Progress events (currently printed inline) are emitted via a callback:

```python
class Observability__Service(Type_Safe):
    on_progress: callable = None          # set by CLI or omitted by API

    def _emit(self, msg: str):
        if self.on_progress:
            self.on_progress(msg)         # CLI passes a Rich print; API ignores it
```

### Tier 2A — CLI layer (typer)

`scripts/observability.py` keeps the typer app but every command becomes a
three-line wrapper:

```python
@app.command('create')
def cmd_create(name: str, ...):
    svc = Observability__Service(on_progress=lambda m: c.print(m))
    result = svc.create(name, region=r, ...)
    _print_create_result(result, c)   # Rich formatting only
```

### Tier 2B — FastAPI layer

New router added to `Fast_API__Playwright__Service`:

```
routes/
  observability/
    Route__Observability__Stack__Create.py
    Route__Observability__Stack__Delete.py
    Route__Observability__Stack__Info.py
    Route__Observability__Stack__List.py
    Route__Observability__Dashboard__Import.py
```

Every route is a pure delegation — no logic:

```python
class Route__Observability__Stack__Create(Type_Safe):
    playwright_service: Playwright__Service

    def handle(self, body: Schema__Stack__Create__Request) -> Response:
        result = Observability__Service().create(**body.as_dict())
        return result.json()
```

## Scheduling via GitHub Actions — full daily lifecycle

Four workflow files drive the ephemeral lifecycle.  No always-on process.
No cron daemon on EC2.  No state to lose if an instance recycles.

**Morning: create + restore** (06:00 UTC Mon-Fri)
```yaml
# .github/workflows/obs-morning.yml
on:
  schedule:
    - cron: '0 6 * * 1-5'
  workflow_dispatch:
    inputs:
      stack_name: {required: false, default: ''}
jobs:
  morning:
    runs-on: ubuntu-latest
    env:
      STACK: ${{ inputs.stack_name || vars.DEFAULT_OB_STACK }}
      URL:   ${{ secrets.SP_LAMBDA_URL }}
    steps:
      - name: Create stack
        run: |
          curl -sf -X POST "$URL/v1/observability/stack" \
            -H "Content-Type: application/json" \
            -d "{\"name\":\"$STACK\",\"no_wait\":false}"
      - name: Restore latest backup
        run: |
          curl -sf -X POST "$URL/v1/observability/stack/$STACK/restore"
```

**Evening: backup + delete** (20:00 UTC Mon-Fri)
```yaml
# .github/workflows/obs-evening.yml
on:
  schedule:
    - cron: '0 20 * * 1-5'
  workflow_dispatch:
    inputs:
      stack_name: {required: false, default: ''}
jobs:
  evening:
    runs-on: ubuntu-latest
    env:
      STACK: ${{ inputs.stack_name || vars.DEFAULT_OB_STACK }}
      URL:   ${{ secrets.SP_LAMBDA_URL }}
    steps:
      - name: Backup
        run: |
          curl -sf -X POST "$URL/v1/observability/stack/$STACK/backup"
      - name: Delete stack
        run: |
          curl -sf -X DELETE "$URL/v1/observability/stack/$STACK"
```

`SP_LAMBDA_URL` is the Lambda Function URL (stored as a GH Actions secret).
`DEFAULT_OB_STACK` is a GH Actions variable (e.g. `sp-observe-prod`).
`workflow_dispatch` allows manual morning/evening runs for one-off stacks.

## Migration plan (incremental — no big-bang)

| Step | What | Risk |
|---|---|---|
| 1 | Extract `Observability__Service` with all logic; CLI calls it | Low — pure refactor, same behaviour |
| 2 | Add `/v1/observability/stacks` GET + `stack/{name}` GET (read-only) | Low — no mutations |
| 3 | Add `POST /v1/observability/stack` (create) — returns immediately if `no_wait=True` | Medium — long-running op needs background task or async status polling |
| 4 | Add `DELETE /v1/observability/stack/{name}` | Low |
| 5 | Add GH Actions workflows | Low |
| 6 | Same pattern for `ec2` operations (`provision_ec2.py`) | Medium |

**Step 3 note:** `ob create` takes ~20 min.  The FastAPI route should accept the
request and return a `202 Accepted` with a job token immediately.  A
`GET /v1/observability/stack/{name}/status` polls the real AWS state.  The CLI
version keeps its synchronous wait loop.

---

## What does NOT change

- CLI behaviour and flags — identical, no regressions
- `typer` as CLI framework — stays
- The `observability.py` / `provision_ec2.py` file names — kept for `sp` entrypoint wiring
- `Type_Safe` / no Pydantic / no Literals — enforced throughout new schemas
- `__init__.py` stays empty in every new package folder

## Open questions for Architect sign-off

1. **Auth on FastAPI routes** — Lambda Function URL + IAM auth is sufficient for
   GH Actions (SigV4); do we need API-key auth for ad-hoc curl calls?
2. **`admin_pass` in API response** — only returned on first creation (never stored).
   Acceptable in JSON body, or should it go in a response header only?
3. **Async create** — 202 + polling vs synchronous (timeout risk on Lambda)?
   Lambda max timeout is 15 min; OS creation is 15-20 min — async is safer.
