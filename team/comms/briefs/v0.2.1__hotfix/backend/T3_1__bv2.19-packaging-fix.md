# T3.1 — Fix BV2.19 packaging glob (UI assets won't ship to Lambda)

⚠ **Tier 3 — integration cleanup.** Standalone PR. Quick fix.

## What's wrong

BV2.19 added `*/ui/**/*` to `package_data` so per-spec UI assets get included in the wheel. The glob landed in the **outer** poetry `pyproject.toml` (root) — which doesn't even package `sg_compute_specs`. The **inner** `sg_compute_specs/pyproject.toml` package-data is missing the glob entirely. **UI files will silently no-op in production Lambda.**

## Tasks

1. **Open both pyprojects:**
   - Root `pyproject.toml` — verify the `*/ui/**/*` glob is here (it shouldn't be).
   - `sg_compute_specs/pyproject.toml` — verify glob is missing.
2. **Move the glob** to the inner `sg_compute_specs/pyproject.toml` `[tool.setuptools.package-data]` (or hatch / poetry equivalent) so it actually applies to the wheel that ships the specs.
3. **Verify wheel contents:**
   ```
   python -m build sg_compute_specs/
   unzip -l dist/sg_compute_specs-*.whl | grep -E "ui/(card|detail)/v0/v0\.1/v0\.1\.0/.*\.(js|html|css)"
   ```
   Should list per-spec UI files.
4. **Remove from outer pyproject** — wherever the glob landed in error.
5. **Add a CI check** — `tests/ci/test_wheel_contains_ui.py` that builds the wheel and asserts at least one spec's `ui/card/.../sg-compute-<spec>-card.js` is present.

## Acceptance criteria

- `unzip -l dist/sg_compute_specs-*.whl` shows per-spec UI assets.
- CI test confirms it.
- Outer `pyproject.toml` no longer contains the misplaced glob.
- Reality doc updated.

## Live smoke test

Build the wheel; `pip install dist/sg_compute_specs-*.whl` in a fresh venv; `python -c "import sg_compute_specs.firefox.ui.card; print('ok')"` (or use `importlib.resources` to read a UI file). Should succeed.
