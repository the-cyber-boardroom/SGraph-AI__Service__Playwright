## The stack shape with `--with-playwright`

```
sp vault-app create --with-playwright --wait
```

spins up a **four-container** stack on a single EC2 host, all sharing one
internal Docker bridge network (`vault-net`):

```
            vault-net  (172.x.0.0/16, bridge)
            │
            ├─ host-plane         (admin / docker-socket; SSM only)
            ├─ sg-send-vault      (TLS :443 to the world; vault UI + API)
            ├─ sg-playwright      (HTTP :8000 internal; browser-automation API)
            └─ agent-mitmproxy    (HTTP :8080 internal — browser proxy
                                  ; HTTP :8081 internal — mitmweb admin)
```

Only **sg-send-vault** publishes a host port (`:443`). Everything else stays on
the internal bridge — reachable by service name on the docker DNS resolver.

## Two ways to reach sg-playwright — internal and external

### From sg-send-vault (internal — preferred for in-stack use)

| What | Value |
|---|---|
| **Base URL**    | `http://sg-playwright:8000`                              |
| **Auth header** | `X-API-Key: <the access_token>`                          |
| **Token source**| `FAST_API__AUTH__API_KEY__VALUE` (already in vault's env)  |
| **TLS verify**  | n/a — internal HTTP on the docker bridge                 |
| **Network**     | `vault-net` — automatic (every service in the compose joins it) |

### From anywhere else (external — for sgit, dashboards, ad-hoc REST clients)

| What | Value |
|---|---|
| **Base URL**    | `http://<public-ip>:11024`                                |
| **Auth header** | `X-API-Key: <the access_token>` (same token; the vault and playwright share `FAST_API__AUTH__API_KEY__VALUE`) |
| **Scheme**      | plain HTTP (for now — TLS would be a §8.2-style follow-up; the same access token is the actual gate) |
| **SG rule**     | `:11024` world-open when `--with-playwright`, opened by `Vault_App__Service.create_stack` |
| **Surfaced on** | `sp vault-app info` → `playwright-url` row (with the `(same X-API-Key)` hint) |

External callers should always use `:11024` (not `:8000`) — `:8000` is the
container-internal port and never published to the host. Inside the compose
stack, the docker service name (`sg-playwright:8000`) is the right form.

Crucially: the **same access token** that authenticates calls *to* the vault is
also the playwright service's API key. They share `FAST_API__AUTH__API_KEY__VALUE`
in compose. The vault doesn't need a second secret — just reuse what it already
has and send it as `X-API-Key`.

### Concrete example (Python)

```python
import os, requests

PLAYWRIGHT_URL = os.environ.get('SG_VAULT_APP__PLAYWRIGHT_URL') or 'http://sg-playwright:8000'
API_KEY        = os.environ['FAST_API__AUTH__API_KEY__VALUE']           # vault already reads this
HEADERS        = {'X-API-Key': API_KEY}

# health probe — works as soon as the playwright container is up
r = requests.get(f'{PLAYWRIGHT_URL}/info/health', headers=HEADERS, timeout=10)
r.raise_for_status()

# typical browser automation — see sg-playwright's OpenAPI for the full surface
r = requests.post(f'{PLAYWRIGHT_URL}/session/screenshot',
                  json    = {'url': 'https://example.com', 'full_page': True},
                  headers = HEADERS,
                  timeout = 60)
```

### Suggested env var name (vault image side)

If the new image wants to read the URL from env (recommended over hard-coding),
the cleanest convention to match the existing `SG_VAULT_APP__*` prefix is:

```
SG_VAULT_APP__PLAYWRIGHT_URL=http://sg-playwright:8000
```

Tell the SG/Compute team this is what you'll consume and the next compose
template update can inject it into the sg-send-vault service. The hard-coded
`http://sg-playwright:8000` works today; an env var just makes Lambda /
laptop / dev-station modes overridable.

## What's wired today (sg-playwright side)

The compose template already configures sg-playwright for the EC2-with-proxy
deployment mode:

| Env on sg-playwright              | Value                            | Purpose |
|-----------------------------------|----------------------------------|---|
| `FAST_API__AUTH__API_KEY__VALUE`  | the shared access token          | API gate; same token vault uses |
| `SG_PLAYWRIGHT__DEPLOYMENT_TARGET`| `container`                      | Disables Lambda-only code paths |
| `SG_PLAYWRIGHT__DEFAULT_PROXY_URL`| `http://agent-mitmproxy:8080`    | Every browser request goes through mitmproxy |
| `SG_PLAYWRIGHT__IGNORE_HTTPS_ERRORS` | `'true'`                      | Required: mitmproxy presents forged certs to intercept TLS |

So when sg-send-vault POSTs a screenshot/scrape job, the browser inside
sg-playwright transparently routes through agent-mitmproxy for upstream
proxying and TLS interception — the vault doesn't have to think about it.

---

## The browser-proxy port — for completeness

agent-mitmproxy is **transparent infrastructure** for the vault — vault doesn't
talk to it. But for the record:

- `:8080` (internal) — the mitmproxy HTTP CONNECT port. Used by sg-playwright's
  browser launches via `SG_PLAYWRIGHT__DEFAULT_PROXY_URL`.
- `:8081` (internal) — the mitmweb admin UI. Used by host-plane's `Routes__Web`
  for `/web/*` forwarding, after this commit also adds `AGENT_MITMPROXY__MITMWEB_HOST=agent-mitmproxy` to host-plane's env (was silently broken before — defaulted to `127.0.0.1` which inside host-plane's container was host-plane itself, not mitmproxy).
- `:8000` (internal) — agent-mitmproxy's own FastAPI admin API.

None of these are reachable from the vault. They're only reachable from
**host-plane** (which has the docker socket and can also act as a reverse
proxy), and host-plane is itself only reachable via SSM port-forward —
intentional: keeps the privileged admin plane off the public internet.

## Known gap (open question, not blocking)

**playwright → vault direction** isn't fully wired yet. The
`SG_PLAYWRIGHT__SG_SEND_BASE_URL` env var exists in `sg_compute_specs/playwright/core/consts/env_vars.py`,
but the compose template doesn't set it because of a real design tension:

- With TLS on (the new default), sg-send-vault binds **only :443** per the §8.2
  contract — it does not also listen on :8080.
- An internal call from sg-playwright would have to go to `https://sg-send-vault`,
  but the served cert is for the EC2 *public IP*, not the docker hostname
  `sg-send-vault` — so cert hostname verification would fail.

Three options for a future slice, all reasonable:

1. **`verify=False` for internal vault calls from sg-playwright.** The token is
   the gate; cert verification adds nothing on a single-host docker network.
   Set `SG_PLAYWRIGHT__SG_SEND_BASE_URL=https://sg-send-vault` and have the
   playwright vault client skip cert verification for `*.local` / docker
   service names.
2. **Playwright loops back through the public IP.** `cert_init.py` already
   demonstrates the IMDSv2 pattern for resolving the EC2 public IP from
   inside a container. sg-playwright could do the same on startup and set
   its vault base URL to `https://<public-ip>` — the served cert *is* for
   that IP, so verification passes naturally. Costs one IMDS round-trip at
   boot; clean.
3. **Extend the §8.2 contract to dual-listen.** A `FAST_API__TLS__ALSO_HTTP_PORT`
   that, when set, binds plain HTTP on a second internal port (e.g. `:8080`) in
   parallel with TLS on `:443`. Cleaner conceptually, costs a port and a code path.

Recommend option (2) — no contract change, no verify=False compromise, and
mirrors the IMDS pattern we're already using elsewhere. If the vault team
wants to use playwright but also have playwright read from the vault, flag it
and we'll wire (2) in a follow-up.
