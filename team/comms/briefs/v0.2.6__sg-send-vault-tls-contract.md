# Brief: Native TLS support in the `sg-send-vault` container

**Version:** v0.2.6
**Date:** 2026-05-14
**Audience:** Vault Dev team (owners of the `sg-send-vault` image), Architect, DevOps
**Status:** PROPOSED — native `sg-send-vault` TLS does not exist yet. The contract
and a working reference implementation **do** exist in `SGraph-AI__Service__Playwright`
as of the P0 TLS PoC (branch `claude/architect-qa-service-ijc61`).
**Source design:** [`team/roles/architect/reviews/05/14/v0.2.6__vault-app-tls-poc-fastapi-sidecar.md`](../../roles/architect/reviews/05/14/v0.2.6__vault-app-tls-poc-fastapi-sidecar.md) §8.2

---

## Why this lands on your plate

The vault UI needs the **Web Crypto API** (`window.crypto.subtle`). Browsers only
expose it in a **secure context** — i.e. the page must be served over HTTPS.
Today `sg-send-vault` serves plain HTTP, so `window.isSecureContext` is `false`
and Web Crypto is unavailable.

The fleet-wide decision (architecture doc Q6, **resolved**) is: **no reverse proxy
anywhere — every FastAPI app terminates its own TLS.** `sg-send-vault` is one of
those apps. There is no Caddy, no nginx, no sidecar proxy coming to do this for
you. The vault container itself must be able to bind an HTTPS port.

The good news: it is a **uvicorn launch concern**, not route code. No request
handler changes. The change is entirely in how the app's server is started.

---

## The contract (§8.2)

Your launch path reads four environment variables:

| Var | Default | Meaning |
|---|---|---|
| `FAST_API__TLS__ENABLED`   | `false`           | Master switch. Default off. |
| `FAST_API__TLS__CERT_FILE` | `/certs/cert.pem` | Path to the cert file. |
| `FAST_API__TLS__KEY_FILE`  | `/certs/key.pem`  | Path to the key file. |
| `FAST_API__TLS__PORT`      | `443`             | Bind port when TLS is on. |

Required behaviour:

- **off** (unset / `false` / `0` / `no`) → `uvicorn.run(app, host='0.0.0.0', port=8000)`.
  Plain HTTP, exactly as today. **This must stay the default** — Lambda, CI, laptop,
  and Claude Web all run plain HTTP off the same image.
- **on** → `uvicorn.run(app, host='0.0.0.0', port=<PORT>, ssl_certfile=<CERT>, ssl_keyfile=<KEY>)`.
- **on but cert/key file missing** → **fail loud**: assert and exit non-zero. Never
  silently fall back to HTTP. The compose `depends_on` ordering (below) guarantees
  the files exist, so a missing file is a real misconfiguration, not a normal state.

The naming is `FAST_API__TLS__*` (not `SG__*`) deliberately — it matches the
`FAST_API__AUTH__API_KEY__*` convention already in `OSBot__Fast_API` and is
destined to be upstreamed there.

## Reference implementation — copy it, don't reinvent it

We have already built and tested this contract in
`SGraph-AI__Service__Playwright`. It is written single-file on purpose so the lift
into your repo (or into `OSBot__Fast_API`) is mechanical:

| File | What it is |
|---|---|
| `sg_compute/fast_api/Fast_API__TLS__Launcher.py` | Reads the env, builds the config, exposes `config_from_env()`, `assert_ready()`, `uvicorn_kwargs()`, and `serve(app)`. |
| `sg_compute/fast_api/Schema__Fast_API__TLS__Config.py` | The `Type_Safe` config schema. |
| `sg_compute/fast_api/tls/lambda_handler.py` | Example handler: `Fast_API__TLS__Launcher().serve(_app)` — three lines. |
| `sg_compute__tests/fast_api/test_Fast_API__TLS__Launcher.py` | Full unit coverage of the env → kwargs → fail-loud paths. |

**The change in `sg-send-vault` is small:** wherever your container entrypoint
currently calls `uvicorn.run(...)`, replace it with the launcher's `serve(app)`
(or copy the ~70-line launcher in and call it). Everything else — routes, auth,
storage — is untouched.

## How the cert files get there (not your job, but you should know)

You do **not** generate the cert. A **one-shot `cert-init` sidecar** runs at boot,
writes `cert.pem` + `key.pem` into a shared `certs` volume, and exits 0. The
vault container is gated behind it in compose:

```yaml
  sg-send-vault:
    # ...
    environment:
      FAST_API__TLS__ENABLED:   "true"
      FAST_API__TLS__CERT_FILE: /certs/cert.pem
      FAST_API__TLS__KEY_FILE:  /certs/key.pem
      FAST_API__TLS__PORT:      "443"
    volumes:
      - certs:/certs:ro
    depends_on:
      cert-init:
        condition: service_completed_successfully
```

P0 ships self-signed certs (browser warning, fine for the PoC). P2 swaps `cert-init`
to issue a publicly-trusted Let's Encrypt **IP cert** via ACME — **same contract,
same files, same volume.** Nothing in `sg-send-vault` changes between P0 and P2.

The compose/volume wiring is owned by `SGraph-AI__Service__Playwright`
(`Vault_App__Compose__Template`, behind the `--with-tls-check` create flag) — you
just need the container to honour the env contract when it is set.

## Explicitly out of scope — do not build these

- **No reverse proxy, no Caddy, no nginx.** The app terminates its own TLS.
- **No cert generation or ACME in `sg-send-vault`.** The sidecar owns acquisition.
- **No renewal logic / no renewal daemon.** Stacks are ephemeral and shorter-lived
  than any cert; if a cert ever expires under a running stack, that is a bug to
  surface, not a thing to automate around.
- **No `:80` listener / HTTP→HTTPS redirect.** Bind `:443` only.
- **No silent HTTP fallback** when TLS is enabled but files are missing — fail loud.

---

## Acceptance

A stack created with `--with-tls-check` and a TLS-enabled `sg-send-vault`:

1. `sg-send-vault` boots only after `cert-init` has written `/certs`.
2. It binds `:443` with the cert; plain `:8000` is not used.
3. A browser hitting `https://<public-ip>/...` reports `window.isSecureContext === true`
   and `window.crypto.subtle` is defined.
4. `sp vault-app cert show` / `cert check` (already shipped in this repo) decode the
   cert the vault is serving and read back the secure-context verdict.

The same image with `FAST_API__TLS__ENABLED` unset still runs plain HTTP on `:8000`
on Lambda / CI / laptop — the "one image, five targets" guarantee is preserved.

## Open items / decisions still pending sign-off

- **Q7 (DevOps/AppSec):** during ACME acquisition (P2) the challenge port must be
  open to `0.0.0.0/0`. Tighten back to caller-`/32` after the sidecar exits, or
  leave `:443` world-open (the access token is the real gate)? Does not block P0
  and does not affect `sg-send-vault` code either way.
- **Upstreaming:** the launcher is intended to land in `OSBot__Fast_API` so every
  `Serverless__Fast_API` app inherits it. If you would rather wait for that release
  than copy the file, say so — but the fleet needs TLS before that release cycle,
  so copying now and de-duplicating later is the recommended path.
