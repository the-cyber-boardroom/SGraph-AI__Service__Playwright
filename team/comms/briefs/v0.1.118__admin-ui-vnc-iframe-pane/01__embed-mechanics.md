# 01 — Embed mechanics

## The URL is literally the IP

For a stack with `public_ip = 18.175.190.150`:

```html
<iframe src="https://18.175.190.150/"
        title="VNC viewer for vnc-clever-noether"
        allow="clipboard-read; clipboard-write"
        style="border:0; width:100%; height:100%"></iframe>
```

That's it. nginx terminates TLS on 443 and reverse-proxies `/` to the noVNC
endpoint inside the chromium container. Same iframe shape works for
`https://{public_ip}/mitmweb/`.

## Self-signed cert — one-time trust prompt

The cert is generated by the EC2 user-data on first boot:

```
openssl req -x509 -nodes -newkey rsa:2048 -days 365 -subj '/CN=sg-vnc/O=sg-vnc/OU=ephemeral'
```

Browsers will REFUSE to load this in an iframe until the user has visited the
URL in a top-level tab once and clicked through "Visit Site / Proceed". After
that the trust persists for the browser session per-origin.

**Recommended UX:**

1. The pane shows a "VNC viewer" placeholder with a clear button:
   `Open viewer in new tab → trust the cert, then come back`.
   The button hits `https://{public_ip}/` in `_blank`.
2. After the user clicks back to the admin UI, the iframe is mounted.
3. Detect "still untrusted" state by listening for `error` on the iframe (or
   by an empty/zero-size content), and show the placeholder again with a
   one-line hint ("If the viewer is blank, the self-signed cert isn't
   trusted yet — open in new tab and accept the warning").

This is the same one-time dance as `https://localhost:443` style dev work —
once trusted, smooth thereafter for the lifetime of the stack.

## Mixed content

Admin UI must be served over **HTTPS** as well (or the iframe will be blocked
as "active mixed content"). If the admin UI is served over HTTP locally
during dev, the iframe still works — the rule only applies when the parent
is HTTPS and the child is HTTP. Our case is HTTPS → HTTPS → fine, modulo
the trust prompt above.

## X-Frame-Options / CSP

nginx default config in this repo does NOT set `X-Frame-Options` or a
restrictive `Content-Security-Policy: frame-ancestors`. The chromium image's
noVNC endpoint also doesn't set them. So the iframe is allowed by default.
If the admin UI runs on a different origin and noVNC ever starts setting
`SAMEORIGIN`, we'd need to add a `frame-ancestors` allowlist in the
nginx config (`Vnc__User_Data__Builder.NGINX_DEFAULT_CONF`). Out of scope
unless it surfaces.

## Sandboxing

Recommend `sandbox="allow-scripts allow-same-origin allow-forms allow-popups
allow-clipboard-read allow-clipboard-write"` to keep the iframe well-scoped.
Drop `allow-top-navigation` so the embedded chromium can't redirect the
admin UI itself.
