## 1. Goal

Add a second spec to `sg_compute_specs/` that bakes the verified-working "Local Claude Code on EC2 GPU" recipe into a `sp <spec> create / list / info / wait / health / connect / exec / delete / ami ‚Ä¶` CLI surface, on the same tech stack and acceptance bar as `sp ollama`.

Concretely: a single `g5.xlarge` ephemeral instance running a `vllm/vllm-openai` Docker container serving `QuantTrio/Qwen3-Coder-30B-A3B-Instruct-AWQ` on `127.0.0.1:8000`, with Claude Code, sgit, and tmux pre-configured so a user lands on a console, runs one launcher, and is talking to a local agentic-coder model in under three minutes (warm) or ~15 minutes (cold).

This is **the second spec migrated onto `Spec__CLI__Builder`** ‚Äî the Docker spec migration (`v0.2.9__docker-on-builder.md`) is the first. Two specs on the Builder is the test of the contract's generality.

## 2. What this spec adds

Capabilities not currently in any sg-compute spec:

| Capability | Notes |
|---|---|
| GPU container runtime via NVIDIA Container Toolkit | `Section__Docker` exists; we need a `Section__NVIDIA_Container_Toolkit`. Ollama runs the daemon natively ‚Äî vLLM runs in Docker and needs the toolkit. |
| `vllm/vllm-openai` Docker run with the verified flag set | Pinned digest, `--tool-call-parser qwen3_coder`, `--kv-cache-dtype fp8`, `--max-model-len 65536`, `--gpu-memory-utilization 0.92`, `--enable-auto-tool-choice`, `--max-num-seqs 1`. |
| HF model cache pre-warm / pre-bake | The 16 GiB model is a first-class artefact. Pre-bake into the AMI for ~3-min warm-recovery; first-boot download as fallback. |
| Claude Code first-boot install | Per `05-ami-publishing.md`, Claude Code is **not** bundled (license-clean). A systemd oneshot runs `curl https://claude.ai/install.sh \| bash` once, marker file at `/var/lib/claude-code-installed`. |
| `~/.claude/settings.json` env block | `CLAUDE_CODE_ATTRIBUTION_HEADER=0`, `CLAUDE_CODE_DISABLE_1M_CONTEXT=1`, `CONTEXT_WINDOW_OVERRIDE=65536`, `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=70`. **These do not work via `export`** ‚Äî must live in the JSON. |
| sgit venv | `python3.12 -m venv ~/claude-session-venv && pip install sgit`. The persistent-memory story for the ephemeral instance. |
| tmux + launcher script | `~/local-llm-claude.sh` configures the env vars and `exec claude`. tmux session `claude` is the user-facing entry point. |

Critically, this spec **is not** "vLLM as a hosted API" ‚Äî it is **single-user, SSM-only, console-driven**. Port 8000 binds to `127.0.0.1`. There is no `--expose-api` equivalent in scope. Anyone needing remote API can either ssh-tunnel or wait for a future `vllm-api` spec.

## 3. Three candidate names

| Candidate | Pros | Cons |
|---|---|---|
| **`vllm`** | Symmetric with `ollama` ‚Äî both name the inference engine. Stable axis: models/parsers/clients change, the engine is the orchestrated artefact. Future-extensible (different `--model` / `--tool-parser` combinations could share the spec). License-clean ‚Äî no Anthropic-product overlap in the spec id. Matches the v0.2.6 contract: spec_id = engine, the recipe lives in defaults + extras. | Vague about the *opinionated recipe* ‚Äî a future "vllm + Llama-3 + hermes parser" stack would either fork this spec or override every default. |
| **`claude-coder`** | Most descriptive of the user-facing capability: a Claude-Code-driven AI coder workstation. The brief itself frames the whole stack around Claude Code. Matches the Qwen3-Coder model name. | Hyphenated spec_ids untested in the Builder (Ollama uses a single token); risks Python identifier-safety issues per CLAUDE.md rule 20 (`Cli__Claude_Coder` etc. requires underscore mapping). Couples the spec name to Anthropic's product, which the AMI-publishing doc explicitly warns against in marketing copy. |
| **`local-claude`** | Mirrors the brief's filename `for-new-stack__claude-offline` and the directory naming the user already uses. Captures the recipe-level identity. | "claude" alone is broad and may collide with future Claude-related specs (claude-batch, claude-agent-sdk-host, etc.). "local" is also broad. Same hyphen-mapping concern. |

### Recommendation: **`vllm`**

Reasons:

1. **Symmetry with `ollama` is the dominant signal.** Both are `LLM_INFERENCE` capability specs serving a local model on a single GPU box. Naming both after their engine keeps the catalogue legible: a user looking at the catalogue sees two LLM-engine choices and knows what they're picking between.
2. **The engine is the most stable axis.** Models, parsers, clients, env-var combinations all churn. The engine container is what the spec actually orchestrates.
3. **License-clean.** The AMI publishing doc is explicit that the spec/AMI must not market Claude Code as bundled. Naming the spec `claude-*` makes that distinction harder to hold in CLI help, tags, and the future Marketplace listing.
4. **Future-proof for variants.** "Different model + different parser" can land as `--model` / `--tool-parser` flags, or ‚Äî if the recipes diverge enough ‚Äî as `vllm-coder` / `vllm-chat` peer specs. Reserving `vllm` as the engine root keeps that namespace open.
5. **Contract fit.** Per v0.2.6, `spec_id` is the value that flows into the SG tag (`sg-compute-spec=vllm`), the CLI subtree (`sp vllm ‚Ä¶`), the Fast_API route (`/api/specs/vllm`), and the AMI tag (`sg-compute-spec=vllm`). All of those read more naturally with the engine name.

If the user prefers a more capability-flavoured name, **`claude-coder`** is the strongest alternative and the rest of this brief works unchanged with that substitution.

Inherited from `Spec__CLI__Builder` for free:

`extra_create_options` on the Builder:

| Option | Type | Default | Notes |
|---|---|---|---|

|---|---|---|

| `Section__Docker` | Docker CE, `usermod -aG docker ssm-user` | unchanged |
| `Section__Agent_Tools` | python3.13 venv + helpers | reuse; the spec's separate sgit venv is python3.12 because that's what the brief specifies |
| `Section__Sidecar` | optional Fast_API host_plane sidecar | reuse, default OFF for v1 |

New sections needed for this spec:

| Section | Renders |
|---|---|
| `Section__NVIDIA_Container_Toolkit` | adds the NVIDIA repo, installs `nvidia-container-toolkit`, `nvidia-ctk runtime configure --runtime=docker`, restarts docker. ~10 lines. |
| `Section__VLLM` | docker rm-f / docker run with the full pinned-digest flag set, including `127.0.0.1:8000` bind, `--ipc=host`, `--restart unless-stopped`, HF cache mount. Until-loop on `/v1/models` for readiness. |
| `Section__SGit_Venv` | `python3.12 -m venv ~/claude-session-venv && pip install sgit`. Optional. |
| `Section__Claude_Code__Firstboot` | writes the systemd `claude-code-firstboot.service` unit; enables but does not start (it runs on first boot of every AMI-launched instance). Also writes `~/.claude/settings.json` with the env block and the `~/local-llm-claude.sh` launcher. |

Order (Section__Shutdown second, per L9):

```
Base ‚Üí Shutdown ‚Üí GPU_Verify ‚Üí Docker ‚Üí NVIDIA_Container_Toolkit
     ‚Üí SGit_Venv ‚Üí Claude_Code__Firstboot ‚Üí VLLM ‚Üí Sidecar
```

VLLM is last among the "real work" sections because it pulls the 16 GiB model weights. Failure during VLLM does not strand the instance because Section__Shutdown registered the timer at position 2.

## 7. AMI strategy

Three modes, all expressible through the existing v0.2.8 AMI surface:

| Mode | What's baked | Boot time | Notes |
|---|---|---|---|
| **Cold** (no `--ami`) | nothing | ~15 min | Full firstboot: dnf, NVIDIA toolkit, vLLM image pull, model download, Claude Code install. |
| **Engine-baked** | Docker, NVIDIA toolkit, vLLM image (digest-pinned) | ~5 min | Model still downloads on first boot. Smaller AMI, easier to ship across regions. |
| **Full-bake** | engine-baked + 16 GiB model in HF cache | ~3 min | The headline mode. `~/.cache/huggingface` survives stop/start on the EBS volume, so spot-reclaim recovery is fastest. |

`sp vllm ami bake <stack>` produces the **full-bake** AMI from a running stack. The bake script in `05-ami-publishing.md` is the reference for what the cleanup phase must scrub:

- `/etc/ssh/ssh_host_*` (security: per-instance keys)
- `/var/lib/cloud/instances/*`, `/var/lib/cloud/instance`
- `/etc/machine-id` ‚Üí empty file
- `~/.bash_history`, `~/.cache/sgit`, `~/.config/sgit`, any `.sg_vault` dirs
- `~/.claude/.credentials*`, `~/.claude/projects`, `~/.claude/conversations`
- `journalctl --vacuum-time=1s`
- secrets-scan grep over `/home/ssm-user`, `/etc`, `/root`

The cleanup is **out of scope for the Builder's `ami bake`** ‚Äî bake produces a snapshot of the live instance, including any session state. Instead:

- Ship a `sp vllm ami prebake-cleanup [stack]` extra that runs the cleanup script over SSM before snapshotting.
- Or: document that AMIs intended for redistribution must be baked from a fresh instance that has never been used interactively.

The first option is more user-friendly and license-safer. Recommend implementing it as part of the v0.2.10 plan.

## 8. IAM, networking, security

Same posture as Ollama:

- Instance profile: `playwright-ec2` (SSM + ECR access). **Required** ‚Äî without it, `aws ssm start-session` fails. (Ollama L4 lesson.)
- Security group: outbound 443 only. **No inbound.** Port 8000 binds to `127.0.0.1` and is never exposed via SG rule.
- SG group name: `vllm-{stack}` (or `{stack}-sg`) ‚Äî never `sg-*` per CLAUDE.md rule 14.
- Caller IP detection: reuse `Caller__IP__Detector`. The brief's recipe doesn't strictly need it (no inbound rules), but the SG-creation flow in the Ollama service expects a caller IP for tagging. Keep the same shape.

## 9. Differences vs Ollama (cheat sheet for the implementer)

| Axis | Ollama | vLLM (this spec) |
|---|---|---|
| Engine | Native daemon (`curl install.sh \| sh`) | Docker container (`vllm/vllm-openai`) |
| GPU access | Direct on host | Via `nvidia-container-toolkit` |
| Health | `http://:11434/api/tags` | `http://127.0.0.1:8000/v1/models` |
| Model store | `~/.ollama/models` (ext4) | `~/.cache/huggingface/hub` (HF) |
| Model size | gpt-oss:20b, ~14 GiB | Qwen3-Coder-30B-A3B-AWQ, ~16 GiB |
| API exposure | `--expose-api` flag binds 0.0.0.0:11434 | **No exposure option in v1** ‚Äî SSM-only |
| Default disk | 250 GiB | 200 GiB (per brief) |
| Default AMI base | DLAMI (GPU drivers preinstalled) | AL2023 (drivers come from container toolkit) |
| Claude Code on the box | `--with-claude` boots tmux, `ollama run <model>` | First-boot install via systemd, env-driven launcher, full settings.json |
| Persistence story | None | sgit venv pre-installed |
| `--max-num-seqs` | not exposed | hardcoded `1` (single user, max KV budget) |

Why **AL2023 not DLAMI**: the brief uses AL2023 + container toolkit because the entire GPU surface is owned by the vLLM container. No driver/userspace clash with the host. This also keeps the AMI smaller for redistribution. If a user later wants DLAMI, add the `Enum__VLLM__AMI__Base.DLAMI` value in v0.2.11 ‚Äî symmetric with Ollama's helper.

## 11. Acceptance gate (non-negotiable)

Per the v0.2.8 Docker plan, every Builder-using spec PR must pass live AWS smoke before merge. For this spec, the acceptance script is:

1. `sp vllm create --region us-east-1 --max-hours 1` ‚Üí response contains the auto-generated stack name.
2. `sp vllm wait <name> --region us-east-1` ‚Üí returns healthy within ~15 min cold (or ~3 min from full-bake AMI).
3. `sp vllm models <name>` ‚Üí returns `local-coder` in the model list.
4. `sp vllm connect <name>` ‚Üí SSM session opens; user can run `~/local-llm-claude.sh` and the canonical `Use the Bash tool to run: pwd` test passes (real `Bash(pwd)` line in the output, not JSON-as-content).
5. `sp vllm ami bake <name> --wait` ‚Üí produces an available AMI (~10‚Äì15 min for the 200 GiB volume).
6. `sp vllm create --ami <id> --region us-east-1 --max-hours 1` ‚Üí fast-boot path returns healthy in <5 min.
7. `sp vllm delete <name>` ‚Üí terminates and confirms SG cleaned up.
8. Verify in the AWS console: spot termination on `shutdown -h now`, billing tags present, no stranded snapshots.

The L1‚ÄìL8 lessons table from the 2026-05-10 debrief applies in full:

- L1 (`__init__` no-op) ‚Äî the Builder already fixes this; verify on a CLI invocation.
- L2 (`health_scheme`) ‚Äî set to `'http'` (not https; vLLM doesn't TLS-terminate).
- L3 (poll inside deadline) ‚Äî inherited from `Spec__Service__Base.health()`.
- L4 (IAM profile) ‚Äî `PROFILE_NAME='playwright-ec2'`.
- L5 (`disk_size_gb` wiring) ‚Äî pass through to `EC2__Launch__Helper.run_instance`.
- L6 (dual-flag bools) ‚Äî Builder auto-emits.
- L7 (`ssm-user` in docker group) ‚Äî `Section__Docker` already does this.
- L8 (no empty `__init__.py` shadowing) ‚Äî direct module imports only; `provision_ec2.py` style.
- L9 (Section__Shutdown at position 2) ‚Äî verified in `User_Data__Builder.parts`.

## 12. What's intentionally out of scope for v1

- **Multi-GPU / sharding.** The brief explicitly excludes it. Stays single-GPU, `--max-num-seqs 1`.
- **`--expose-api` flag.** No remote API exposure in v1. Future spec can add it with the same SG-scoped pattern Ollama uses.
- **LiteLLM proxy.** vLLM has native Anthropic Messages support; LiteLLM is an extra hop the brief explicitly removes.
- **Open WebUI.** Not bundled, console-only.
- **Marketplace listing.** Tier 3 from the brief is a separate work-stream ‚Äî not blocking the spec landing.
- **Other clients (Codex, Cursor, Cline).** The recipe is Claude-Code-specific. Other clients may work but aren't smoke-tested.
- **Cross-region fast-launch.** Each region needs its own AMI bake. Multi-region copy is a v0.3.x story.

## 15. Open questions

1. **Spec name confirmation.** Recommendation: `vllm`. Alternatives: `claude-coder`, `local-claude`. **Decision needed before any code lands.**
2. **`Section__Claude_Code__Firstboot` placement.** Reusable across specs ‚Äî should live under `sg_compute/platforms/ec2/user_data/` not under `sg_compute_specs/vllm/`. Architect sign-off.
3. **`Enum__Spec__Capability.AGENTIC_CODER` ‚Äî yes/no?** If no, the spec ships under `LLM_INFERENCE` and the agentic-coder framing is just nav-group + display copy.
4. **Pre-bake cleanup as a CLI verb** (`sp vllm ami prebake-cleanup`) vs. a manual SSM script run by the user. Recommendation: ship the verb in v1, makes the AMI pipeline reproducible.
5. **`served_model_name` is hardcoded `'local-coder'` in the brief.** It must match what's exported as `ANTHROPIC_MODEL` in the launcher. If the user wants to override the model later, both have to move together. Probably fine to keep as a Builder extra with that default.
6. **Default region.** `Spec__CLI__Defaults.DEFAULT_REGION` is `eu-west-2`. The user's claude+vllm work is in `us-east-1`. Don't change the default; document that `--region us-east-1` is the live-test region for this spec.
