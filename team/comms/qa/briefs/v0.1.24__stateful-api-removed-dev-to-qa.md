## TL;DR

**Breaking change, pre-GA.** The stateful session API is deleted. `/session/*` is gone. `/quick/*` is gone. `/browser/*` is reborn as six stateless one-shot endpoints that each launch a fresh Chromium, run one action, tear down. Every `/browser/*` body accepts an optional `browser_config` (with proxy — Bug #1 fix applies everywhere, not just `/sequence/execute`). Endpoint count: 25 → 10.

Your acceptance work:

1. Re-verify Bug #1 (proxy auth) against the new one-shot surface — the same mitmproxy probe against `POST /browser/get-content` instead of the old `/session/create` + `/browser/navigate` pair.
2. Confirm that two back-to-back requests to the same warm container no longer share state (Lambda cold/warm mixed traffic).
3. Confirm `/quick/*` 404s cleanly (route genuinely removed, not just returning 500).
4. Re-run your 2026-04-17 proxy-testing suite against `/sequence/execute` with `session_id` + `close_session_after` removed — both fields are gone from the schema.

---

## Why this happened

Your v0.1.23 proxy-testing debrief flagged (§Bad failures) that the session API was "a grenade in a Lambda box — warm containers hold state across invocations". The 404-session-not-found bug (fixed in v0.1.23) was the first symptom. Rather than paper over it with cross-container state, we pulled the API out.

Decision thread: user's message in this session — *"this state design doesn't make sense in a lambda function right? specially since we want all requests to lose all state and always start from scratch (for a number of reasons, including security)"*. Agreed. Removed.

## New /browser/* surface — exhaustive

Every endpoint is `POST`. Every body has the same shape template:

```json
{
  "url": "https://example.com",
  "browser_config": { ... optional; proxy lives here ... },
  "wait_until": "load",
  "timeout_ms": 30000,
  ...action-specific fields...
}
```

| Endpoint                   | Required body fields   | Optional body fields                                          | Response                        |
|----------------------------|------------------------|---------------------------------------------------------------|---------------------------------|
| `/browser/navigate`        | `url`                  | `browser_config`, `wait_until`, `timeout_ms`                  | `Schema__Browser__One_Shot__Response` (JSON) |
| `/browser/click`           | `url`, `selector`      | `browser_config`, `wait_until`, `timeout_ms`                  | JSON                            |
| `/browser/fill`            | `url`, `selector`, `value` | `browser_config`, `wait_until`, `timeout_ms`              | JSON                            |
| `/browser/screenshot`      | `url`                  | `selector`, `click`, `full_page`, `browser_config`, `wait_until`, `timeout_ms` | `image/png` + `X-*-Ms` headers |
| `/browser/get-content`     | `url`                  | `click`, `browser_config`, `wait_until`, `timeout_ms`         | JSON incl. `html`, `final_url`  |
| `/browser/get-url`         | `url`                  | `click`, `browser_config`, `wait_until`, `timeout_ms`         | JSON incl. `final_url`          |

`Schema__Browser__One_Shot__Response` — canonical JSON body:

```json
{
  "url": "https://example.com",
  "final_url": "https://example.com/",
  "duration_ms": 1247,
  "timings": {
    "playwright_start_ms": 120,
    "browser_launch_ms": 680,
    "steps_ms": 340,
    "browser_close_ms": 107,
    "total_ms": 1247
  },
  "content": "...HTML or null...",
  "error_message": null
}
```

Proxy usage (Bug #1 fix, applies on every endpoint):

```json
{
  "url": "https://api.ipify.org/",
  "browser_config": {
    "proxy": {
      "server": "http://mitmproxy.dev.example:8080",
      "ignore_https_errors": true,
      "auth": { "username": "qa-user", "password": "qa-pass-x7" }
    }
  }
}
```

## What we need from QA

### 1. Bug #1 regression on the new surface (blocks v0.1.24 sign-off)

Re-run your 2026-04-17 authenticated-proxy probe against `POST /browser/get-content` (the direct analogue of `/quick/html`) with the proxy body above. The Bug #1 fix was wired through `Proxy__Auth__Binder`, which now runs on every `/browser/*` call via the shared `Sequence__Runner` path. Pass = correct creds navigate in <2 s, wrong creds fail fast (<10 s, distinct error surface).

**Also test:** `POST /browser/screenshot` with proxy auth. That endpoint goes through the same runner, but the response is raw bytes — confirm the `X-*-Ms` timing headers still come through under proxy conditions.

### 2. Warm-container state-leak regression

Simulate: on a deployed Lambda, two back-to-back requests from different clients. Previously this was:

```
Client A → POST /session/create                → session_id=A
Client A → POST /browser/navigate {session_id: A}
Client B → POST /session/create                → session_id=B (maybe same warm container)
Client B → POST /browser/navigate {session_id: B}
```

State leak risk: Client A's cookies / proxy creds in the Playwright process when Client B's request arrives.

New surface:

```
Client A → POST /browser/get-content {url: ...A-login...}
Client B → POST /browser/get-content {url: ...B-login...}
```

Every request launches a fresh Chromium process. Your test: confirm that if Client A logs in (cookies set) and Client B requests the same origin, Client B sees no logged-in state. No `Set-Cookie` leakage, no `document.cookie` carryover.

The fresh-per-call design makes this structurally safe — but test it empirically once, and lock it down as a deploy-via-pytest assertion.

### 3. `/quick/*` + `/session/*` 404 confirmation

```
POST /quick/html      → 404 (not 500)
POST /quick/screenshot → 404 (not 500)
POST /session/create  → 404
GET  /session/list    → 404
...
```

FastAPI emits 404 for unknown routes by default; this is just a sanity check that the removals happened cleanly. If any of these come back 200 or 500, the route didn't get unregistered properly.

### 4. `/sequence/execute` schema change

`Schema__Sequence__Request` loses two fields:
- `session_id` (was `Session_Id = None`)
- `close_session_after` (was `bool = True`)

If your probes send either field, Pydantic will either ignore or reject depending on extras config. Update the probes to drop both fields. The request still takes `browser_config` + `steps` + `capture_config` + `sequence_config` + `trace_id`.

The v0.1.23 `resolve_session` reorder fix (commit `90db20d`) is now dead code — removed in v0.1.24. The `test__http_autogen_session_id_without_browser_config_raises_422` regression test has been rewritten as `test__sequence_request_requires_browser_config` (same intent, different schema).

### 5. Capabilities endpoint

`GET /health/capabilities` no longer reports `supports_session_lifecycle` or `max_concurrent_sessions`. New fields:

- `supports_one_shot_browser_actions: bool` (always true on this version)
- `supports_proxy_auth_basic: bool` (carried from v0.1.23)

If your probes assert on old keys, update them.

## Open caveats

- **No CI integration test for one-shots yet** — same gap as v0.1.23. Local-only L-tier tests cover the code, but CI does not exercise the stateless one-shot flow against a real proxy. Same mitmproxy-sidecar ask as before — QA owns the test spec, DevOps owns the runner wiring.
- **Warm-browser pool not added** — if cold-start latency (200–400 ms per request) becomes a problem, a per-container pool can be added without breaking the wire contract. Not needed yet.
- **`Schema__Session__Credentials` survives** — still used for vault-glue on `/sequence/execute`. Not renamed in this slice to keep the diff small; candidate for `Schema__Vault__Credentials` rename in a follow-up.
