# v0.1.96 — Licensing & AWS Marketplace Assessment

**Series:** 8 of 8 (final).
**Status:** RESEARCH — requires legal review before any AMI is published to AWS Marketplace.

This file enumerates the open-source components shipped in each AMI, classifies each by license, and assesses AWS Marketplace eligibility per AMI. **It is a starting point for legal review, not a legal opinion.** Final marketplace-eligibility decisions must be made with counsel.

---

## Why this matters

Decision #8 in [doc 1](v0.1.96__playwright-stack-split__01__overview.md) commits every EC2/AMI to be fully self-contained — no runtime fetches from S3, vault, or Parameter Store. Combined with the clean-slate refactor, this is a one-time opportunity to make each AMI **distributable**, including (where licensing permits) on the AWS Marketplace.

The operator already flagged that **Elasticsearch + Kibana cannot be placed on AWS Marketplace** since Elastic re-licensed under SSPL/ELv2 in 2021. This doc maps the same question across every other component we ship.

---

## License classification — quick reference

| Class | Examples | Marketplace eligible? |
|---|---|---|
| **Permissive** (Apache 2.0, BSD-2/3, MIT, ISC, MPL 2.0) | Apache, OpenSearch, Prometheus, mitmproxy | **Yes**, with attribution. |
| **Weak copyleft** (LGPL 2.1, LGPL 3.0) | WebKit (in Playwright), some libs | **Usually yes** — dynamic linking is fine; static linking triggers source disclosure. |
| **Strong copyleft** (GPL v2, GPL v3, AGPL v3) | Some kernel modules, certain GUI tools | **Case by case** — typically permitted for the OS layer; AGPL is the riskiest. |
| **Source-available, non-OSS** (SSPL, ELv2, BSL) | Elasticsearch ≥7.11, Kibana ≥7.11, MongoDB | **No** — explicitly excluded by AWS Marketplace seller policy. |
| **Commercial / proprietary** | Microsoft Playwright base image | **Depends** — usually requires the upstream vendor's redistribution agreement. |

## Recommendations

1. **Three AMIs are clean today**: `sp os`, `sp prom`, and the OS layer. These can be marketplace-targeted from the first bake.
2. **One AMI is conditional on a vendor check**: `sp ami` (Playwright) needs a Microsoft container-image-terms review. If the answer is "redistribution requires our agreement", we either get the agreement, build our own Playwright base image from source (Apache 2.0), or scope `sp ami` out of marketplace.
3. **One AMI is conditional on a swap**: `sp vnc ami`. The fastest path to clean marketplace eligibility is replacing `linuxserver/chromium` with a permissive-only image. Doc 5 P6 already deferred this work; if marketplace listing is on the roadmap, P6 should be re-prioritised for `sp vnc`.
4. **One AMI is permanently ineligible**: `sp el ami`. Document this clearly in the section's README so contributors don't waste cycles.
5. **Embed third-party licenses in every AMI** under `/usr/share/licenses/` (or equivalent) — required by attribution-style licenses (Apache 2.0, BSD, MIT) regardless of marketplace status. The `sp {section} ami create` flow should automate this from a manifest.

## Open questions (legal / business — not technical)

1. **Microsoft Playwright base image redistribution** — does the existing Microsoft Container Image Terms permit AMI distribution on AWS Marketplace? Needs a one-time legal answer.
2. **Marketplace seller registration** — is this org already registered as an AWS Marketplace seller, or is that a separate workstream?
3. **Pricing model** — BYOL / hourly / annual? Affects how the AMI is packaged but not which components ship.
4. **Branding / naming on Marketplace** — "SG Playwright" vs a parent product name. Out of scope for engineering, but the AMI metadata (`Name` tag, description) is set by `sp {section} ami create`, so the operator should pick conventions before the first bake.
5. **GPL source-availability** — if `sp vnc ami` ships with GPL components, do we host the source bundle ourselves or point at upstream? Affects ops, not licensing legality.

## How this gets implemented

This doc is **research, not a build task**. The legal review it triggers may produce constraints that ripple back into doc 4–6 (e.g. "swap `linuxserver/chromium` to a permissive image before any AMI bake"). Until those answers come back, treat marketplace eligibility as a future-state goal, not a current-phase deliverable.

The engineering hooks are minimal:
- AMI-bake flow embeds a generated `LICENSES.txt` per section.
- Each section's README declares its marketplace status (one of: ✅ eligible / ⚠ conditional / ❌ ineligible) so it's discoverable without re-reading this doc.
