## 1. Drivers

Three forces push this reorganisation, in order of priority:

1. **API independence.** The Playwright API is generic Python code — it should run anywhere Playwright is installed (laptop, CI, Claude Web, container). Today it's entangled with `lambda_handler.py`, the docker build classes, and the S3-zip boot logic. Clean separation means `api/` knows nothing about S3, zip, Lambda, or the image.
2. **Always-up admin surface.** When the user-code FastAPI fails to load (bad zip, missing env var, import error), operators need a diagnostic endpoint that's still up. Today the container dies and the Function URL returns an empty body — impossible to debug without CloudWatch. The container should ship its OWN thin FastAPI that's always listening, with the user FastAPI mounted on top when healthy.
3. **Generic code-loader.** The "load Python code from an S3 zip on Lambda boot" pattern is not Playwright-specific — only the base image is. Once the loader lives in a package that doesn't depend on `sgraph_ai_service_playwright`, it can be extracted to `osbot_aws` and reused by every future Lambda in the estate.

## 2. Package boundaries

Two installable packages, each with its own `pyproject.toml`. No circular dependency.

### `api/` — the Playwright service

- **Contents:** `sgraph_ai_service_playwright/` (the package), `tests/unit/`, `tests/integration/`, `tests/local/`, `scripts/package_code.py`.
- **Installable as:** `sgraph-ai-service-playwright` (unchanged distribution name).
- **Depends on:** `osbot-utils`, `osbot-fast-api-serverless`, `playwright`, `osbot-aws` (only for `S3__Zip_Bytes` used by `scripts/package_code.py`).
- **MUST NOT import from:** `container.*`.
- **Entry points:**
  - In-process: `Fast_API__Playwright__Service().setup().app()` (unchanged).
  - Via container: the boot shim loads this package onto `sys.path` and mounts its FastAPI.
  - Laptop/CI: `uvicorn api.sgraph_ai_service_playwright.fast_api.Fast_API__Playwright__Service:app` works with no container, no boot shim.

### `container/` — the runtime chassis

- **Contents:** `Dockerfile`, `lambda_entry.py` (boot shim), `image_version`, `sg_lambda_runtime/` (Python package with the loader + admin FastAPI), `docker/` (Build/ECR/Lambda/Base/Local orchestration classes), `tests/`.
- **Installable as:** `sg-lambda-runtime` (new distribution).
- **Depends on:** `osbot-utils`, `osbot-aws`, `osbot-fast-api-serverless`.
- **MUST NOT import from:** `api.*` or `sgraph_ai_service_playwright.*`. The boot shim loads the user package dynamically via `sys.path.insert`; it never imports it at module level.
- **Entry point:** `python3 lambda_entry.py` (the Dockerfile CMD).

### Shared at repo root

- `pyproject.toml` — a thin workspace root that declares both packages as dev-installable via `pip install -e api -e container`.
- `requirements.txt` — shared lock. Each package's own `pyproject.toml` re-declares its real deps; `requirements.txt` is the pinned union used by CI.
- `.github/workflows/`, `.claude/`, `library/`, `team/`, `README.md` — neutral.

## 4. S3 path scheme

### Old (v0.1.28, shipped but brittle):
```
s3://{account}--sg-playwright--{region}/{lambda_name}/code/v{X.Y.Z}.zip
```
The `code/` segment is a dead tier, and there's no separation of dev/main/prod for the same lambda.

### New (v0.1.29):
```
s3://{account}--sg-playwright--{region}/lambdas/{lambda_name}/{stage}/v{X.Y.Z}.zip
```

Examples:
```
s3://745506449035--sg-playwright--eu-west-2/lambdas/sg-playwright-api/dev/v0.1.29.zip
s3://745506449035--sg-playwright--eu-west-2/lambdas/sg-playwright-api/main/v0.1.29.zip
s3://745506449035--sg-playwright--eu-west-2/lambdas/sg-playwright-api/prod/v0.1.29.zip
```

### Why this layout

- **`lambdas/` prefix** — leaves room for future peer tiers (`scripts/`, `layers/`, `datasets/`) without re-partitioning.
- **`{lambda_name}`** — the logical name. One lambda can run multiple stages. Name is NOT the AWS function name (which is stage-suffixed); it's the application identity.
- **`{stage}`** — `dev`, `main`, `prod`. Same code can be pinned at different versions per stage. Rollback = flip the Lambda's `SG_PLAYWRIGHT__CODE_S3_VERSION` to any previously-uploaded `v{X.Y.Z}.zip` under that stage's folder.
- **`v{X.Y.Z}.zip`** — immutable. Never overwritten. Never a `latest.zip`.

### Env vars (Lambda-side)

Three env vars, all required on any stage Lambda:

| Env var | Value | Example |
|---|---|---|
| `SG_PLAYWRIGHT__LAMBDA_NAME` | Logical name | `sg-playwright-api` |
| `SG_PLAYWRIGHT__CODE_S3_STAGE` | Stage | `dev` |
| `SG_PLAYWRIGHT__CODE_S3_VERSION` | Pinned version | `v0.1.29` |

Boot-shim builds the key: `lambdas/{name}/{stage}/{version}.zip`.

### Generic rename (deferred to OSBot_AWS extraction)

When the loader is extracted to `osbot_aws`, the env var prefix becomes project-neutral:
- `SG_PLAYWRIGHT__LAMBDA_NAME` → `OSBOT__LAMBDA_NAME`
- `SG_PLAYWRIGHT__CODE_S3_STAGE` → `OSBOT__CODE_S3_STAGE`
- `SG_PLAYWRIGHT__CODE_S3_VERSION` → `OSBOT__CODE_S3_VERSION`

For v0.1.29 we keep the `SG_PLAYWRIGHT__` prefix for continuity. The rename is a later mechanical pass, not part of this plan.
