# v0.1.96 — `sp prom` — Prometheus Sister Section

**Series:** 5 of 7.
**Status:** PROPOSED.
**Template:** `sgraph_ai_service_playwright__cli/elastic/`.

This file specifies the new `sp prom` section: an EC2-hosted Prometheus + cAdvisor + node-exporter stack that scrapes the Playwright EC2 (and any other targets we point it at).

---

## Why a section, not a Lambda / managed AMP

- **AMP (Amazon Managed Prometheus)** is the right long-term home but adds IAM-signing complexity to every scraper. The v0.1.72 brief already plans for AMP via the `observability` section.
- **`sp prom`** is the lightweight, single-EC2 alternative — same pattern as `sp el` and `sp os`, used for dev/CI stacks where AMP is overkill.
- Both can coexist; targets choose which to remote-write to.

## CLI commands (under `sp prom`)

| Command | Purpose |
|---|---|
| `sp prom create [name]` | Provision an EC2 running Prometheus + cAdvisor + node-exporter. |
| `sp prom list` | List `sg:purpose=prometheus` stacks. |
| `sp prom info <name>` | Tag metadata + live `/-/healthy` + target count. |
| `sp prom delete <name>` | Terminate. |
| `sp prom connect <name>` | SSM shell. |
| `sp prom exec <name> <cmd>` | Run via SSM. |
| `sp prom health <name>` | Health check. |
| `sp prom wait <name>` | Poll until `/api/v1/targets` returns. |
| `sp prom forward <name> [--port 9090]` | Local port-forward via SSM. **Replaces** the old top-level `sp forward-prometheus`. |
| `sp prom targets <name>` | List configured scrape targets (from tags + config). |
| `sp prom add-target <name> --target <url> --job <name>` | Add a target; regenerate `prometheus.yml`; reload. |
| `sp prom query <name> '<promql>'` | Run an instant PromQL query. |
| `sp prom metrics <url> [--key <api-key>]` | Fetch metrics text from any target. **Replaces** the old top-level `sp metrics`. |

## FastAPI routes

```
POST   /v1/prometheus/stack                         create
GET    /v1/prometheus/stacks                        list
GET    /v1/prometheus/stack/{name}                  info
DELETE /v1/prometheus/stack/{name}                  delete
GET    /v1/prometheus/stack/{name}/targets          list scrape targets
POST   /v1/prometheus/stack/{name}/targets          add a target
POST   /v1/prometheus/stack/{name}/query            ?query=<promql>  → forwards to /api/v1/query
GET    /v1/prometheus/stack/{name}/health
```

`query` and `metrics` exposing PromQL through the API enables Grafana-less dashboards and lets the Playwright service (or any caller) pull live metrics without a port-forward.

## What moves from the Playwright compose

Three containers + their volumes:

| Container | Source | New location |
|---|---|---|
| `prometheus` | `provision_ec2.py:294–308` (`COMPOSE_SVC_PROMETHEUS`) | `prom/compose/fragments/prometheus.service.yaml` |
| `cadvisor` | `provision_ec2.py:263–275` (`COMPOSE_SVC_CADVISOR`) | `prom/compose/fragments/cadvisor.service.yaml` |
| `node-exporter` | `provision_ec2.py:277–292` (`COMPOSE_SVC_NODE_EXPORTER`) | `prom/compose/fragments/node-exporter.service.yaml` |

The `prometheus.yml` config is no longer hardcoded in the Playwright compose; it's generated by `Prometheus__Config__Generator` from a list of `Schema__Prom__Scrape__Target`. The Playwright EC2 becomes one such target — added to a `sp prom` stack via `sp prom add-target`.

## Scraping the Playwright EC2

After the strip, Playwright still emits Prometheus metrics on `/metrics` (the FastAPI app, via `prometheus-client`). The flow becomes:

```
Playwright EC2 ──── /metrics ────►  sp prom EC2 (Prometheus scrape)
                                   ▲
                        sp prom add-target --target http://<playwright-ip>:8000/metrics --job playwright
```

Allowlist the `sp prom` SG-IP on the Playwright host's port 8000.

## Open questions

1. **Grafana** — bundle it into `sp prom` for an instant dashboard story, or keep it out for now (operator runs Grafana locally / in `sp el`)? Bundling adds ~80 MB and a UI auth surface; not bundling means an extra step but a cleaner stack.
2. **Retention** — default to 7 d on EBS, or 24 h? Aligns with v0.1.72's "ephemeral observability" principle.
3. **Multiple Playwright targets** — does `sp prom` support multi-target out of the box, or one stack-per-Playwright? Multi-target is cheap, but the `add-target` UX becomes the operator's main workflow.
4. **Federation** — once AMP is real, does `sp prom` push to AMP via remote-write? Probably yes; out of scope for this plan.
