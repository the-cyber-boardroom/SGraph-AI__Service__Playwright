- N5 ✅ — Interceptor model is **default-off + ship examples + provision-time choice via env vars**:
  - **Default:** mitmproxy starts with no interceptor.
  - **Examples baked in:** `vnc/mitmproxy/interceptors/examples/` ships a small library of reference scripts (e.g. `header_logger.py`, `header_injector.py`, `flow_recorder.py`).
  - **At `sp vnc create` time, the operator picks one of:**
    - `--interceptor <example_name>` — load one of the baked examples by name.
    - `--interceptor-script <local_path>` — read a local Python file; its source is embedded in the EC2 user-data and written into the container as the active interceptor.
    - neither — start with no interceptor.
  - This pattern stays inside the self-contained AMI principle (no runtime fetches) while keeping the section flexible. Same model can be retrofitted to `agent_mitmproxy` (the headless-API mitmproxy in `sp`) in a follow-up if desired.

This file specifies `sp vnc` — the human-debug "streaming browser" stack. A separate EC2 instance running a Chromium-with-VNC container fronted by nginx and routed through its **own** mitmproxy. This is the "operator looks at a browser" use case, not the "headless API captures traffic" use case.

## Why a separate mitmproxy from `sp`

Two distinct use cases, two distinct mitmproxy instances:

| | `sp` mitmproxy (in main Playwright EC2) | `sp vnc` mitmproxy |
|---|---|---|
| Purpose | Capture/transform traffic for **headless API** sessions | Capture/transform traffic for **human-driven VNC** sessions |
| Lifetime | Same as the Playwright host | Same as the VNC host |
| Auth | API-key (admin FastAPI) | Basic proxy auth (operator-owned creds) |
| Interceptor | Programmatic / per-session (today: baked default) | Default-off; opt-in via baked example or env-var script (N5) |
| Audit destination | Per-session artefact bundle | Local file on the EC2; no external sink (N4) |

Co-locating them on one EC2 (today's PoC shape) couples two different lifecycles and confuses the audit story.

## CLI commands (under `sp vnc`)

| Command | Purpose |
|---|---|
| `sp vnc create [name] [--interceptor <example>] [--interceptor-script <path>]` | Provision an EC2 running chromium+VNC + nginx + mitmproxy. Interceptor choice per N5. |
| `sp vnc list` | List `sg:purpose=vnc` stacks. |
| `sp vnc info <name>` | Tag metadata + nginx URL + mitmweb URL + active interceptor name (if any). |
| `sp vnc delete <name>` | Terminate. All state on the EC2 is wiped (N3). |
| `sp vnc connect <name>` | SSM shell. |
| `sp vnc health <name>` | Health: chromium up, nginx 200, mitmweb reachable. |
| `sp vnc open <name>` | Open the viewer URL in a local browser. **Replaces** the old top-level `sp forward-browser`. |
| `sp vnc forward <name> [--port 3000]` | Local port-forward to the viewer via SSM. |
| `sp vnc flows <name>` | List recent mitmproxy flows (via mitmweb API). |
| `sp vnc clear-flows <name>` | Clear the flow buffer. |
| `sp vnc interceptors` | List the example interceptors baked into the AMI. |
| `sp vnc ami {list,create,delete,wait}` | AMI subgroup, matching `sp el ami`. |

The mitmproxy fragment reuses the `agent_mitmproxy` image but with a different env config (interceptor source per N5):

```yaml
mitmproxy:
  image: ${ECR_REPO}/agent-mitmproxy:${IMAGE_TAG}
  environment:
    MITM_MODE: regular
    MITM_PROXYAUTH: ${OPERATOR_PROXY_USER}:${OPERATOR_PROXY_PASS}
    # N5: one of these is set (or none for the default-off case)
    VNC_MITM_INTERCEPTOR_NAME:   ""    # e.g. header_logger  → resolved to /opt/interceptors/examples/header_logger.py
    VNC_MITM_INTERCEPTOR_SOURCE: ""    # raw Python source written to /opt/interceptors/runtime/active.py at boot
  ports:
    - "8080:8080"           # behind nginx + IP allowlist
    - "8081:8081"           # mitmweb UI, behind nginx
```

The chromium container is configured with `HTTPS_PROXY=http://mitmproxy:8080` so all browser traffic flows through the local mitmproxy.

## Auth surface

Three layers of auth, one per port:

1. **nginx (port 443)** — Basic auth or operator IP allowlist; serves the chromium-VNC UI.
2. **mitmweb UI (proxied via nginx)** — same nginx auth.
3. **mitmproxy proxy port (8080)** — Basic proxy auth via `MITM_PROXYAUTH`; only chromium uses it (loopback on the docker network), but if it's exposed it stays SG-restricted.
