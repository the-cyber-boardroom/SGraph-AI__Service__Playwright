## Top-level `sp` after the cleanup

Source for current state: `scripts/provision_ec2.py:1513–3031` and the `--help` output the operator pasted.

Three groups: **stack lifecycle**, **vault tooling**, **AMI tooling**, plus the four section subcommands.

### Stays at top level (stack lifecycle for the Playwright EC2)

`create`, `list`, `info`, `delete`, `connect`, `shell`, `env`, `exec`, `exec-c`, `logs`, `diagnose`, `forward`, `wait`, `clean`, `open`, `screenshot`, `smoke`, `health`, `ensure-passrole`.

### Vault tooling — stays as a subgroup

`vault-clone`, `vault-list`, `vault-run`, `vault-commit`, `vault-push`, `vault-pull`, `vault-status` regroup under `sp vault` (long alias) / `sp v` (short). Top-level flat `vault-*` aliases are dropped.

| `sp vault-list` | `sp vault list` |
| `sp vault-run` | `sp vault run` |

### AMI tooling — stays as a subgroup

`bake-ami`, `wait-ami`, `tag-ami`, `list-amis`, `create-from-ami` regroup under `sp ami`. **Verbs match the just-shipped `sp el ami {list, create, delete, wait}` pattern** (commits `9e076e3`, `008dae1` on dev) so all four sister sections present the same AMI surface.

| Old | New |
|---|---|
| `sp bake-ami` | `sp ami create` |
| `sp wait-ami` | `sp ami wait` |
| `sp tag-ami` | `sp ami tag` |
| `sp list-amis` | `sp ami list` |
| `sp create-from-ami` | **stays as `sp create-from-ami`** (top-level, mirrors `sp el create-from-ami` pattern — different action: launches an instance from an AMI, not an AMI op itself). |

The convention emerging from `sp el`: **`sp {section} ami {verb}` for AMI ops, `sp {section} create-from-ami` at the section top level for the "launch instance from baked AMI" shortcut.** Doc 4-6 sister sections inherit this from day one.

## Migration order

Per the operator's agreement, the order is:

### Phase A — API consolidation (foundation)

1. Promote `Stack__Naming` helpers from `Elastic__AWS__Client` to a shared `cli/aws/` module.
2. Land `Image__Build__Service` + Type_Safe result schemas; reduce `Build__Docker__SGraph_AI__Service__Playwright` and `Docker__SP__CLI` to thin wrappers.
3. Migrate the rest of `provision_ec2.py` userdata/SG/IAM logic into `Ec2__Service` + `Ec2__AWS__Client`. Reduce typer commands to wrappers.
4. Add the missing FastAPI routes for any EC2 op the CLI exposes (so the duality is complete before sister sections arrive).

Exit criterion: every existing `sp` command for the Playwright host has a FastAPI counterpart.

### Phase B — Carve out sister sections

Order chosen for **lowest blast radius first** — sister sections can be created before the Playwright EC2 is stripped, because they're additive.

5. **`sp os`** — folder, service, AWS client, schemas, compose, routes, tests. Land green on its own. fluent-bit moves later.
6. **`sp prom`** — same. Add the Playwright EC2 as a scrape target while it still has its embedded Prometheus, to validate the cross-host scrape works.
7. **`sp nvm`** — same. Clone `agent_mitmproxy` for the human-debug mitmproxy.

Each sister section is its own PR / commit + debrief, mirroring the `v0.1.32__agent-mitmproxy` pattern.

### Phase C — Strip the Playwright EC2

8. Confirm `sp prom` (or the operator's local Prometheus) is scraping Playwright successfully.
9. Confirm `sp nvm` is usable for the streaming-browser workflow.
10. Delete the moved containers from `provision_ec2.py:226–337`.
11. Bake a new AMI; tag the previous AMI as `superseded`.
12. Update the reality doc.

### Phase D — Command cleanup

13. Regroup `vault-*` under `sp vault`; `*-ami` under `sp ami`.
14. Delete `forward-prometheus`, `forward-browser`, `forward-dockge`, `metrics`.
15. Update operator docs / runbook references.

Each phase ships green and observable; any phase can be paused without blocking earlier ones.
