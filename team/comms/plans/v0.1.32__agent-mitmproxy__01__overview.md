# v0.1.32 — Agent Mitmproxy (Overview)

**Status:** PROPOSED. Phase 1 scoped; Phases 2-5 deferred.
**Series:** 1 of 6 (overview → architecture → folder layout → CI/provisioning → observability → Phase 1 checklist).
**Owner:** Dev + DevOps (shared).

---

## Motivation

The Playwright service is a browser-automation capability for agents. To run safely, agents need a **controlled, auditable network-egress gateway** so we can:

- Allowlist destinations per session.
- Inject + strip headers (observability, tracing, user-agent normalisation).
- Log every request for post-hoc forensics.
- Terminate TLS for deep inspection where policy allows it.
- Revoke access per agent without redeploying the agent.

Running mitmproxy directly on Lambda is not viable — the EC2 spike (v0.1.31) confirmed Firecracker's `/dev/shm` + fork() constraints hang Firefox + WebKit the same way they'd hang any long-lived proxy process. EC2 is the correct target; the Docker image story we built for Playwright ports cleanly.

## Scope (Phase 1 only)

In scope:

- A new containerised service `agent_mitmproxy` packaged the same way as `sgraph_ai_service_playwright` (single ECR image, EC2 host, `provision_*_ec2.py` one-shot).
- mitmweb (with web UI) fronted by Basic proxy auth — mirrors the existing systemd-unit pattern the operator already runs.
- Admin FastAPI (`Fast_API__Serverless` + X-API-Key) for:
  - CA cert distribution (`GET /ca/cert`)
  - Interceptor-script inspection (`GET /config/interceptor`)
  - Mitmweb UI proxying (`GET /web/**` forwards to localhost:8081)
  - Health / info routes that mirror Playwright's `Routes__Health`.
- A default Python interceptor (`default_interceptor.py`) that injects debug headers on request + response.
- `scripts/provision_mitmproxy_ec2.py` — sibling of `provision_ec2.py`, reuses the SSM/IAM/SG pattern.
- `.github/workflows/ci__agent_mitmproxy.yml` — builds + pushes `agent-mitmproxy:latest` to a new ECR repo, paths-filtered to `agent_mitmproxy/**`.
- Unit tests following the existing `tests/unit/` pattern.

Out of scope (deferred phases):

- **Phase 2** — per-agent bearer tokens + upload/replace interceptor via admin API.
- **Phase 3** — AMP/AMG/OpenSearch wiring (observability plan lives in doc 5).
- **Phase 4** — Playwright-EC2 ↔ mitmproxy-EC2 cert-fetch integration.
- **Phase 5** — hardening (rate-limiting, S3 audit export, Cognito/IAM on admin API).

## Decisions locked

| # | Decision | Rationale |
|---|---|---|
| 1 | Sibling folder `agent_mitmproxy/` in the same repo | Fastest iteration; shares CI/IAM/ECR patterns. Graduate to separate repo when API stabilises. |
| 2 | HTTP Basic auth via `--proxyauth user:pass` | Matches operator's existing systemd unit; simplest; agents already know how. |
| 3 | Admin API behind `Fast_API__Serverless` + X-API-Key | Matches Playwright; one middleware, one header style. |
| 4 | CA cert + interceptor served via admin API | Single auth surface; no CA baked into client images. |
| 5 | SG ingress `0.0.0.0/0` on ALL exposed ports | Auth on every port; VPC complexity deferred. |
| 6 | mitmweb UI bound to localhost, proxied via admin API | Single auth layer (X-API-Key) instead of managing mitmweb's `web_password` separately. |
| 7 | Secrets via GHA secrets → UserData | Matches Playwright's cleartext-in-UserData pattern for the spike. SSM Parameter Store deferred. |
| 8 | Default interceptor baked in image; upload via admin API deferred to Phase 2 | Keeps Phase 1 small; no persistence concerns. |
| 9 | mitmweb process started via a small `supervisord` OR shell wrapper inside the container | Single `docker run`; we already run multi-process via user-data in Playwright. |

## Success criteria for Phase 1

1. `python scripts/provision_mitmproxy_ec2.py --stage dev` brings up a single EC2 instance. Returns `{instance_id, public_ip, proxy_url, admin_url}`.
2. Agent with the basic-auth creds can `curl -x http://user:pass@<ip>:8080 https://example.com` and get a response WITH the injected `X-Debug-*` headers.
3. `curl -H "X-API-Key: <key>" http://<ip>:8000/ca/cert` returns the mitmproxy CA in PEM.
4. `curl -H "X-API-Key: <key>" http://<ip>:8000/config/interceptor` returns the current Python interceptor source.
5. `python scripts/provision_mitmproxy_ec2.py --terminate` cleans up every instance carrying the spike Name-tag.
6. CI workflow passes on PR: lint, unit tests, Docker build+push, provision (on merge to dev).

## What this unlocks

Once Phase 1 is green, the Playwright EC2 spike can be re-pointed at the agent-mitmproxy instance and we get an end-to-end controlled browser-via-proxy path. That's the first real environment in which the Firefox + MITM-auth bug from the v0.1.31 brief can be reproduced with full telemetry — whether we fix the Firefox auth path or add a workaround, we'll have the data to decide.
