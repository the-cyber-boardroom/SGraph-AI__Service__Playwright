# v0.2.26 — `sg aws lambda` — info, config, logs, invocations + fuzzy name resolution

Extends `sg aws lambda` from a deploy-only surface into a complete day-to-day
Lambda management CLI: list functions, inspect config, stream CloudWatch
Logs, and view recent invocations. Also restructures the command shape from
`sg aws lambda <verb> <name>` to `sg aws lambda <name> <verb>` (with `list`
as the one verb that doesn't take a name), and adds fuzzy partial-name
resolution so operators can type `sg aws lambda waker info` instead of the
full 32-character function name.

---

## 1. TL;DR

| Item | Decision |
|------|----------|
| **Goal** | Replace ~80% of the trips an operator makes to the AWS console for Lambda day-to-day work (inspect config, read logs, view invocations) with a fast CLI. |
| **Command shape** | `sg aws lambda <name> <verb>`. The single exception: `sg aws lambda list` (no name). |
| **Fuzzy name resolution** | Operator types any unique substring; the resolver picks it. `sg aws lambda wa info` works if "waker" is the only function in the region containing "wa". |
| **New verbs** | `info`, `details`, `config`, `logs`, `invocations`, `invoke`, plus migrated `deploy`, `delete`, `url create/show/delete`. |
| **CloudWatch Logs** | Default last-1h fetch; `--tail` streams; `--since`, `--until`, `--filter`, `--stream`. Uses `logs:FilterLogEvents`. |
| **Invocations table** | CloudWatch Logs Insights query (`logs:StartQuery` → poll) over the `REPORT` lines. Same data the AWS console shows. |
| **New AWS client** | `Logs__AWS__Client` (boto3 EXCEPTION — `osbot_aws` doesn't cover CloudWatch Logs at the level we need). |
| **Tests** | In-memory fakes for both `Lambda__AWS__Client` (extended) and new `Logs__AWS__Client`. No mocks. |
| **Migration** | Current commands (`sg aws lambda deployment …`, `sg aws lambda url …`) keep working with a deprecation warning for one release, then are removed. |

## 2. Problem & current state

### 2.1 What exists today

`sg aws lambda` has six commands, all deploy/url focused:

```
sg aws lambda deployment list
sg aws lambda deployment deploy <name> --code-path <path> --handler <h>
sg aws lambda deployment delete <name>
sg aws lambda url create <name>
sg aws lambda url show <name>
sg aws lambda url delete <name>
```

No way to inspect a function's runtime/memory/handler/env config, no way to
read CloudWatch Logs, no way to see recent invocations. For any of those,
operators open the AWS console.

### 2.2 What the recent `sg vp bootstrap` debugging needed

Three screenshots captured this:

- **pic1** — "Recent invocations" table: timestamp, requestId, log stream,
  duration, billed duration, memory set, memory used. The standard
  Lambda > Monitor > Logs panel.
- **pic2** — CloudWatch Logs event stream: START / REPORT / END entries
  plus `[ERROR] Runtime.ImportModuleError: Unable to import module
  'lambda_entry': No module named 'sg_compute_specs'`. The data that
  actually identifies the bug.
- **pic3** — Functions list: 14 functions in the region with name,
  description, package type, runtime, last-modified.

All three live in the AWS console. None of them are reachable from `sg`
today. After landing this plan they are: `sg aws lambda waker invocations`
(pic1), `sg aws lambda waker logs --since 10m` (pic2),
`sg aws lambda list` (pic3).

### 2.3 The command-shape problem

The current shape (`<verb> <name>`) is fine when there are 2 verbs. With
10+, it breaks down — `sg aws lambda deployment deploy <name>` is more
typing than `sg aws lambda <name> deploy`, and the operator's mental model
is "the function" first, "the action" second. The user requested
`<name> <verb>` and that's what we'll build.

### 2.4 The full-name problem

Lambda function names like `sg-compute-vault-publish-waker` are 32+ chars.
Operators won't type them. Fuzzy substring resolution is table-stakes for
a CLI that's meant to replace the console.

## 3. New command structure

```
sg aws lambda list                              # show all functions in region (pic3 equivalent)
sg aws lambda list --json
sg aws lambda list --runtime python3.12         # filter

sg aws lambda <name> info                       # one-screen summary (name, runtime, memory, last-modified, state)
sg aws lambda <name> details                    # full GetFunction response, env vars, layers, role ARN, VPC, etc.
sg aws lambda <name> config                     # just the config block (Handler, Runtime, Timeout, MemorySize, Architecture)
sg aws lambda <name> config set --memory 512 --timeout 60       # update specific config fields
sg aws lambda <name> config set --env KEY=value                 # set/update one env var (interactive 'remove' too)

sg aws lambda <name> logs                       # last 1h, all streams, tail of 100 events
sg aws lambda <name> logs --since 10m           # human-friendly duration: 30s/5m/2h/1d
sg aws lambda <name> logs --since 2026-05-17T14:00:00Z          # absolute ISO timestamp
sg aws lambda <name> logs --until <time>        # bounded window
sg aws lambda <name> logs --filter "ERROR"      # CloudWatch filter pattern
sg aws lambda <name> logs --tail                # poll-loop streaming (every 2s)
sg aws lambda <name> logs --stream <stream-id>  # focus on one log stream
sg aws lambda <name> logs --json

sg aws lambda <name> invocations                # last 20 invocations, pic1 shape
sg aws lambda <name> invocations --limit 100
sg aws lambda <name> invocations --since 1h
sg aws lambda <name> invocations --failed       # only errors / timeouts
sg aws lambda <name> invocations --json

sg aws lambda <name> invoke                     # synchronous invoke; payload via stdin or --payload
sg aws lambda <name> invoke --payload '{"key":"value"}'
sg aws lambda <name> invoke --payload-file path.json
sg aws lambda <name> invoke --type Event        # async fire-and-forget

sg aws lambda <name> deploy --code-path <path> --handler <h>    # replaces 'deployment deploy'
sg aws lambda <name> delete --yes                                # replaces 'deployment delete'

sg aws lambda <name> url create [--auth-type NONE|AWS_IAM]      # replaces 'url create'
sg aws lambda <name> url show
sg aws lambda <name> url delete --yes

sg aws lambda <name> tags list
sg aws lambda <name> tags set --kv "Owner=dinis"
sg aws lambda <name> tags remove --key Owner

sg aws lambda <name> versions list              # ListVersionsByFunction
sg aws lambda <name> aliases list               # ListAliases
```

**Reserved names.** `list` is the only verb that doesn't take a name. To
avoid clash, no function may be resolved to the literal string `"list"` —
the parser intercepts it. Other potentially confusing tokens (e.g. a
function actually named "info") are disambiguated by the rule that the
*first* positional after `lambda` is always the function name, the
*second* is always the verb. If you genuinely have a function named "info",
`sg aws lambda info info` works.

## 4. Design

### 4.1 Fuzzy name resolution — `Lambda__Name__Resolver`

```python
class Lambda__Name__Resolver(Type_Safe):
    cache_ttl_seconds: int = 300                         # 5-min cache of function-list

    def resolve(self, query: str) -> Safe_Str__Lambda__Name:
        names = self._all_function_names()               # cached list
        matches = self._rank(query, names)
        if len(matches) == 0: raise self._not_found(query, names)
        if len(matches) == 1: return matches[0]
        # multiple matches
        if matches[0].score > matches[1].score + 10:    # clear winner
            return matches[0].name
        raise self._ambiguous(query, matches)
```

**Ranking (in order — first non-empty wins, returns ALL ties at that level):**

| Tier | Rule | Score |
|------|------|-------|
| 1 | Exact match | 1000 |
| 2 | Case-insensitive exact | 900 |
| 3 | Starts-with (case-insensitive) | 800 - position of mismatch |
| 4 | Contains as substring (case-insensitive) | 600 - position of match |
| 5 | All query chars appear in order (subsequence) | 400 - len(name) |

**Examples (using pic3's 14 functions, filtered to lambdas only):**

| Query | Matches | Resolved |
|-------|---------|----------|
| `sg-compute-vault-publish-waker` | tier 1 | exact |
| `sg-compute` | tier 3 (prefix) → `sg-compute-vault-publish-waker` only | unique |
| `sg-c` | tier 3 → `sg-compute-vault-publish-waker` only | unique |
| `waker` | tier 4 → `sg-compute-vault-publish-waker` only | unique |
| `wa` | tier 4 → `sg-compute-vault-publish-waker` only | unique |
| `sg` | tier 3 → matches `sg-compute-vault-publish-waker`, `sgraph_ai_app_send__*`, `sg-playwright-*`, `sp-playwright-*` | **ambiguous** — error listing top-5 candidates |
| `playwright` | tier 4 → 5 matches | **ambiguous** |
| `xyz123` | nothing | **not found** — suggests 3 closest names by edit distance |

**Error messages:**

```
$ sg aws lambda sg info
  ✗  Ambiguous function name "sg" — 8 matches:
       sg-compute-vault-publish-waker
       sg-playwright-dev
       sg-playwright-baseline-dev
       sgraph_ai_app_send__admin-dev
       sgraph_ai_app_send__user-dev
       sgraph_ai_app_send__user-prod
       sp-playwright-cli-dev
       sp-playwright-cli-prod
     Add more characters to disambiguate.
```

**Caching.** Function list cached at `~/.sg/cache/lambda-functions-<region>.json`
for 5 minutes (configurable via `--no-cache` flag on any subcommand).
Invalidated automatically on `deploy` / `delete` actions in the same
process.

### 4.2 `info` / `details` / `config` — read paths

All three are pure-read; one `lambda:GetFunction` call covers them all.
Differences are presentation:

- **`info`** — table with name, runtime, state, last-modified, memory,
  timeout, handler, code-size, function-URL (if any). Six rows, one column,
  human-friendly. The "I want to see what this thing IS" view.
- **`details`** — full nested response: env vars (values masked unless
  `--show-env-values`), layer ARNs, VPC config, dead-letter config,
  tracing config, ephemeral-storage size, architecture, signing profile,
  KMS key ARN, role ARN. The "I want to see EVERYTHING" view.
- **`config`** — just the editable config block: runtime, handler,
  memory, timeout, environment, architecture. The "I want to see / change
  this" view. `sg aws lambda <name> config set ...` mutates these via
  `lambda:UpdateFunctionConfiguration` (gated by `SG_AWS__LAMBDA__ALLOW_MUTATIONS=1`,
  same as existing deploy/delete).

### 4.3 `logs` — CloudWatch Logs

**API used:** `logs:FilterLogEvents` over log group
`/aws/lambda/<function-name>`. For `--tail`, poll every 2s with
`startTime` advancing to the last-seen event timestamp, dedup by `eventId`.

**Defaults:**
- `--since 1h` (last hour)
- Last 100 events shown (configurable via `--limit`)
- All log streams in the group
- Pretty output: `<timestamp> [<level>] <message>` with colour by level
  (RED for ERROR/Runtime.*, YELLOW for WARNING, default for the rest)

**`--tail` mode:**
- Banner: `→ tailing /aws/lambda/<name> (Ctrl-C to stop)`
- Each poll: `filter_log_events(startTime=last_ts+1)`
- Dedup by `eventId` (CloudWatch can return the same event across polls)
- Graceful Ctrl-C handler that prints a one-line summary
  (`✓ tailed for 4m32s, 87 events shown`)

**Filter syntax** (`--filter`):
- Passed straight to CloudWatch — same syntax as the console
  ([CloudWatch Logs filter docs](https://docs.aws.amazon.com/AmazonCloudWatch/latest/logs/FilterAndPatternSyntax.html))
- Quick examples shown in `--help`: `"ERROR"`, `"[level=ERROR]"`,
  `"{$.eventName = AssumeRole}"`

**Time parsing (`--since` / `--until`):**
- Relative: `30s`, `5m`, `2h`, `1d`, `7d`
- Absolute ISO 8601: `2026-05-17T14:00:00Z` (UTC required)
- Defaults to UTC; refuses ambiguous formats with a clear error

### 4.4 `invocations` — CloudWatch Logs Insights

Reproduces the AWS console's "Recent invocations" panel (pic1) using a
Logs Insights query:

```
fields @timestamp, @requestId, @logStream, @duration, @billedDuration,
       @memorySize / 1000 / 1000 as memorySetMB,
       @maxMemoryUsed / 1000 / 1000 as memoryUsedMB,
       @initDuration
| filter @type = "REPORT"
| sort @timestamp desc
| limit 20
```

**API flow:**

1. `logs:StartQuery` with the query string + time window
2. Poll `logs:GetQueryResults` every 500ms until status is `Complete`
3. Render the rows in a Rich table:

```
┌─────────────────────────┬──────────────┬──────────┬────────┬──────────┐
│ Timestamp               │ RequestId    │ Duration │ Billed │ MemUsed  │
├─────────────────────────┼──────────────┼──────────┼────────┼──────────┤
│ 2026-05-17 14:22:32.773 │ dd767896…    │  96.4 ms │ 97 ms  │ 46 / 512 │
│ 2026-05-17 14:22:32.605 │ b919d67e…    │ 105.2 ms │ 106 ms │ 46 / 512 │
└─────────────────────────┴──────────────┴──────────┴────────┴──────────┘
```

**`--failed`** modifies the query: `| filter @type = "REPORT" and ispresent(@error)`
or similar (TBD — needs validation against real data).

**Cost note.** Insights queries are billed per GB scanned (~$0.005/GB). A
default 20-row query over the last hour on a low-traffic function is
fractions of a cent. Mention this in operator docs but don't block on it.

### 4.5 `invoke` — direct invocation

`lambda:Invoke` with `InvocationType=RequestResponse` (sync, default) or
`Event` (async). Payload sources, in priority order:

1. `--payload '<json>'` — inline
2. `--payload-file <path>` — read from disk
3. stdin (if not a tty)
4. Empty `{}` if none provided

Output: response body printed to stdout (JSON), execution log to stderr
under `--log` flag (uses `LogType=Tail` in the invoke API which returns
the last 4 KB of execution log base64-encoded).

Gated by `SG_AWS__LAMBDA__ALLOW_MUTATIONS=1` only for async (`--type Event`)
since sync invocation is read-shaped from the operator's perspective.

### 4.6 Custom command parser

Standard Typer registers commands as `app.command('<name>')` — that
doesn't accept a leading positional that's then dispatched to a
sub-command. Two options:

**Option A — single Typer command, manual sub-dispatch:**

```python
@lambda_app.command(name='', invoke_without_command=False, no_args_is_help=False)
def lambda_root(
    ctx: typer.Context,
    name_or_list: str = typer.Argument(..., help='Function name (fuzzy) or "list".'),
    verb:         Optional[str] = typer.Argument(None, help='info|details|config|logs|invocations|invoke|deploy|delete|url|tags|versions|aliases'),
):
    if name_or_list == 'list':
        return cmd_list(...)
    name = Lambda__Name__Resolver().resolve(name_or_list)
    return _dispatch(name, verb, ctx.args)
```

But this loses Typer's auto-generated `--help` per sub-verb. Bad UX.

**Option B — Click MultiCommand subclass (chosen):**

A custom Click `Group` subclass that intercepts `get_command()`. If the
first positional looks like a function-name query (anything other than
`list`), it consumes that arg, resolves it via `Lambda__Name__Resolver`,
and dispatches to a per-verb Typer command that takes `name` as a
context-injected dependency. Generated help is per-verb, full Typer
behaviour preserved.

```python
class Lambda__App(typer.Typer):
    def __init__(self):
        super().__init__(cls=Lambda__Click__Group, ...)

class Lambda__Click__Group(click.Group):
    def resolve_command(self, ctx, args):
        if args and args[0] != 'list' and not args[0].startswith('-'):
            query = args[0]
            verb  = args[1] if len(args) > 1 else None
            resolved = Lambda__Name__Resolver().resolve(query)
            ctx.obj = {'function_name': resolved}
            # rewrite args to: <verb> [remaining]
            return super().resolve_command(ctx, args[1:])
        return super().resolve_command(ctx, args)
```

Each verb command reads the resolved name from `ctx.obj['function_name']`.
Test coverage for the parser itself is a small fixture set.

### 4.7 New AWS client — `Logs__AWS__Client`

Lives at `sgraph_ai_service_playwright__cli/aws/logs/service/Logs__AWS__Client.py`.
boto3 EXCEPTION (header comment) — `osbot_aws` doesn't cover CloudWatch
Logs at the level we need (filter events, start query, get query results).

Surface:

```python
class Logs__AWS__Client(Type_Safe):
    region: str = ''

    def client(self): return boto3.client('logs', ...)    # seam

    def filter_events(self,
                      log_group:    str,
                      start_time:   int,                  # epoch ms
                      end_time:     Optional[int] = None,
                      filter_pattern: str = '',
                      log_streams:  Optional[List[str]] = None,
                      limit:        int = 100,
                      ) -> Schema__Logs__Events__Response: ...

    def tail_events(self,
                    log_group:    str,
                    filter_pattern: str = '',
                    poll_interval_ms: int = 2000,
                    ) -> Iterator[Schema__Logs__Event]: ...     # generator; caller breaks on Ctrl-C

    def start_query(self, log_group: str, query: str, start_time: int, end_time: int) -> str: ...   # returns queryId
    def get_query_results(self, query_id: str) -> Schema__Logs__Query__Result: ...                  # status + rows
    def wait_query(self, query_id: str, timeout_sec: int = 30) -> Schema__Logs__Query__Result: ...  # poll until Complete
```

In-memory test fake (`Logs__AWS__Client__In_Memory`) — dict-backed event
store keyed by `(log_group, log_stream, timestamp)`; supports filter
patterns by simple substring match (Insights query support is mocked at
the result level — tests provide canned rows for `start_query` /
`get_query_results` pairs).

## 7. Risks

| Risk | Mitigation |
|------|------------|
| Fuzzy resolver returns wrong function in production | "Clear winner" rule requires +10 score margin; otherwise error with full list. Ambiguous-by-default rather than guess-and-hope. |
| Lambda function name cache goes stale after deploy in a different terminal | 5-min TTL; `--no-cache` flag for one-off bust; cache file is per-region so multi-region work doesn't collide |
| `logs --tail` polling rate hammers CloudWatch | Default 2s poll interval (≈ 1800 calls/hour, well under default account limit of 25 TPS for FilterLogEvents); configurable via `--poll-interval` |
| Insights query cost surprises | Default time window 1h, limit 20 rows; `--since` accepts max 24h without `--yes-cost` confirmation flag; operator docs explain the $0.005/GB pricing |
| Custom Click parser breaks standard `--help` output | Phase B test fixture verifies `sg aws lambda --help`, `sg aws lambda <name> --help`, `sg aws lambda <name> logs --help` all render correctly |
| Backwards incompatibility breaks existing scripts | One-release deprecation window with stderr warning; old commands still work; release notes call out the migration explicitly |
| `invoke` returns large payloads that overflow terminal | Output is JSON-formatted; `--output-file` flag writes to disk; truncated with a clear marker if > 10 MB |
| Operator runs `logs --tail` and forgets, eating credits | Ctrl-C summary makes it visible; no auto-timeout (operator's call) |

## 8. Acceptance criteria

Merge to dev requires:

1. `sg aws lambda list` returns the same 14 functions visible in pic3.
2. `sg aws lambda waker info` resolves to `sg-compute-vault-publish-waker`
   without prompting.
3. `sg aws lambda sg info` fails with an ambiguity error listing all
   matching functions.
4. `sg aws lambda xyz info` fails with a "not found" error and 3 closest
   suggestions.
5. `sg aws lambda waker logs --since 1h` shows the same events as the
   CloudWatch Logs console for that time range.
6. `sg aws lambda waker logs --tail` streams events as they arrive;
   Ctrl-C exits cleanly with a summary.
7. `sg aws lambda waker invocations` reproduces the pic1 table (timestamp,
   requestId, duration, billed duration, memory).
8. `sg aws lambda waker config` shows handler/runtime/timeout/memory and
   `... config set --memory 1024` updates it (with mutation guard).
9. `sg aws lambda waker invoke --payload '{}' --log` returns the response
   AND the last 4 KB of execution log.
10. Existing `sg aws lambda deployment deploy` still works AND prints a
    one-line deprecation warning naming the new shape.
11. ~75 new tests pass; existing Lambda CLI tests still pass.
12. `sg aws lambda --help` and `sg aws lambda <name> --help` and
    `sg aws lambda <name> <verb> --help` all render correctly.

## 9. Anti-scope (NOT in this plan)

- **Lambda layer management** (`sg aws lambda <name> layers …`) — useful
  but separate, lots of edge cases (cross-account, version pinning).
- **Reserved-concurrency / provisioned-concurrency configuration** — niche;
  ship if/when needed.
- **CloudWatch Logs subscriptions** (firehose / Kinesis exports) — out of
  scope for a CLI day-to-day tool.
- **Function-URL CORS configuration** — current `url create` doesn't touch
  it either; ship later when there's a concrete need.
- **VPC config editing** — too easy to break networking from a CLI without
  a "are you sure" flow; read-only in `details` is enough for v1.
- **EventSourceMapping management** (SQS / DynamoDB streams / Kinesis
  triggers) — separate command family entirely (`sg aws lambda <name>
  triggers …` or `sg aws events …`).
- **Cross-region search** — `sg aws lambda list` is region-scoped; explicit
  `--region` flag if needed.
- **Insights queries beyond the REPORT-line one** — only the built-in
  "invocations" query in v1; user-defined queries are post-v1.

## 10. Open questions (for the human before Phase A starts)

| Q | Question | Architect's default |
|---|----------|---------------------|
| Q1 | Should fuzzy resolution be opt-out (i.e. `--exact` flag for scripts)? | Yes. Scripts shouldn't fail intermittently when someone adds a function whose name collides. `--exact` skips the resolver. |
| Q2 | Should `list` be `sg aws lambda list` or `sg aws lambdas` (plural)? | `sg aws lambda list`. Plural-noun-as-top-level breaks the `<name>` slot. The current `sg aws cf distributions list` precedent is similar but `cf distributions` is a separate subcommand tree, not a fuzzy-name slot — different problem. |
| Q3 | Default time window for `logs` — 1h or 15m? | 1h. Lambda traffic is bursty; 15m often shows nothing on low-traffic functions. |
| Q4 | Should `logs` colourise based on parsed level? | Yes (RED on ERROR/Runtime.*, YELLOW on WARNING). Operators read logs visually; colour pays off. |
| Q5 | `invoke` default — sync or async? | Sync (RequestResponse). It's the safer default; matches the AWS CLI; async is opt-in via `--type Event`. |
| Q6 | `config set` interactive confirmation before applying? | Yes — show a before/after diff and ask y/N (with `--yes` for scripts). Configuration changes can break a function silently. |
| Q7 | Name-resolver cache: per-region file, per-account, or per-credential-role? | Per-region per-account (filename `lambda-functions-<account>-<region>.json`). Avoids leaking function names between identities. |
| Q8 | If the resolver matches by subsequence (tier 5) AND there's a single match, should we accept silently or warn? | Warn. Subsequence is a wide net; printing "→ resolved 'wkr' to sg-compute-vault-publish-waker (subsequence match)" lets the operator catch surprises. |
| Q9 | Should `logs --tail` have a `--max-events` safety cap to auto-exit? | No, but `Ctrl-C` summary is mandatory. Trusting the operator to break the loop. |
| Q10 | Insights query for `--failed` invocations — define now or punt to Phase D? | Define now (in §4.4): `filter @type = "REPORT" and (ispresent(@error) or @duration >= @timeout * 1000)`. Validates against real data in Phase D. |
