---

## Why a section, not a replacement

`sp el` and `sp os` solve overlapping but distinct problems:

- `sp el` — Elasticsearch + Kibana, the operator's existing dev-pack target.
- `sp os` — OpenSearch + Dashboards, the destination for `fluent-bit` log shipping (today fluent-bit lives in the Playwright compose; after the strip it needs a host).

Keeping both lets the team move at independent cadences and avoids a forced migration.

## CLI commands (under `sp os`)

Mirror the elastic command surface 1:1 so the operator can transfer muscle memory:

| Command | Purpose |
|---|---|
| `sp os create [name]` | Provision an EC2 running OpenSearch + Dashboards + fluent-bit. |
| `sp os list` | List `sg:purpose=opensearch` stacks. |
| `sp os info <name>` | Tag-based metadata + live health. |
| `sp os delete <name>` | Terminate. |
| `sp os connect <name>` | SSM shell on the host. |
| `sp os exec <name> <cmd>` | Run command via SSM. |
| `sp os health <name>` | Health check (cluster status, node count, doc count). |
| `sp os wait <name>` | Poll until the dashboards URL responds. |
| `sp os harden <name>` | Enable basic auth / IP allowlist. |
| `sp os wipe <name>` | Drop all indices, keep the host. |
| `sp os seed <name>` | Load a known fixture index for smoke testing. |
| `sp os dashboard <name> <file>` | Import a saved-object NDJSON. |

`sp os` and `sp opensearch` both register as aliases for the same typer app, matching `sp el` / `sp elastic`.

## FastAPI routes (under `Fast_API__SP__CLI`)

Same operations, JSON return shapes:

```
POST   /v1/opensearch/stack                         create
GET    /v1/opensearch/stacks                        list
GET    /v1/opensearch/stack/{name}                  info
DELETE /v1/opensearch/stack/{name}                  delete
GET    /v1/opensearch/stack/{name}/health           health
POST   /v1/opensearch/stack/{name}/dashboards       dashboard import
POST   /v1/opensearch/stack/{name}/wipe             wipe
POST   /v1/opensearch/stack/{name}/seed             seed
```

Long-running ops (`create` ~10–15 min) follow v0.1.72 step-3 pattern: `202 Accepted` immediately, status via `GET /v1/opensearch/stack/{name}` polling against tags.

## Tags + naming

Same conventions as elastic, with `purpose=opensearch`:

```
sg:purpose      = opensearch
sg:stack-name   = sp-os-{slug}
sg:allowed-ip   = <creator IP>/32
sg:creator      = <iam-user>
sg:created-at   = <iso8601>
sg:section      = os
```

`aws_name_for_stack()` and `sg_name_for_stack()` helpers from `Elastic__AWS__Client` graduate to a shared utility (proposed location: `sgraph_ai_service_playwright__cli/aws/Stack__Naming.py`) so all four sections (`el`, `os`, `prom`, `nvm`) share one implementation.

## fluent-bit — the log shipper move

fluent-bit is removed from the Playwright compose and added to `os/compose/fragments/fluent-bit.service.yaml`. Configuration changes:

- Input: docker-compose log driver on the **OpenSearch host itself** (not the Playwright host).
- Output: `https://localhost:9200` with cluster certs.
- The Playwright host loses fluent-bit entirely; if log shipping is needed from there, that is a v0.1.97+ concern (likely a CloudWatch agent or a remote fluent-bit pointing at this section's endpoint).
