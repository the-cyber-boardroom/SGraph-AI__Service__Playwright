# v0.1.96 — Playwright Stack Split (Overview)

**Status:** PROPOSED. Nothing described here exists yet.
**Series:** 1 of 7 (overview → API consolidation → strip Playwright EC2 → `sp os` → `sp prom` → `sp nvm` → command cleanup + migration order).
**Owner:** Architect + Dev + DevOps (shared).
**Predecessor:** [v0.1.72 — SP CLI ↔ FastAPI Duality](../briefs/v0.1.72__sp-cli-fastapi-duality.md).

---

## Motivation

The Playwright EC2 host today bundles ~9 containers in a single docker-compose: Playwright + agent-mitmproxy + linuxserver/chromium (KasmVNC) + nginx-for-VNC + Prometheus + cAdvisor + node-exporter + fluent-bit + Dockge. That bundling was useful while the operator was PoC-ing the moving parts on a single instance. It is now a liability:

- The "main" image carries observability and human-debug tooling that production never needs.
- Two unrelated mitmproxy use cases (headless API traffic capture vs. human VNC viewer) collapse into one container, masking which one a given flow belongs to.
- Top-level `sp` commands have grown sideways (`forward-prometheus`, `forward-browser`, `forward-dockge`, `metrics`) — operations that belong to the embedded subsystems, not to the Playwright host.
- Promising patterns from the existing `sp el` / elastic section (one folder per concern, Type_Safe service layer, narrow boto3 boundary) are not being applied to the rest of the stack.

The v0.1.31 reality doc explicitly flags the observability bundle as an "extended" spike, not a long-term shape — this plan executes that intent.

## Scope

In scope:

1. **API-first consolidation.** Every `sp` operation goes through a single `*__Service` class that the typer CLI and the FastAPI routes both call. Extends the v0.1.72 duality pattern to EC2 lifecycle, image build, and the three new sections below.
2. **Strip the main Playwright EC2** to **two containers**: `playwright` + `agent-mitmproxy`. Nothing else.
3. **Carve out three sister sections** that mirror the `sp el` / elastic pattern:
   - **`sp os`** — OpenSearch (sister to `sp el` Elasticsearch+Kibana; both kept).
   - **`sp prom`** — Prometheus + cAdvisor + node-exporter (and Grafana if/when needed).
   - **`sp nvm`** — Nginx + VNC + Mitmproxy as a human-debug streaming-browser stack. Mitmproxy here is a **separate instance** from the one bundled with `sp` (different use case: human viewer, not API-side capture).
4. **Prune the `sp` command list** — drop the embedded-subsystem forwarders, move metrics under `sp prom`, regroup the survivors.

Out of scope (explicitly deferred):

- Migrating the Playwright FastAPI service itself (`Fast_API__Playwright__Service`) — only the CLI-side `Fast_API__SP__CLI` and its service layer change.
- Multi-tenant or cross-account stack support; same single-account assumption as today.
- Replacing typer or Rich; the CLI framework stays.
- Backwards compatibility with running EC2s or AMIs — the operator confirmed clean-slate is acceptable.

## Decisions locked

| # | Decision | Rationale |
|---|---|---|
| 1 | Main Playwright EC2 = `playwright` + `agent-mitmproxy` only | Single responsibility per image; production has no use for VNC/Prometheus/Dockge. |
| 2 | Three sister sections (`os`, `prom`, `nvm`) modeled on `sp el` | The elastic folder is the proven template (50 files, service + AWS client + schemas). Reuse the shape. |
| 3 | `sp el` (Elasticsearch+Kibana) and `sp os` (OpenSearch) **both kept** | They serve different downstream consumers; not a rename. |
| 4 | Two distinct mitmproxy instances (one in `sp`, one in `sp nvm`) | Different use cases, different lifetimes, different auth surfaces. Co-locating them was a PoC artefact. |
| 5 | API-first for every new section from day one | Avoid the elastic-style late-API problem; new sections get FastAPI routes alongside CLI commands in the same PR. |
| 6 | `sp el` keeps its current shape (CLI-only) | Out of scope for this plan; documented as a follow-up in `07`. |
| 7 | Clean slate, no backwards compat | Operator-confirmed. Removes a large class of migration questions. |

## Success criteria

1. `sp create` provisions an EC2 running exactly two containers — `playwright` and `agent-mitmproxy`. No prometheus, no VNC, no dockge.
2. `sp prom create`, `sp os create`, `sp nvm create` each provision their own EC2, with the same lifecycle commands as `sp el` (`create / list / info / delete / connect / health`).
3. Every command listed under any section can be invoked equivalently as a FastAPI POST/GET against `Fast_API__SP__CLI`.
4. `sp --help` shows a clean top-level command list with no `forward-prometheus`, `forward-browser`, `forward-dockge`, `metrics`, or other embedded-subsystem leakage.
5. Existing tests pass; new sections each have unit tests in their own `tests/unit/.../{section}/` folder following the elastic pattern.
6. Reality doc updated in the same commit that lands the change.

## What this unlocks

- Smaller, faster Playwright AMI bake (fewer layers, fewer security-relevant containers).
- Clean separation of "production" (Playwright host) from "operator tooling" (VNC, Dockge) and "platform" (Prometheus, OpenSearch).
- A copy-paste template for any future sister stack: clone an existing `{section}/` folder, swap the AWS client and the docker-compose, ship.
