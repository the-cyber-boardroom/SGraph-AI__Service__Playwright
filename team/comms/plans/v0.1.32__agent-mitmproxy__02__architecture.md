## Port surface (SG ingress — all authenticated)

| Port | Process | Binding | Auth |
|---|---|---|---|
| **8080** | mitmweb proxy | `0.0.0.0` | `--proxyauth user:pass` (HTTP Basic on CONNECT) |
| **8000** | uvicorn (admin FastAPI) | `0.0.0.0` | `X-API-Key` header middleware |
| **8081** | mitmweb UI | **`127.0.0.1`** — NOT exposed via SG | Reached via admin-API proxy `GET /web/**` |
| 9090 | prometheus /metrics (Phase 3) | `127.0.0.1` | ADOT collector scrapes locally |

**No unauthenticated port.** If a caller hits `8080` without Basic auth → `407 Proxy Authentication Required`. If a caller hits `8000` without the API key → `401 Unauthorized`.

## Container boot sequence

```
1. docker entrypoint: /app/entrypoint.sh
2. entrypoint.sh writes /app/current_interceptor.py from the default baked in the image
3. entrypoint.sh execs supervisord -c /app/supervisord.conf
4. supervisord starts two programs:
   a. mitmweb --proxyauth "$PROXY_AUTH_USER:$PROXY_AUTH_PASS"
              --web-host 127.0.0.1
              --web-port 8081
              --set block_global=false
              --set confdir=/root/.mitmproxy
              -s /app/addons/addon_registry.py          # loads default_interceptor + audit_log + metrics
   b. uvicorn agent_mitmproxy.fast_api.app:app --host 0.0.0.0 --port 8000
5. mitmweb on first run generates /root/.mitmproxy/mitmproxy-ca-cert.pem
6. Admin API reads that path on demand (no startup coupling)
```

If supervisord feels heavy, a two-process shell wrapper with `trap`-based cleanup is equivalent and is what the Playwright image uses today — we'll mirror that pattern.

## How clients use it

### 1. Agent as an HTTP/HTTPS client

```bash
# One-shot
curl -x http://<proxy-user>:<proxy-pass>@<ip>:8080 https://api.example.com

# Persistent (env-var style — most libraries honour these)
export HTTP_PROXY=http://<proxy-user>:<proxy-pass>@<ip>:8080
export HTTPS_PROXY=http://<proxy-user>:<proxy-pass>@<ip>:8080
# To trust the MITM cert:
curl -H 'X-API-Key: <admin-key>' http://<ip>:8000/ca/cert > /etc/ssl/certs/mitmproxy.pem
update-ca-certificates
```

### 2. Playwright service as a client (integration target)

The Playwright spike's request payload already supports this shape:

```json
{
  "proxy": {
    "server": "http://<ip>:8080",
    "auth": {"username": "<proxy-user>", "password": "<proxy-pass>"},
    "ignore_https_errors": true
  }
}
```

Phase 4 will replace `ignore_https_errors: true` with real CA-cert installation at Playwright-container boot (fetched from this admin API). Out of scope for Phase 1.

### 3. Operator viewing the mitmweb UI

```
curl -H 'X-API-Key: <admin-key>' http://<ip>:8000/web/ > index.html
# Or open a browser pointed at http://<ip>:8000/web/ with the API key passed as a header
# (browser plugins like ModHeader make this trivial; a /web?api_key=... query-string
# variant is in scope for Phase 1 to avoid needing a plugin)
```

## Observability surface (Phase 1 — log-only)

- Every mitmproxy flow emits one JSON line to stdout (`audit_log_addon.py`):
  ```json
  {"ts":"2026-04-19T...","flow_id":"...","method":"GET","url":"https://...","status":200,
   "bytes_request":..., "bytes_response":..., "elapsed_ms":..., "client_addr":"...",
   "proxy_user":"..."}
  ```
- `docker logs -f agent-mitmproxy` shows both mitmweb's own output and the audit lines.
- Phase 3 adds Prometheus metrics + fluent-bit shipping the audit lines to OpenSearch.

## Integration with Playwright service

Loose coupling — agent-mitmproxy does not know about Playwright. Playwright opts in by adding:

```python
proxy_url = get_env('AGENT_MITMPROXY__URL')                  # http://user:pass@<ip>:8080
ca_cert_url = get_env('AGENT_MITMPROXY__CA_CERT_URL')        # http://<ip>:8000/ca/cert
```

and fetching + installing the cert at its own boot (Phase 4). Phase 1 ships neither of those env vars — the integration is manual for now.

## Why mitmweb instead of mitmdump?

Operator's existing systemd unit runs `mitmweb`. Keeping mitmweb:

- Gives a usable UI for debug (gated behind admin-API auth).
- Same invocation as operator's current script — zero learning delta.
- Same flag support (`--proxyauth`, `-s <script>`, `--set block_global=false`).

Trade-off: mitmweb pulls in a minimal web server on port 8081 that we bind to localhost. One extra process, acceptable.

## Why not run mitmproxy as a sidecar of the Playwright container?

Considered and rejected:

- Coupled lifecycle — proxy restart affects Playwright; separate scaling is useful.
- Separate audit domain — "what did the proxy see" is a different concern from "what did the browser do".
- The operator already runs mitmweb as a standalone service on a dedicated EC2 — this plan aligns with their existing mental model.
