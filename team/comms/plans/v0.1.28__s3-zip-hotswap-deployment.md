## 1. Why a separate plan

The source brief ("brief v2") is already a detailed technical memo. This file does three things on top of it:

1. Locks down the open questions (brief §338–347) with the answers agreed in the 04/17 exchange.
2. Expands the CI pipeline split (brief §218–233) into a concrete per-workflow diff against today's `.github/workflows/ci-pipeline.yml`.
3. Adds a test plan + rollback mechanic that the brief didn't formalise.

Where the brief is already correct, this plan cites it rather than restating it. Treat the brief as the *what*; this plan is the *how* and *in-what-order*.

## 2. Scope

**In scope:**
- New `lambda_entry.py` boot shim at the top of the container image.
- New `scripts/package_code.py` that uses `osbot_aws.aws.s3.S3__Zip_Bytes`.
- Dockerfile entrypoint switch from `sgraph_ai_service_playwright.fast_api.lambda_handler` to `lambda_entry.run`.
- New `image_version` file + exposure in `/health/info` alongside `service_version` and `code_source`.
- CI pipeline split into **Track A** (image) and **Track B** (code).
- Lambda recycle trigger via env-var bump.
- Deploy tests updated to drive two tracks.

**Out of scope (deliberate):**
- `load_dependencies` pattern — deps stay in the image (brief §15).
- `clear_osbot_aws_modules()` purge — we're not loading deps from S3 (see §6).
- `main` / `prod` rollout — dev first, promote afterwards.
- Cold-start cache optimisation beyond a single ETag check (see §6).

## 3. Design decisions — answers to brief §338–347

| Brief question | Decision | Rationale |
|---|---|---|
| Q1: `clear_osbot_aws_modules()`? | **Skip.** | No deps loaded from the zip, so no shadowing. Removing it surfaces real `ImportError`s instead of masking them. |
| Q2: `lambda_entry.py` path inside the image? | **Top-level at `/var/task/lambda_entry.py`.** | Matches `sgraph_ai_app_send` pattern; cosmetic placement inside `bootloader/` adds no value. |
| Q3: Shim in the zip or the image? | **Baked into the image.** | The shim is the rollback escape hatch. Putting it in the zip is chicken-and-egg. |
| Q4: Cold-start cache? | **Ship with a one-line ETag check.** | `/tmp` survives warm invocations; skipping the 2–4 MB download on warm invokes is ~200 ms saved per call. |
| (New) `code_source` on `/health/info`? | **Redact account ID.** | Surface `{lambda_name}/code/{version}.zip` and `{bucket_tail}` only. No secret value, but no reason to volunteer it either. |
| (New) Mutable `latest.zip` pointer? | **DROPPED — always pin to a versioned key.** | Every zip in S3 is immutable: `code/v0.1.28.zip`, `code/v0.1.29.zip`, … No `latest.zip`, no `last-known-good.zip`. Deployment = flip `SG_PLAYWRIGHT__CODE_S3_VERSION` env var on the Lambda. Rollback is the same operation with an older version string. The Lambda env var IS the source of truth for "what is running". |

## 4. Architecture recap (condensed)

Precedence from brief §36–47:

```
SG_PLAYWRIGHT__CODE_LOCAL_PATH → sys.path.insert(0, ...)          # laptop or volume mount
else AWS_REGION                → download s3://.../<version>.zip  # Lambda — <version> is
                                                                  #   always a pinned value
                                                                  #   like "v0.1.28" (never
                                                                  #   "latest"), sourced from
                                                                  #   SG_PLAYWRIGHT__CODE_S3_VERSION
else                           → passthrough sys.path             # pytest / uvicorn
```

Two version axes surfaced in `/health/info`:

```
service_version : v0.1.28        (from the extracted zip's version file)
image_version   : runtime-v1     (from /var/task/image_version baked at build time)
code_source     : sg-playwright-dev/code/v0.1.28.zip (redacted; see §3)
```

## 6. CI pipeline split — Track A vs Track B

### 6.1 Current state

`.github/workflows/ci-pipeline.yml` (the reusable workflow) has **seven jobs, linear**:

1. `run-unit-tests`
2. `build-docker-image`
3. `check-aws-credentials`
4. `push-to-ecr`
5. `run-integration-tests`
6. `increment-tag`
7. `deploy-lambda`
8. `smoke-test`

Every commit runs all of them — ~7 minutes of end-to-end time, dominated by the Docker build + ECR push, even when only one Python line changed.

### 6.2 Target state — two reusable workflows

Replace the single `ci-pipeline.yml` with two reusable workflows plus a thin dispatcher:

```
.github/workflows/
  ci-pipeline.yml                 ← dispatcher: decides Track A vs Track B based on changed paths
  ci-pipeline__track-a-image.yml  ← runs ~monthly when the image changes
  ci-pipeline__track-b-code.yml   ← runs every commit (fast path)
  ci-pipeline__dev.yml            ← unchanged trigger (pushes to dev)
  ci-pipeline__main.yml           ← unchanged trigger (pushes to main)
  ci-pipeline__prod.yml           ← unchanged trigger (manual)
```

### 6.3 Track A — runtime image (slow, infrequent)

**Trigger:** `paths:` filter on one of:
- `sgraph_ai_service_playwright/docker/**`
- `requirements.txt`
- `lambda_entry.py`
- `image_version`
- manual `workflow_dispatch`

**Jobs (sequential):**

| Job | Notes |
|---|---|
| `run-unit-tests` | Unchanged |
| `build-docker-image` | Unchanged |
| `run-integration-tests` | Unchanged (still validates the image itself) |
| `push-to-ecr` | Unchanged |
| `bump-image-version-tag` | Reads `image_version` file, tags ECR image with both `runtime-vN` and `latest` |
| `update-lambda-image` | Points dev Lambda at the new image URI. Env vars unchanged. |
| `smoke-test` | Unchanged |

**Expected wall time:** ~7 min (same as today — we're not optimising this path; it runs rarely).

### 6.4 Track B — code deploy (fast, every commit)

**Trigger:** everything NOT caught by Track A's `paths:` filter.

**Jobs (sequential):**

| Job | Notes |
|---|---|
| `run-unit-tests` | Unchanged — the safety net |
| `check-aws-credentials` | Unchanged |
| `package-code` | `python scripts/package_code.py` — uploads `code/{version}.zip` (immutable; only one key per version) |
| `update-lambda-env-var` | Bumps `SG_PLAYWRIGHT__CODE_S3_VERSION` on the dev Lambda to the new version → forces cold-start recycle |
| `run-deploy-tests-track-b` | `pytest tests/deploy/test_Deploy_Code__to__dev.py` — asserts the new version reports on `/health/info` |
| `smoke-test` | `pytest tests/deploy/test_Smoke__Playwright__Service__dev.py` — unchanged |
| `increment-tag` | Unchanged |

**Expected wall time:** ~90 s (unit tests + ~15 s S3 upload + ~10 s Lambda env-var bump + ~30 s cold start + ~15 s smoke).

### 6.5 Dispatcher (`ci-pipeline.yml` replacement)

```yaml
# Pseudocode — concrete YAML written during implementation
on: workflow_call
jobs:
  detect-changes:
    outputs: { track: ${{ steps.diff.outputs.track }} }
    steps:
      - uses: actions/checkout@v5
      - id: diff
        run: |
          if git diff --name-only $BEFORE..$AFTER | grep -Eq '^(Dockerfile|requirements\.txt|lambda_entry\.py|image_version|sgraph_ai_service_playwright/docker/)'; then
            echo "track=a" >> $GITHUB_OUTPUT
          else
            echo "track=b" >> $GITHUB_OUTPUT
          fi
  track-a:
    needs: detect-changes
    if: needs.detect-changes.outputs.track == 'a'
    uses: ./.github/workflows/ci-pipeline__track-a-image.yml
    secrets: inherit
  track-b:
    needs: detect-changes
    if: needs.detect-changes.outputs.track == 'b'
    uses: ./.github/workflows/ci-pipeline__track-b-code.yml
    secrets: inherit
```

Manual `workflow_dispatch` on `ci-pipeline__dev.yml` gets a new `force_track` input (`a` / `b` / `auto`) so ops can override.

### 6.6 First-time migration

The split creates a bootstrap paradox: the first Track B run needs the image to already know about `lambda_entry.py`. Sequence:

1. Land `lambda_entry.py` + Dockerfile change on `dev` → this triggers **Track A** (path filter matches). Image built + pushed + Lambda re-pointed.
2. Land `scripts/package_code.py` + first code zip manually (not via CI) on `dev` → validate S3 upload and extraction work.
3. Land the CI split itself on `dev` → this triggers **Track A** again because `.github/workflows/**` changes.
4. Land any small Python change → first Track B run fires. If green, the split is live.

## 7. Rollback mechanics

Deployments are just env-var flips. Rollback = deploy an older version string.

| Scenario | Command | Time to recover |
|---|---|---|
| Bad code version currently set on the Lambda | Bump `SG_PLAYWRIGHT__CODE_S3_VERSION` to the previous known-good value (every prior version remains in S3, immutable) | ~15 s (env-var update + cold start) |
| Image itself broken | Update Lambda `ImageUri` back to previous ECR tag | ~30 s |
| Total meltdown — boot shim fails | Update Lambda `ImageUri` to the pre-refactor tag | ~30 s |

The third case is the reason `lambda_entry.py` ships **in the image**, not in the zip. The image is the last line of defence.

**Why no `latest.zip` pointer:** the Lambda env var IS the authoritative pointer. "What code is the dev Lambda running?" → `aws lambda get-function-configuration ... | jq .Environment.Variables.SG_PLAYWRIGHT__CODE_S3_VERSION`. One read, no S3 key indirection. Every zip in S3 is an immutable artefact and every deployment event is visible in CloudTrail as an `UpdateFunctionConfiguration` call.

## 10. Success criteria

1. Typical code-only change: commit → green smoke test in **under 2 minutes** (vs ~7 min today).
2. `/health/info` on dev shows distinct `service_version`, `image_version`, and redacted `code_source`.
3. Rollback to previous code version takes **under 30 seconds** end-to-end (env-var flip + cold start; previous zips stay in S3 forever).
4. Unit tests stay green at 305+ (new boot-shim tests raise the count to ~315).
5. Image rebuild runs **only** when Track A's paths change — default path is Track B.
