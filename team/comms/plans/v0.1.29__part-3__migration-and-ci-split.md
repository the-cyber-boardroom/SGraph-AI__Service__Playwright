## 2. CI pipeline split

Three new workflow files replace the current monolithic `ci-pipeline.yml`.

### 2.1 `ci-pipeline__dev.yml` — the dispatcher

```yaml
name: CI Pipeline — dev (dispatcher)
on:
  push:
    branches: [dev]
  workflow_dispatch:

jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      container: ${{ steps.filter.outputs.container }}
      api:       ${{ steps.filter.outputs.api       }}
    steps:
      - uses: actions/checkout@v5
      - uses: dorny/paths-filter@v3
        id: filter
        with:
          filters: |
            container:
              - 'container/**'
              - '.github/workflows/**'
            api:
              - 'api/**'
              - 'requirements.txt'

  track-a-image:
    needs: detect-changes
    if: needs.detect-changes.outputs.container == 'true'
    uses: ./.github/workflows/ci-pipeline__track-a-image.yml
    secrets: inherit
    with:
      stage: dev

  track-b-code:
    needs: detect-changes
    if: needs.detect-changes.outputs.api == 'true'
    uses: ./.github/workflows/ci-pipeline__track-b-code.yml
    secrets: inherit
    with:
      stage: dev
```

**Rule:** a push can trigger both tracks. Track A runs first; Track B waits on its completion (add `needs: track-a-image` if A ran).

### 2.2 `ci-pipeline__track-a-image.yml` — container track

| Job | Tests run | AWS mutation |
|---|---|---|
| `unit-tests` | `container/tests/unit/` | — |
| `build-image` | `container/tests/docker/test_Build__Docker__SG_Lambda.py::test_build_docker_image` | — |
| `integration-tests` | `container/tests/integration/` | — |
| `push-to-ecr` | `container/tests/docker/test_ECR__Docker__SG_Lambda.py` | ECR push |
| `update-lambda-image` | `container/tests/deploy/test_Deploy_Image__to__dev.py::test_1__update_image_uri` | `Lambda.update_image_uri(...)` |
| `smoke-test-image` | `container/tests/deploy/test_Deploy_Image__to__dev.py::test_2__assert_image_version_on_info` | — |

**Bumps:** `container/image_version` (manual bump when the image needs to change).

### 2.3 `ci-pipeline__track-b-code.yml` — code track

| Job | Tests run | AWS mutation |
|---|---|---|
| `unit-tests` | `api/tests/unit/` | — |
| `integration-tests` | `api/tests/integration/` | — |
| `increment-tag` | — | git tag + bump `api/sgraph_ai_service_playwright/version` |
| `package-code` | — | `python api/scripts/package_code.py --lambda-name sg-playwright-api --stage dev` — uploads `v{new}.zip` to S3 |
| `update-lambda-env-var` | `api/tests/deploy/test_Deploy_Code__to__dev.py::test_1__bump_version_env_var` | `Lambda.set_env(SG_PLAYWRIGHT__CODE_S3_VERSION=v{new})` → cold-start |
| `smoke-test-code` | `api/tests/deploy/test_Deploy_Code__to__dev.py::test_2__assert_code_version_on_info` | — hits `/admin/info`, asserts `code_version==v{new}` AND `user_app_loaded==true` |

**Bumps:** `api/sgraph_ai_service_playwright/version` (every push via `increment-tag`).

### 2.4 Timings (target)

- Track A: ~5 min (image build dominates). Rare.
- Track B: ~2 min (no image build, no integration container). Every push.

## 3. Phasing against the live dev Lambda

The dev Lambda is currently running the v0.1.28 image — boot shim single-file, env vars NOT set, Function URL returning empty. This plan transitions it to v0.1.29 without losing the ability to roll back.

### Step 0 — Unblock the v0.1.28 Lambda (already discussed)

Before touching anything else, set on the dev Lambda:
```
SG_PLAYWRIGHT__CODE_LOCAL_PATH = /var/task
```
That makes the v0.1.28 shim use the baked code. `/health/info` starts responding. This is a TEMPORARY hack — removed in Step 3.

### Step 1 — Land Parts 1 + 2 design docs + Phase 1 file moves (api/)

- Merge `v0.1.29__part-{1,2,3}.md` to `dev`.
- Move files into `api/` per Phase 1.
- Create `api/pyproject.toml`, verify `pip install -e api` works.
- Run `pytest api/tests/unit/` — green.
- **No image rebuild yet.** The container still ships the old layout because `Build__Docker` still reads from the old path (updated in Step 2).
- **Dev Lambda: unaffected** — no image push, no env-var change.

### Step 2 — Phase 2 file moves (container/)

- Move files into `container/` per Phase 2.
- Update `Build__Docker__SG_Lambda.build_docker_image()` to stage from the new paths (both `container/` and `api/` sources).
- Write the new `lambda_entry.py` using `Code_Loader` + `Admin_API` (per Part 2).
- Write the admin FastAPI (`container/sg_lambda_runtime/admin_api/`).
- Run `pytest container/tests/unit/` — green.
- **Dev Lambda: still unaffected** — no image push.

### Step 3 — First v0.1.29 image build + manual deploy

- CI still runs the OLD monolithic `ci-pipeline.yml` (workflow split not landed yet). Push to `dev`. The old pipeline builds the image, but it's building the new layout — should succeed because `Build__Docker` was updated in Step 2.
- Manually (not CI) on dev Lambda console:
  1. Upload first `v0.1.29.zip` via `python api/scripts/package_code.py --lambda-name sg-playwright-api --stage dev`.
  2. Set env vars: `SG_PLAYWRIGHT__LAMBDA_NAME=sg-playwright-api`, `SG_PLAYWRIGHT__CODE_S3_STAGE=dev`, `SG_PLAYWRIGHT__CODE_S3_VERSION=v0.1.29`.
  3. Remove the Step-0 hack: unset `SG_PLAYWRIGHT__CODE_LOCAL_PATH`.
- Hit `https://<function-url>/admin/info` — assert `user_app_loaded==true`, `code_version==v0.1.29`.
- Hit `/health/info` (user app) — assert `service_version==v0.1.29`.
- If anything's broken, `/admin/info` tells us why.

### Step 4 — Land the CI split

- Merge `ci-pipeline__dev.yml` + `ci-pipeline__track-a-image.yml` + `ci-pipeline__track-b-code.yml`.
- Delete `ci-pipeline.yml` (the monolith).
- Next push to `dev`: the dispatcher fires, picks the right track based on changed paths. Both tracks can run in parallel (or in sequence if both are needed).

### Step 5 — Validation

- Run baseline: Dinis's proxy / curl harness from the v0.1.24 deploy, hitting the new Function URL. Must pass unchanged.
- Flip `SG_PLAYWRIGHT__CODE_S3_VERSION` back to `v0.1.28` → `/admin/info` should show `user_app_loaded==false` (because v0.1.28 zip never existed under the new path scheme) OR succeed if we back-fill a v0.1.28 zip. Either way, the rollback mechanism is exercised.

## 5. Risks and mitigations

| Risk | Mitigation |
|---|---|
| File moves break imports silently (no test covers the import path) | Grep for `from sgraph_ai_service_playwright.docker` before Phase 1; expect zero matches outside `container/` |
| `git mv` history gets lost because rename-detection threshold is too high | Use `git log --follow` to verify; if broken, split into smaller renames |
| Track A + Track B race on the same push | Dispatcher sequences them: Track B waits on Track A if both changed. Worst case, Track B picks up the new image on its next run |
| Dev Lambda outage during Step 3 | Step 0 hack keeps v0.1.28 image functional. The Step 3 cutover is reversible: flip env var back, remove the new zip, restore Step 0 hack |
| `osbot_docker` missing in CI (already a known issue for tests under `container/tests/docker/`) | Add to `requirements.txt` or skip-gate those tests on its presence. Same fix as today |
| Admin FastAPI hides user FastAPI bug (operator thinks `/admin` = `/health`) | Distinct path prefixes + explicit `user_app_loaded` flag in `/admin/info`. Document in README |

## 7. Success criteria

1. `api/` and `container/` each installable independently via `pip install -e .`.
2. `pytest api/tests/unit/` and `pytest container/tests/unit/` both green, no cross-package imports.
3. Dev Lambda serves `/admin/info` with `user_app_loaded==true`, `code_version==v0.1.29`.
4. Flipping `SG_PLAYWRIGHT__CODE_S3_VERSION` on dev Lambda changes `code_version` on `/admin/info` after a cold start.
5. Track A pipeline runs only when `container/**` changes; Track B only when `api/**` changes.
6. No file in `api/` imports from `container/`; no file in `container/` imports from `sgraph_ai_service_playwright.*`.
