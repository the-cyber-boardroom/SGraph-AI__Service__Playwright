# v0.2.28 — `sg credentials`: deferred work + REPL substring fall-back

Follow-up to v0.2.25 (Phases A–E shipped) and v0.2.27 (REPL substring). Folds
the three deferred v0.2.25 phases (C edit mode, F migration, G transparency +
backup) and the small `sg repl` substring enhancement into a single release.

---

## 1. TL;DR

| Item | Decision |
|------|----------|
| **Goal** | Finish the v0.2.25 contract: edit-mode UX, transparency layer, backup/restore, and migrate every existing `*__AWS__Client.client()` to the role-aware seam. Land the v0.2.27 REPL substring fall-back at the same time. |
| **Scope** | 4 phases. ~50 new files (mostly schemas, audit-event records, primitives, edit-mode helpers); ~15 modified files (existing `*__AWS__Client.client()` callers). |
| **Compatibility** | Pure additions on the credential surface. Phase F (client migration) is behaviour-preserving when no role is configured — the new seam falls through to the same `boto3.client(service)` it does today. |

| **Cost** | ≈ 1.5–2 days (or two parallel agent runs: A+B together, then C+D). |
| **Dependency** | v0.2.25 (Phases A–E) shipped. v0.2.26 (sg aws lambda) shipped — gives us a non-trivial `Lambda__AWS__Client` + `Logs__AWS__Client` to migrate as part of Phase F. |

## 2. What's missing from v0.2.25 (review findings)

The Phase A–E ship covered the keyring backend, the standard CLI commands (`list/add/remove/switch/show/status/log/trace/export`), the `sg osx keyring.*` low-level surface, the `as <role>` REPL pseudo-command, and `Sg__Aws__Session.boto3_client()` + STS AssumeRole. Independent review surfaced these gaps against the v0.2.25 (rev 4) brief:

1. **7 of ~16 specified CLI commands missing** — `init`, `test`, `whoami`, `route`, `wipe`, `backup`, `restore`.
2. **§3.8 transparency layer essentially absent.** The audit log records `{action, timestamp, role}` only; the brief specifies `identity_arn`, `session_id`, `resolved_via` (chain), `command_args` (allowlist-redacted), `session_expiry`, `duration_ms`. No identity banner on command startup. No correlatable STS session-name factory (`sg-<role>-<ts>-<uuid>`) — current code uses placeholder `'sg-cli'`. `sg credentials trace` exists but is an audit-log grep, not the dry-run resolver §3.8.4 specified.
3. **Phase F migration not started.** Zero of the ~7 existing `*__AWS__Client.client()` callers (IAM, ACM, CF, Lambda, EC2, Route53, Elastic, Logs) were migrated to `Sg__Aws__Session.boto3_client()`. The seam exists but is unused — meaning the `as <role>` REPL pseudo-command currently has zero effect on actual AWS calls.
4. **Edit mode (§4.2) entirely absent.** No `sg credentials edit`, no temp `.toml.sg-edit` projection, no `$EDITOR` round-trip, no parse-error retry loop, no orphan recovery.
5. **Backup/restore (Phase G / §7) absent.** No `sg credentials backup` / `restore`.
6. **§3.7 invariant 5 not enforced.** Audit log file is created with default umask; spec mandates 0600.
7. **`Credentials__Store` lacks recursive `source_role` chain resolution** for nested assume-role chains.

## 3. Design — phase by phase

### 3.1 Phase A — Transparency & audit (§3.8 of v0.2.25)

This is the most important deferred piece. Without it, `sg --as admin foo` has no operator-visible feedback explaining *which* identity actually ran, and the audit log can't be cross-referenced with CloudTrail.

#### 3.1.1 Identity banner

Print a single dim stderr line before every credential-resolved command:

```
[sg] role=admin → arn:aws:sts::123456789012:assumed-role/sg-admin/sg-cli-1715961234-a1b2c3d4 (expires in 58m)
```

Suppression rules:
- `--quiet` flag on the root `sg` group suppresses it
- Auto-suppressed when stderr is not a TTY (CI logs stay clean)
- Auto-suppressed for read-only verbs that didn't actually call AWS (banner is post-resolution, not pre-)

Implementation: a `Sg__Aws__Session` callback (`on_resolved`) fires once per session creation. `Cli__SG.py` registers a banner-printer at top-level startup; tests register a list-appender.

#### 3.1.2 Correlatable STS session names

Replace the `'sg-cli'` placeholder in `Sg__Aws__Session._assume_role()` (current code, line 39) with:

```python
def _session_name(role_name: str) -> str:                                          # sg-<role>-<ts>-<uuid8>
    return f'sg-{role_name}-{int(time.time())}-{uuid.uuid4().hex[:8]}'
```

64-char IAM SessionName limit means we truncate the role component to ~40 chars. The full session name is included in every audit-log event for CloudTrail correlation.

#### 3.1.3 Rich audit-log events

Replace `Schema__Audit__Event` with the full v0.2.25 §3.8.2 shape:

```python
class Schema__Audit__Event(Type_Safe):
    timestamp        : Safe_Str__Iso8601_Timestamp
    role             : Safe_Str__Role__Name
    identity_arn     : Safe_Str__Aws__Arn                    # STS GetCallerIdentity result
    session_id       : Safe_Str__Sts__Session_Name           # correlatable with CloudTrail
    session_expiry   : Safe_Str__Iso8601_Timestamp
    action           : Enum__Audit__Action
    command_args     : Safe_Str__Audit__Command_Args         # allowlist-redacted
    resolved_via     : List__Safe_Str__Role__Name            # e.g. ['default', 'admin'] for chained assume-role
    duration_ms      : Safe_Int__Duration__Millis
    success          : bool
    error            : Safe_Str__Error__Message              # empty on success
```

**Command-args redaction allowlist:** the redactor takes a typed argv list and emits a string where every value not in `ALLOWED_ARG_NAMES` is replaced with `<redacted>`. The allowlist is per-command and lives next to the command definition (e.g. `sg credentials add` allows `<role-name>` to appear; `<access-key>` and `<secret-key>` are always redacted). Default-deny.

Append-only JSONL at `~/.sg/audit.jsonl`. Append failures must never block the primary operation; they go to stderr as a dim warning.

#### 3.1.4 File mode 0600 enforcement (§3.7 inv 5)

`Audit__Log._ensure_file()` calls `os.chmod(path, 0o600)` after creation and asserts mode on every append (warn on stderr if widened). 50 MB rotation: `~/.sg/audit.jsonl` → `~/.sg/audit.jsonl.1` (single roll, no archive history).

#### 3.1.5 `sg credentials trace` — dry-run resolver

```
$ sg credentials trace aws lambda waker info
[trace] command path:    aws lambda waker info
[trace] routing match:   aws lambda *      → role: deploy-lambda
[trace] role chain:      default → deploy-lambda (assume)
[trace] would assume:    arn:aws:iam::123456789012:role/sg-deploy-lambda
[trace] session name:    sg-deploy-lambda-<ts>-<uuid> (dry-run; not generated)
[trace] effective creds: source=keyring (sg.aws.default.access_key)
```

No AWS calls. Pure resolver dump. Implemented as `Credentials__Resolver.trace(command_path)` returning a `Schema__Trace__Result` that the CLI formats.

### 3.2 Phase B — Edit mode (§4.2)

`sg credentials edit` projects the current keyring state to a temp `.toml.sg-edit` file, opens `$EDITOR` (blocking), parses the saved file, diffs against the in-memory snapshot, prints a confirmation table, and applies + shreds.

#### 3.2.1 Files

```
sgraph_ai_service_playwright__cli/credentials/edit/
├── Credentials__Editor.py            # orchestrator
├── Editor__Launcher.py               # subprocess wrapper around $EDITOR (test seam)
├── Toml__Projector.py                # in-memory → TOML text
├── Toml__Parser.py                   # TOML text → in-memory snapshot
├── Edit__Diff.py                     # snapshot vs snapshot → typed diff
├── Edit__Session__Journal.py         # ~/.sg/edit-sessions/<uuid>.json (orphan recovery)
└── Temp__File__Manager.py            # secure mkstemp + shred (Safe_Str__Edit__Session__Id)
```

#### 3.2.2 Flow

1. **Session start.** Generate UUID, create journal file `~/.sg/edit-sessions/<uuid>.json` with `{started_at, status: 'projecting'}`. mkstemp `.toml.sg-edit`.
2. **Project.** Read every keyring entry under `sg.config.*`, `sg.aws.*`, `sg.vault.*`, `sg.secret.*` via `Credentials__Store`. Render as commented TOML (sections per role, vault, secret namespace). Secrets show as `********` placeholder + `# (sealed — replace to change)` comment.
3. **Launch.** `Editor__Launcher.launch(path)` — `subprocess.run([editor, path], check=True)`. Blocks until editor exits. `$VISUAL` > `$EDITOR` > `vi` fallback.
4. **Parse.** `Toml__Parser` returns either `Schema__Edit__Snapshot` or a parse error with `(line, col, message)`. On parse error: print the error, offer retry up to 3 times (`Re-open in editor? [Y/n]`), then abort.
5. **Diff.** `Edit__Diff` produces `Schema__Edit__Diff` listing additions, removals, modifications grouped by namespace. Modified secrets (those whose value is no longer `********`) are diffed by presence-only — values never appear in the diff output.
6. **Confirm.** Print a rich table with the diff. Ask `Apply these changes? [y/N]`. Default no.
7. **Apply.** Each diff item maps to a `Credentials__Store` mutation. Apply in order: deletes first, then additions, then modifications. On any failure, rollback by re-projecting the pre-edit snapshot to keyring (Phase B follows the standard "snapshot → ops → maybe rollback" pattern).
8. **Shred.** `Temp__File__Manager.shred(path)` overwrites with random bytes then unlinks. Journal status → `'completed'`.

#### 3.2.3 Orphan recovery

On `sg credentials edit` startup, scan `~/.sg/edit-sessions/`. Any entry with `status != 'completed'` and `started_at > 1h ago` is an orphan (editor crashed or process killed). Offer:

```
Found orphaned edit session <uuid> started <X> ago.
Options: (r)esume — re-open the temp file; (d)iscard — shred and journal as discarded; (s)kip
```

If the temp file is missing, only `discard` and `skip` are available.

### 3.3 Phase C — Missing standard commands

| Command | Behaviour |
|---------|-----------|
| `sg credentials init` | First-time wizard. Prompts for: default role name, AWS account, region. Creates `sg.config.role.default` + writes access/secret to `sg.aws.default.*` (prompted, never on argv). Idempotent. |
| `sg credentials test [<role>]` | Calls `sts:GetCallerIdentity` via `Sg__Aws__Session.boto3_client(role, 'sts')`. Prints `arn`, `account`, `userid`, `session_expiry`. Non-zero exit on failure. |
| `sg credentials whoami` | Prints the active role (from `Sg__Aws__Context.current_role` if set, else routing default). Does NOT call AWS. For "what would happen now if I ran a command" use `trace`. |
| `sg credentials route <command...>` | Shorthand for `trace` that only prints the resolved role name (single line, scriptable). `sg credentials route aws lambda waker` → `deploy-lambda`. |
| `sg credentials wipe` | Deletes every `sg.*` keyring entry. Refuses without `--yes-i-really-mean-it`. Audited. |
| `sg credentials backup --to <path>` | Dumps all `sg.*` keyring entries to an age-encrypted tarball. Requires `--passphrase` (prompt). No secrets in argv. |
| `sg credentials restore --from <path>` | Inverse. Refuses if any target `sg.*` entry already exists unless `--overwrite`. |

`age` is a single static binary — bundled via `python-age` (small pure-Python age implementation) to avoid adding a system dependency. If python-age proves unstable we can fall back to wrapping the `/usr/bin/age` binary (homebrew on mac).

### 3.4 Phase D — AWS client migration (Phase F of v0.2.25)

Every existing `*__AWS__Client` currently has:

```python
def client(self):
    kwargs = {}
    if self.region:
        kwargs['region_name'] = self.region
    return boto3.client('lambda', **kwargs)
```

Migrate to:

```python
def client(self):
    return Sg__Aws__Session.from_context().boto3_client(service_name='lambda', region=self.region)
```

`Sg__Aws__Session.from_context()` reads `Sg__Aws__Context.current_role` (REPL `as` pin) or the routing-table default. When no role is configured, `Sg__Aws__Session` falls through to bare `boto3.client(service)` — behaviour-preserving for users who never set up credentials.

**Clients to migrate:**

| Client | Owner module | Notes |
|--------|--------------|-------|
| `Lambda__AWS__Client` | `aws/lambda_/service/` | v0.2.26 |
| `Logs__AWS__Client` | `aws/logs/service/` | v0.2.26 |
| `Lambda__Deployer` | `aws/lambda_/service/` | uses boto3 directly — same pattern |
| `Cloudfront__AWS__Client` | `aws/cf/service/` | |
| `IAM__AWS__Client` | `aws/iam/service/` | |
| `ACM__AWS__Client` | `aws/acm/service/` | |
| `Route53__AWS__Client` | `aws/dns/service/` | |
| `Cost_Explorer__AWS__Client` | `aws/billing/service/` | |
| `Elastic__AWS__Client` | `sgraph_ai_service_playwright__cli/elastic/service/` | EC2 + SG + KP, multi-service |
| `EC2__Resource__Resolver` and similar in `firefox/`, `neko/`, `vnc/`, `podman/` | various | grep `boto3.client(` to find all |

`Sg__Aws__Session.from_context()` caches sessions per `(role, region)` tuple for ≤ session_expiry minus 5-min refresh window. Each session creation fires the `on_resolved` callback (Phase A banner).

### 3.5 Phase E — REPL substring fall-back (was v0.2.27)

Fold v0.2.27 in: extend `Cli__SG__Repl._match()` so substring matching kicks in when prefix matching returns zero hits. ~15 lines.

```python
def _match(prefix, options):                                                       # prefix first; substring fall-back
    prefix_hits    = sorted(o for o in options if o.startswith(prefix))
    if prefix_hits:
        return prefix_hits, 'prefix'
    substring_hits = sorted(o for o in options if prefix in o)
    return substring_hits, 'substring'
```

`_resolve()` consumes the `(hits, kind)` tuple and prints a dim visibility line when a substring match wins:

```
sg/aws/lambda> waker info
  → matched 'waker' as 'sg-compute-vault-publish-waker' (substring)
sg/aws/lambda/sg-compute-vault-publish-waker> info
```

Conservative tie-breaking: 2+ substring hits returns the ambiguity list unchanged (no auto-pick).

## 4. Implementation order

| # | Phase | Why this order |
|---|-------|---------------|
| 1 | **A — Transparency** | Cheapest and most independent. Every other phase benefits from real audit events being correlatable in CloudTrail. |
| 2 | **D — Client migration** | Once banner + audit are real, migration becomes load-bearing. Migrate `Lambda__AWS__Client` first (small, well-tested), then sweep the rest. Each client migration is one commit + a passing test run. |
| 3 | **C — Missing commands** | `init`, `test`, `whoami`, `route` slot in naturally once A+D are done because they all use the new session infrastructure. `wipe`/`backup`/`restore` are independent. |
| 4 | **B — Edit mode** | Biggest UX surface; benefits from C's `Credentials__Store` already being battle-tested. |
| 5 | **E — REPL substring** | Tiny, last. ~1 hour. |

Phases A+D can run in parallel. Phase B is best done solo (lot of file surface, easy to introduce subtle bugs). Phase C splits into "needs Sg__Aws__Session" (init/test/whoami/route) and "doesn't" (wipe/backup/restore) — those two halves can also run in parallel.

## 5. Acceptance criteria

1. `sg credentials init` walks a fresh user through setting up `default` role; the resulting `sg.config.role.default` + `sg.aws.default.access_key` + `sg.aws.default.secret_key` round-trip via `sg credentials show default`.
2. `sg credentials test default` returns the `sts:GetCallerIdentity` result (real STS call); `sg credentials test bogus-role` exits non-zero.
3. `sg credentials whoami` prints the active role without making any AWS call.
4. `sg credentials route aws lambda waker` prints exactly one role name on stdout.
5. `sg credentials trace aws lambda waker info` prints the dry-run resolver output (path, route match, chain, would-assume ARN, session-name template) — no AWS calls.
6. `sg credentials edit` opens `$EDITOR`, accepts edits, diffs, confirms, applies; round-trips with no semantic change when saved unmodified. Parse errors retry up to 3 times. Crashed editor → orphan recovery offer on next run.
7. `sg credentials backup --to /tmp/b.tar.age` writes an encrypted tarball; `sg credentials wipe --yes-i-really-mean-it && sg credentials restore --from /tmp/b.tar.age` restores all entries.
8. `sg credentials wipe` refuses without confirmation flag.
9. Audit log entries include `identity_arn`, `session_id`, `resolved_via`, `command_args` (with secrets replaced by `<redacted>`), `session_expiry`, `duration_ms`. File mode is 0600 after creation and after every append.
10. STS session names match pattern `sg-<role>-<unix_ts>-<8_hex>`.
11. Identity banner prints once per resolved-session command; suppressed by `--quiet` and when stderr is not a TTY.
12. Every existing `*__AWS__Client.client()` (≥ 8 clients) is migrated to `Sg__Aws__Session.boto3_client()` with no behavioural change when no role is configured.
13. `sg repl` → `aws lambda waker` resolves uniquely to `sg-compute-vault-publish-waker` (substring) and prints the dim visibility line.
14. `sg repl` → `aws lambda playwright` returns the ambiguity list when multiple functions contain `playwright`.
15. All existing v0.2.25 + v0.2.26 + v0.2.27 (now folded) tests still pass; ≥ 80 new tests added.

## 6. Open questions

| Q | Question | Architect's default |
|---|----------|---------------------|
| Q1 | `age` backend — `python-age` library or `/usr/bin/age` subprocess? | python-age (no system dep). Reconsider if it has bugs at scale. |
| Q2 | Should the identity banner appear on `--help` and `--version`? | No — those don't resolve credentials. Banner is post-resolution. |
| Q3 | Audit log rotation: single roll at 50 MB, or archive history with `audit.jsonl.N` ring? | Single roll. Operators who need long-term retention should ship to S3/CloudWatch via an external sink (out of scope). |
| Q4 | Should `sg credentials wipe` also delete `~/.sg/audit.jsonl` and edit-session journals? | No — wipe targets credentials only. Audit log is forensic and should outlive a wipe. Document this clearly. |
| Q5 | Edit-mode shred: 1-pass random overwrite enough on SSD, or do we need 3-pass + sync? | 1-pass + sync + unlink. SSDs make multi-pass theatre; the FDE on the laptop is the real guarantee. |
| Q6 | Should `Sg__Aws__Session.from_context()` cache include the access-key fingerprint, so rotating the underlying key invalidates the cache without an explicit `--reload`? | Yes — cache key = `(role, region, access_key_fingerprint)`. Avoids a class of "I rotated my key but the old session still works" surprises. |
| Q7 | When `sg --as admin` is used outside the REPL, should the banner still print (it's not a "session" in the REPL sense)? | Yes — banner is per-command. Sessions only matter for cache lifetime. |
| Q8 | Phase F migration: bundle all 8+ client migrations in one PR or split per client? | One PR per client, each shipping its own commit. Easier to bisect if something regresses. |

## 7. Anti-scope

- Cross-platform keyring (Linux Secret Service, Windows Credential Manager) — still macOS-only
- Hardware token / YubiKey integration
- AWS SSO / IAM Identity Center login flow
- Web UI for credentials management
- Audit-log shipping (S3, CloudWatch) — operators who need this build their own sink
- TAB-completion in REPL prompt (would need `readline` integration)
- Fuzzy ranking with scores in the REPL substring matcher (subsequence, etc. — kept conservative)

## 8. Migration notes

- Operators upgrading from v0.2.25 Phase A–E will see new keyring keys appear under `sg.config.*` only if they use `init`. Existing entries continue to work unchanged.
- Audit log format changes: the new fields are additive; readers should tolerate missing fields on pre-v0.2.28 events. A v0.2.28 audit-log compaction tool is OUT of scope.
- Phase F migration of `*__AWS__Client.client()` is behaviour-preserving when no role is configured. The first operator to actually configure a role and observe a wrong behaviour should file a bug.
