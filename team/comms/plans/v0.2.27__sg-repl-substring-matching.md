# v0.2.27 — `sg repl`: substring matching fall-back

A small enhancement to the REPL's per-segment resolver so that operators
can use substring matches (not just prefixes) when navigating fuzzy-name
slots like `sg aws lambda <name>` or `sg aws ec2 <instance>`.

---

## 1. TL;DR

| Item | Decision |
|------|----------|
| **Goal** | In `sg repl`, when prefix matching finds zero hits, fall back to substring matching. Behaviour stays identical when prefix matching finds one or more hits. |
| **Scope** | Single file: `sg_compute/cli/Cli__SG__Repl.py` — extend the `_match()` helper and `_resolve()` loop. ~15 lines. |
| **Compatibility** | Pure addition. Existing prefix-resolves keep working unchanged. New substring-resolves print a one-line `→ matched 'waker' as 'sg-compute-vault-publish-waker' (substring)` to make it visible. |
| **Tests** | ~6 new tests in the existing REPL test suite. |
| **Cost** | ≈ 1 hour |
| **Dependency** | None — independent of v0.2.25 and v0.2.26; can ship before, after, or alongside either. |

---

## 2. Problem

The REPL's `_match()` function does pure prefix matching:

```python
def _match(prefix, options):
    return sorted(o for o in options if o.startswith(prefix))
```

With v0.2.26 landed, REPL navigation into a Lambda function works:

```
sg/aws/lambda> sg-c<Enter>                ← matches sg-compute-vault-publish-waker (prefix)
sg/aws/lambda/sg-compute-vault-publish-waker>
```

But the v0.2.26 CLI surface accepts substring fuzzy:

```
$ sg aws lambda waker info                ← works (substring)
$ sg aws lambda wa info                   ← works (substring)
```

The same shortcuts should work in REPL. Today they don't:

```
sg/aws/lambda> waker<Enter>               ← no match (waker is not a prefix of any function)
sg/aws/lambda> wa<Enter>                  ← no match
```

Operators have to remember "REPL is prefix-only, CLI is substring-fuzzy",
which is gratuitous mental overhead.

---

## 3. Design

### 3.1 The `_match()` change

Two-stage match with explicit visibility when the second stage kicks in:

```python
def _match(prefix, options):
    """Prefix match first; fall back to substring match if prefix returns zero."""
    prefix_hits    = sorted(o for o in options if o.startswith(prefix))
    if prefix_hits:
        return prefix_hits, 'prefix'
    substring_hits = sorted(o for o in options if prefix in o)
    return substring_hits, 'substring'
```

Returns the matches AND the match-kind so the caller can announce
substring hits.

### 3.2 The `_resolve()` change

```python
def _resolve(sg_app, base_path, words):
    current = list(base_path)
    for i, word in enumerate(words):
        if word.startswith('-') or not _is_group(sg_app, current):
            return current, list(words[i:])
        available = _children(sg_app, current) - ({'repl'} if not current else set())
        hits, kind = _match(word, available)
        if len(hits) == 1:
            current.append(hits[0])
            if kind == 'substring':
                console.print(f'  [dim]→ matched {word!r} as {hits[0]!r} (substring)[/dim]')
        elif len(hits) > 1:
            return None, hits
        else:
            return current, list(words[i:])
    return current, []
```

Visibility line uses `[dim]` so it doesn't compete with command output but
operators can still see WHY the resolver picked what it picked. Important
for the same reason §3.8 (transparency) matters in v0.2.25 — magic that
silently picks the wrong thing is worse than no magic.

### 3.3 Case sensitivity

Both stages stay case-sensitive for v1 (matches the current prefix
behaviour). If operators want `LAMBDA` to match `lambda`, that's a separate
follow-up — most CLI tools keep case sensitivity at this level and it
avoids surprising matches.

### 3.4 Tie-breaking

If substring matching returns 2+ hits, the existing ambiguity path applies
unchanged — the resolver returns `(None, hits)` and the REPL prints the
candidate list. Substring match doesn't get a "best of" auto-pick like the
v0.2.26 fuzzy resolver — REPL stays conservative. The fuzzy CLI resolver
in `Lambda__Name__Resolver` has its own tie-breaking (the +10 score margin)
and that's appropriate for direct CLI use, less so for incremental REPL
navigation where the operator can simply type one more character.

---

## 4. Implementation

Single PR, single file:

- Update `sg_compute/cli/Cli__SG__Repl.py`:
  - `_match()` returns `(hits, kind)` tuple
  - `_resolve()` consumes the tuple and prints the dim substring-match line
- Add ~6 tests to the existing REPL test suite covering:
  1. Prefix match still works (regression)
  2. Substring match resolves uniquely
  3. Substring match with multiple hits returns the ambiguity list
  4. Visibility line is printed for substring matches but not prefix matches
  5. Substring fall-back is per-segment (a substring match at one level
     doesn't affect prefix behaviour at the next level)
  6. Word starting with `-` (option flag) still short-circuits as before

---

## 5. Acceptance criteria

1. `sg repl` then `aws lambda waker` resolves uniquely to
   `sg-compute-vault-publish-waker` and prints the visibility line.
2. `sg repl` then `aws lambda playwright` returns the ambiguity list
   (multiple functions contain "playwright").
3. `sg repl` then `aws lambda sg-c` still works as before (prefix match,
   no visibility line printed).
4. `sg repl` then `aws lambda xyz` returns "not found" / falls through as
   trailing args — same as today.
5. All existing REPL tests pass without modification.

## 6. Open questions

| Q | Question | Architect's default |
|---|----------|---------------------|
| Q1 | Make matching case-insensitive at the same time? | No — separate follow-up. Keep this PR minimal. |
| Q2 | Should the visibility line be opt-out via `--quiet`? | No — it's already dim. If operators object after using it, we can add an opt-out then. |
| Q3 | Should substring matching apply to verbs as well as fuzzy-name slots? | Yes — `_match` is segment-agnostic, the behaviour is uniform. Means `aws lambda waker inf` would substring-match `info`. Slight risk of surprise but consistent. |
| Q4 | Should this also subsume the v0.2.26 §4.1 fuzzy resolver into the REPL? | No — keep them separate. CLI fuzzy uses subsequence matching and tie-breaking heuristics that are too aggressive for incremental REPL navigation. |

---

## 7. Anti-scope

- Case-insensitive matching (separate follow-up if desired)
- Subsequence / character-skip matching (the v0.2.26 §4.1 resolver tier 5)
- TAB-completion in the REPL prompt (would need `readline` integration; separate effort)
- Fuzzy ranking with scores (over-engineering for the REPL use case)
