# v0.2.25 — sg credentials & multi-role workflow

This document proposes how `sg` should manage multiple AWS identities locally so
operators can use least-privilege roles per command without ever giving the
day-to-day deploy user dangerous powers like `iam:*`.

It is the foundation for the future "credentials server for agents" — once the
local file path is stable, the server is a drop-in replacement for the loader.

---

## 1. TL;DR

| Item | Decision (gating answers captured 2026-05-17) |
|------|------------------------------------------------|
| File location | **Project-local only:** `./sg-compute/aws/.credentials.toml` |
| File format | **TOML** (sections, comments, stdlib `tomllib`) |
| Credential types | **Both** from v1: static access keys AND STS `AssumeRole` |
| Role selection | (1) explicit `--as <name>` flag, (2) auto-route from `[command_routes]`, (3) `default` fallback |
| Single seam | One `Sg__Aws__Session` factory; every `*__AWS__Client.client()` consults it |
| Safety | File-mode 0600 enforced; credentials never logged; `.gitignore` updated; `sg credentials check` verifies hygiene |
| Future | Same loader interface, swap file backend for HTTP credentials-server backend |

## 2. Problem & current state

### 2.1 The trigger

```
$ sg aws iam role list
  ✗  ClientError: AccessDenied — User: arn:aws:iam::745506449035:user/SG-Deploy-User
     is not authorized to perform: iam:ListRoles
```

This is the **correct** failure mode — `SG-Deploy-User` should NOT have `iam:*`.
But operators currently have only two options:

1. Add `iam:*` to `SG-Deploy-User` (bad — most powerful user gets more powerful)
2. Manually export `AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY` from a different user
   for every IAM command (tedious, error-prone, easy to leave the wrong creds
   in the env)

Neither matches the "smooth + secure + least-privilege" workflow we want.

### 2.2 What's missing

- **No notion of named credentials** inside `sg`. Today every `boto3.client(...)`
  call inherits the ambient environment.
- **No per-command role routing.** `sg aws iam *` could auto-pick an
  `iam-admin` identity; `sg vp *` could auto-pick `vault-publish-operator`.
- **No audit trail** of which identity ran which command. With multiple keys
  in flight, this becomes essential.
- **No isolation from `~/.aws/credentials`.** Mixing day-job AWS profiles with
  project-specific keys is a recipe for cross-contamination — and per CLAUDE.md
  rule 12 (no AWS creds in Git), we can't lean on the repo to enforce isolation
  either.

## 3. Design

### 3.1 The credentials file

**Path** (project-local): `./sg-compute/aws/.credentials.toml`
**Permissions**: must be `0600`; `sg` refuses to read if `mode & 0o077 != 0`.
**`.gitignore`**: add `sg-compute/aws/.credentials.toml` and
`sg-compute/aws/.sessions/` on first run if not already ignored.

Example file:

```toml
# ─────────────────────────────────────────────────────────────────────────────
# sg-compute / aws / credentials                                  (chmod 0600)
# Each [roles.<name>] block is one AWS identity. Either:
#   • static keys:    access_key + secret_key
#   • assume-role:    source_role + role_arn (+ optional external_id, session_name)
# Region is per-role; default region is "eu-west-2" if unset.
# ─────────────────────────────────────────────────────────────────────────────

default_role = "deploy"

[roles.deploy]                                    # day-to-day deploys
type        = "static"
access_key  = "AKIA…"
secret_key  = "…"
region      = "eu-west-2"
account_id  = "745506449035"
note        = "SG-Deploy-User — least-privilege deploy creds"

[roles.iam-admin]                                 # IAM-only role
type            = "assume_role"
source_role     = "deploy"                        # base creds used to call sts:AssumeRole
role_arn        = "arn:aws:iam::745506449035:role/sg-admin-iam"
session_name    = "sg-iam-admin"
duration_seconds= 3600
region          = "eu-west-2"
note            = "AssumeRole into IAM-only admin; trust policy restricts to deploy user"

[roles.vault-publish-operator]
type        = "static"
access_key  = "AKIA…"
secret_key  = "…"
region      = "eu-west-2"

# ─────────────────────────────────────────────────────────────────────────────
# Auto-routing — first matching glob wins.
# Override with `sg --as <role> …` at any time.
# ─────────────────────────────────────────────────────────────────────────────
[command_routes]
"aws iam *"            = "iam-admin"
"aws lambda *"         = "vault-publish-operator"
"aws cf *"             = "vault-publish-operator"
"vp *"                 = "vault-publish-operator"
"*"                    = "deploy"                 # explicit fallback
```

### 3.2 Schema (in code)

```
sgraph_ai_service_playwright__cli/aws/credentials/
├── enums/
│   └── Enum__Credentials__Type.py             # STATIC / ASSUME_ROLE
├── primitives/
│   ├── Safe_Str__Aws__Access_Key.py           # ^AKIA[0-9A-Z]{16}$
│   ├── Safe_Str__Aws__Secret_Key.py           # 40-char base64-ish (NOT logged)
│   ├── Safe_Str__Aws__Account_Id.py           # ^\d{12}$
│   └── Safe_Str__Credentials__Role_Name.py    # ^[a-z][a-z0-9-]{1,40}$
├── schemas/
│   ├── Schema__Credentials__Role__Static.py
│   ├── Schema__Credentials__Role__Assume_Role.py
│   ├── Schema__Credentials__Route.py          # pattern → role_name
│   ├── Schema__Credentials__File.py           # parsed top-level
│   └── Schema__Credentials__Session__Cache.py # cached STS session (expiry, token)
├── service/
│   ├── Credentials__Store.py                  # load + validate TOML
│   ├── Credentials__Router.py                 # command-string → role_name
│   ├── Sg__Aws__Session.py                    # SINGLE SEAM: returns boto3.Session
│   ├── Sg__Aws__Context.py                    # CLI-set current role (thread-local)
│   └── Session__Cache.py                      # disk-backed STS session cache
└── cli/
    └── Cli__Credentials.py                    # sg credentials add/list/show/test/check/whoami
```

### 3.3 The single seam — `Sg__Aws__Session`

Every existing `*__AWS__Client.client()` method goes from this:

```python
def client(self):
    return boto3.client('iam')
```

to this:

```python
def client(self):
    return Sg__Aws__Session().boto3_client('iam')
```

`Sg__Aws__Session.boto3_client(service)`:

1. Reads the active role name from `Sg__Aws__Context.current_role()`
2. Asks `Credentials__Store` for the resolved credentials of that role
   (recursively resolving `source_role` for `assume_role` entries)
3. If `type=assume_role`: checks `Session__Cache`; if cache miss or
   expiring within 5 min, calls `sts:AssumeRole` and caches the result
4. Returns `boto3.Session(...).client(service)`

In-memory test composition stays unchanged — tests subclass the `*__AWS__Client`
and override `client()`, bypassing the session factory.

### 3.4 Role selection — three layers, deterministic

```
   sg --as iam-admin aws iam role list      ← (1) explicit flag wins
   sg aws iam role list                      ← (2) routes match "aws iam *" → iam-admin
   sg some-other-command                     ← (3) "*" route → "deploy" (or default_role)
```

`Credentials__Router.route(command_path: str) -> role_name`:

- `command_path` is the dot/space-joined Typer path (e.g. `"aws iam role list"`).
- Routes are tried **in file order** (first match wins). `fnmatch` semantics.
- If no route matches and no `default_role` is set, the command **exits 2** with
  a clear "no credential available" message — never silently uses ambient AWS
  env vars (that defeats the whole isolation goal).

`Sg__Aws__Context` is set once per CLI invocation via a root `@app.callback()`
that reads the `--as` flag and the routed command path. Tests can override it
with a context manager.

### 3.5 New CLI surface

```
sg credentials list                          # all configured roles (no secrets shown)
sg credentials show <role>                   # full entry (still no secrets — shows  ****)
sg credentials add  <role> --type static \   # interactive: prompts for key+secret
                          --region eu-west-2
sg credentials remove <role> --yes
sg credentials test <role>                   # calls sts:GetCallerIdentity; reports identity + age
sg credentials whoami                        # shows: active role for current command path
                                             # (or use --as to test specifically)
sg credentials check                         # hygiene audit:
                                             #   • file mode 0600?
                                             #   • file in .gitignore?
                                             #   • every role pingable?
                                             #   • any role granting iam:* / *:*?
                                             #   (last check delegates to IAM__Policy__Auditor)
sg credentials route <command-path>          # debug: which role would handle this command?
```

Plus a **global flag** at the root `sg` typer:

```
sg --as <role> …                             # one-shot override
sg --whoami …                                # print resolved identity before executing
```

### 3.6 Safety invariants

1. **Never log secrets.** `Schema__Credentials__Role__Static.secret_key` has a
   `__repr__` / `__str__` that returns `"****"`. The auditor / dump paths
   are forbidden from JSON-serialising secret fields.
2. **File mode 0600.** Loader refuses world/group-readable files with a hard error.
3. **`.gitignore` enforcement.** `sg credentials check` warns CRITICAL if the
   file is tracked by git or not in `.gitignore`.
4. **Pre-commit hook (proposed).** Optional `.claude/hooks/` or
   `pre-commit` config that blocks commits introducing the credentials file path.
5. **Session cache mode 0600** in `./sg-compute/aws/.sessions/<role>.json`,
   gitignored, auto-expires.
6. **No fallback to ambient AWS env vars** unless the operator explicitly opts
   in via `sg --as env aws …` (a magic role name that means "use whatever the
   shell has"). Default behaviour requires an explicit named role.

## 4. User journey (worked example)

The operator wants to run `sg aws iam role check sg-compute-vault-publish-waker-role`
without giving their default user `iam:*`.

**One-time setup (≤ 5 min):**

```bash
# 1. Create a dedicated IAM admin role in the AWS console (or use existing).
#    Trust policy: SG-Deploy-User can sts:AssumeRole into it.
#    Permissions:  iam:Get*, iam:List*, iam:Pass* (read-mostly).

# 2. Initialise the credentials file with the existing deploy creds.
sg credentials init   # writes a starter ./sg-compute/aws/.credentials.toml (chmod 0600)
                      # adds 'sg-compute/aws/.credentials.toml' + '.sessions/' to .gitignore

sg credentials add deploy --type static --region eu-west-2
# (interactive — prompts AWS_ACCESS_KEY_ID + AWS_SECRET_ACCESS_KEY; secret echoed as ****)

# 3. Add the assume-role entry.
sg credentials add iam-admin --type assume_role \
   --source-role deploy \
   --role-arn arn:aws:iam::745506449035:role/sg-admin-iam

# 4. Verify both work.
sg credentials test deploy       # ✓  identity: arn:aws:iam::…:user/SG-Deploy-User
sg credentials test iam-admin    # ✓  identity: arn:…:assumed-role/sg-admin-iam/sg-iam-admin (cached for 3593s)

# 5. Add the auto-route so future iam commands JUST WORK.
#    Edit ./sg-compute/aws/.credentials.toml:
#      [command_routes]
#      "aws iam *" = "iam-admin"
#      "*"        = "deploy"
```

**Day-to-day:**

```bash
sg aws iam role check sg-compute-vault-publish-waker-role
  → Sg__Aws__Context auto-resolved role: iam-admin
  ✓  Audit: sg-compute-vault-publish-waker-role — severity: INFO

sg vp register hello-cv --vault-key dev-pack
  → Sg__Aws__Context auto-resolved role: deploy
  ✓  …

sg --as deploy aws iam role list                      # explicit override
  ✗  AccessDenied — expected, deploy user can't list IAM
```

## 5. Phased implementation

### Phase A — file loader + single seam (≈ ½ day)

- Primitives + schemas (§3.2 file tree)
- `Credentials__Store.load(path)` — TOML parse, mode check
- `Sg__Aws__Session.boto3_client()` — static-only path first
- `Sg__Aws__Context` thread-local
- One `*__AWS__Client` migrated as proof (pick the IAM client — smallest)
- Tests: ~20

### Phase B — CLI surface (≈ ½ day)

- `sg credentials init / add / list / show / remove / test / whoami / check / route`
- All commands chmod-guard the file on write
- Pretty Rich output with secrets masked
- Tests: ~15 (golden-file CLI per existing pattern)

### Phase C — auto-routing + global `--as` (≈ ¼ day)

- Root `@app.callback()` adds `--as` and `--whoami`
- `Credentials__Router` with fnmatch routing
- Tests: ~8

### Phase D — STS AssumeRole + session cache (≈ ½ day)

- `Schema__Credentials__Role__Assume_Role` resolution
- `Session__Cache` disk persistence + 5-min refresh window
- `sg credentials test` reports session expiry
- Tests: ~12 (using in-memory STS fake)

### Phase E — migrate all `*__AWS__Client.client()` to the seam (≈ ¼ day)

- IAM (already done in A), ACM, CF, Lambda, EC2, Route53, Elastic — same one-line change each
- Existing in-memory test subclasses keep working unchanged (they override `client()`)
- Tests: existing suites must still pass (no new tests needed)

### Phase F — safety net (≈ ¼ day)

- `sg credentials check` (mode, gitignore, ping, IAM audit)
- `.gitignore` patcher run by `credentials init`
- Documentation: `team/roles/devops/CREDENTIALS.md` (operator guide)

**Total: ~2 days. ~55 net-new tests. ~7 `client()` migrations.**

## 6. Future — vault-backed credentials server

The whole design hinges on **`Credentials__Store` being the only thing that
knows where credentials come from**. Today it loads a TOML file. Tomorrow
it could:

1. **Vault backend:** `Credentials__Store__Vault` — fetches role definitions
   from an SGit vault, decrypts in-process, never persists to disk.
2. **HTTP credentials-server backend:** `Credentials__Store__Server` — talks to
   a long-running agent-credentials service that holds creds in memory and
   issues STS sessions on demand. Auth via mTLS or short-lived bearer tokens.
3. **Hybrid:** static roles from a local file, sensitive roles via the server.

The `Sg__Aws__Session` interface doesn't change. CLI commands don't change.
Tests don't change. Only the loader implementation swaps.

The "agents on-demand" story collapses to:
- Agent starts → asks credentials-server for `role=ci-deployer`
- Server checks agent identity + ACL → returns a 15-min STS session
- Agent runs `sg vp …` → `Sg__Aws__Session` uses the cached session
- Session expires → next call triggers a re-fetch

## 7. Risks

| Risk | Mitigation |
|------|------------|
| Operator accidentally commits the credentials file | `sg credentials init` patches `.gitignore`; `sg credentials check` flags CRITICAL if tracked; optional pre-commit hook |
| Credentials leak into logs / tracebacks via `repr()` | Override `__repr__`/`__str__` on secret-bearing primitives; explicit test asserts repr is `****` |
| `Sg__Aws__Context` thread-local breaks with async/threading | Tests cover concurrent invocations; for sync CLI it's a non-issue; document the contract |
| AssumeRole sessions expire mid-command | Refresh-on-use: every `boto3_client()` call checks expiry, refreshes if < 5 min remaining |
| Loss of ambient-AWS-env fallback breaks existing scripts | Phase A keeps ambient fallback if no credentials file present; Phase C makes the file required (with a one-line opt-in for ambient via `sg --as env`) |
| Performance: STS round-trip adds latency to first call | Session cache makes it a one-time cost per role per hour |

## 8. Acceptance criteria

Merge to dev requires:

1. `sg aws iam role list` works without modifying `SG-Deploy-User` perms,
   using the auto-routed `iam-admin` role.
2. `sg credentials check` flags a CRITICAL finding if the credentials file is
   ever staged for commit.
3. `sg credentials test <role>` returns identity + (for assume-role) session
   age in seconds.
4. All existing tests pass without modification (the seam is transparent).
5. ~55 new tests for the credentials package.
6. `sg credentials whoami` shows the resolved role for any command path.
7. Operator docs at `team/roles/devops/CREDENTIALS.md` (one-page onboarding).

## 9. Open questions (for the human before Phase A starts)

| Q | Question | Architect's default |
|---|----------|---------------------|
| Q1 | `sg credentials init` should auto-write a starter file with `deploy` role placeholder, or refuse to write and just instruct the operator? | Auto-write a stub with comments + placeholder values; refuse to overwrite if file exists |
| Q2 | Should `sg credentials add` actually write the secret to the file, or should it ALSO offer to put the secret into the OS keyring (macOS Keychain / Linux Secret Service)? | v1: just the file. Keyring is a v2 (separate plan) — it complicates cross-machine workflows |
| Q3 | Pre-commit hook to block staging `sg-compute/aws/.credentials.toml`: opt-in via `sg credentials install-hook`, or auto-installed by `sg credentials init`? | Auto-install by `init`, with a friendly message and an opt-out flag |
| Q4 | Should `sg credentials check` also detect roles whose static keys are over 90 days old (rotation hygiene)? | Yes, emit INFO; CRITICAL at 180 days. Same auditor pattern as `IAM__Policy__Auditor` |
| Q5 | What happens to existing `region` env var (`AWS_DEFAULT_REGION`) — does the role's `region` override it, or vice versa? | Role's `region` wins. The whole point of named roles is determinism |
| Q6 | Should `command_routes` also support negation (e.g. `"!aws iam *"`)? | No — first-match-wins ordering covers it; negation adds parser complexity |
| Q7 | Session cache location: in `./sg-compute/aws/.sessions/` (project-local, ephemeral) or `~/.cache/sg/sessions/` (XDG, shared across project clones)? | Project-local. Cleaner blast radius, easier to wipe with `rm -rf sg-compute/aws/.sessions/` |

## 10. Anti-scope (NOT in this plan)

- **OS keyring integration** — Q2; deferred to a future plan
- **MFA flows** — STS `AssumeRole` will support `--token-code` later; v1 assumes no-MFA roles for automation
- **AWS SSO** — separate auth model, deserves its own plan
- **Credentials rotation automation** — `sg credentials rotate` is post-v1
- **Cross-account assume-role chains** — single-hop only in v1 (source_role → role_arn; no role_arn → role_arn)
- **The credentials server itself** — §6 is forward-looking; this plan only ships the local file backend
