# v0.2.25 — sg credentials, multi-role & macOS Keychain

This document proposes how `sg` should manage multiple AWS identities locally
on a developer laptop so operators can use least-privilege roles per command
without ever giving the day-to-day deploy user dangerous powers like `iam:*`.

It is the foundation for the future "credentials server for agents" — once the
local interface is stable, the server is a drop-in replacement for the loader.

**Design pivot from revision 1 (2026-05-17):** there is no credentials file.
All state — role catalogue, routes, secrets — lives in the macOS Keychain.
The killer-feature for editing is a temp-file projection mode (§4.2) that
hands operators their normal editor without any config file ever persisting
on disk.

---

## 1. TL;DR

| Item | Decision |
|------|----------|
| **Storage** | **macOS Keychain only.** No `.credentials.toml`, no file on disk. |
| Platform | macOS-only for v1 (CI/Linux/containers use IAM instance profiles or env vars — they don't need `sg credentials`) |
| Backend access | **Default: shell out to macOS `security` CLI** (zero new Python deps; see Q12). Fallback: `keyring` Python lib (small dep, avoids argv exposure of secrets). |
| CLI namespacing | `sg credentials.*` is top-level (user-facing AWS abstraction). `sg osx.*` is a new namespace for OS-specific surfaces — `sg osx keyring.*` lives there. Mirrors `sg aws.*` cleanly. |
| Credential types | **Both first-class from v1:** static access keys AND STS `AssumeRole`. AssumeRole is the recommended least-privilege primitive — trust policies can restrict source principal, source IP, MFA, time-of-day, external ID. |
| Role selection | (1) explicit `--as <name>`, (2) auto-routed from stored rules, (3) `default_role` |
| **Transparency** | Default one-line identity banner before every command; persistent JSONL audit log at `~/.sg/audit.jsonl`; correlatable STS session names that join with CloudTrail; `sg credentials trace` dry-run. See §3.8. |
| Editing UX | Standard CLI commands (`set`, `add`, `remove`) PLUS an **edit mode**: `sg credentials edit` opens a temp TOML projection in `$EDITOR`, parses on save, diffs, applies, shreds the file |
| Single seam | One `Sg__Aws__Session` factory; every `*__AWS__Client.client()` consults it |
| Safety | Secrets never on argv, never in logs; keyring health-checked at startup; temp files always shredded on edit-mode exit |
| Recovery | `sg credentials backup --to <path>` writes an age-encrypted dump |
| Future | Same loader interface, swap keyring backend for HTTP credentials-server (agent flow) |

## 2. Problem & current state

### 2.1 The trigger

```
$ sg aws iam role list
  ✗  ClientError: AccessDenied — User: arn:aws:iam::745506449035:user/SG-Deploy-User
     is not authorized to perform: iam:ListRoles
```

This is the **correct** failure mode — `SG-Deploy-User` should NOT have `iam:*`.
But operators currently have only two options:

1. Add `iam:*` to `SG-Deploy-User` (bad — most powerful user gets more powerful).
2. Manually export `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` from a different
   user for every IAM command (tedious, error-prone, easy to leave the wrong
   creds in the env).

### 2.2 What's missing

- **No notion of named credentials** inside `sg`. Every `boto3.client(...)`
  call inherits the ambient environment.
- **No per-command role routing.** `sg aws iam *` could auto-pick an
  `iam-admin` identity; `sg vp *` could auto-pick `vault-publish-operator`.
- **No audit trail** of which identity ran which command.
- **No isolation from `~/.aws/credentials`.** Mixing day-job AWS profiles with
  project-specific keys invites cross-contamination.
- **Nowhere safe to store secondary secrets** — sgit vault keys, GitHub PATs,
  Anthropic API keys live in private notes / shell history.

### 2.3 Why no TOML file (revised)

Revision 1 of this brief proposed a `./sg-compute/aws/.credentials.toml`
catalogue alongside keyring secrets. Sign-off discussion ruled it out:

- **CI / containers don't need it.** The whole point of generating
  least-privilege IAM roles is so containers run as a single locked-down
  identity (instance profile / Lambda exec role / env vars). They never call
  `sg credentials`.
- **Multi-role switching is a dev/laptop concern only.** There is no team
  use-case where multiple developers share a checked-in role catalogue.
- **Anything a file does, keyring + good CLI does.** "Editability" was the
  only remaining argument; §4.2 (edit mode) solves it without persisting
  anything on disk.

Net effect: no `.gitignore` patching, no file-mode 0600 enforcement, no
pre-commit hook for stray TOMLs, no forensic question of "where did my
secrets leak from." The file simply doesn't exist.

## 3. Design

### 3.1 Storage — macOS Keychain only

All state lives as generic-password items in the user's login Keychain via the
Python `keyring` library. Every item has:

- `service` — namespaced string (see §3.2)
- `account` — always `"sg"` (we don't multiplex users; the OS account is the user)
- `password` — either a single secret value or a JSON blob (for catalogue entries)

No file. No directory. No `.gitignore` change. Operators wipe everything with
`sg credentials wipe --yes`, which iterates known prefixes and deletes them.

### 3.2 Namespacing scheme

| Bucket | Key pattern | Holds | Example |
|--------|-------------|-------|---------|
| **Catalogue** | `sg.config.role.<name>` | JSON role definition (type, region, account_id, assume-role wiring) | `sg.config.role.deploy` |
| **Catalogue** | `sg.config.routes` | JSON ordered list of `{pattern, role}` | `sg.config.routes` |
| **Catalogue** | `sg.config.default_role` | role name string | `sg.config.default_role` |
| **AWS secrets** | `sg.aws.<role>.access_key`<br>`sg.aws.<role>.secret_key`<br>`sg.aws.<role>.session_token` (assumed-role only) | single secret per item | `sg.aws.deploy.access_key` |
| **Vault keys** | `sg.vault.<name>` | sgit vault key | `sg.vault.dev-pack` |
| **Generic** | `sg.secret.<ns>.<name>` | free-form operator secret | `sg.secret.github.dinis-pat` |

Splitting catalogue (JSON blobs) from secrets (single values) keeps inspection
simple: `sg credentials show` only reads `sg.config.*` (no Keychain access
prompts for secrets); secret retrieval only touches `sg.aws.*` / `sg.vault.*`
/ `sg.secret.*`.

### 3.3 The role catalogue (in keyring)

Each role is one `sg.config.role.<name>` entry. Stored value:

```json
{
  "type": "static",
  "region": "eu-west-2",
  "account_id": "745506449035",
  "note": "SG-Deploy-User — least-privilege deploy creds"
}
```

Or for assume-role:

```json
{
  "type": "assume_role",
  "source_role": "deploy",
  "role_arn": "arn:aws:iam::745506449035:role/sg-admin-iam",
  "session_name": "sg-iam-admin",
  "duration_seconds": 3600,
  "region": "eu-west-2",
  "note": "AssumeRole into IAM-only admin; trust restricts to deploy user"
}
```

Secret material is NEVER stored in the catalogue JSON — only in
`sg.aws.<role>.*` items. The catalogue tells the loader which secret items
to fetch.

### 3.4 Command routing (in keyring)

A single `sg.config.routes` entry holds the ordered routing table:

```json
[
  {"pattern": "aws iam *",    "role": "iam-admin"},
  {"pattern": "aws lambda *", "role": "vault-publish-operator"},
  {"pattern": "vp *",         "role": "vault-publish-operator"},
  {"pattern": "*",            "role": "deploy"}
]
```

First match wins (`fnmatch` semantics). Edited via `sg routes edit` (which
uses the §4.2 edit-mode for just this table) or `sg routes add / remove`.

### 3.5 The single seam — `Sg__Aws__Session`

Every existing `*__AWS__Client.client()` goes from this:

```python
def client(self):
    return boto3.client('iam')
```

to this:

```python
def client(self):
    return Sg__Aws__Session().boto3_client('iam')
```

`Sg__Aws__Session.boto3_client(service)`:

1. Reads active role from `Sg__Aws__Context.current_role()`.
2. Asks `Credentials__Store` for the resolved credentials of that role
   (recursively resolving `source_role` for `assume_role`).
3. If `type=assume_role`: checks `Session__Cache`; if cache miss or
   expiring within 5 min, calls `sts:AssumeRole` and caches the result.
4. Returns `boto3.Session(...).client(service)`.

In-memory test composition is unchanged — tests subclass `*__AWS__Client`
and override `client()`, bypassing the session factory.

### 3.6 Role selection — three layers

```
sg --as iam-admin aws iam role list       (1) explicit flag wins
sg aws iam role list                       (2) routes match "aws iam *" → iam-admin
sg some-other-command                      (3) "*" route → "deploy" (or default_role)
```

`Credentials__Router.route(command_path)`:

- `command_path` is the dot/space-joined Typer path (e.g. `"aws iam role list"`).
- Routes tried in stored order, first match wins.
- No match + no `default_role` → exit 2 with clear error. Never silently
  falls back to ambient AWS env vars; that defeats the whole isolation goal.

`Sg__Aws__Context` is set once per CLI invocation via a root `@app.callback()`.

### 3.7 Safety invariants

1. **Secrets never on argv.** `sg credentials set` and `sg keyring set` always
   prompt interactively (terminal echo off). Positional secret args are refused.
2. **Secrets never in logs.** Schema fields holding secrets override
   `__repr__` / `__str__` to return `"****"`. Explicit test asserts this.
3. **No ambient AWS env fallback** unless the operator types `sg --as env …`
   (magic role name meaning "use whatever the shell has").
4. **Keyring backend probed at startup.** If `keyring.get_keyring()` is
   anything other than macOS Keychain (e.g. `keyrings.alt.file.PlaintextKeyring`
   accidentally installed via dependency creep), `sg` refuses to operate
   with a clear error and remediation step.
5. **Temp files always shredded.** §4.2 edit mode uses `try/finally` to
   overwrite-then-unlink temp files even on crash. Temp files live in
   `$TMPDIR` (per-user) with mode 0600.
6. **Edit-mode session journal** at `~/.sg/edit-sessions/<uuid>.json`
   (no secrets — only timestamps, status, ops digest). On next `edit`
   invocation, orphans from previous crashes are detected and cleaned up.

### 3.8 Transparency & audit

Auto-routing makes role selection invisible to operators. That's good for
UX but bad for forensics — "which role just ran that command?" needs an
answer that doesn't depend on the operator remembering. Four mechanisms,
all enabled by default:

#### 3.8.1 Identity banner

Before any command that consults `Sg__Aws__Session`, the resolved identity
is printed to stderr as a single dim line:

```
$ sg aws iam role check sg-compute-vault-publish-waker-role
  → role: iam-admin  (assumed via deploy → arn:aws:iam::745506449035:role/sg-admin-iam, expires in 59m)
  ✓ Audit: ... severity: INFO
```

Silenced with `sg --quiet …` or globally via `sg.config.banner_enabled = false`
in the catalogue. Banner is on by default — explicit visibility is the right
default for a tool that holds privileged credentials.

#### 3.8.2 Audit log (`~/.sg/audit.jsonl`)

Every command that takes a credential-resolved path appends one JSONL line:

```json
{
  "ts":             "2026-05-17T14:32:11.847Z",
  "sg_version":     "0.2.25",
  "command_path":   "aws iam role check",
  "command_args":   ["sg-compute-vault-publish-waker-role"],
  "role":           "iam-admin",
  "role_source":    "route:aws iam *",
  "resolved_via":   ["deploy", "assumed:sg-admin-iam"],
  "identity_arn":   "arn:aws:sts::745506449035:assumed-role/sg-admin-iam/sg-iam-admin-1747491131-7a3f9b2c",
  "session_expiry": "2026-05-17T15:32:11Z",
  "duration_ms":    1284,
  "exit_code":      0,
  "session_id":     "sg-iam-admin-1747491131-7a3f9b2c"
}
```

Rules:
- **No secrets, ever.** Access keys, secret keys, session tokens MUST NOT
  appear in the log. Schema-level guard with an explicit test.
- **Append-only.** File opened `'a'` with `O_APPEND`; never truncated by `sg`.
- **Mode 0600** at creation. Existing-file mode is asserted on every write;
  if it has loosened, the log line is dropped with a stderr WARN.
- **Size-bounded.** When `audit.jsonl` exceeds 50 MB, rotate to
  `audit.jsonl.N` (gzip), keep last 5. Operator can rewire via
  `sg.config.audit_log_max_size`.

`command_args` is included for forensic value but is **redacted via
allowlist**: only argument values matching `Safe_Str__*` types known to be
non-secret are kept; everything else becomes `"***REDACTED***"`. Default-deny,
not default-allow.

#### 3.8.3 Correlatable STS session names

Session names are constructed as:

```
sg-<role>-<unix-ts>-<8-char-uuid>
e.g. sg-iam-admin-1747491131-7a3f9b2c
```

Two consequences:

1. **CloudTrail join.** Every `sts:AssumeRole` and every downstream API call
   made with the resulting session appears in CloudTrail tagged with this
   exact session name. The session name is also written to the local audit
   log line above. So a question like "what did `sg vp register hello-cv`
   actually call in AWS at 14:32 on Tuesday?" answers with a CloudTrail
   filter on `sessionName=sg-iam-admin-1747491131-7a3f9b2c`.
2. **No collisions.** Two parallel `sg` invocations with the same role get
   distinct session names, distinct cached sessions, no race.

The 8-char UUID is also surfaced as `session_id` in the audit log so the
local + remote audit trails join exactly.

#### 3.8.4 `sg credentials trace` and `sg credentials log`

```
sg credentials trace <command-path>          # dry-run: prints the full resolution chain WITHOUT executing
sg credentials log    [--tail] [--role X]    # pretty-print recent audit entries
sg credentials log --json                    # raw JSONL for piping into jq
```

`trace` example:

```
$ sg credentials trace "aws iam role check foo"
  route:           "aws iam *"  →  role 'iam-admin'
  role 'iam-admin' (assume_role)
    source_role:    'deploy' (static)
      access_key:   sg.aws.deploy.access_key  ✓ (keychain)
      secret_key:   sg.aws.deploy.secret_key  ✓ (keychain)
      region:       eu-west-2
    role_arn:       arn:aws:iam::745506449035:role/sg-admin-iam
    session_cache:  ~/.sg/sessions/iam-admin.json (expires in 23m, no refresh needed)
    effective:      arn:aws:sts::745506449035:assumed-role/sg-admin-iam/<would-be-new-session>
  resolved in:      4 ms (cached)
```

This is the answer to "why is `sg X` using role Y" — operators don't have
to read code to debug routing.

## 4. CLI surface

### 4.1 Standard commands

```
sg credentials init                          # first-time wizard; creates first role + default_role interactively
sg credentials list                          # all roles + routes (no secrets shown)
sg credentials show <role>                   # full role JSON (secrets masked)
sg credentials show --as-toml                # whole catalogue as TOML to stdout (for piping to less / diff)
sg credentials add <role> --type static --region eu-west-2     # interactive: prompts AKID + secret
sg credentials add <role> --type assume_role --source-role deploy --role-arn ...
sg credentials remove <role> --yes
sg credentials test <role>                   # sts:GetCallerIdentity; reports identity + (for assume-role) session age
sg credentials whoami                        # active role for current command path + secret source
sg credentials route <command-path>          # debug: which role would handle this command?
sg credentials trace <command-path>          # full resolution-chain dry-run (see §3.8.4)
sg credentials log    [--tail] [--role X]    # pretty-print recent audit entries from ~/.sg/audit.jsonl
sg credentials log    --json                 # raw JSONL for piping into jq
sg credentials wipe   --yes                  # delete ALL sg.config.* and sg.aws.* keyring entries

sg routes list
sg routes add <pattern> <role> [--position N]
sg routes remove <pattern> --yes
sg routes edit                               # edit-mode for just the routes table

# ── sg osx namespace — OS-specific surfaces ──────────────────────────────────
sg osx keyring backend                       # show: macOS Keychain (✓) / other (refuse)
sg osx keyring set <bucket> <key>            # interactive prompt; never on argv
sg osx keyring get <bucket> <key>            # value to stdout (refuses tty without --yes)
sg osx keyring list  [<bucket>]              # list keys (names only)
sg osx keyring delete <bucket> <key> --yes

sg credentials backup  --to <path>           # age-encrypted dump (passphrase prompt)
sg credentials restore --from <path> [--merge|--replace]
```

Plus global flags at the root `sg` typer:

```
sg --as <role> …                             # one-shot role override
sg --whoami …                                # print resolved identity before executing
sg --quiet …                                 # suppress the §3.8.1 identity banner
sg --trace …                                 # also print the full §3.8.4 resolution chain
```

**Namespacing rationale.** `sg credentials.*` stays top-level because it's
the user-facing abstraction (named AWS roles, routing, audit) — the same
way `sg aws.*` is top-level even though it internally uses boto3. The
storage backend is an implementation detail that §8 will eventually swap.
`sg osx keyring.*` is the low-level escape-hatch surface for storing
non-AWS secrets (vault keys, GitHub PATs); it lives under `sg osx` because
it's literally a macOS Keychain wrapper and won't apply to non-macOS
backends. See Q13 if you'd prefer to move `sg credentials.*` under
`sg osx` too.

### 4.2 Edit mode — the killer feature

**Problem solved.** Operators want the "edit a config file" muscle memory
(open in vim/VS Code, scroll, tweak, save, done) without a config file
actually persisting on disk.

**Flow.**

```
sg credentials edit                          # whole catalogue (roles + routes + default_role)
sg credentials edit <role>                   # one role
sg routes edit                               # just routes
```

1. **Project.** `Credentials__Editor` reads the current keyring state and
   projects it to a TOML string. Secrets appear as the literal placeholder
   `"***SECRET — replace to change, leave unchanged to keep***"`.
2. **Write.** The projection is written to a per-user temp file
   (`$TMPDIR/sg-edit-<uuid>.toml`, mode 0600), plus a session journal at
   `~/.sg/edit-sessions/<uuid>.json` recording `status=in_progress`,
   timestamp, and a SHA-256 of the projection (no secrets).
3. **Launch.** `Editor__Launcher.launch(path)` picks the editor:
   - `$VISUAL` if set
   - else `$EDITOR` if set
   - else `code --wait <path>` if `code` is in `$PATH` and `$TERM_PROGRAM=vscode`
   - else `vim <path>`
   All variants block until the editor exits.
4. **Parse & diff.** On editor exit:
   - Re-read the file → `tomllib.loads`
   - Parse into `Schema__Catalogue__Snapshot`
   - Diff against pre-edit snapshot → list of operations:
     `add_role` / `update_role` / `remove_role` / `set_secret` /
     `add_route` / `remove_route` / `reorder_routes` / `set_default_role`
   - Render a confirmation table:
     ```
     About to apply:
       + role 'iam-admin'        (assume_role, region eu-west-2)
       ~ role 'deploy'           region: eu-west-1 → eu-west-2
       - role 'old-test'
       + route '*'               → deploy
       ~ secret sg.aws.deploy.access_key  (changed)
     Proceed? [y/N]
     ```
5. **Apply or abort.**
   - `y` → apply ops sequentially against the keyring; on any error, attempt
     reverse-order rollback of the just-applied ops; if rollback also fails,
     leave the journal pointer with `status=needs_attention` and the
     remaining ops so the operator can `sg credentials edit --resume`.
   - `n` → no changes.
6. **Cleanup (always, try/finally).**
   - Overwrite temp file with random bytes
   - `os.unlink` temp file
   - Update session journal to `status=completed | aborted | needs_attention`

**Recovery from editor crashes.** On next `sg credentials edit`, the
`Credentials__Editor` scans `~/.sg/edit-sessions/` for `status=in_progress`
entries older than 1 hour. If any are found, it prompts:

```
Orphan edit session detected (uuid abc123, started 2h ago).
The temp file no longer exists (likely cleared by the OS).
Discard this session? [Y/n]
```

On `Y` the journal is removed; on `n` it's preserved so a manual inspection
can happen.

**Parse-error loop.** If the TOML on save fails to parse or fails schema
validation, edit-mode re-opens the editor with an error comment block
prepended to the file:

```toml
# ─────────────────────────────────────────────────────────────────────────
# ✗ Could not apply changes:
#   line 14: 'region' is required for static roles
#   line 22: route pattern '' is empty
# Fix the issues above, save, and close to retry. Close without saving to abort.
# ─────────────────────────────────────────────────────────────────────────
```

After 3 failed parses in a row, edit-mode aborts with a clear message and
shreds the file. The operator can re-run `sg credentials edit` whenever
they're ready.

**Why a temp file rather than an inline TUI?** Operators already have an
editor they like; reusing it is zero learning curve. A TUI would need its
own keybindings, scroll model, multi-file handling, and themed display —
too much surface for a v1 feature. The temp-file trick gives 95% of the
benefit at 5% of the cost.

**Implementation seams (for testing).**

- `Editor__Launcher.launch(path)` — overrideable; tests inject a fake
  launcher that pre-writes a desired post-edit TOML and returns immediately.
- `Temp__File__Manager` — handles `mkstemp`, chmod, shred, unlink. Tests
  use a real `$TMPDIR` (already isolated by pytest tmp_path fixtures).
- `Credentials__Editor.project / parse / diff / apply` — pure functions
  except `apply`, which delegates to `Keyring__Store`.

### 4.3 REPL friendliness

The `sg repl` (defined in `sg_compute/cli/Cli__SG__Repl.py`) walks the
Click command tree by name with per-segment prefix matching. Every command
in §4.1 is a standard Typer subcommand, so they all work natively in REPL:

```
sg> credentials
sg/credentials> list
sg/credentials> show deploy
sg/credentials> add log-reader --type static --region eu-west-2
sg/credentials> trace "aws iam role check foo"
sg/credentials> ..
sg> osx keyring
sg/osx/keyring> set vault dev-pack
sg/osx/keyring> list
```

Two REPL-specific concerns that need explicit handling in Phase B:

#### 4.3.1 The `as <role>` pseudo-command

`sg --as <role>` is a root-level callback flag. In REPL each invocation is a
fresh `sg_app` dispatch, so an operator who wants to run several commands
as `iam-admin` would have to retype `--as iam-admin` every line. That
defeats the REPL ergonomic.

**Fix.** Add a REPL-only pseudo-command `as <role>` that pins the role
for the rest of the REPL session by setting `Sg__Aws__Context.current_role`
in the REPL's process state. The prompt updates to reflect the pin:

```
sg> as iam-admin
sg [as iam-admin]> aws iam role list           ← runs as iam-admin
sg [as iam-admin]> aws iam role show foo       ← still iam-admin
sg [as iam-admin]> as deploy                   ← switch
sg [as deploy]> as --                          ← clear pin; back to routing default
sg> aws iam role list                          ← back to routed role (or default)
```

`as` is implemented as a special-case in `Cli__SG__Repl.py:run_repl()`,
intercepted before normal command resolution. It does NOT exist as a top-level
`sg` command in non-REPL mode (where `sg --as <role> …` already covers it).

#### 4.3.2 Subprocess and Ctrl-C handling

Two commands need verification that they cooperate with the REPL loop:

- **`sg credentials edit`** drops into `$EDITOR` via `subprocess.run`.
  Required behaviour: editor takes over the terminal, REPL prompt suspends,
  on editor exit the REPL prompt returns. The existing REPL loop already
  works this way for any subprocess-spawning command, but Phase C must
  include an explicit test (using a fake `$EDITOR` script) that the REPL
  prompt is intact after edit-mode completes.
- **`sg credentials log --tail`** polls in a loop. Ctrl-C must exit the
  tail loop and return to the REPL prompt — NOT exit the REPL itself. The
  REPL's `try/except KeyboardInterrupt` at the input layer must NOT swallow
  the signal until the command finishes. Phase G test: simulate a `--tail`
  invocation with SIGINT and assert the REPL receives `EOFError`/`prompt`
  again, not exit.

#### 4.3.3 REPL acceptance criteria (extends §10)

These extend the §10 list:

15. `sg repl` then `as iam-admin <Enter>` pins the role; subsequent
    commands route via `iam-admin`; `as --` clears the pin.
16. `sg repl` then `credentials edit <Enter>` opens `$EDITOR`, returns
    to the REPL prompt on save.
17. `sg repl` then `credentials log --tail <Enter>` followed by SIGINT
    returns to the REPL prompt, not OS shell.

## 5. User journey (worked examples)

### One-time setup (≤ 3 min)

```bash
sg credentials init
# > Wizard: keychain backend detected ✓
# > Enter name for first role [deploy]:
# > AWS access key ID: AKIA...
# > AWS secret access key: (echo off)
# > Region [eu-west-2]:
# > Set 'deploy' as default_role? [Y/n] Y
# ✓ Stored role 'deploy' in macOS Keychain

sg credentials add iam-admin --type assume_role \
   --source-role deploy \
   --role-arn arn:aws:iam::745506449035:role/sg-admin-iam

sg routes add "aws iam *" iam-admin --position 0
```

## 7. Backup & restore

The cost of going file-less: a corrupted Keychain or a new laptop loses the
role catalogue. Mitigations:

- **`sg credentials backup --to ~/Documents/sg-creds.age`** dumps the entire
  `sg.*` keyring state to an age-encrypted file using a user-prompted
  passphrase. Safe to store in iCloud Drive / Dropbox / vault.
- **`sg credentials restore --from <path>`** prompts for the passphrase,
  decrypts, and writes back. `--merge` keeps existing entries; `--replace`
  wipes first.
- Backup format is the same TOML projection used by edit mode, wrapped in
  an age payload — so a determined operator can decrypt with the `age` CLI
  directly (no `sg`-specific format lock-in).

iCloud Keychain sync of generic-password items is NOT relied on. It requires
`kSecAttrSynchronizable`, which the `keyring` lib doesn't set, and only works
Apple-to-Apple. Operators who want automatic cross-device sync use the
backup-to-iCloud-Drive workflow above.

## 8. Future — vault-backed credentials server

`Credentials__Store` being the only thing that knows where credentials come
from is the whole point. v1 ships a keyring backend; v2 can grow:

1. **Vault backend:** `Credentials__Store__Vault` — fetches role definitions
   from an SGit vault, decrypts in-process, never persists.
2. **HTTP credentials-server backend:** `Credentials__Store__Server` — talks
   to a long-running agent-credentials service that holds creds in memory
   and issues STS sessions on demand. Auth via mTLS or short-lived bearer
   tokens. This is what unlocks the "agent on a remote container" workflow.

The `Sg__Aws__Session` interface doesn't change. CLI commands don't change.
Tests don't change. Only the loader implementation swaps.

## 9. Risks

| Risk | Mitigation |
|------|------------|
| Keychain corruption / new laptop = lost catalogue | `sg credentials backup` to age-encrypted file; init wizard nags about backup |
| Wrong `keyring` backend silently loaded (e.g. plaintext fallback via dependency creep) | `Keyring__Health.probe()` refuses anything other than macOS Keychain on startup |
| Operator copies secret from `sg keyring get` into shell history | `get` refuses to print to a tty without `--yes`; default output assumes a pipe |
| Temp file from edit-mode persists after crash | Session journal + orphan detection on next `edit`; `$TMPDIR` gets cleared on reboot anyway |
| Partial apply during edit-mode (keyring throws mid-batch) | Track applied ops; attempt reverse-order rollback; on rollback failure, leave `needs_attention` journal + `--resume` path |
| Credentials leak via `repr()` in tracebacks | Secret-bearing primitives override `__repr__`/`__str__` to `"****"`; explicit test |
| `Sg__Aws__Context` thread-local breaks with async | Tests cover concurrent invocations; sync CLI is the only consumer in v1 |
| Loss of ambient-AWS-env fallback breaks existing scripts | Phase A keeps ambient fallback if NO catalogue is configured; once a catalogue exists, the opt-in is `sg --as env` |
| Performance: STS round-trip adds latency to first call | Session cache makes it one-time per role per hour |
| Operator's editor pulls in a language-server that indexes the temp file | Temp file extension is `.toml.sg-edit` (not standard `.toml`) to avoid LSP attachment; lives in `$TMPDIR` not the project root |
| Audit log accidentally captures a secret (e.g. an arg that happens to be a token) | `command_args` is redacted via allowlist — only known-safe `Safe_Str__*` types pass through; everything else becomes `***REDACTED***`. Phase G fuzz test asserts no known-secret substring ever appears in the log. |
| Audit log grows unbounded on a busy laptop | 50 MB rotation with gzip + keep-last-5; configurable via `sg.config.audit_log_max_size` |
| Identity banner clutters scripted output | `--quiet` global flag; auto-suppressed when stderr is not a tty AND `SG__AUDIT_BANNER=auto` (default) |
| Subprocess to `/usr/bin/security` exposes password in `ps` for ~10ms | Acceptable for v1 (window is tiny, ps requires same-user privileges). If unacceptable, flip to `keyring` lib via Q12 — that uses the Security framework C API and never exposes via argv. |

## 10. Acceptance criteria

Merge to dev requires:

1. `sg aws iam role list` works without modifying `SG-Deploy-User` perms,
   using the auto-routed `iam-admin` role.
2. `sg credentials test <role>` returns identity + (for assume-role) session
   age in seconds.
3. `sg credentials edit` round-trip: add a role, save, see it in
   `sg credentials list` — no temp file left on disk afterwards.
4. `sg credentials edit` orphan recovery: kill the editor mid-session,
   re-run `sg credentials edit`, see the cleanup prompt.
5. `sg credentials edit` parse-error retry: introduce a syntax error, save,
   see the annotated re-open, fix, save, success.
6. `sg osx keyring backend` reports "macOS Keychain ✓" on macOS; refuses to
   operate if any other backend is active.
7. `sg credentials backup` + `restore` round-trips a 3-role catalogue
   through an age-encrypted file.
8. `sg vp …` resolves vault keys from `sg.vault.<name>` keyring entries
   before falling back to `SG_VAULT_KEY__*` env vars.
9. **Transparency:** every credential-resolved command produces one line in
   `~/.sg/audit.jsonl` containing the resolved role, source, identity ARN,
   session id, exit code, and duration. Fuzz test (50 commands) asserts the
   log file never contains any known secret value.
10. **CloudTrail correlation:** an STS session name written to the audit log
    matches an entry in CloudTrail with the same `sessionName`. Verified by
    a deploy-via-pytest test that runs an assume-role command then queries
    CloudTrail with the captured session id.
11. **`sg credentials trace`** prints the full resolution chain for any
    command path without invoking AWS (zero `boto3` calls — verified by
    asserting no `Sg__Aws__Session.boto3_client()` invocation occurs).
12. All existing tests pass without modification.
13. ~95 new tests for the credentials + osx-keyring + audit packages.
14. Operator docs at `team/roles/devops/CREDENTIALS.md` (one-page onboarding,
    including a section on reading the audit log).

## 11. Open questions (for the human before Phase A starts)

| Q | Question | Architect's default |
|---|----------|---------------------|
| Q1 | `sg credentials init` runs a wizard interactively, or just creates an empty catalogue and points you at `sg credentials add`? | Interactive wizard. First-time UX matters; an empty catalogue is a silent dead-end. |
| Q2 | Edit-mode projection format: TOML, YAML, or JSON? | TOML — matches existing brief aesthetics; preserves comments through round-trip; `tomllib` is stdlib. |
| Q3 | If `$EDITOR` is unset and `code` not in PATH, fall back to `vim` or refuse? | `vim` (always present on macOS); refuse only if vim is also missing. |
| Q4 | Backup encryption: `age` (third-party brew tool) or built-in symmetric crypto via `cryptography` lib? | `age` — simple operator story (`brew install age`), well-known format, no in-house crypto. |
| Q5 | Should `sg credentials check` detect static-key age > 90 days (rotation hygiene)? | Yes. INFO at 90 days, CRITICAL at 180. |
| Q6 | `AWS_DEFAULT_REGION` env var — does role's `region` override it, or vice versa? | Role's `region` wins. The whole point of named roles is determinism. |
| Q7 | Should `command_routes` support negation (e.g. `"!aws iam *"`)? | No — first-match-wins ordering covers it; negation adds parser complexity. |
| Q8 | Session cache location: `~/.sg/sessions/` or project-local? | `~/.sg/sessions/` (per-user, survives project deletion). Mode 0600. |
| Q9 | Edit-mode confirmation: always require y/N, or allow `--yes` to skip? | Always require by default; `--yes` exists for scripted/agent use. |
| Q10 | On TOML parse error, re-open editor with annotations, or abort? | Re-open with annotations; abort after 3 consecutive failed parses. |
| Q11 | Edit-mode temp file extension — `.toml`, `.toml.sg-edit`, or `.sg-edit`? | `.toml.sg-edit` — keeps syntax highlighting in most editors while avoiding LSP attachment that indexes the file. |
| Q12 | Backend access: subprocess to `/usr/bin/security` (zero new deps, password briefly in argv) OR add the `keyring` Python lib (~50 KB, ~1 transitive dep on macOS, uses Security framework C API so no argv exposure)? | **Default to `/usr/bin/security`** — no new dependencies, the binary ships with every macOS, and the argv-exposure window is ~10ms in a same-user process (real but minor). Behind a feature flag, allow opt-in to `keyring` for operators who want stricter argv hygiene. |
| Q13 | Should `sg credentials.*` also move under `sg osx.*` (i.e. become `sg osx credentials.*`) for full v1 honesty, or stay top-level as the backend-agnostic user-facing abstraction? | **Stay top-level.** `sg credentials` is the user mental model ("named AWS roles"); the storage backend is incidental. Mirrors `sg aws.*` (top-level even though boto3-backed). `sg osx.*` is the right home for low-level OS surfaces like raw keyring access. Open to reversing if you'd prefer strict honesty. |
| Q14 | Identity banner enabled by default, or opt-in via `--banner` / config? | Enabled by default. The whole point is "no surprises about which role just ran" — making it opt-in defeats the goal. `--quiet` is the escape hatch. |
| Q15 | Audit log location: `~/.sg/audit.jsonl` or `~/Library/Logs/sg/audit.jsonl` (macOS convention)? | `~/.sg/audit.jsonl` — keeps all sg state under one root that's easy to inspect / nuke. Symlink can be created by ops if they want `~/Library/Logs` visibility. |

## 12. Anti-scope (NOT in this plan)

- **Windows / Linux dev support** — macOS-only for v1.
- **`.credentials.toml` file on disk** — removed in rev 2; was rev-1 design.
- **MFA flows** — STS `AssumeRole` will support `--token-code` later; v1 assumes no-MFA roles for automation.
- **AWS SSO** — separate auth model, deserves its own plan.
- **Credentials rotation automation** — `sg credentials rotate` is post-v1.
- **Cross-account assume-role chains** — single-hop only in v1.
- **The credentials server itself** — §8 is forward-looking; this plan ships only the local keyring backend.
- **Cross-machine keyring sync** — backup/restore via age-encrypted file is the answer; iCloud Keychain item sync is too unreliable to depend on.
- **Custom keyring backends** (1Password CLI, Bitwarden) — `keyring` lib supports them via plugins; v1 sticks to macOS Keychain for predictability.
- **An interactive TUI for editing** — temp-file-in-`$EDITOR` is the chosen UX; TUI is too much surface area for v1.
- **Remote shipping of the audit log** — `~/.sg/audit.jsonl` is local-only in v1. No SIEM integration, no S3 upload, no telemetry. Operators who want this can `tail -F` the file into whatever they like.
- **Cross-tool audit correlation beyond CloudTrail** — the session-name trick joins local + CloudTrail. Joining with other tools (Datadog, OpenTelemetry, etc.) is post-v1.
