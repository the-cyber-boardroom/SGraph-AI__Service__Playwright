# v0.1.96 — `sp nvm` — Nginx + VNC + Mitmproxy Sister Section

**Series:** 6 of 7.
**Status:** PROPOSED.
**Template:** `sgraph_ai_service_playwright__cli/elastic/`.

This file specifies `sp nvm` — the human-debug "streaming browser" stack. A separate EC2 instance running a Chromium-with-VNC container fronted by nginx and routed through its **own** mitmproxy. This is the "operator looks at a browser" use case, not the "headless API captures traffic" use case.

---

## Naming

`nvm` = **N**ginx + **V**NC + **M**itmproxy. Short alias; no clash with `sp el`, `sp os`, `sp prom`. Long alias: `sp browser-viewer` (typer registers both).

If `nvm` reads too much like Node Version Manager for muscle-memory clashes, alternative names to consider in the open-questions section.

## Why a separate mitmproxy from `sp`

Two distinct use cases, two distinct mitmproxy instances:

| | `sp` mitmproxy (in main Playwright EC2) | `sp nvm` mitmproxy |
|---|---|---|
| Purpose | Capture/transform traffic for **headless API** sessions | Capture/transform traffic for **human-driven VNC** sessions |
| Lifetime | Same as the Playwright host | Same as the VNC host |
| Auth | API-key (admin FastAPI) | Basic proxy auth (operator-owned creds) |
| Interceptor | Programmatic / per-session | Long-lived operator config |
| Audit destination | Per-session artefact bundle | Operator's local file or S3 dump |

Co-locating them on one EC2 (today's PoC shape) couples two different lifecycles and confuses the audit story.

## CLI commands (under `sp nvm`)

| Command | Purpose |
|---|---|
| `sp nvm create [name]` | Provision an EC2 running chromium+VNC + nginx + mitmproxy. |
| `sp nvm list` | List `sg:purpose=browser-viewer` stacks. |
| `sp nvm info <name>` | Tag metadata + nginx URL + mitmweb URL. |
| `sp nvm delete <name>` | Terminate. |
| `sp nvm connect <name>` | SSM shell. |
| `sp nvm health <name>` | Health: chromium up, nginx 200, mitmweb reachable. |
| `sp nvm open <name>` | Open the viewer URL in a local browser. **Replaces** the old top-level `sp forward-browser`. |
| `sp nvm forward <name> [--port 3000]` | Local port-forward to the viewer via SSM. |
| `sp nvm flows <name>` | List recent mitmproxy flows (via mitmweb API). |
| `sp nvm clear-flows <name>` | Clear the flow buffer. |

The mitmproxy fragment reuses the `agent_mitmproxy` image but with a different env config:

```yaml
mitmproxy:
  image: ${ECR_REPO}/agent-mitmproxy:${IMAGE_TAG}
  environment:
    MITM_MODE: regular
    MITM_PROXYAUTH: ${OPERATOR_PROXY_USER}:${OPERATOR_PROXY_PASS}
    MITM_INTERCEPTOR: /opt/interceptors/operator_default.py
  ports:
    - "8080:8080"           # behind nginx + IP allowlist
    - "8081:8081"           # mitmweb UI, behind nginx
```

The chromium container is configured with `HTTPS_PROXY=http://mitmproxy:8080` so all browser traffic flows through the local mitmproxy.

## Auth surface

Three layers of auth, one per port:

1. **nginx (port 443)** — Basic auth or operator IP allowlist; serves the chromium-VNC UI.
2. **mitmweb UI (proxied via nginx)** — same nginx auth.
3. **mitmproxy proxy port (8080)** — Basic proxy auth via `MITM_PROXYAUTH`; only chromium uses it (loopback on the docker network), but if it's exposed it stays SG-restricted.

## Open questions

1. **Section name** — is `nvm` clear enough, or rename to `bv` (browser-viewer) / `vnc` / `viewer`? `nvm` is concise but collides culturally with Node Version Manager.
2. **Multiple browsers** — Phase 1 ships chromium only. Firefox / Webkit later? If so, `Enum__Nvm__Browser__Type` is on the schema from day one.
3. **Persistent profile** — is the chromium user profile baked into the AMI, kept on EBS, or wiped per session? Operator preference.
4. **Audit export** — does `sp nvm` push mitmproxy flows to S3 on `delete`, or only on explicit `flows export`? Defaulting to `export-on-delete` aligns with "no data lost on teardown".
5. **Sharing the operator-mitmproxy interceptor** — does `agent_mitmproxy` ship a "default operator" interceptor profile, or does each `sp nvm` stack get its own?
