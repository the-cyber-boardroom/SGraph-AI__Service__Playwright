## What we want to see

For every request passing through the proxy:

- **WHO** — basic-auth user + client IP
- **WHAT** — method, host, path, response code, byte sizes
- **WHEN** — request ts, response ts, elapsed ms
- **HOW** — TLS version, cipher (from mitmproxy flow metadata)
- **WHY NOT** — if the request was blocked, which allowlist rule rejected it

And aggregates for operators:

- Req/sec per proxy user
- p50/p95/p99 latency
- Blocked vs allowed counts (Phase 2 adds the allowlist, but the metric schema is ready in Phase 1)
- Top destinations
- Bytes in/out (billing signal)

## Three backends, three jobs

| Backend | What lands there | Why |
|---|---|---|
| **AMP (Managed Prometheus)** | Counters + histograms from `metrics_addon.py` | Time-series alerting, dashboards |
| **AMG (Managed Grafana)** | Dashboards sourced from AMP + OpenSearch | Human UI |
| **OpenSearch (managed)** | Raw per-flow audit JSON from `audit_log_addon.py` | Forensics; full-text search; per-user breakdowns |

Each has its own shipper. We do NOT invent a new observability framework — standard tooling.

## Architecture — Phase 3

```
EC2 instance
│
├─ agent-mitmproxy container (Phase 1)
│    ├─ mitmweb → addons emit:
│    │    • Prometheus counters on :9090/metrics (localhost-only)
│    │    • Audit JSON lines on stdout → docker logs
│    └─ admin FastAPI (unchanged)
│
├─ ADOT collector container (Phase 3 addition)
│    • Scrapes localhost:9090/metrics every 15s
│    • remote_write to AMP workspace (SigV4-signed via EC2 instance profile)
│
└─ fluent-bit container (Phase 3 addition)
     • Tails `/var/lib/docker/containers/<agent-mitmproxy>/*.log`
     • Parses JSON lines from stdout
     • POSTs to OpenSearch `agent-mitmproxy-logs-YYYY.MM.DD` (SigV4-signed)
```

All three containers share the EC2 host. No sidecar network, no service mesh — simplest possible.

Append to the instance role `agent-mitmproxy-ec2-spike`:

1. **Managed**: `AmazonPrometheusRemoteWriteAccess` — lets the ADOT collector push to any AMP workspace in the account.
2. **Inline** for OpenSearch (narrower — one specific domain):

   ```json
   {
     "Version": "2012-10-17",
     "Statement": [{
       "Effect": "Allow",
       "Action": ["es:ESHttpPost", "es:ESHttpPut"],
       "Resource": "arn:aws:es:<region>:<account>:domain/<domain-name>/*"
     }]
   }
   ```

Attached in `ensure_instance_profile()` in `provision_mitmproxy_ec2.py`. Because `attach_role_policy` is idempotent, re-running the provisioner picks these up.

## Prometheus addon — what it emits

`agent_mitmproxy/addons/metrics_addon.py` uses `prometheus_client`:

```python
# Counters
mitmproxy_requests_total{method,scheme,host,status,proxy_user}
mitmproxy_bytes_request_total{proxy_user,direction="in"}
mitmproxy_bytes_response_total{proxy_user,direction="out"}
mitmproxy_blocked_total{proxy_user,reason}           # Phase 2 — allowlist

# Histograms
mitmproxy_request_duration_seconds{method,host,proxy_user}

# Gauges
mitmproxy_active_flows{proxy_user}
```

Exposed at `127.0.0.1:9090/metrics` by `prometheus_client.start_http_server(9090, addr='127.0.0.1')` on addon init.

Cardinality control — `host` label is deliberately not bucketed into "allowed/denied" buckets; that's what OpenSearch is for. We'll pre-allow-list known high-value hosts (`openai.com`, `anthropic.com`, `*.example.com` from configured allowlist) and lump everything else into `host="other"` to bound the cardinality. Concrete rule: top-20 hosts tracked individually, remainder collapsed.

## ADOT collector — how we run it

On the EC2 host, `docker run` a sibling container:

```bash
docker run -d --restart=always --name adot-collector \
  --network host \
  -v /etc/adot/collector.yaml:/etc/otel-config.yaml \
  public.ecr.aws/aws-observability/aws-otel-collector:latest \
  --config=/etc/otel-config.yaml
```

`/etc/adot/collector.yaml` (dropped by UserData in Phase 3):

```yaml
receivers:
  prometheus:
    config:
      scrape_configs:
        - job_name: 'agent-mitmproxy'
          scrape_interval: 15s
          static_configs:
            - targets: ['127.0.0.1:9090']

exporters:
  prometheusremotewrite:
    endpoint: "${AMP_REMOTE_WRITE_URL}"
    auth:
      authenticator: sigv4auth

extensions:
  sigv4auth:
    region: "${AWS_DEFAULT_REGION}"
    service: "aps"

service:
  extensions: [sigv4auth]
  pipelines:
    metrics:
      receivers: [prometheus]
      exporters: [prometheusremotewrite]
```

UserData sets `AMP_REMOTE_WRITE_URL` + `AWS_DEFAULT_REGION` from provisioner flags. SigV4 uses the instance profile — no static creds.

## Fluent-bit — how we run it

Another sibling container:

```bash
docker run -d --restart=always --name fluent-bit \
  --network host \
  -v /var/lib/docker/containers:/var/lib/docker/containers:ro \
  -v /etc/fluent-bit/fluent-bit.conf:/fluent-bit/etc/fluent-bit.conf:ro \
  fluent/fluent-bit:latest
```

`/etc/fluent-bit/fluent-bit.conf` — tail the agent-mitmproxy container's docker log, filter for JSON lines (ignore mitmweb's own output), ship to OpenSearch:

```ini
[INPUT]
  Name        tail
  Path        /var/lib/docker/containers/*/agent-mitmproxy*.log
  Parser      docker
  Tag         agent.mitmproxy

[FILTER]
  Name    grep
  Match   agent.mitmproxy
  Regex   log ^\{                                # keep only lines starting with '{' (audit JSON)

[FILTER]
  Name    parser
  Match   agent.mitmproxy
  Key_Name log
  Parser   json

[OUTPUT]
  Name                aws_opensearch
  Match               agent.mitmproxy
  Host                ${OPENSEARCH_HOST}
  Port                443
  tls                 On
  Index               agent-mitmproxy-logs
  Logstash_Format     On
  Logstash_Prefix     agent-mitmproxy-logs
  AWS_Auth            On
  AWS_Region          ${AWS_DEFAULT_REGION}
```

(The aws_opensearch plugin SigV4-signs from the instance profile automatically.)

## Grafana dashboards

Two dashboards, checked into `agent_mitmproxy/observability/dashboards/` as JSON:

### 1. `traffic-overview.json`
- Panel 1: Req/sec, last 1h, per proxy_user
- Panel 2: p95 latency, per host
- Panel 3: 2xx/4xx/5xx response codes stacked
- Panel 4: Bytes in/out per user (stacked area)
- Panel 5: Top 10 hosts by request count

### 2. `security-forensics.json`
- Panel 1: OpenSearch query — blocked requests in the last hour, table
- Panel 2: OpenSearch query — TLS versions seen, pie chart
- Panel 3: OpenSearch query — requests from any proxy_user that is NEW in the last 24h (anomaly hint)

Operator imports these once via AMG's "Import dashboard" UI (or via Grafana API if the workspace has it enabled). We do NOT auto-provision dashboards in Phase 3 because AMG provisioning-as-code requires extra setup (Grafana Cloud-style agent); for the spike, manual import is fine.

## Validation for Phase 3

1. After provision: `aws aps list-workspaces --region <region>` shows traffic in the workspace within 2 minutes.
2. Grafana query `rate(mitmproxy_requests_total[1m])` returns a non-zero series after a few curl hits through the proxy.
3. OpenSearch index `agent-mitmproxy-logs-YYYY.MM.DD` exists after the first few requests.
4. The two dashboard JSONs render cleanly when imported into AMG.

## What's out of scope even for Phase 3

- Alerting (Slack/email) — one more Grafana config step; low priority until we know what's flaky.
- Distributed tracing — the `X-Agent-Mitmproxy-Request-Id` header we inject is a seed; if downstream services (Playwright) emit it too, OpenSearch gives us per-request correlation via simple search. True OTel tracing is Phase 5.
- Cost monitoring — CloudWatch billing alarms on the AMP/AMG/OpenSearch line items. Operator concern, not agent-mitmproxy concern.
