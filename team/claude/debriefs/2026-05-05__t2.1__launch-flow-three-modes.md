## What was wrong

FV2.5 required a three-mode launch selector (FRESH / BAKE_AMI / FROM_AMI), AMI picker, cost preview banner, and validation. What shipped was FRESH-only with no mode selector, no AMI picker, no cost preview — 25% of scope, marked done.

## Changes delivered

### New component — `sg-compute-ami-picker`

`components/sp-cli/_shared/sg-compute-ami-picker/v0/v0.1/v0.1.0/` (3 files)

The picker is wired and ready: `setSpecId(specId)` is the activation hook. When `GET /api/amis` ships, implement the fetch there. Currently renders a warning placeholder ("Backend route GET /api/amis not yet available"). Dispatches `sg-compute:ami.selected` with `{ spec_id, ami_id }` on selection.

### Updated — `sg-compute-launch-form`

**HTML:** Added mode selector radio group (FRESH / BAKE_AMI / FROM_AMI) at top; cost preview banner (`.bake-only`); AMI name field for BAKE_AMI (`.bake-only`); AMI picker slot for FROM_AMI (`.from-ami-only`); inline error for FROM_AMI without AMI.

**CSS:** CSS-only show/hide — `.bake-only` and `.from-ami-only` default `display:none`; `.mode-bake-ami .bake-only` and `.mode-from-ami .from-ami-only` reveal via class on `.launch-form` root. Mode selector styled as a 3-column grid of bordered cards with accent highlight on selected radio. WCAG AA compliant: focus-visible ring on radio cards; mode descriptions use `:hover` + checked state for affordance.

**JS:** 
- `_setMode(mode)` — toggles `mode-bake-ami` / `mode-from-ami` class on form root; calls `amiPicker.setSpecId()`
- `getValues()` — extended with `creation_mode`, `ami_id`, `ami_name`
- `validate()` — FROM_AMI without selected AMI returns `false` + shows inline error; panel calls this before submitting
- `_resetMode()` — resets radio to FRESH, removes classes
- `$$()` helper added for multi-element queries in Shadow DOM

### Updated — `sg-compute-launch-panel`

`_launch()` now:
1. Calls `this._form.validate()` before proceeding — blocks submit if FROM_AMI has no AMI
2. Includes `creation_mode` and `ami_id` in the POST body to match `Schema__Node__Create__Request__Base`

### Filed — backend follow-up brief

`team/comms/briefs/v0.2.1__hotfix/backend/BV__ami-list-endpoint.md`

Documents the required `GET /api/amis?spec_id=<id>` endpoint shape, filtering, ordering, and the exact JS integration hook in `sg-compute-ami-picker.setSpecId()` to wire once it ships.

## PARTIAL status — what's blocked

The AMI picker cannot be live-tested until `GET /api/amis` exists. The BAKE_AMI and FROM_AMI launch paths also cannot be end-to-end smoke-tested (the form UI renders correctly, but the backend won't accept `creation_mode: bake-ami` yet — EC2 launch helper doesn't implement bake/from-ami variants).

FRESH mode is fully functional. Live smoke test: open launch form → FRESH selected by default → submit → `POST /api/nodes` with `creation_mode: fresh` → node appears in list.
