# Debrief — T1.7: Fix /ui/ absolute imports in co-located spec detail JS

**Date:** 2026-05-05
**Branch:** `claude/sgcompute-frontend-v0.2-5sjIO`
**Commit:** (backfilled by Historian)
**Hotfix tier:** T1 — runtime breakage

---

## Root cause

FV2.6 co-located 8 spec detail JS files into `sg_compute_specs/<spec>/ui/detail/v0/v0.1/v0.1.0/`. When rewriting their relative imports, the dev used absolute `/ui/...` paths based on the assumption that `sgraph_ai_service_playwright__api_site/` was mounted at `/ui/` on the control plane.

That assumption was valid at FV2.6 ship time — `Fast_API__SP__CLI` mounted the api_site at `/ui/`. But BV2.10 folded SP-CLI under `/legacy/`, so the mount moved to `/legacy/ui/`. `Fast_API__Compute` had no `/ui/` mount of its own. Every `import { apiClient } from '/ui/shared/api-client.js'` (and all shared component imports) became a 404 at load time.

## Why Option A (relative imports) was rejected

The T1.7 brief recommended "Option A — relative imports" as a frontend-only fix. On inspection, this is not feasible:

- Detail files are served from `/api/specs/<spec>/ui/detail/v0/v0.1/v0.1.0/`
- Shared assets are at `/legacy/ui/shared/` and `/legacy/ui/components/sp-cli/_shared/`
- These are on completely different URL branches — no `../` traversal can cross from `/api/specs/...` to `/legacy/...`

Relative imports would only work if the shared utilities were duplicated into every spec's `ui/shared/` directory — 8 copies of `api-client.js` and `node-state.js`. This was rejected as maintenance debt.

## Fix: Option B — `/ui/` mount on `Fast_API__Compute`

Added `_mount_dashboard_ui()` to `Fast_API__Compute.setup()`:

```python
def _mount_dashboard_ui(self):
    from starlette.staticfiles import StaticFiles
    if self.dashboard_root_override:
        path = self.dashboard_root_override
    else:
        import sgraph_ai_service_playwright__api_site
        path = sgraph_ai_service_playwright__api_site.path
    self.app().mount('/ui', StaticFiles(directory=path, html=True), name='dashboard-ui')
```

This mounts `sgraph_ai_service_playwright__api_site/` at `/ui/` directly on `Fast_API__Compute`, making it independent of the `/legacy/` sub-app lifecycle. The `/ui/` path is now a stable, non-deprecated URL prefix for dashboard shared assets.

**The 8 spec detail JS files require zero changes** — their `/ui/shared/...` and `/ui/components/sp-cli/_shared/...` imports resolve correctly again.

## Why this is the right long-term decision

- `/legacy/` carries `X-Deprecated` headers and will eventually be removed (v0.3.x). Coupling spec UI imports to `/legacy/ui/...` would require another fix sweep at that point.
- `/ui/` on `Fast_API__Compute` is stable, documented in the route comment block, and survives the SP-CLI deprecation.
- `dashboard_root_override` attribute on `Fast_API__Compute` allows test injection without mocks, consistent with the existing `ui_root_override` pattern.

## Process failure captured

This is a **bad failure**: the import rewrite in FV2.6 made an assumption about the URL structure that was broken by a concurrent backend phase (BV2.10). The FV2.6 brief did not include a "live browser smoke test" acceptance criterion — if it had, this would have been caught before merge.

The new process rule (live smoke test as acceptance gate) would have caught this. Applies from now.
