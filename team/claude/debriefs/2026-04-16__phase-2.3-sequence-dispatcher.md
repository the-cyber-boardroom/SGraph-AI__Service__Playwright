## What was delivered

### `service/Sequence__Dispatcher`

Thin class wrapper over the parse helpers that already live in `dispatcher/step_schema_registry.py` (Phase 1.3). Gives the runners a consistent DI surface for step parsing:

| Method | Purpose |
|--------|---------|
| `parse_single_step(step_dict, step_index)` | One wire-format dict → typed `Schema__Step__*` (via `parse_step` from the registry) |
| `parse_steps(step_dicts)` | Bulk-parse; preserves order and per-step index for ID backfill / error messages |

The module-level registry functions (`parse_step`, `result_schema_for`, `STEP_SCHEMAS`, `STEP_RESULT_SCHEMAS`) stay as-is. The class adds no parsing logic — it's a DI seam, not a re-implementation.

### Scope decision: `execute_step` deferred to Phase 2.5

Spec shows `Sequence__Dispatcher.execute_step(step, page, capture_config, capabilities, artefact_writer)` wrapping `Step__Executor.execute_{action}`. That method needs:

1. `Step__Executor` — the only class allowed to touch `page.*` (not yet built)
2. `Artefact__Writer` — for sink routing (not yet built)
3. A real `playwright.sync_api.Page` — which requires running Chromium (integration test territory)

Rather than ship a stub with a `NotImplementedError` body that lies about the surface, `execute_step` lands in Phase 2.5 alongside `Step__Executor`. Same rationale as the `Browser__Launcher` pivot in Phase 2.2.
