## What was delivered

### `service/Artefact__Writer`

The only class permitted to write to artefact sinks (spec §10 bullet). Surface:

| Method | Purpose |
|--------|---------|
| `write_artefact(artefact_type, data: bytes, sink_config)` | Dispatch bytes to the configured sink; return `Schema__Artefact__Ref` with the correct ref variant populated. Returns `None` when `sink_config.enabled=False`. |
| `encode_inline(data)` | `base64` → `Safe_Str__Text__Dangerous` (used by INLINE sink; exposed for reuse). |
| `write_bytes_to_local(folder, artefact_type, data)` | Filesystem write; creates folder, names file `{type}_{hash}.{ext}`. Fully implemented — no integration-test needed. |
| `write_bytes_to_vault(vault_ref, artefact_type, data)` | **Seam** — `NotImplementedError` default. Subclass / DI provides vault HTTP client. |
| `write_bytes_to_s3(bucket, prefix, artefact_type, data)` | **Seam** — `NotImplementedError` default. Subclass / DI provides `osbot-aws` S3 adapter. |
| `read_from_vault(vault_ref) -> dict` | **Seam** — JSON-in helper for `Credentials__Loader.apply`. |
| `write_to_vault(vault_ref, data: dict) -> None` | **Seam** — JSON-out helper for `Credentials__Loader.save_state_to_vault`. |

Module-level `FILENAME_EXTENSIONS` maps each `Enum__Artefact__Type` to a file extension (png/webm/pdf/har/zip/log/log/html). `HASH_LEN = 10` — `Safe_Str__Hash` is exactly 10 hex chars (40 bits, identity not crypto).

### Design — seams, not adapters

The real vault HTTP client and `osbot-aws` S3 client don't exist yet in this repo. Rather than stub wrong shapes for them, the class exposes four subclass-overridable seams. The unit test uses an in-memory `_InMemoryWriter(Artefact__Writer)` subclass — three dicts (vault JSON / vault bytes / S3) and a trivial `write_bytes_to_*` override each. No mocks, no patches. Same pattern as `_FrozenSessionManager` in Phase 2.2.

`write_bytes_to_local` is fully implemented because it's just `os.makedirs` + `open(...).write`, no external service.

### Scope decision: `capture_*` methods deferred

Spec flow diagram (line 1159) shows `Step__Executor.execute_{action}() → Artefact__Writer.capture_*()`. Those methods need:

1. A real `playwright.sync_api.Page`
2. Per-artefact-type Playwright APIs (`page.screenshot()`, `page.pdf()`, `context.tracing.start/stop`, HAR recording at context creation, etc.)

They land alongside `Step__Executor` in Phase 2.5. `write_artefact` already takes raw `bytes` — so once `capture_*` produces the bytes (from `page.screenshot()` etc.), routing to the sink is already solved.

## Deviations from brief

- v1 source not in pack — method surface derived from three callsites (Credentials__Loader on lines 988/993/1002 + the `capture_*` reference on line 1159).
- `capture_*(page, …)` deliberately omitted — needs Playwright `Page`; lands in Phase 2.5 with `Step__Executor`.
- `Safe_Str__Hash` is exactly 10 hex chars. Used as a truncated SHA-256 identity for artefact content — not for integrity/cryptographic verification. If the spec later requires a full SHA-256, a wider `Safe_Str__Hash` variant (or separate primitive) is needed.
