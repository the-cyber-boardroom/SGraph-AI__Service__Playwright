# Debrief: OpenSearch Index Pattern Creation
**Slice:** v0.1.72 — observability stack  
**Branch:** `claude/secure-nginx-vnc-proxy-FE75Q`  
**Date:** 2026-04-24  
**Outcome:** Working — `sp ob create` now creates the index pattern end-to-end with zero manual steps.

---

## What we were trying to do

Every new `sp-observe-*` stack requires an index pattern (`sg-playwright-logs`, time field `@timestamp`) in OpenSearch Dashboards before any visualisation can render. The goal was to create it automatically during `sp ob create` so a fresh stack is dashboard-ready with no manual UI steps.

---

## What went wrong (root causes, in order discovered)

### 1. Wrong API endpoint (`data_views` silently no-ops)

`POST /api/data_views/data_view` returns HTTP 200 on AWS-managed OpenSearch 3.5 but does not persist the object. No error, no body indication — pure silent no-op. This masked the failure for several iterations.

**Good failure** — caught by the `_verify_index_pattern_exists()` read-back check once we added it.

### 2. Verify function checked status code, not body

The original `_verify_index_pattern_exists()` returned `True` on any HTTP 200, including the silent no-op above. Replaced with a body-aware check: reads back the saved object and checks `.attributes.title == OPENSEARCH_INDEX`.

**Good failure** — straightforward to fix once the no-op was identified.

### 3. IAM / SigV4 cannot write saved objects (FGAC + system-index restriction)

Attempts to write directly to `.opensearch_dashboards*` via SigV4 are blocked even for `all_access` roles on AWS-managed OpenSearch with FGAC enabled. Only internal users with `security_manager` permission can write saved objects. The `playwright-ec2` IAM role has `all_access` mapped but not `security_manager`. Every SigV4 system-index PUT returned 403.

**Bad failure** — two separate `_sys_idx_write_sigv4()` and `_sys_idx_write()` functions were written, tried, and silently fell back without surfacing the 403. Both are now deleted.

### 4. `fields` key in POST body causes 400

Sending `"fields": "[]"` (or any value) in the `attributes` object causes:
```
HTTP 400: {"message":": definition for this key is missing"}
```
on AOS. The working curl omits `fields` entirely. OSD auto-discovers fields from the live index mapping after the pattern is saved. Removed `_fetch_index_fields()` and the `fields` attribute from the body.

**Good failure** — caught immediately once we added full debug logging.

### 5. Tenant set in both header AND query string causes 400

Our URL included `&security_tenant=global` as a query parameter *and* `securitytenant: global` as a header. The working curl sets the tenant via header only (`?overwrite=true` with no tenant param). AOS rejects the request when the tenant appears in both places.

**Bad failure** — this was the final blocker and took multiple iterations to find because the error message (`": definition for this key is missing"`) gives no indication it is about the query string rather than the JSON body. It was only identified by adding full debug logging and comparing the exact URL to the working curl.

## What actually works (confirmed on `sp-observe-2`, 2026-04-24)

```
POST /_dashboards/api/saved_objects/index-pattern/{id}?overwrite=true
  Auth:    Basic <admin>:<master-password>
  Headers: Content-Type: application/json
           osd-xsrf: true
           securitytenant: global        ← header only, NOT query param
  Body:    {"attributes": {"title": "<id>", "timeFieldName": "@timestamp"}}
           ← no 'fields' key at all
```

Verify by reading back and checking `.attributes.title`:
```
GET /_dashboards/api/saved_objects/index-pattern/{id}
  → check r.json()['attributes']['title'] == id
```

## Lessons

1. **"Definition for this key is missing" does not mean what it says.** It surfaced for two completely different root causes — a query-string conflict and a body key AOS rejects. Never trust AOS error messages as literal descriptions.
2. **Silent no-ops are the worst failure mode.** `POST /api/data_views/data_view` returning 200 without persisting wasted the most time. Read-back verification after every write is non-negotiable.
3. **Compare against a known-working curl before debugging Python.** The working curl was available from the other debug session. Comparing the two directly (via the debug logging we added) found the query-string bug in one run.
4. **Delete dead code immediately.** `_sys_idx_write` and `_sys_idx_write_sigv4` were confirmed non-functional but stayed in the file for several commits, adding noise to every read.

## Acceptance criteria met

- `sp ob create <name>` creates the `sg-playwright-logs` index pattern in a single run with no manual steps.
- `sp ob dashboard-import` fails fast with a clear error if admin credentials are missing.
- `_verify_index_pattern_exists()` checks the response body, not just the status code.
- All silent fallback paths removed.
