## What was done

Migrated the `docker` spec's card and detail web components out of the dashboard tree and into `sg_compute_specs/docker/ui/` — the canonical co-location structure defined in FV2.6.

### Files moved (git mv)

| Source | Destination |
|--------|-------------|
| `api_site/plugins/docker/v0/v0.1/v0.1.0/sg-compute-docker-card.{js,html,css}` | `sg_compute_specs/docker/ui/card/v0/v0.1/v0.1.0/` |
| `api_site/components/sp-cli/sg-compute-docker-detail/v0/v0.1/v0.1.0/sg-compute-docker-detail.{js,html,css}` | `sg_compute_specs/docker/ui/detail/v0/v0.1/v0.1.0/` |

### Detail JS — relative → absolute imports

The detail component had 7 relative `import` statements that pointed into the dashboard tree. These would have broken after the file moved out of `components/sp-cli/`. They were rewritten to absolute `/ui/` paths:

- `../../../../../../shared/api-client.js` → `/ui/shared/api-client.js`
- `../../../../_shared/sg-compute-stack-header/...` → `/ui/components/sp-cli/_shared/sg-compute-stack-header/...`
- (and 5 other shared components similarly)

The `/ui` prefix is where `Fast_API__SP__CLI` mounts the `sgraph_ai_service_playwright__api_site` directory. This is a known temporary coupling; it will resolve when shared components also move to `tools.sgraph.ai`.

### admin/index.html

Two script tags updated from dashboard-relative paths to API paths:
- `../plugins/docker/...` → `/api/specs/docker/ui/card/v0/v0.1/v0.1.0/sg-compute-docker-card.js`
- `../components/sp-cli/sg-compute-docker-detail/...` → `/api/specs/docker/ui/detail/v0/v0.1/v0.1.0/sg-compute-docker-detail.js`

Comment block updated to document the mixed-mode loading (migrated specs use API paths, others still use dashboard-relative paths).

---

## Decisions made

**Absolute `/ui/` paths in detail imports** — chosen over:
1. Removing the imports entirely (would require admin/index.html to explicitly load all 4 previously-implicit shared components)
2. Using `https://dev.tools.sgraph.ai/...` URLs (those components aren't there yet)

The `/ui/` coupling is accepted as a deliberate temporary state. The same pattern will be used for subsequent spec migrations (podman, vnc, etc.) until the shared components move to `tools.sgraph.ai`.

---

## What was NOT changed

- Card JS: no imports to fix — it only imports from `dev.tools.sgraph.ai` (already absolute).
- HTML/CSS files for both card and detail: no relative references, pure markup/CSS.
- Other spec cards/details in admin/index.html: left unchanged (podman, elastic, vnc, etc. — their FV2.6 sessions follow).
- Shared components in `api_site/components/sp-cli/_shared/`: remain in dashboard tree; not part of this phase.
