## The symptom

```
docker run --rm -p 8000:8000 \
  -e FAST_API__AUTH__API_KEY__VALUE=local-dev-key \
  diniscruz/sg-playwright:latest

# UI → Execute → screenshot_simple:
"detail": "screenshot_simple failed: ModuleNotFoundError: No module named 'playwright'"
```

The `from playwright.sync_api import sync_playwright` lazy import at
`Browser__Launcher.py:64` fired the moment a screenshot was requested.

## What the old `requirements.txt` claimed

```
# playwright intentionally omitted — the base image mcr.microsoft.com/playwright/python:v1.58.0-noble
# already ships playwright + matching Chromium binaries. Re-installing via pip would upgrade the
# Python library past the pre-installed Chromium revision, causing BrowserType.launch to fail.
```

Two assertions in that comment:

1. *"the base image already ships playwright"* — **false** for the `v1.58.0-noble` tag.
2. *"re-installing via pip would upgrade past the bundled Chromium"* — **true** when the install is unpinned, but the fix is to pin, not to omit.

## Evidence chain

Inside a fresh container of the published image:

| Step | Command | Result |
|------|---------|--------|
| 1 | `pip install playwright` (unpinned) | Successfully installs `playwright-1.59.0`. Proves no preinstalled playwright was present in the python interpreter's `dist-packages`. |
| 2 | Re-run `/screenshot/simple` | `BrowserType.launch: Executable doesn't exist at /ms-playwright/chromium_headless_shell-1217/...` — playwright 1.59.0 demanded a Chromium revision the v1.58.0-noble base image doesn't ship. |
| 3 | `pip uninstall playwright`; `pip install playwright==1.58.0` (pinned to match the base image) | Module imports cleanly. `/screenshot/simple` returns a valid PNG of `sgraph.ai` in 1988 ms. |

That's the whole proof: the python module was missing; the version pinned to the base-image tag is what makes the bundled `/ms-playwright/` Chromium binaries usable.

## The fix

`sg_compute_specs/playwright/requirements.txt`:
```
playwright==1.58.0
# MUST match the base-image tag mcr.microsoft.com/playwright/python:v1.58.0-noble.
# … bumping the base image (FROM in the Dockerfile) MUST bump this pin AND the
#   literal in the Dockerfile's verify step in lockstep.
```

`sg_compute_specs/playwright/Dockerfile` — install with `python3 -m pip` (aligned with `CMD ["python3", ...]`) + a build-time guard:
```dockerfile
RUN  python3 -m pip install -r requirements.txt --no-cache-dir
RUN  python3 -m pip show playwright
RUN  python3 -c "import sys, importlib.metadata; \
                 print('python:', sys.executable, sys.version); \
                 import playwright; print('playwright at:', playwright.__file__); \
                 v = importlib.metadata.version('playwright'); \
                 print('playwright version:', v); \
                 from playwright.sync_api import sync_playwright; \
                 assert v == '1.58.0', \
                        f'playwright {v} != 1.58.0 (base image v1.58.0-noble) — bump requirements.txt and this literal in lockstep'; \
                 print('build-time guard OK')"
```

Three things that the guard now proves at build time:
1. The module **imports** in the same interpreter as `CMD`.
2. The installed version is **exactly** `1.58.0` (matches the bundled Chromium).
3. `sync_playwright` is loadable (the Node-driver entry point is reachable).

A successful build means a fresh `docker run` will serve `/screenshot/simple` without any in-container `pip install` ritual.

## Lockstep contract

Three places now carry the `1.58.0` literal:

| File | Line |
|------|------|
| `sg_compute_specs/playwright/Dockerfile` | `FROM mcr.microsoft.com/playwright/python:v1.58.0-noble` |
| `sg_compute_specs/playwright/Dockerfile` | `assert v == '1.58.0', …` |
| `sg_compute_specs/playwright/requirements.txt` | `playwright==1.58.0` |

Bumping the base image **must** move all three in the same commit. The build-time guard refuses to ship if any drift.

## Good failures

- **Empirical reproduction inside the live container.** The dev ran `pip install playwright` to see what would happen, observed the 1.59.0/1217-Chromium mismatch, then `pip install playwright==1.58.0` and watched `/screenshot/simple` work. That's the proof that drove the fix.
- **The build-time guard caught its own bugs three times.** Each broken iteration (`playwright.__version__` doesn't exist; `packaging` isn't installed) failed loudly with a build error rather than silently shipping a broken image.

## Bad failures

- **The original "intentionally omitted" comment in `requirements.txt`** — wrong on both factual claims (the base image *doesn't* ship the python module for this tag; the bundled-Chromium concern is solved by pinning, not by omitting). The comment was a `# trust me` that nobody verified, and it survived long enough to ship a broken image. The new comments are deliberately verbose with the *why* and the *what fails if you change this* so the next maintainer can't make the same call.
- **Three guard-fix commits in a row** (`9b699a2`, `e4c9140`, `16e4cab`). The actual fix in `d33483f` was correct on the first try; everything after was chasing my own guard mistakes:
  - `9b699a2` — defensive `python3 -m pip` change that turned out not to matter (the python/python3 hypothesis was a red herring, but the change is still correct hygiene).
  - `e4c9140` — discovered `playwright.__version__` doesn't exist; switched to `importlib.metadata.version`.
  - `16e4cab` — discovered the base image doesn't ship the top-level `packaging` module; dropped it for a plain string compare.

  Each step was a "good failure" in isolation (BuildKit caught it) but the cumulative roundtrip was avoidable. Should have asked the dev for the full BuildKit log earlier instead of stacking defensive layers.
