## The Problem

Every request to the Playwright service was taking 28-30 seconds — matching the configured service timeout exactly. The smoke suite confirmed it: 4 URLs, 3 requests each, was taking longer than a minute instead of the expected 20s.

The `x-browser-close-ms` response header was the first clue: `28498`. Browser teardown was blocking for 28.5 seconds per call, then the asyncio error was preventing the next request from starting cleanly.

Two distinct bugs compounded to produce this:

---

## Bug 1 — Browser Close Blocking (28-30s)

### Root Cause

`browser.close()` and `playwright.stop()` both trigger Chromium's graceful network drain. The service runs mitmproxy as a sidecar proxy; mitmproxy holds HTTP keep-alive connections open for connection reuse. Playwright's internal close timeout is 30 seconds — it waits for those keep-alive connections to drain before allowing Chromium to exit, which they never do.

This was observable via the `x-browser-close-ms` response header (added in the Prometheus/metrics slice) showing consistently 28,000–29,000ms on every teardown.

The ordering of `playwright.stop()` vs `browser.close()` made no difference — both code paths hit the same Chromium drain wait.

### Investigation Path

1. Initially suspected browser *startup* delay (asyncio blocking `sync_playwright().start()`)
2. Confirmed via `x-browser-close-ms: 28498` that teardown was the bottleneck, not startup
3. Tried `playwright.stop()` before `browser.close()` — no improvement
4. Tried `threading.Thread(target=teardown, daemon=True).join(timeout=2.0)` — capped the block at 2s but did not eliminate it cleanly
5. Final fix: SIGKILL the Node.js subprocess and its Chromium children directly

### Fix — SIGKILL Process Tree (`Browser__Launcher.py`)

The Playwright sync API wraps a Node.js subprocess. The PID is accessible via:
```python
playwright_handle._impl_obj._connection._transport._proc.pid
```

Chromium runs as a child of that Node.js process. The fix scans `/proc` for children, kills them first (to prevent re-parenting to init), then kills the Node.js parent:

```python
def stop(self, session_id: Session_Id) -> Safe_UInt__Milliseconds:
    result = self.browsers.pop(session_id, None)
    if result is None:
        return Safe_UInt__Milliseconds(0)
    ts_before_close = self.now_ms()
    if not self._sigkill_tree(result.playwright):      # Fast path: instant
        import threading                               # Slow fallback: capped at 2s
        def _teardown():
            try:    result.playwright.stop()
            except: pass
            try:    result.browser.close()
            except: pass
        t = threading.Thread(target=_teardown, daemon=True)
        t.start()
        t.join(timeout=2.0)
    return Safe_UInt__Milliseconds(self.now_ms() - ts_before_close)

def _sigkill_tree(self, playwright_handle) -> bool:
    import os, signal
    node_pid = self._node_pid(playwright_handle)
    if not node_pid:
        return False
    for child_pid in self._proc_children(node_pid):
        try:    os.kill(child_pid, signal.SIGKILL)
        except: pass
    try:    os.kill(node_pid, signal.SIGKILL)
    except: pass
    return True
```

**Browser isolation is not affected.** Each request calls `sync_playwright().start()` (fresh Node.js subprocess) + `browser_type.launch()` with no `user_data_dir` (fresh temp profile). Cookies, localStorage, and cache are fully isolated per request regardless of how teardown happens.

### Result

Browser teardown time dropped from 28,000ms to effectively 0ms (SIGKILL is instant).

## Bug 2 — Lambda Asyncio Conflict (requests failing after first success)

### Root Cause

Playwright's sync API (`sync_playwright().start()`) raises an error if called from inside a running asyncio event loop:

```
playwright._impl_py_exception_classes.Error:
  It looks like you are using Playwright Sync API inside the asyncio loop.
  Please use the Async API instead.
```

Lambda runs its ASGI app inside `asyncio.run()` (via the Lambda Web Adapter). Even `def` (sync) FastAPI route handlers execute within that running event loop — so the Playwright sync API blows up on the second or later invocation once the Lambda warm path is established.

This was the "final piece of the puzzle": even with SIGKILL bringing teardown to zero, the next request would fail with this asyncio error because `sequence_runner.execute()` was called from inside the event loop.

### Fix — `_run_sequence()` Wrapper (`Playwright__Service.py`)

A thin wrapper detects whether a running event loop exists and, if so, dispatches the sequence execution to a fresh `ThreadPoolExecutor` thread (which has no running loop):

```python
def _run_sequence(self, request: Schema__Sequence__Request) -> Schema__Sequence__Response:
    import asyncio
    try:
        asyncio.get_running_loop()
        from concurrent.futures import ThreadPoolExecutor
        with ThreadPoolExecutor(max_workers=1) as pool:        # Fresh thread has no running event loop
            return pool.submit(self.sequence_runner.execute, request).result()
    except RuntimeError:
        return self.sequence_runner.execute(request)           # No loop running — EC2 / local, call directly
```

On EC2 and local development there is no running asyncio loop, so `asyncio.get_running_loop()` raises `RuntimeError` and the fast path (`execute()` directly) is taken — zero overhead.

All three `sequence_runner.execute()` call sites in `Playwright__Service` were updated to use `_run_sequence()`:
- `execute_sequence()`
- `browser_screenshot()`
- `run_one_shot()` (which covers navigate, click, fill, get-content, get-url)

## Supporting Fixes

### CI: EC2 Image Rebuild Gap

The `detect-changes` path filter only watched `Dockerfile`, `requirements.txt`, and `lambda_entry.py`. Lambda hot-loads Python from an S3 zip overlay, so this was correct for Lambda. But EC2 pulls the ECR image directly — any Python source change requires a rebuild.

Fixed by adding an `ec2-image` filter to `detect-changes`:
```yaml
ec2-image:
  - 'sgraph_ai_service_playwright/**'
  - '!sgraph_ai_service_playwright/docker/**'
```

`image-rebuild-needed = true` when either `lambda-image` OR `ec2-image` matches.

### CI: AWS Public ECR Rate Limiting

Unauthenticated pulls to `public.ecr.aws` (used for the Lambda Web Adapter base image) are throttled. Added an authentication step before the Docker build:

```yaml
- name: Authenticate to AWS Public ECR
  run: |
    aws ecr-public get-login-password --region us-east-1 | \
      docker login --username AWS --password-stdin public.ecr.aws
```

Requires two IAM permissions on the CI role:
- `ecr-public:GetAuthorizationToken`
- `sts:GetServiceBearerToken` (STS dependency — required even when the ECR permission is present)

## Good Failures

- **Order-swap didn't help** — reversing `playwright.stop()` / `browser.close()` order was a reasonable hypothesis and was tried. The response headers disproved it quickly (still 28s). Good failure: surfaced early, didn't waste time.
- **Thread timeout as intermediate step** — the 2s timeout cap was a legitimate incremental fix but still left a 2s penalty. Prompted the SIGKILL approach.

## Bad Failures

- **EC2 image rebuild gap** — Python source changes were silently not baked into the EC2 image. This was invisible because EC2 wasn't the primary deployment target when the filter was written (Lambda S3-zip path was the focus). Not caught by CI — it passed green while the EC2 container was running stale code. Fixed but should have been caught at the time the EC2 target was added.

## Open Items / Follow-ups

1. **mitmproxy keep-alive root fix** — the real root cause of the 28s close is mitmproxy holding HTTP keep-alive connections open. A `Connection: close` configuration on the mitmproxy sidecar would eliminate the Chromium drain wait entirely. The SIGKILL approach is the correct production workaround; the mitmproxy fix would make teardown clean without force-killing.

2. **Version surfaced on /health/info** — the Docker image does not currently expose the git SHA baked at build time. When debugging "is the EC2 running the right code?", there is no programmatic way to confirm. Baking `GIT_SHA` as a build arg and returning it on `/health/info` would close this gap.

3. **Smoke tests in CI** — the Lambda smoke test job was re-enabled in this session, wired to run after `provision-lambdas`.
