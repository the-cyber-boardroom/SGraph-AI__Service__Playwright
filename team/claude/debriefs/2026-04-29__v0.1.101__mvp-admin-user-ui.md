## What shipped

Three backend PRs and a static UI, implemented by a chain of sub-agents in one
session.

### PR-1 — Linux + Docker routes mounted

`Routes__Linux__Stack` and `Routes__Docker__Stack` already existed but were not
wired into `Fast_API__SP__CLI`. Two lines of setup + two service attributes +
4 mounting tests.

**Schema fix discovered during PR-1:** `extra_ports: List__Port` in both
`Schema__Linux__Create__Request` and `Schema__Docker__Create__Request` had to
become `List[int]` — `Type_Safe__List` subclasses are not handled by
`Type_Safe__To__BaseModel`, which caused a `PydanticSchemaGenerationError` at
FastAPI startup. Runtime behaviour is identical; `List[int]` is what the AWS
helper already accepted.

### PR-2 — `catalog/` sub-package

New package: `sgraph_ai_service_playwright__cli/catalog/`

- `Enum__Stack__Type` — 5 values: LINUX, DOCKER, ELASTIC, OPENSEARCH, VNC
- 4 schemas, 2 collections (one-class-per-file throughout)
- `Stack__Catalog__Service__Entries` mixin — builds the 5 catalog entries with
  endpoint paths derived from `type_id.value` (one helper, no hard-coded paths)
- `Stack__Catalog__Service` — composes Linux/Docker/Elastic services;
  `get_catalog()` + `list_all_stacks(type_filter)`
- `Routes__Stack__Catalog` — `GET /catalog/types` + `GET /catalog/stacks?type=`
- Full test suite: schema round-trips, `_Fake_*` service subclasses, TestClient
  route tests, mounting test

Correct `Fast_API__Routes` import path found by checking the codebase:
`osbot_fast_api.api.routes.Fast_API__Routes` (not `osbot_fast_api.api.Fast_API__Routes`
as the brief assumed).

### PR-3 — `Routes__Elastic__Stack`

New folder: `sgraph_ai_service_playwright__cli/elastic/fast_api/routes/`

5 lifecycle endpoints: list, info (404 on miss), create, delete (404 on miss),
health. Elastic__Service has no `.setup()` method — its `aws_client` is a proper
`Type_Safe`-typed attribute that auto-initialises (unlike Linux/Docker).

404 detection for delete uses `not response.target` (empty `Safe_Str__Id`);
`Schema__Elastic__Delete__Response` has no `.deleted` field. The brief's sketch
assumed `.deleted` — the route adapts to the real service shape as specified.

`Elastic__Service` arg-order deviations (verified before coding):
- `get_stack_info(stack_name, region)` — name-first (opposite of linux/docker)
- `delete_stack(stack_name, region)` — name-first
- `health(stack_name, password, check_ssm)` — no region param

## Post-ship bug fix

**`GET /linux/stacks` → 500 `'NoneType' has no attribute 'instance'`**

Root cause: `Linux__Service` and `Docker__Service` use a lazy init pattern —
`aws_client = None` until `.setup()` is called. `Type_Safe` auto-constructs
the service objects but never calls `.setup()`, so the first live request
hit `None.instance`.

Fix: added `self.linux_service.setup()` and `self.docker_service.setup()`
inside `Fast_API__SP__CLI.setup()` before `super().setup()`. `Elastic__Service`
was not affected — its `aws_client` is a proper `Type_Safe`-typed attribute.

## Failures

### Good failures (surfaced early, fixed)

- **`List__Port` → `List[int]`** — caught at import time by a FastAPI startup
  error before any test ran. Informed the fix immediately.
- **`Fast_API__Routes` import path** — caught by a `ModuleNotFoundError` on
  first test run. One-line fix, no design impact.
- **`Elastic__Service` has no `.setup()` method** — caught by a unit test
  `AttributeError` before the fix was deployed. Removed the call.
- **`GET /linux/stacks` 500 in production** — caught on first live HTTP call
  after deployment. Fixed and re-pushed within minutes.

### Bad failures (none)

No failures were silenced or worked around. All were surfaced by tests or live
calls and fixed at root cause.
