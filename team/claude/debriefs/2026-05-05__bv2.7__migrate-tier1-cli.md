## What Was Blocked (BV2.8 Task 2)

### Goal
Replace `: object = None` DI field annotations with typed annotations in the
22 spec service files (e.g. `Docker__Service`, `Elastic__Service`, etc.).

### Blocker 1 — Type_Safe union types not supported

`X | None = None` union annotations raise a `ValueError` at attribute assignment:

```
ValueError: On Docker__Service, invalid type for attribute 'aws_client'.
Expected 'Docker__AWS__Client | None' but got '<class 'Docker__AWS__Client'>'
```

Type_Safe validates the attribute type strictly — a `Docker__AWS__Client` instance
does not satisfy `Docker__AWS__Client | None` in its current implementation.

### Blocker 2 — Circular imports in `*__AWS__Client.py` files

Moving helper imports to module level in e.g. `Docker__AWS__Client.py` breaks
on import:

```
ImportError: cannot import name 'DOCKER_NAMING' from partially initialized module
'sg_compute_specs.docker.service.Docker__AWS__Client' (circular import)
```

The cycle: `Docker__AWS__Client` → `Docker__SG__Helper` → `DOCKER_NAMING` from
`Docker__AWS__Client`. The constants (`DOCKER_NAMING`, `TAG_*`) are in the same
module as the class, so moving helpers to module scope creates the cycle.

### Resolution

All BV2.8 Task 2 changes were reverted. The `: object = None` pattern remains
as a Type_Safe-compatible DI idiom for now.

**Failure classification:**
- **Good failure** — both blockers surfaced immediately via test runs, not in
  production. No broken code committed.

### Path forward for BV2.8 Task 2

Options (not implemented):
1. Move tag constants out of `*__AWS__Client.py` into a separate `*__Constants.py`,
   breaking the circular import. Then typed annotations become possible once
   Type_Safe gains union support.
2. Upgrade Type_Safe to support `X | None` — upstream `osbot-utils` change.
3. Accept `: object = None` permanently and document the pattern in the style guide.
