## What was wrong

FV2.4 brief Task 3 required a `<sg-compute-spec-detail>` component for viewing a spec's full manifest. FV2.4 only shipped the spec grid — no detail panel, no click handler, not acknowledged in the debrief.

## Changes delivered

### New component — `sg-compute-spec-detail`

`components/sp-cli/sg-compute-spec-detail/v0/v0.1/v0.1.0/` (3 files)

Sections:
- **Header**: icon, display_name, spec_id code, stability badge (coloured), nav_group chip, "Launch node →" header button
- **Manifest panel**: version, boot time, nav group, capability chips, create_endpoint_path — rendered as a `<dl>` property grid
- **Extends lineage**: shows parent spec_ids as `<code>` tags; falls back to "No parent specs — this is a root spec." (all current specs have `extends=[]`)
- **Baked AMIs**: placeholder warning ("GET /api/amis not yet available") — consistent with T2.1 AMI picker
- **Documentation**: anchor to `/api/specs/{id}/readme` — the link is generated even though the backend route isn't wired yet; if the route 404s the user sees a broken link in a new tab (acceptable — better than hiding it)
- **Launch section**: "Launch node →" footer button dispatches `sp-cli:catalog-launch` with `{ entry: spec }`

### `sg-compute-specs-view.js` — "View details" button added

Each spec card gains a "View details" secondary button alongside the existing "Launch node" button. Click dispatches `sp-cli:spec.selected` with `{ spec }`.

### `sg-compute-specs-view.css` — `.btn-detail` style added

Secondary button style (border outline, no fill) — visually subordinate to the primary `.btn-launch`.

### `admin.js` wired

- `_specDetailTabIds: {}` state map (spec_id → panelId)
- `_openSpecDetailTab(spec)` — opens `sg-compute-spec-detail` tab, one per spec_id, focus-if-exists behaviour matching `_openDetailTab`
- Listener for `sp-cli:spec.selected`

### `admin/index.html` — script tag added

`<script type="module" src="../components/sp-cli/sg-compute-spec-detail/...">` loaded alongside the other view shells.

## Also in this commit: linux spec removal

`sp linux` was a backwards-compat alias for `sp podman` in `provision_ec2.py` (line 828). Per user instruction, Docker and Podman are the two canonical execution primitives; a bare-Linux alias adds confusion. Removed:
- `app.add_typer(_podman_app, name='linux')` from `provision_ec2.py`
- `scripts/linux.py` (orphan file, not a console entry point)
- Stale `linux.py` comments in `scripts/vnc.py`
