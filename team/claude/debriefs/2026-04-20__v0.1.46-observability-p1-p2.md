## Design Decisions

### IAM policy separation
`IAM__OBSERVABILITY_POLICY_ARNS` is a separate tuple from `IAM__POLICY_ARNS`. Reason: `test_constants.test__stack_pins_match_design` hard-codes `IAM__POLICY_ARNS == (ECR, SSM)` and this cannot be changed without breaking the test and the semantic assertion it captures.

### `restart: always` test fix
`yaml.count('restart: always') == 2` → `>= 2`. The old assertion was only technically correct for 2 services; extending to 10 services makes `>= 2` the right invariant.

### `render_user_data()` backward compatibility
The test calls `render_user_data(playwright_image_uri=..., sidecar_image_uri=..., compose_content=...)` with no `api_key_value`. New signature adds `api_key_value: str = ''` as an optional parameter — existing test call continues to work.

### Prometheus scrape auth gap
Prometheus does not natively support custom header names (`X-API-Key`) in scrape config for v2.51.0. The scrape config only covers infrastructure targets (cadvisor, node-exporter, loki, grafana, prometheus self). App-level metrics (`sg_playwright_*`, `sg_mitmproxy_*`) are accessible via `sg-ec2 metrics` CLI command which runs `curl` on the EC2 host via SSM. Phase 2 resolution options: (a) auth bridge middleware in FastAPI accepting `Authorization: Bearer <token>`; (b) dedicated unauthed internal metrics port; (c) push gateway.

### Config file injection pattern
All config files are written by single-quoted heredocs inside `USER_DATA_TEMPLATE` via the `{observability_configs_section}` placeholder. Single-quoted heredoc markers prevent shell expansion of `$` in CloudWatch agent config (`${aws:InstanceId}`). Config content strings are substituted as VALUES into the bash script (not processed by Python's `.format()`), so `{}` inside JSON/YAML is safe.

## Good Failures

- **Format string escaping for CloudWatch agent JSON** — `{instance_id}`, `${aws:InstanceId}` etc. in JSON configs would be misinterpreted by Python's `.format()` if the configs were embedded in the template string directly. Resolved by using the `{observability_configs_section}` VALUE substitution pattern.
- **16KB user-data limit** — initially uncertain whether the expanded template would exceed the EC2 limit. Verified: total ~5KB (well within limit).
- **`IAM__POLICY_ARNS` immutability** — attempting to add CloudWatch/X-Ray to `IAM__POLICY_ARNS` would break the test assertion. Identified and resolved by creating a separate tuple.

## Bad Failures

None — all design constraints were respected and existing tests were not broken.
