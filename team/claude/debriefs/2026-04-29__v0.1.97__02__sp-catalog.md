## What shipped

A new `sp catalog` typer command with two subcommands. Mirrors the existing `Routes__Stack__Catalog` HTTP surface but reads directly from `Stack__Catalog__Service` (no FastAPI server required) ‚Äî per Q5 default.

```
sp catalog types                  # List all known stack types + metadata
sp catalog stacks [--type X]      # Cross-section list of live stacks
```

## How

| File | Role |
|---|---|
| `cli/catalog/cli/Renderers.py` | Tier-2A Rich renderers ‚Äî `render_types` / `render_stacks`. Reads `display_name` (not `label`), shows `available` column. |
| `scripts/catalog.py` | Typer entry ‚Äî 2 commands. `_service()` constructs `Stack__Catalog__Service` and calls `setup()` on `linux_service` + `docker_service` (lazy-init pattern). `--type` validates against `Enum__Stack__Type`; unknown values raise `typer.BadParameter`. |
| `scripts/provision_ec2.py` (modified) | Mounts `sp catalog` on the main `sp` app via `add_typer`. |

## What `sp catalog --help` looks like

```
Usage: sp catalog [OPTIONS] COMMAND [ARGS]...

  Cross-section enumeration ‚Äî list known stack types or live stacks across all sections.

Commands:
  stacks   List live stacks across every section. Pass --type to filter by stack type.
  types    List all registered stack types (linux / docker / elastic / opensearch / vnc / ‚Ä¶).
```

## Note: catalog scope today

`Stack__Catalog__Service.list_all_stacks()` only iterates `linux_service` / `docker_service` / `elastic_service`. `OpenSearch__Service`, `Prometheus__Service`, `Vnc__Service` are not yet composed ‚Äî see `team/claude/debriefs/vnc-handover/05__catalog-integration.md`.

So today, `sp catalog stacks --type vnc` will return an empty list even when there are real `sp pw vnc` stacks running. The catalog **types** call already lists VNC + OpenSearch (the entries methods exist), so `sp catalog types` works fully.

When the catalog gets the missing service composition, `sp catalog stacks` automatically picks them up ‚Äî no CLI changes needed.
