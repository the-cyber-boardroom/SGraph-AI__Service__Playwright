### Service classes (10 of 12)
- `service/JS__Expression__Allowlist` — deny-all by default; exact-match gate for the EVALUATE step.
- `service/Request__Validator` — single home for cross-schema validation (spec §9). Methods: `validate_session_create`, `validate_step`, `validate_sink_configs`, `validate_step_ids_unique`, `validate_sequence_timeout`, `reject`. Raises `HTTPException(422)` carrying `Schema__Error__Response`.
- `service/Capability__Detector` — env-driven deployment target detection + per-target `Schema__Service__Capabilities` profiles (lambda, claude_web, laptop, ci, container). Builds `Schema__Service__Info` for `/info` and `Schema__Health__Check` for `/health`.
- `service/Session__Manager` — in-memory session lifecycle (create/get/close, capture config, start time, artefact accumulation, action counters, active-session listing, healthcheck). No Playwright calls; Browser objects are held as opaque refs.
- `service/Sequence__Dispatcher` — parse surface (`parse_single_step`, `parse_steps`) wrapping `dispatcher/step_schema_registry.parse_step`. `execute_step` deferred to Phase 2.5 (needs `Step__Executor`).
- `service/Artefact__Writer` — sink routing for INLINE (base64), VAULT, S3, LOCAL_FILE. Assembles `Schema__Artefact__Ref` with size + 10-hex sha256 identity hash. `write_bytes_to_local` fully implemented; `write_bytes_to_vault` / `write_bytes_to_s3` / `read_from_vault` / `write_to_vault` are subclass-overridable seams (real vault + osbot-aws clients not yet wired). Typed capture wrappers `capture_screenshot(data, sink_config)` / `capture_page_content(...)` / `capture_pdf(...)` (Phase 2.9) pin the artefact type so Step__Executor callers don't repeat it; each returns `None` when the sink_config is disabled.
- `service/Credentials__Loader` — vault ↔ Playwright `BrowserContext` glue. `apply(session_id, session_manager, credentials)` loads cookies / storage state via the writer's vault seams and sets `extra_http_headers`. `save_state_to_vault` persists state back. No direct vault client; no `page.*` calls (context-level only).
- `service/Browser__Launcher` — real Chromium process lifecycle. `launch(browser_config)` spawns via `sync_playwright().chromium.launch(...)`; lazy runtime start; per-session `browsers` registry; `stop(session_id)` / `stop_all()`; healthcheck. `ENV_VAR__CHROMIUM_EXECUTABLE` escape hatch for pre-installed Chromium builds. Only LOCAL_SUBPROCESS + CHROMIUM supported today (CDP_CONNECT, BROWSERLESS, Firefox, WebKit raise `NotImplementedError`). The only non-`Step__Executor` class permitted to import `playwright.sync_api`.
- `service/Step__Executor` — **only class (alongside `Browser__Launcher`) permitted to import `playwright.sync_api`**. `execute(page, step, step_index, capture_config)` dispatches on `step.action`. First-pass handlers (Phase 2.9): `execute_navigate` / `execute_click` / `execute_fill` / `execute_screenshot` / `execute_get_content` / `execute_get_url`. Screenshot and GET_CONTENT bytes are handed to `artefact_writer.capture_*()` — Step__Executor never writes to a sink directly. Every handler catches `Exception`, times the step, and returns a uniformly-shaped `Schema__Step__Result__*` (PASSED or FAILED with error_message). Deferred handlers (`execute_press / select / hover / scroll / wait_for / video_start / video_stop / evaluate / dispatch_event / set_viewport`, 10 total) raise `NotImplementedError` with a `Phase 2.11` marker rather than falling through to AttributeError. Helpers: `passed_result`, `failed_result`, `resolve_id` (falls back to `Step_Id(str(step_index))`), `filter_refs` (drops None refs from disabled sinks), `now_ms` (wall-clock seam for tests).
- `service/Playwright__Service` — thin orchestrator, Phase 2.7 health surface only. Composes Capability__Detector + Session__Manager + Browser__Launcher + Request__Validator. `setup()` primes detection (idempotent). Methods: `get_service_info()`, `get_capabilities()`, `get_health()` (aggregates browser_launcher + session_manager + connectivity sub-checks into `Schema__Health`). Session/action/sequence methods are NOT on this class yet — they land in Phase 2.10 when Routes__Session / Routes__Browser / Routes__Sequence arrive.

### FastAPI app (Phase 2.7 wiring)
- `fast_api/Fast_API__Playwright__Service` — extends `Serverless__Fast_API` (osbot-fast-api-serverless). Holds one `Playwright__Service`. `setup()` primes the service then defers to the base class; the default API-key middleware stays on (reads `FAST_API__AUTH__API_KEY__NAME` / `FAST_API__AUTH__API_KEY__VALUE` from the env — the Lambda receives both via `set_lambda_env_vars`). `setup_routes()` registers `Routes__Health` and `Routes__Set_Cookie` (the osbot-fast-api class that exposes `/auth/set-cookie-form` HTML UI + `/auth/set-auth-cookie` POST — both in `AUTH__EXCLUDED_PATHS` so they bypass the API-key middleware).
- `fast_api/routes/Routes__Health` — `Fast_API__Routes` subclass with `tag='health'`. Three `GET` endpoints: `/health/info` (→ `Schema__Service__Info`), `/health/status` (→ `Schema__Health`), `/health/capabilities` (→ `Schema__Service__Capabilities`). Pure delegation to `Playwright__Service`. Module exports `TAG__ROUTES_HEALTH` + `ROUTES_PATHS__HEALTH` constants.
- `fast_api/lambda_handler.run()` — works end-to-end: `python -m sgraph_ai_service_playwright.fast_api.lambda_handler` boots uvicorn on :8000 and all three `/health/*` endpoints return 200 + valid Schema__* JSON (verified locally 2026-04-17).

### Docker infrastructure (Phase 2.8)
- `docker/Docker__SGraph_AI__Service__Playwright__Base` — shared init for all four Docker classes. `setup()` instantiates `Create_Image_ECR` against `docker/images/sgraph_ai_service_playwright/` and `Deploy_Lambda` against `fast_api.lambda_handler.run`. Helpers: `api_docker()`, `image_uri()` (`:latest`), `repository()`, `path_docker()`, `path_dockerfile()`, `lambda_function()`. Module constants: `IMAGE_NAME='sgraph_ai_service_playwright'`, `LOCAL_PORT=8888`, `CONTAINER_PORT=8000`. (Spec uses hyphenated `SGraph-AI`; Python normalises to `SGraph_AI` per CLAUDE.md §16.)
- `docker/Build__Docker__SGraph_AI__Service__Playwright` — `build_docker_image()` (→ `create_image_ecr.build_image()`), `create_container()` / `start_container()` / `created_containers()` (container lifecycle), `dockerfile()` (file content), `image_architecture()`.
- `docker/ECR__Docker__SGraph_AI__Service__Playwright` — `ecr_setup()` (idempotent repo create), `publish_docker_image()` (runs `check_for_docker_config_json()` first, then `push_image()`). `check_for_docker_config_json()` carries forward the Docker Desktop credsStore workaround (`{'credsStore': 'desktop'}` breaks the Python Docker API ECR login → delete the file).
- `docker/Lambda__Docker__SGraph_AI__Service__Playwright` — `create_lambda(delete_existing, wait_for_active)` builds Lambda at `memory_size=5120 MB` (osbot_aws reads `memory_size`, **not** `memory` — wrong attr silently falls back to 512 MB and Playwright OOMs at cold start with `Runtime.ExitError`), `architecture=x86_64`, `timeout=300s`, wires env vars, deletes existing if asked, creates, waits for `Active`, then creates the Function URL. `set_lambda_env_vars()` propagates seven env vars from the process env to the Lambda env (five `SG_PLAYWRIGHT__*` + two `FAST_API__AUTH__API_KEY__*` for the middleware) and pins `SG_PLAYWRIGHT__DEPLOYMENT_TARGET='lambda'`. `create_lambda_function_url()` does the delete → osbot_aws `function_url_create_with_public_access()` (statement 1: `lambda:InvokeFunctionUrl` + `StringEquals lambda:FunctionUrlAuthType=NONE`) → boto3 `add_permission(InvokedViaFunctionUrl=True)` (statement 2: `lambda:InvokeFunction` + `Bool lambda:InvokedViaFunctionUrl=true`) dance — AWS requires **both** statements for public Function URLs created after October 2025 (osbot_aws only emits the first; without the second AWS returns 403 at the URL). `update_lambda_function()` + `function_url()` for post-deploy ops. Tuning constants: `LAMBDA_MEMORY_MB=5120`, `LAMBDA_ARCHITECTURE='x86_64'`, `LAMBDA_TIMEOUT_SECS=300`.
- `docker/Local__Docker__SGraph_AI__Service__Playwright` — label-based container reuse (`source=local__docker__sgraph_ai_service_playwright`). `create_or_reuse_container()` / `delete_container()`, `GET()` / `POST()` helpers via `requests`, `local_url()` prefixes `http://localhost:8888`, `uvicorn_server_running()` + `wait_for_uvicorn_server_running()` poll container logs for `Uvicorn running on `. Module constant: `LABEL_SOURCE`.
