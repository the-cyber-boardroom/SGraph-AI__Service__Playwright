# Review — v0.2.29 `sg aws` primitives expansion

The v0.2.29 umbrella + 8 sibling packs are a strong shape: locked decisions, Foundation-first orchestration, one-folder-per-slice, deferred items called out explicitly. The author clearly internalised the v0.2.28 pattern. **The substantive issues below are all fixable in the briefs before Foundation starts.**

This review groups findings by severity. Each `RIPPLE` is a brief edit; each `RETHINK` is a scoping conversation; each `NOTE` is informational.

---

## Top-3 things to fix before Foundation starts

1. **Co-locate the new `aws/ec2/` with the existing top-level `__cli/ec2/` package** — or explicitly move it. There are already an `Ec2__AWS__Client` (347 LOC), `Ec2__Service` (203 LOC), schemas, primitives, and a `Safe_Str__AWS__Account_Id` at the top-level `ec2/`. Slice B mentions `scripts/provision_ec2.py` but is silent on the existing `__cli/ec2/` package. This is the single biggest source of confusion in the pack as it stands.
2. **The reality-doc paths are wrong.** The packs reference `team/roles/librarian/reality/aws-and-infrastructure/` and `security-and-audit/` ~12 times — neither domain exists. The actual structure has `cli/`, `infra/`, `security/`, `vault/`, etc. The existing aws content lives under `cli/aws-dns.md` and `cli/observability.md`.
3. **The `credentials` / `creds` / `aws credentials` naming is genuinely confusing and `08__credentials.md` documents a command that does not exist as written.** Today it's `sg credentials *`, not `sg aws credentials *`. Slice G adds `sg aws creds`. The user-guide page needs to either reflect reality or the existing CLI needs to be re-mounted (see §3.3).

## 1. Naming & location collisions

### 1.1 `aws/ec2/` vs existing `__cli/ec2/` — RIPPLE (high)

The pack proposes `sgraph_ai_service_playwright__cli/aws/ec2/` as a brand-new folder. But:

```
sgraph_ai_service_playwright__cli/ec2/                  (EXISTS, ~550 LOC)
├── service/
│   ├── Ec2__AWS__Client.py            347 LOC — find / describe / terminate / IAM helpers
│   └── Ec2__Service.py                203 LOC — preflight, instance creation orchestration
├── primitives/
│   ├── Safe_Str__Deploy_Name.py
│   ├── Safe_Str__Instance__Id.py
│   ├── Safe_Str__AMI__Id.py
│   ├── Safe_Str__AWS__Account_Id.py   ← duplicate (see §1.4)
│   ├── Safe_UInt__Max_Hours.py
│   └── Safe_Str__Stage.py
├── enums/   Enum__Instance__State.py
└── schemas/ Schema__Ec2__Preflight.py + many more
```

It is **mounted on the FastAPI duality** (`Routes__Ec2__Playwright` on `Fast_API__SP__CLI`) and is the backing for `sp create / list / info / delete`. It is also the place `default_playwright_image_uri()` and `default_sidecar_image_uri()` live — both consumed by `Docker__Service`, `Firefox__Service`, and many specs.

**Required brief edit** (Slice B README, Production files section): explicitly choose one of:

- **(A) Consolidate**: move existing `__cli/ec2/` into `__cli/aws/ec2/`, update the ~15 importers (firefox, docker, ec2_service, scripts/provision_ec2 et al.) in the same PR. Slice B size becomes M-L not M.
- **(B) Coexist**: keep `__cli/ec2/` as-is (it powers the FastAPI duality), add `__cli/aws/ec2/` for the new CLI surface, and **import** `Ec2__AWS__Client` from the existing location rather than reinventing it. Slice B size stays M.

I recommend **B** in this milestone (least blast radius; coexistence already exists between `aws/cf/` and `sg_compute_specs/vault_publish/`'s CF helpers) with a follow-up v0.2.30 hygiene pack for consolidation.

### 1.2 `aws/observability/` vs existing `__cli/observability/` — RIPPLE (high)

The pack proposes `sgraph_ai_service_playwright__cli/aws/observability/` for Slice H. But there is already:

```
sgraph_ai_service_playwright__cli/observability/        (EXISTS, ~310 LOC)
├── service/
│   ├── Observability__AWS__Client.py    206 LOC — AMP / OpenSearch / Grafana workspace CRUD
│   └── Observability__Service.py        101 LOC
└── schemas/ Schema__Stack__Component__{AMP,OpenSearch,Grafana}, ...
```

These are **different semantics**: the existing package manages the *infrastructure that stores observability data* (Amazon Managed Prometheus workspaces, OpenSearch domains, Amazon Managed Grafana workspaces); Slice H is the *unified READ surface* over the data already there (S3 logs + CloudWatch logs + CloudTrail).

The naming collision will confuse readers and split the surface.

**Required brief edit:**

- Rename Slice H's folder to `aws/observe/` (matches the locked CLI verb `sg aws observe`, avoids package name clash). The umbrella's Surface Map already uses `observe`; the Foundation file index just needs to match.
- Add one line to Slice H README pointing at the existing top-level `observability/` package and stating "this is the read surface; that is the infrastructure surface — they're complementary".
- A v0.2.30 follow-up pack can absorb the existing top-level `observability/` into `aws/observability-infra/` (or similar) once Slice H is stable.

### 1.3 `sg credentials` vs `sg aws credentials` vs `sg aws creds` — RETHINK (high)

The pack ships `library/docs/cli/sg-aws/08__credentials.md` "covering the existing v0.2.28 `sg aws credentials` work" and Slice G adds `sg aws creds`. But the actual mount today is:

```python
# sg_compute/cli/Cli__SG.py:34-35
from sgraph_ai_service_playwright__cli.credentials.cli.Cli__Credentials import app as _credentials_app
app.add_typer(_credentials_app, name='credentials', ...)
```

So today the verb is `sg credentials add ...`, NOT `sg aws credentials add ...`. The user-guide page is documenting a command surface that doesn't exist.

Two coherent options — pick one:

- **(A) Re-mount existing `credentials` under `sg aws`.** Foundation adds `Cli__Aws.add_typer(_credentials_app, name='credentials')` and keeps the top-level `sg credentials` as a hidden alias. The user-guide page becomes accurate; long-lived AWS keys live under `sg aws credentials`; short-lived scoped credentials live under `sg aws creds`. Reasonable consolidation.
- **(B) Keep `sg credentials` at the top level.** Drop or rewrite `08__credentials.md` to document `sg credentials *`. Slice G's `sg aws creds` continues — the two are intentionally separate (one is "manage your keyring", the other is "give me a scoped session").

Both are defensible. (A) is cleaner UX; (B) is zero-risk to the v0.2.28 work. **Whichever is chosen needs to be explicit in the umbrella's locked decisions.**

### 1.4 Duplicate `Safe_Str__AWS__*` primitives — RIPPLE (medium)

There are currently **three places** the AWS account-id primitive lives or is proposed:

- `sgraph_ai_service_playwright__cli/ec2/primitives/Safe_Str__AWS__Account_Id.py` — existing
- `sgraph_ai_service_playwright__cli/credentials/primitives/Safe_Str__AWS__Account_Id.py` — added this week (v0.2.29 prep, account_id caching work)
- Foundation proposes `aws/_shared/primitives/Safe_Str__AWS__Account_Id.py` — proposed

Same pattern likely true for `Safe_Str__AWS__Region` (~3 places already), `Safe_Str__AWS__Role__ARN` (~2 places), and `Safe_Str__AWS__Tag_*`. Foundation will create a 4th if no consolidation happens.

**Required brief edit** (`02__common-foundation.md` §1.1 + §1.5):

- Foundation owns the canonical home and **migrates the existing duplicates** to import from `aws/_shared/primitives/` in the same PR.
- ~8 files touched (Type_Safe primitives have very narrow blast radius — they're imported by 1-2 callers each typically). Adds ~1-2 hours to Foundation; halves the maintenance surface for every later pack.

If the migration is judged risky for one PR, ship Foundation with **deprecation re-exports** from the old locations and make the migration a Foundation-follow-up.

### 1.5 `aws/iam/graph/` sub-folder vs existing `aws/iam/` — RIPPLE (low)

Foundation §1.2 says "existing `aws/iam/` package gains a `graph/` subfolder" — OK. But the existing `aws/iam/` has `cli/`, `service/`, `templates/` directly (not `aws/iam/cli/Cli__Iam.py` as the only verb). The new `graph/` sub-tree should be:

```
aws/iam/graph/
├── cli/Cli__Iam__Graph.py
├── service/IAM__Graph__Builder.py / IAM__Graph__Walker.py / ...
└── schemas/ enums/ primitives/ collections/
```

Then wired via `iam_app.add_typer(graph_app, name='graph')` inside the existing `Cli__Iam.py`. Brief implies this but doesn't show it explicitly — worth adding to the Slice D pack.

## 2. Missed acknowledgements of existing infrastructure

### 2.1 `aws/lambda_/` has a sophisticated dynamic Click group already — NOTE / RIPPLE (medium)

Foundation and Slice E (Bedrock) talk about `agent invoke <name>` and `tool <session-id>` style verbs without acknowledging the existing pattern in `aws/lambda_/cli/Lambda__Click__Group.py`:

```python
# Two-level dynamic Click group: function-name navigation + per-function verb tree.
# Wires REPL prefix navigation by registering real Command objects under
# node.commands.items() — what makes `sg-c info` style prefix lookup work.
```

This is a **mature pattern in this repo** for "select a resource, then a verb on it". The Bedrock `agent {invoke,get,memory,stop,tools}` tree and the EC2 `<id-or-name>` resolver in Slice B both want this shape.

**Brief edit**: add a line in Foundation §1.1 (or §2) pointing at `Lambda__Click__Group` as the canonical pattern for "named-resource → verb" sub-trees, and reference it in Slice B README (EC2 `<id-or-name>`) and Slice E README (Bedrock `agent <name>`, `tool browser session <id>`).

### 2.2 `sg aws cf` + `sg_compute_specs/vault_publish/` already have CloudFront CRUD — NOTE (medium)

The vault-publish spec (v0.2.23) ships its own CloudFront management via `sgraph_ai_service_playwright__cli/aws/cf/` (`CloudFront__AWS__Client`, `Distribution__Builder`, `Origin__Failover__Builder`). No Slice in v0.2.29 directly touches CF, but **Slice H (observability) and the CloudFront-via-Firehose mention in locked decision #8** are adjacent.

Worth a one-line acknowledgement in the umbrella so the Foundation PR doesn't accidentally reinvent CF tagging or naming.

### 2.3 Existing `aws/logs/` is `CloudWatch__Logs` — RIPPLE (low)

Slice H README treats `aws/logs/` as "existing CloudWatch source". Confirmed — `Logs__AWS__Client` does `filter_events`, `tail_events`, `start_query`, `get_query_results` against CloudWatch Logs Insights. The wording is right; just confirming.

The `Source__Contract` adapter for CloudWatch in Slice H is then `Logs__Source__Adapter` wrapping `Logs__AWS__Client` — explicitly call this out in Slice H so the agent knows there's no need to write the boto3 calls.

### 2.4 `boto3` already used directly in elastic for S3 — RIPPLE (low)

`__cli/elastic/lets/cf/{consolidate,events,inventory}/service/S3__*.py` use `boto3.client('s3')` directly (4 files). CLAUDE.md rule #14 (no direct boto3) is broken here today. Slice A (S3) should either:

- Migrate these to use the new `S3__AWS__Client` in the same PR; OR
- Explicitly defer to a v0.2.30 hygiene pack and note them as known exceptions.

This is the kind of thing where a foundation-time grep avoids a "wait, why is there still a `boto3.client('s3')` lying around?" PR comment six weeks later.

## 3. Scope concerns

### 3.1 Slice B's relationship to its source brief — RETHINK (high)

The source brief is [`v0.27.43__dev-brief__sg-compute-container-hosts-primitive.md`](../../../../humans/dinis_cruz/briefs/05/17/from__daily-briefs/v0.27.43__dev-brief__sg-compute-container-hosts-primitive.md). Its central ask is the **container hosts primitive** ("Kubernetes-shaped without being Kubernetes"). Slice B delivers EC2 CRUD verbs — useful, but the source brief's *primary intent* is deferred to v0.2.30 per locked decision #11.

That's a defensible scoping choice; **just be honest about it in Slice B's README**:

- Say up front: "This slice is the **EC2 primitive layer** the container-hosts primitive will build on. The container-hosts primitive itself is a separate v0.2.30 pack."
- Don't open with "Promotes the existing `scripts/provision_ec2.py`" — that's the implementation mechanism, not the user-facing intent.

A reader of just the source brief and Slice B would have legitimate trouble seeing the connection without §11 of the umbrella.

### 3.2 Pricing API verb pulls in a separate AWS service — RIPPLE (low)

`sg aws ec2 pricing` calls AWS Pricing API. The README correctly notes "Pricing API only lives in `us-east-1`". But Pricing API is a separate service with its own IAM permissions (`pricing:GetProducts`). Worth:

- Adding `pricing:GetProducts` to the scoped-creds catalogue for the read-only EC2 scope (Slice G).
- A `--cache-ttl` on `pricing` (Pricing API is slow and rate-limited).

### 3.3 Slice G's local-only scoping — NOTE (high)

The brief and the pack agree: Phase 1 + Phase 2 only. Phase 5 (deployed service) and Phase 6 (audit dashboard) defer.

Good carve-out, BUT: the source brief frames the deployed version as the eventual goal ("the local version is a stepping stone"). Slice G should explicitly **commit to API stability** so Phase 5 is a server swap, not a rewrite. The brief implies this ("The clients do not change. The same `sg-creds get --scope X` call now hits the deployed service.") — bake it into Slice G as an acceptance criterion: `sg aws creds get --scope X` returns a `Schema__Scoped__Credentials` whose shape will not change between local and deployed.

### 3.4 Bedrock Slice E size (XL, ~4 days) — NOTE

The largest slice. Worth flagging that v0.2.30 should plan a 2nd-pass review of just this slice; the scoping (chat + agent + tool, no kb/guardrail/eval) is sensible but a 2800-line slice has the most surface for "we missed a Bedrock SDK quirk". The brief carves out the right escape hatch ("if AgentCore blocks, chat ships standalone").

## 4. Foundation-PR specific issues

### 4.1 `Source__Contract` in Foundation may be premature — RETHINK (medium)

Foundation §2.5 ships the `Source__Contract` ABC. But its **only consumer** (Slice H) is also the last slice planned. Slice A and F implement their adapters — but those adapters are *additions*, not *behaviour Foundation needs in order to compile*.

Alternative: ship the ABC in Slice H, have Slices A and F implement adapters as the very last step of their respective PRs (or as a 30-min final-rebase commit). Reduces Foundation surface.

The risk of the current approach: Slice A and F land their adapters against a Foundation ABC that turns out to need a small signature change when Slice H actually wires it up. Three slices need a rebase instead of one.

**Defensible either way**; just be explicit about the choice. If you keep the Foundation-owned ABC, **lock its signatures hard** (the Foundation acceptance test should include an adapter contract test that exercises every method).

### 4.2 Foundation tests reference `Aws__Tagger` for resources Foundation doesn't create — RIPPLE (low)

§1.4 mentions `Test__Aws__Tagger.py`. The tagger has a public API (`tags_for(surface, verb)`) — what is the test asserting on? Suggest:

- Each `Enum__AWS__Surface` value produces a 5-tag set with the right keys
- The `sg:created-by` value follows the `<sg-version>:<surface>:<verb>` shape  
- Round-trip via `Schema__AWS__Tag.json()` preserves data

(The pack might already assume this; just naming it explicitly so the agent doesn't have to guess.)

### 4.3 `Aws__Region__Resolver` precedence misses one source — RIPPLE (low)

§3.7 precedence:

```
--region <flag>  >  $AWS_REGION  >  bucket/resource region  >  $SG_AWS__DEFAULT_REGION  >  us-east-1
```

Missing: **the active sg role's region** (from `Schema__AWS__Role__Config.region`). Today `Sg__Aws__Session` already uses this; the resolver should honour it. Probably sits between `$AWS_REGION` and resource hint, or just after `--region`.

**Required addition**:

```
--region <flag>
  > $AWS_REGION
  > active sg role's region (Sg__Aws__Context.get_current_role() → Schema__AWS__Role__Config.region)
  > bucket/resource region (when applicable)
  > $SG_AWS__DEFAULT_REGION
  > us-east-1
```

## 6. Smaller things worth fixing now

- **Umbrella ratification check** says "All 12 locked decisions accepted by Dinis" — the list only contains 12 items numbered 1-12. Good. But each Slice pack repeats the locked decisions inline rather than pointing at the umbrella; if any get amended it will be in 8 places. **Suggest**: each sibling pack's "Where this fits" section says "the umbrella's locked decisions are non-negotiable; this pack adds slice-specific decisions only". Then each pack's body only carries its own decisions.
- **Slice H's `agent-trace <vault-session-id>`**: depends on the vault recording the session ID with the same key the S3 and CloudWatch sources can correlate against. **The tagging convention in Foundation §3.6 doesn't include `sg:session-id`.** Add it (Foundation can leave the value optional; Slice H asserts presence at the point it tries to correlate).
- **Slice B `wait` polling backoff** (2s → 5s → 10s → 15s capped, 5 min total) is conservative but reasonable. Note that `Ec2__Service` already has wait logic — reuse, don't reinvent.
- **Slice E's vault layout** mentions `<vault>/bedrock/chat/<ISO-day>/<run-id>.json`. The `Vault__Spec__Writer` (v2.9 canonical at `sg_compute/vault/`) requires `(spec_id, stack_id, handle)`. Bedrock is not a spec in `sg_compute_specs/` — Slice E needs to either (a) register a `vault_publish`-style namespace with the spec writer or (b) use the raw `sg_compute/vault/` primitives directly. Either is fine but the brief glosses over the integration.
- **Slice G's vault paths** (`<vault-root>/aws/creds-catalogue/`) similarly need the integration story with `Vault__Spec__Writer`.
- **Slice E's `chat` "list-models"** verb is described but its data source isn't named. `boto3 bedrock list-foundation-models` is the right call; worth saying so explicitly.

## 8. What the packs got right (worth preserving)

So I don't get rolled into the "negative review" category — these are the parts to keep:

- **Eight independent sibling packs** is the correct shape. v0.2.28's two-PR shape (`credentials` + the consumer migrations) showed the seams; eight-way parallelism is a natural extension.
- **Foundation-first**, with hard signature locking, is the right play. The orchestration plan §6.3 ("if a sibling needs a signature change, open Architect-review request") is exactly the right posture.
- **Per-namespace mutation gates** — `SG_AWS__<SURFACE>__ALLOW_MUTATIONS` — preserves the existing convention without overloading a single gate. ✅
- **`Sg__Aws__Session` as the single client seam** — this is the rule that v0.2.28 paid the cost of establishing, and the packs honour it consistently. ✅
- **`Source__Contract`** as the abstraction for unified observability is the right shape (even if §4.1 questions where it lives) — Slice H's design wouldn't be possible without it.
- **Locked scoping for Slice E (Bedrock chat+agent+tool only) and Slice D (IAM graph phases 1+3 only)** — disciplined carve-outs.
- **Tagging convention (`sg:owner`, `sg:source`, `sg:created-by`, `sg:created-at`, `sg:surface`)** is well-thought-out and the precondition for both Slice H and the future leak-sweeper.
- **The user-guide page is a per-slice deliverable, not a follow-up** — this is the right pressure to keep the user-facing surface in lock-step with code.
- **Acceptance commands inline in each sibling pack's README** — turns the sibling pack into the actual handoff document. ✅
