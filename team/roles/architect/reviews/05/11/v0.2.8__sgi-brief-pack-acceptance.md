## What the pack gets right

- **Side-effect boundary table** (`02__architecture` §Side-effect boundaries) is the kind of architectural discipline the Playwright service has, mirrored: one class owns each side effect (`Capture__Filesystem` reads live FS, `Bundle__Publisher`/`Bundle__Resolver` own Storage I/O, `Load__Extractor` writes the target, `Exec_Provider__*` is the only command-issuer).
- **Provider abstraction is honest**, not aspirational. Two real families (`Storage`, `Exec_Provider`), three implementations each, with the In_Memory variant designed for test composition (mirrors `register_playwright_service__in_memory()`).
- **`Storage__Local_Disk` mirrors the S3 tree exactly.** This is what makes "zip the registry, drop on a USB stick" actually work (P19 offline-first). Migration is a tree-copy.
- **IFD path immutability + content-addressed manifest** (P4 + P8): cache-control-forever on every leaf, rollback by URI, no versioning gymnastics.
- **Twin pattern** lifted from `SG_Send__Deploy` for both `Exec_Provider__Twin` and `Type__Twin__Bundle` — patterns transfer.
- **Strip workflow is split** into analyse → plan → execute → verify with immutable plans, restore-data captured before each removal, and no `--force` to skip verification before `bake --publish`.
- **Forbidden-patterns table (B.14)** is a paste-ready code-review checklist.
- **No mocks / no patches (B.5)** — same rule as Playwright service. Tests assert on recorded calls.

## Open architectural concerns

### A1 — `Exec_Provider__SSH` as default is a real reversal from sg-compute

Pack's P10 justification is sound (synchronous, ~50× faster, cloud-agnostic, offline-capable), but it trades the SSM bug class for a new one: **port-22 + per-run keypair + per-run SG ingress lifecycle**. If the teardown path leaks, allow-list entries accumulate.

**Mitigation required:** `EC2__Key_Pair__Lifecycle` and `EC2__Security_Group__Ingress` (under `providers/aws/`) need the same rigour the SSM `run_command` path got during the May 10 / May 11 sessions — context-manager based, idempotent revoke, failure-during-body still triggers cleanup. The pack mentions the pattern (B.8) but the implementation needs explicit ownership.

### A2 — `catalog.json` is a single-writer choke point

S3 PUT is atomic per-object but not across the read-modify-write of an index file. Two parallel publishes would clobber each other.

**Mitigation required:** pin the mechanism before M4. Three plausible options:
1. Conditional PUT via `If-Match`/`If-None-Match` (S3 supports this since 2024). Simple, retry on conflict.
2. `catalog/index-<sha>.json` + pointer object pattern. More work, fully atomic.
3. Single-writer constraint enforced by tooling (publish goes through CI only).

Recommend option 1 for v0.1; revisit if we ever have concurrent agents publishing.

### A3 — `state.json` has no concurrency guard

Two `sgi` invocations in the same cwd would race on `tracked`/`current`. Probably fine for single-user CLI in v1, but should be acknowledged.

**Mitigation:** a `state.lock` (advisory `flock(2)`) is the minimal fix. File when M2 lands.

### A4 — GPU cold-start budget vs DLAMI floor is tight

Pack targets `cold_start_target_ms: 90_000` for `vllm_docker` but commits to DLAMI AL2023 OSS as the GPU base (`08__specs-and-recipes` §Architectural decision). Previous session's findings: DLAMI boot floor ~37s vs AL2023 ~17s. So 90s leaves ~53s for first_load + boot + execution.

**Plausible but tight.** NVMe + parallel multipart S3 GET should hit it. The validation spike (companion brief) must measure this end-to-end early — if we're at 120s, the rollout plan changes.

### A5 — Strip `minimal` mode is the most ambitious + risky part

File-access logging on a real instance has gotchas: mmap'd shared libraries don't always show up in inotify; tmpfs noise; transient files. Pack acknowledges this as v1 simplification (debug-only static list shipped first).

**Decision:** strip is **out of scope for the validation spike**. M7 in the pack stays as planned, but ships after the spike validates the core capture/load path. Don't let strip complexity gate the cold-start measurement.

### A6 — `Exec_Provider__Sg_Compute` couples to sg-compute's CLI output format

Pack documents the fallback to text parsing → `feedback-to-sg-compute.md`. Reasonable for v0.1 but every UX change in `sg lc list` could break sgi's parser.

**Recommendation:** file `sg lc list --json`, `sg lc info --json`, `sg lc create --json` requests early. Even with D3 (no sg-compute mutation during spike), these are *additive* flags — they don't change existing behaviour and could land in parallel with the spike if a separate session picks them up.

### A7 — Sidecar `tests/` overlaps with spec `tests/`

Pack defines `sidecar/tests/{unit,integration,end_to_end}/` (`05__bundles` §Sidecar) AND `sg_image_builder_specs/<spec>/tests/{unit,integration,end_to_end}/` (`08__specs-and-recipes` §Each spec is a Python package). The relationship between the two is not pinned.

**Question to resolve before M3/M4:** is the sidecar `tests/` a *copy* of the spec test suite (so bundles travel with their tests), a *bundle-scoped subset* (only what tests this bundle, not the whole recipe), or *both* with a rule for choosing?

**Recommendation:** sidecar `tests/` should be **bundle-scoped only** — the spec's test suite stays in code, registered with the spec package. The sidecar's tests prove "this bundle's contents are intact and runnable in isolation". This avoids the duplication and makes the strip workflow's test arbiter unambiguous (P16 = spec tests in code, not sidecar tests).

### A8 — `Schema__Bundle__Module.modules` is declared-only in v1

Carrying the field forward without runtime support means the first person who populates it creates dead metadata.

**Recommendation:** ship the schema field with a comment marking it informational; either the `Bundle__Packer` defaults it to `[]` and never populates, or we punt the field to v0.2. Lean toward the latter — easier to add later than to remove.

### A9 — `Storage` interface needs a streaming API spec

Pack's interface declares `get_stream` / `put_stream` but doesn't pin the iteration unit (bytes? chunks of N MiB? S3 multipart-aligned?).

**Recommendation:** pin to 8 MiB chunks for `Storage__S3` to match its multipart threshold; `Storage__Local_Disk` and `Storage__In_Memory` match for consistency. Adds one line to the interface contract; saves a future refactor.

## Conditions on acceptance

These must be addressed during or alongside implementation; they are not blockers for starting M0.

1. **The validation spike (companion brief) runs first**, before any work on the full ladder or the strip workflow. If the spike doesn't deliver a credible cold-start win, we revisit the whole pack.
2. **`providers/aws/` AWS-helper context managers** (key pair, SG ingress, ephemeral launcher) get the same rigour as the SSM path did. Explicit teardown tests with simulated failures.
3. **`catalog.json` write strategy is pinned** in a code comment + schema doc before M4 publishes the first bundle.
4. **A7 is resolved** (sidecar tests = bundle-scoped only, my recommendation) before M3 packages the first bundle.
5. **Each milestone produces a debrief** under `team/claude/debriefs/`, indexed normally. The pack's "release readiness" checklist (`09__implementation-plan` §Acceptance for v0.1.0) is the final gate; per-milestone gates are the debriefs.
