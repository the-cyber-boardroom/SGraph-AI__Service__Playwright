# Architecture Design — FastAPI TLS Sidecar + `cert` CLI Section (PoC)

**Reviewer:** Architect
**Date:** 2026-05-14
**Repo version:** `v0.2.6`
**Status:** DESIGN — PoC scope + phased build-out, decisions flagged for human sign-off.
**Parent doc:** [`v0.2.6__vault-app-tls-options.md`](v0.2.6__vault-app-tls-options.md) — established that vault-app must serve HTTPS to unblock Web Crypto, recommended a Caddy sidecar (Option A). This doc explores the user's follow-up: *could a FastAPI sidecar do what Caddy does, and grow a `cert`/`https` command surface?*

---

## 1. The reframe: we already have a FastAPI sidecar

The user's question — *"how hard would it be to write a simple FastAPI sidecar that does what Caddy is doing?"* — has a better answer than "write one":

> **`Fast_API__Host__Control` already exists and already runs on every vault-app stack.**
> It is the `host-plane` container — `sgraph_ai_service_playwright_host` — sitting on
> `vault-net` with the docker socket mounted and the access token wired in as
> `FAST_API__AUTH__API_KEY__VALUE`. It just doesn't publish a host port today.

`sg_compute/host_plane/fast_api/Fast_API__Host__Control.py` is a `Serverless__Fast_API`
subclass with 8 route classes (Auth, Containers, Docs, Images, Logs, Pods, Shell, Status),
always-on API-key auth, and — notably — it **already serves `/auth/set-cookie-form` and
`/auth/set-auth-cookie`**, the exact cookie machinery the vault UI needs.

So the real question is not "build a sidecar" — it's **"what part of Caddy's job should
move into our FastAPI world, and what should stay with a purpose-built proxy?"** That
decomposition is the whole design.

Node is a non-starter here — it would be the only non-Python runtime in the fleet. "Ideally
FastAPI" is the right instinct; this doc assumes Python/FastAPI throughout.

---

## 2. Decomposing "what Caddy does"

Caddy in our context is really five jobs. They have *very* different FastAPI feasibility:

| Caddy job | FastAPI / Python feasibility | Verdict |
|---|---|---|
| **Terminate TLS on :443** | uvicorn does this natively (`--ssl-keyfile/--ssl-certfile`, or a programmatic `ssl.SSLContext`) | **Easy** |
| **Mint a self-signed cert** | `cryptography` lib — `Routes__CA` (`sg_compute_specs/mitmproxy/api/routes/Routes__CA.py:48-62`) already *parses* x509; generation is the same library, ~30 lines | **Easy** |
| **Inspect / test a cert** | `cryptography` again — fingerprint, validity, SAN, self-signed? `Routes__CA.info()` is the working precedent | **Easy** |
| **ACME client** (real certs, incl. the new `shortlived` IP certs) | the `acme` Python library — certbot's own library — drives `http-01` / `tls-alpn-01` issuance | **Medium — and this is the high-value piece** |
| **Reverse-proxy bytes to an upstream** | *not* built in — a hand-rolled `httpx` streaming proxy: websockets, SSE, chunked encoding, hop-by-hop headers, buffering. Our own `Section__Nginx` header flags SSE-safety / `proxy_buffering off` as real concerns | **Hard / risky — do not hand-roll for production** |

The pattern is stark: **four of the five jobs are FastAPI's strengths or already have
in-repo precedent. The fifth — being a production reverse proxy — is the one thing you
genuinely should not reimplement.**

---

## 3. The core principle: split the data plane from the control plane

> **Data plane** = terminate TLS, move bytes to the upstream. Latency-sensitive,
> protocol-fiddly, security-critical, *boring* — a solved problem. Caddy/nginx own this.
>
> **Control plane** = mint / request / renew / inspect / *test* certificates, and expose
> that lifecycle as an API and a CLI. This is where the durable, reusable, **dogfoodable**
> value is — especially now, because **publicly-trusted IP-address certs are brand new
> (LE GA Jan 2026) and tooling for them is thin.** Owning a clean control plane for them
> is a real asset, not a yak-shave.

**Recommendation:** build the **control plane in FastAPI**; leave the **data plane** to a
purpose-built proxy. Concretely:

- A reusable **`sg_compute/platforms/tls/` library** owns all cert logic (generate,
  inspect, ACME, secure-context probe) — pure Python, no FastAPI, no AWS, unit-testable
  in-memory.
- A **FastAPI surface** exposes that library over HTTP (which app — see §4).
- A **`cert` CLI section** drives the surface (see §6).
- The **data plane** stays Caddy (per the parent doc) — but Caddy is now *fed* by our
  control plane: host-plane writes the cert files, Caddy reloads. Caddy does the boring
  proven thing; we own the interesting thing.

This also means the PoC and the long-term design are the same shape — no throwaway work.

---

## 4. Which FastAPI app exposes it? (a security-boundary decision)

Two candidates, and this is **not** a free choice — it is a security-boundary call that
needs AppSec eyes (flagged Q2):

### Candidate A — extend `host-plane` (`Fast_API__Host__Control`)

Add a `Routes__Host__Cert` route class. Maximum reuse: auth, CORS, the cookie-form, the
`Serverless__Fast_API` base, the boot-time key — all already there.

- **The catch:** host-plane is **privileged** — it has the docker socket and can control
  every container on the box. Today it is **internal-only** (no published host port).
  Putting host-plane on a public `:443` listener makes the privileged control plane
  internet-facing. That is a real escalation of blast radius.

### Candidate B — a new slim, unprivileged app (`Fast_API__TLS` / `Fast_API__Cert`)

A purpose-built `Serverless__Fast_API` subclass with **only** the cert routes, no docker
socket, no privilege. It is the thing that may live on a public port.

- More code, but a **clean boundary**: the public-facing surface can do cert things and
  *nothing else*. host-plane stays internal.

### Architect's call

**The cert *logic* lives in the `sg_compute/platforms/tls/` library regardless** — that
decision is independent and safe. **The *exposure* should be Candidate B for anything
public-facing.** host-plane *may* also mount the same route class for *internal* operator
convenience (reachable via SSM port-forward, like everything else on host-plane today),
but host-plane does **not** go on a public port. This keeps the privileged plane internal
and gives the public surface a minimal attack area.

For the **PoC specifically**, Candidate B is also simply *smaller* — see §5.

## 5. The PoC: prove `window.isSecureContext === true`

The PoC's single job is the acceptance test from the parent doc's C10: **a real browser,
hitting this stack over HTTPS, reports `window.isSecureContext === true` and exposes
`crypto.subtle`.** Crucially —

> **The PoC needs no reverse proxy at all.** Proving secure-context only needs *something*
> served over TLS. The hard/risky data-plane job is entirely out of PoC scope.

### PoC components

| # | Component | What it is |
|---|---|---|
| P1 | `sg_compute/platforms/tls/Cert__Generator.py` | `cryptography`-based self-signed cert+key generation (CN/SAN = the instance public IP). ~30 lines. |
| P2 | `sg_compute/platforms/tls/Cert__Inspector.py` | given a PEM *or* a live `host:443`, return issuer / SAN / validity / fingerprint / `is_self_signed`. Mirrors `Routes__CA.info()`. |
| P3 | `sg_compute/platforms/tls/Schema__Cert__Info.py` | Type_Safe schema for P2's output (one class per file; no raw primitives). |
| P4 | `Fast_API__TLS` slim app + `Routes__TLS` | endpoints: `GET /tls/cert-info`, `GET /tls/secure-context-check` (serves a tiny self-contained HTML page that evaluates `window.isSecureContext` + `!!window.crypto?.subtle` and renders the result + POSTs it to…), `POST /tls/secure-context-result` (records what the browser reported). uvicorn started with the P1 cert on `:443`. |
| P5 | compose + user-data wiring | one new service in `Vault_App__Compose__Template`, publishes `443`, SG opens `443` — exactly the C1–C5 surface from the parent doc, just pointing at `Fast_API__TLS` instead of Caddy for the PoC. |
| P6 | `sp vault-app cert check` | the CLI command that runs the acceptance test — see §6. |

### The dogfooding angle (this is the elegant part)

**This repo *is* a browser-automation service.** The secure-context check does not need a
human with a browser — it is a one-line `page.evaluate('window.isSecureContext')`. So
`sp vault-app cert check` should **drive the Playwright service against the vault-app
stack**:

1. CLI calls the Playwright service (EVALUATE step) → load `https://<public-ip>/tls/secure-context-check`
2. evaluate `window.isSecureContext` and `!!(window.crypto && window.crypto.subtle)`
3. assert both `true` → the PoC passes, automatically, in CI, no human

The SG Playwright Service validating its own sibling's TLS posture is a genuinely strong
architectural story — and it exercises the EVALUATE path end-to-end as a side benefit.

> **Allowlist note (boundary check):** `window.isSecureContext` is **not** on the
> `JS__Expression__Allowlist` (deny-all by default — `Step__Executor.py` /
> `JS__Expression__Allowlist.py`). The PoC must either add these two exact expressions to
> the allowlist (preferred — they are constant, side-effect-free reads) or use a trusted
> `allow_all` surface. **Adding two read-only expressions to the allowlist is a
> deliberate, logged allowlist change** — small, but it crosses the security boundary the
> Architect guards, so it is called out as decision Q3, not done silently.

## 6. The `cert` / `https` CLI section

Per the research, the precedent is `Spec__CLI__Builder._register_ami` — a sub-typer
mounted onto *every* spec's app. A `cert` sub-typer added the same way (`_register_cert`)
gives every spec `sp <spec> cert …` for free, which directly serves the parent doc's
fleet-wide "all exposed ports use SSL" goal.

### Proposed command surface

| Command | What it does | Reaches |
|---|---|---|
| `sp <spec> cert show` | TLS-handshake the stack's `:443`, decode the presented cert — issuer, SAN, validity, days-left, fingerprint, `self-signed?` | `Cert__Inspector` against `host:443` |
| `sp <spec> cert check` | the **PoC acceptance test** — handshake **+** dogfood the Playwright service for `window.isSecureContext` / `crypto.subtle` → pass/fail | Playwright service + `:443` |
| `sp <spec> cert request` | trigger issuance on the sidecar — `--mode self-signed \| letsencrypt-ip \| letsencrypt-dns` | `POST /tls/cert/request` |
| `sp <spec> cert renew` | force a renewal cycle (relevant once ACME `shortlived` IP certs land — ~6-day rotation) | `POST /tls/cert/renew` |
| `sp <spec> cert local` | purely local: generate a self-signed cert, or inspect a PEM file on disk — **no AWS, works offline** | `Cert__Generator` / `Cert__Inspector` |

`--mode` is a fixed-value set → an `Enum__Cert__Mode` class (no Literals — CLAUDE.md rule
3). Each command maps 1:1 onto a `Routes__TLS` method, so the CLI is a thin client over
the same library the sidecar uses.

The user floated "maybe later moved to a `cert`/`https` section." Recommendation: **start
as the per-spec `cert` sub-typer** (consistent with `ami`, zero new machinery), and *if* a
cross-spec top-level `sp cert …` proves wanted, mount it in the top-level `sp` app
(`scripts/provision_ec2.py` already mounts cross-cutting typers). Don't build the
top-level group speculatively.

## 7. The reusable TLS library — `sg_compute/platforms/tls/`

Cert logic is **cross-cutting**, not vault-app-specific — it belongs under `sg_compute/`,
not `sg_compute_specs/vault_app/`. Proposed module (one class per file, all `Type_Safe`):

| File | Responsibility |
|---|---|
| `Cert__Generator.py` | self-signed cert+key generation (`cryptography`) |
| `Cert__Inspector.py` | decode a PEM or live `host:443` → `Schema__Cert__Info` |
| `Cert__ACME__Client.py` | ACME issuance via the `acme` lib — incl. the `shortlived` IP-cert profile (the parent doc's Option B1) and standard hostname certs (B2) |
| `Secure_Context__Probe.py` | drives the Playwright-service dogfood check; returns a Type_Safe result |
| `Schema__Cert__Info.py` / `Schema__Secure_Context__Result.py` / `Enum__Cert__Mode.py` | pure-data schemas + the mode enum |

This module is the **single home** the FastAPI surface, the CLI section, and the
user-data builder all import from. It is unit-testable in-memory with no AWS and no
network (generate → inspect round-trips locally) — satisfying the no-mocks rule.

## 8. Phasing

| Phase | Scope | Outcome |
|---|---|---|
| **P0 — the PoC** | §5 (P1–P6). Slim `Fast_API__TLS` on `:443` with a self-signed cert, the secure-context test page, `sp vault-app cert check` dogfooding Playwright. **No proxy.** | A green, automated `window.isSecureContext === true` on a real EC2 stack. Proves the whole approach. |
| **P1 — control plane feeds the data plane** | `sg_compute/platforms/tls/` library hardened; **Caddy** added as the actual reverse proxy for the vault UI (parent doc Option A); host-plane/`Fast_API__TLS` *manages the cert files Caddy reads* and triggers reload. Full `cert` CLI section live. | vault UI served over HTTPS through a proven proxy; we own cert lifecycle. |
| **P2 — ACME** | `Cert__ACME__Client` issues `shortlived` IP certs (parent doc B1) and/or hostname certs (B2). `cert request` / `cert renew` become real. | Publicly-trusted HTTPS, no browser warning. |
| **P3 — (optional, NOT committed)** | *If and only if* the data-plane proxy need stays modest, evaluate a FastAPI/`httpx` proxy to retire Caddy. Requires the proxy to be proven against SSE/websockets/streaming first. | Single Python sidecar — *maybe*. Explicitly deferred and conditional. |

P0 is small and self-contained. P3 is the only place "FastAPI replaces Caddy entirely"
lives — and it is **conditional and last**, precisely because §2 says the data plane is
the one job not to hand-roll on a deadline.

## 9. Decisions flagged for human sign-off

| # | Decision needed | Architect's recommendation |
|---|---|---|
| Q1 | Build the cert **control plane** in FastAPI at all, or just use Caddy's built-in ACME and skip the custom surface? | **Build it.** IP-cert tooling is immature; a clean, dogfoodable, CLI-driven control plane is a real asset and the CLI section was explicitly requested. Caddy stays the *data plane*. |
| Q2 | Public TLS surface: extend privileged `host-plane`, or a new slim unprivileged `Fast_API__TLS`? | **New slim app (Candidate B)** for anything public-facing; host-plane *may* mount the same routes but stays internal-only. Needs AppSec confirmation. |
| Q3 | Add `window.isSecureContext` + `crypto.subtle` to `JS__Expression__Allowlist`? | **Yes** — two constant, side-effect-free read expressions. Logged allowlist change, per the boundary rule. |
| Q4 | `cert` as a per-spec sub-typer (like `ami`) or a top-level `sp cert` group? | **Per-spec sub-typer first** — consistent, zero new machinery. Promote to top-level only if cross-spec demand appears. |
| Q5 | Does P0 (the PoC) go on the `claude/architect-qa-service-ijc61` branch as the next slice, or wait for the parent doc's Caddy decision first? | **PoC can start independently** — it shares no code with the Caddy choice (it *is* the alternative data plane for P0 only) and its real deliverable is the reusable `tls/` library + the dogfood check, both of which survive any later Caddy decision. |

## 10. Handoff

- **If Q1–Q5 land as recommended**, P0 is a well-bounded Dev slice: the `sg_compute/platforms/tls/` library (5 small classes), a slim `Fast_API__TLS` app + one route class, the compose/user-data wiring from the parent doc's C1–C5, a `_register_cert` sub-typer with `check` + `show` + `local`, and the allowlist addition. No application-boundary violations — `Step__Executor` is *used*, not bypassed; the single-image guarantee is untouched (the sidecar is composition, not image change).
- **The data-plane (Caddy) decision** stays with the parent doc — this doc deliberately does not re-litigate it; P0 just doesn't need it.
- **ACME (P2)** depends on the parent doc's Q4 (B1 vs B2) — keep it downstream.
- **Reality doc:** when P0 ships, record under `team/roles/librarian/reality/` (cli/ + infra/ domains): a new `sg_compute/platforms/tls/` library, a `Fast_API__TLS` sidecar, and a `cert` CLI sub-typer exist; vault-app can serve HTTPS for the secure-context check.
- **This doc + the parent** should both feed `library/reference/v{version}__decisions-log.md` once Q1–Q5 are ratified — the Historian's trail.
