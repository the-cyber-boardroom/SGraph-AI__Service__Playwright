## Status

**ACCEPTED** as the target deployment architecture for the Playwright service. The current Lambda machinery (agentic S3-zip code-loader, dual-Lambda split, ECR-specific build pipeline) is **retired** in favour of a single version-tagged Docker Hub image, matching the `diniscruz/sg-send-vault` pattern. Backwards compatibility is explicitly **not** a constraint — there are no live customers.

This review records the decision and the target. It does **not** change code. A follow-up Dev brief carries the migration.

## 1. Current reality (verified against code, not briefs)

The Playwright service today carries **three parallel deployment mechanisms** and **two Lambda functions** — significantly more machinery than the workload needs.

### 1.1 The Lambda path

- **One ECR image** — built from `sg_compute_specs/playwright/core/docker/images/sgraph_ai_service_playwright/dockerfile`. Base: `mcr.microsoft.com/playwright/python:v1.58.0-noble` + AWS Lambda Web Adapter `1.0.0`.
- **Two Lambda functions from that one image** — `sg-playwright-<stage>` ("agentic" variant) and `sg-playwright-baseline-<stage>`. They differ only in environment variables (`Lambda__Docker__SGraph_AI__Service__Playwright.py:99-159`).
- **A dynamic code loader** — `Agentic_Code_Loader.py:51-52` implements a 3-tier fallback at cold start: local path (dev) → S3 versioned zip (`s3://{account}--sgraph-ai--{region}/apps/sg-playwright/{stage}/v{x.y.z}.zip`) → passthrough (baked image code). Soft-fails on any S3 error.
- **A boot shim** — `Agentic_Boot_Shim.py` reads `/var/task/image_version`, runs the code loader, dynamically imports the FastAPI class, pins the handler. Wired by a repo-root `lambda_entry.py` that pins `PLAYWRIGHT_FAST_API_CLASS_PATH`.
- **Three deploy mechanisms** — full image→ECR→Lambda (`scripts/provision_lambdas.py`); code-only S3-zip hot-swap (`scripts/deploy_code.py` + `scripts/package_code.py`); local docker-compose (`Local__Docker__SGraph_AI__Service__Playwright.py`).

### 1.2 The vault contrast (the model to copy)

The `v2-delta` arch-plan filed 2026-05-14 reframes the whole effort as a **vault-app execution substrate**: a 4-container stack (`sg-send-vault` + `sg-playwright` + `mitmproxy` + host-control-plane) composed from **pullable Docker images**. `diniscruz/sg-send-vault:latest` is already published, multi-arch, on Docker Hub. The vault app does not have a code-loader, an S3 zip pipeline, or a dual-function split. It is one image, one tag.

The Playwright service should be the same shape.

## 2. Target architecture

```
┌─────────────────────────────────────────────────────────────┐
│  CI: build once  →  push diniscruz/sg-playwright:<version>  │
│       multi-arch (amd64 + arm64), version-tagged             │
└───────────────────────────┬─────────────────────────────────┘
                            │  docker pull
        ┌───────────────────┼───────────────────┬─────────────┐
        ▼                   ▼                   ▼             ▼
   laptop / CI        vault-app stack       host-plane     Fargate
   docker-compose     docker-compose        pods API       task
                      (4-container)         (EC2)
```

- **One image.** `diniscruz/sg-playwright:<version>`. Built from `sg_compute_specs/playwright/Dockerfile`. No LWA, no `lambda_entry.py`, no `image_version` file copy, no S3-zip extraction. Entry point: `uvicorn` serving `Fast_API__Playwright__Service`.
- **One run model.** A container. Consumed identically by docker-compose (laptop/CI/vault-app stack) and the host-plane pods API (EC2). Fargate is the same image if/when needed.
- **Versioning.** The image tag *is* the version, sourced from `sg_compute_specs/playwright/version`. Rollback = deploy the previous tag.
- **No AWS-specific deploy code.** No ECR, no Lambda, no S3. CI has exactly one deploy job: build + push to Docker Hub.

### 2.1 Target spec layout (D4)

Bring `sg_compute_specs/playwright/` onto the contract in `brief/05__stack_contract.md` — flatten the parallel `core/` tree up one level:

```
sg_compute_specs/playwright/
  manifest.py            # exists
  version                # exists — drives the image tag
  Dockerfile             # the one image; uvicorn entry, no LWA
  docker-compose.yml     # local + stack run
  service/               # Playwright__Service, Step__Executor, Browser__Launcher, …
  schemas/               # primitives, enums, schemas, collections
  dispatcher/  fast_api/  client/  metrics/  skills/
```

Both the `agentic_fastapi/` and `agentic_fastapi_aws/` packages are deleted (D7). `Fast_API__Playwright__Service` reparents directly onto `Serverless__Fast_API`. The agentic self-description surface is not carried forward here — see §5 Q2.

## 4. Deployment-target impact review

The project's historical guarantee was "one image, five targets (laptop / CI / Claude Web / Fargate / Lambda)". This review **drops Lambda** from that set. Revised guarantee: **one image, run as a container anywhere** — laptop, CI, EC2 (host-plane pods), Fargate, Claude Web sandbox. The image is *more* portable after this change, not less: removing the LWA layer and the `/var/task` Lambda conventions makes it a plain OCI container with no AWS coupling.

`.claude/CLAUDE.md` and the Architect `ROLE.md` both still assert the five-target / "one image, 5 deployment targets" guarantee. Those need a one-line amendment once Slice 4 lands — flagged here for the Librarian.

## 6. Summary

The Playwright Lambda is an experiment that predates both the SG/Compute spec model and the vault-app substrate reframe. It is the single largest source of accidental complexity in the service's deployment story — three deploy paths, two functions, a dynamic code-loader, an ECR pipeline. None of it has a current consumer that the simpler model cannot serve.

**Target:** `sg-playwright` becomes a version-tagged Docker Hub image, exactly like `sg-send-vault`, consumed by docker-compose and the host-plane pods API. Build once, pull anywhere. The migration is mostly deletion.
