# Architecture Research — TLS for the vault-app EC2 stack

**Reviewer:** Architect
**Date:** 2026-05-14
**Repo version:** `v0.2.6`
**Status:** RESEARCH — options laid out, one recommendation, decisions flagged for human sign-off before Dev picks it up.
**Revised 2026-05-14:** §2 and Option B corrected — Let's Encrypt *does* now issue
short-lived certs for bare IP addresses (GA Jan 2026). The Option A recommendation is
unchanged; the *later* path is now richer (B1 / B2).

**Trigger:** A live `sp vault-app create` produced a working stack, but the sg-send-vault
browser UI fails with:

> `Failed to create vault: Web Crypto API not available. Requires secure context (HTTPS or localhost).`

This is a *good failure* (per the CLAUDE.md convention): it surfaced a real gap early —
every exposed web port on our ephemeral stacks currently serves plain HTTP, and the
browser-side crypto the vault depends on is gated behind a [secure context](https://developer.mozilla.org/en-US/docs/Web/Security/Secure_Contexts).

---

## 1. The problem, precisely

`window.crypto.subtle` (Web Crypto API) is only exposed to **secure contexts**. A secure
context is:

- `https://` of *any* kind — **including a self-signed / untrusted cert the user has
  click-through accepted**, or
- `http://localhost` / `http://127.0.0.1` / `http://*.localhost`.

The vault-app instance is reached at `http://<public-ip>:8080` — neither. So the fix is
not "get a *trusted* cert"; the fix is "**serve over `https://` at all**". A trusted cert
removes the browser warning; an untrusted cert that the user has click-through accepted
will, in most browser flows, still expose secure-context-only APIs — but that is not a
contract to lean on blindly. The robust check is to verify `window.isSecureContext ===
true` in the target browsers (and, if it comes back false, trust the internal CA rather
than rely on a warning bypass). That distinction drives every option below.

## 2. The certificate constraint: trusted HTTPS on raw IPs is now possible — but short-lived

> **Correction (2026-05-14):** an earlier draft of this doc claimed Let's Encrypt would
> not issue certificates for bare IP addresses. That was historically true but is **no
> longer accurate** — Let's Encrypt took short-lived and IP-address certificates to
> general availability in January 2026. Section 2 is rewritten below; the correction
> brief was supplied by the user the same day.

The user asked specifically about Let's Encrypt. The current (2026 GA) position:

> **Let's Encrypt now issues publicly-trusted certificates for bare IPv4 and IPv6
> addresses** — but only as **short-lived certificates** via the `shortlived` ACME
> profile, valid ~160 hours (just over six days). Validation is `http-01` or
> `tls-alpn-01` only — **`dns-01` cannot prove control of an IP**. Renewal automation is
> therefore mandatory for any stack that outlives one cert.

So DNS is **no longer an absolute blocker** for a *trusted* cert on `https://<public-ip>`.
But IP-address certs are operationally different from the familiar 90-day domain cert:

- the ACME client must support IP identifiers **and** the `shortlived` profile (Certbot
  ≥5.3 added `--ip-address`, ≥5.4 for webroot; **Caddy `shortlived`-IP support is an open
  question** — see Q4);
- validation is `http-01` / `tls-alpn-01`, never `dns-01` — issuance needs port 80 or the
  TLS port reachable by the ACME servers during `create`, which means briefly widening the
  caller-`/32` security-group rule;
- the cert rotates every few days, so the proxy needs a reliable reload / deploy-hook path;
- our stacks are *ephemeral* — most live ~1 h, far under one cert lifetime — so for the
  common case **renewal never even fires**; the automation only earns its keep on the
  longer-lived stacks.

Our stacks today still have **no DNS**: no Route53 helper anywhere in `sg_compute*`, every
spec hands out a raw `http://<ip>:<port>` URL. That now bears on *Option B2* (hostname
certs) but no longer blocks *trusted HTTPS as such* (*Option B1*).

The work still splits cleanly into **"unblock now"** and **"trusted certs later"** — and
the recommendation is unchanged in shape: pick an option whose *now* and *later* are the
same component, just reconfigured.

## 3. Options

### Option A — Reverse-proxy sidecar with a self-signed cert *(unblocks today)*

Add a TLS-terminating reverse proxy as a **container in the compose stack**, listening on
`:443`, proxying to `sg-send-vault:8080` over the internal `vault-net`. Cert is
self-signed, generated at boot (or `tls internal` if Caddy).

- **Engine choice — Caddy.** `caddy:2.7-alpine`, zero build, a 4-line Caddyfile:
  ```
  :443 {
      tls internal
      reverse_proxy sg-send-vault:8080
  }
  ```
  `tls internal` makes Caddy mint and rotate its own cert — no `openssl` step, no cert
  files to manage. nginx works too (we have `Section__Nginx`) but needs the explicit
  `openssl` block and a cert volume; Caddy is less moving parts and we already run it in
  the VNC spec.
- **Secure context:** ✅ satisfied. Web Crypto works. Browser shows a one-time
  "not private" warning the user click-throughs.
- **DNS:** none needed. Works with raw IPs **today**.
- **Upgrade path:** the *same Caddy container* flips to real Let's Encrypt certs by
  changing `:443 { tls internal }` → `vault-xyz.<domain> { }` once a DNS story exists.
  Automatic HTTPS is Caddy's default behaviour. **Now and later are one component.**
- **Bonus:** the sidecar is the natural home for the auth gate (see §5).
- **Cost:** one more container (~15 MB image), one more published port, ~3 schema/CLI
  touch-points (§4).

### Option B1 — Let's Encrypt IP-address certificate *(trusted certs, no DNS, automation-heavy)*

Have the sidecar (Caddy, or a Certbot/ACME flow) obtain a publicly-trusted **`shortlived`
IP-address certificate** for the instance's public IPv4/IPv6, validated via `http-01` or
`tls-alpn-01`.

- **Secure context:** ✅, **and** no browser warning — fully trusted, no DNS.
- **DNS:** none required.
- **Constraints:** needs `shortlived`-profile support in the chosen ACME client, ~6-day
  renewal cadence with a reload/deploy hook on rotation, and `http-01`/`tls-alpn-01`
  reachable by the ACME servers during issuance — the SG rule must widen briefly beyond
  the caller `/32`.
- **Fit for ephemeral stacks:** good for the *short* common case (a stack living < 160 h
  never renews); the renewal machinery only matters for long-lived stacks.
- **Verdict:** viable as a **phase-1.5** follow-up *if* trusted HTTPS on raw IPs is judged
  valuable — but more moving parts than self-signed Caddy for the *immediate* Web Crypto
  unblock. Open question: confirm Caddy can drive the `shortlived` IP-cert profile (Q4).

### Option B2 — DNS automation + hostname certificate *(cleanest long-term, biggest lift)*

Build the missing DNS layer: a Route53 hosted zone (e.g. `vault.sgraph.internal` or a real
delegated domain), a `*__DNS__Helper` that creates `vault-<stack>.<zone>` → instance IP on
`create` and deletes it on `delete`, then let the sidecar issue a normal hostname cert via
Caddy automatic HTTPS.

- **Secure context:** ✅, no browser warning, **and** stable named URLs — the nicest UX.
- **DNS:** requires a brand-new shared component (`sg_compute/platforms/aws/dns/` or
  similar), a registered/delegated zone, and lifecycle wiring in **every** spec that wants
  it. Standard 90-day certs, `dns-01` available, rate-limit-aware design needed — a
  wildcard cert per zone sidesteps per-stack issuance entirely.
- **Verdict:** the correct *long-term fleet standard* if we want durable hostnames, but it
  is its own brief and must **not** block the Web Crypto fix. If we ship Option A's
  component, B2 is a reconfiguration of that component plus the DNS layer — not a rewrite.

### Option C — TLS inside the FastAPI container (uvicorn `--ssl-keyfile`)

Terminate TLS in the app process itself — `sg-send-vault` (and friends) run uvicorn with
`--ssl-keyfile/--ssl-certfile`.

- **Rejected.** It violates our boundaries and the "one image, five targets" guarantee:
  - The same image runs on laptop/CI/Lambda/Fargate where TLS is terminated *elsewhere*
    (or not wanted). Baking TLS into the app process couples the image to one deployment
    shape.
  - It does **not generalise** — the 4-container stack would need certs distributed to
    every container, and there is still no single edge to attach auth to.
  - Cert provisioning/rotation logic leaks into application code.
  - It is also **not our code to change** for `sg-send-vault` — that image is
    `diniscruz/sg-send-vault:latest`, an external dependency. We can change the
    *Playwright* FastAPI app, but the vault UI throwing the error is the external one.

  A *small* app-side change is still worth making (see §5, "defence in depth") but it is
  not the TLS-termination strategy.

### Option D — AWS-native: ALB / CloudFront + ACM certificate

Put an Application Load Balancer (or CloudFront distribution) with an ACM cert in front
of the instance.

- **Rejected for ephemeral stacks.** ACM still needs a domain for a useful cert; an ALB
  is a heavyweight, slow-to-provision, separately-billed resource per ephemeral stack;
  CloudFront adds minutes to `create`/`delete`. This is the right shape for a
  *long-lived* production service, not for `sp vault-app create` stacks that live for an
  hour. Out of scope.

### Option E — SSH/SSM port-forward to `localhost` *(stop-gap, not a fix)*

`localhost` is a secure context, so `sp vault-app connect`-style port-forwarding
(`localhost:8080` → instance `:8080`) makes Web Crypto work **with zero server changes**.

- **Useful as a documented workaround** the moment someone hits this — worth a line in
  the `info` output. But it is not "all exposed web ports use SSL"; it just avoids
  exposing the port. Keep as a fallback, not the answer.

## 4. Recommendation

> **Adopt Option A now: a Caddy reverse-proxy sidecar with `tls internal`, added to the
> vault-app compose stack. Treat Options B1 (Let's Encrypt IP cert) and B2 (DNS + hostname
> cert) as explicitly-deferred follow-ups that reconfigure the *same* Caddy container —
> B1 the lighter phase-1.5, B2 the long-term fleet standard.**

Rationale:

1. It fixes the actual bug (secure context) **today**, with **no DNS dependency**.
2. The *now* component and the *later* component are identical — Caddy's automatic HTTPS
   is one config line away from real certs. No throwaway work.
3. It matches prior art (`Vnc__Caddy__Template`) — patterns transfer, no new tech
   introduced.
4. It gives us the **single edge** to consolidate auth onto (§5) — directly answering the
   user's "could also add some more features like the auth".
5. It generalises into a shared component so the "**all exposed web ports use SSL**"
   principle becomes the default for every spec (§6).

## 5. The sidecar as the auth edge (the user's "could also add the auth")

Today the access token is enforced **per-container** via `FAST_API__AUTH__API_KEY__VALUE`
on each FastAPI app, and the browser cookie is `x-sgraph-access-token`. The Caddy sidecar
is the natural place to **consolidate** this:

- **Minimum (ship with Option A):** Caddy just terminates TLS and proxies. Auth stays
  where it is. Zero auth risk added.
- **Phase 1.5 (cheap win):** Caddy enforces the access token at the edge — a
  `@authed` matcher checking the `X-API-Key` header *or* the `x-sgraph-access-token`
  cookie, returning `401` before the request ever reaches a container. Containers keep
  their own check as defence-in-depth.
- **Phase 2 (the VNC pattern):** swap `caddy:2.7-alpine` for the custom
  `caddy-security` build (already proven in `Vnc__Caddy__Template`) — a real login
  portal, JWT cookies, a `/auth/set-cookie-form` equivalent served by the proxy itself.
  This would also let us **retire the "paste this `javascript:` snippet" workaround** we
  added to `sp vault-app info` this session — the portal sets the cookie properly.

**Recommendation:** ship Option A with **TLS-only Caddy** first (smallest safe change),
then file a *separate* brief for the auth-edge work. Do not bundle them — the TLS fix is
urgent and low-risk; the auth consolidation touches the security boundary and wants its
own review + AppSec eyes.

**Small app-side change still worth doing (defence in depth, not the strategy):** the
*Playwright* FastAPI app (the one in *this* repo) should detect a non-secure context far
more gracefully than an opaque "Web Crypto API not available" — but note the error in the
live report came from `sg-send-vault`, an **external image** we don't control. File that
as a courtesy issue upstream; it is not in our remit to fix here.

## 6. The general principle: "all exposed web ports use SSL"

The user's framing — *"we really should be in a position where all exposed web ports use
SSL"* — is an architectural standard, not a vault-app feature. The repo is **inconsistent
today**:

| Spec | Exposed scheme | TLS today |
|---|---|---|
| elastic | https:443 | self-signed (nginx) |
| opensearch | https:443 | yes |
| vnc | https:443 | self-signed (Caddy + auth) |
| **vault-app** | **http:8080** | **none** ← the gap |
| prometheus | http:9090 | none |
| ollama | http:11434 | none |
| local-claude | http:8000 | none |

**Architectural proposal (for ratification):**

1. Promote the Caddy-sidecar pattern into a **shared, reusable component** —
   `sg_compute/platforms/ec2/user_data/Section__Caddy.py` (sibling to the existing
   `Section__Nginx`), parameterised by `upstream_host:port`. vault-app is its first
   consumer; prometheus / ollama / local-claude adopt it next.
2. Establish the standard: **no spec publishes a plain-HTTP port to the world.** Either it
   sits behind the Caddy section on `:443`, or it stays internal and is reached via SSM
   port-forward. New specs inherit this by default.
3. `Section__Nginx` and the bespoke elastic nginx block become **candidates to migrate**
   onto `Section__Caddy` for consistency — lower priority, tracked, not urgent.

This is the path where the vault-app fix pays for itself across the whole fleet.
