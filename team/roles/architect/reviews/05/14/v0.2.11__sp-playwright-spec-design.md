1. Goal
═══════════════════════════════════════════════════════════════════════════════

Make `sp playwright` a first-class spec CLI so a caller can spin up an
ephemeral, isolated instance of the Playwright FastAPI service on demand,
use it, and throw it away — the same shape as `sp vnc`, `sp docker`,
`sp elastic` today.

"Ephemeral" = no long-lived shared service. Each caller gets their own
instance, auto-terminating, billed only for the minutes it ran.

The thing being launched is just a container: `diniscruz/sg-playwright`.
Everything in this doc is plumbing around that one immutable image.

2. Two ephemeral models — pick per call, not per design
═══════════════════════════════════════════════════════════════════════════════

The spec supports BOTH. They share schemas and the CLI surface; only the
launch backend differs.

2.1 Pod-on-host  (fast, cheap, shared host)
  - The container runs as a *pod* on an already-running host via the
    host-plane pods API (`POST /pods` on `Fast_API__Host__Control`).
  - Boot time: seconds (image already pulled, or one pull then cached).
  - Use when: you have a host EC2 up (e.g. the vault-app stack box) and
    want N short-lived Playwright instances on it cheaply.
  - Isolation: process/container-level, shared kernel + host.
  - This is the default for "give me a Playwright API for 10 minutes".

2.2 EC2-node  (slow, fully isolated, dedicated box)
  - The container runs on its own dedicated EC2 instance, launched via
    the standard SG/Compute node path (`Spec__Service.create_node()` →
    `EC2__Platform` → user-data pulls + runs the image).
  - Boot time: ~30-60 s from a baked AMI, 3-10 min cold.
  - Use when: you need hard isolation, a public Function-URL-equivalent,
    or a box that outlives any single host.
  - Carries the standard auto-terminate timer (`max_hours`, default 1).

Architect note: model 2.1 is the one that makes "ephemeral versions of
this Playwright FastAPI" cheap and instant. Model 2.2 is the fallback for
isolation-sensitive callers. Build 2.1 first.

    with_mitmproxy   : bool                      = False      # True → launch a paired mitmproxy sidecar + wire playwright through it

4.3 `Playwright__Stack__Service`  (the launcher — mandatory methods)

    def setup(self) -> 'Playwright__Stack__Service':
        # wire host-plane client + EC2 platform; return self

    def create_stack(self, request, creator='') -> ...__Create__Response:
        # backend == POD:
        #   build Schema__Pod__Start__Request via Playwright__Pod__Spec:
        #     image = 'diniscruz/sg-playwright:' + request.image_tag
        #     ports = { '8000/tcp': '<allocated>' }
        #     env   = { 'FAST_API__AUTH__API_KEY__VALUE': request.api_key,
        #               'SG_PLAYWRIGHT__DEFAULT_PROXY_URL': request.proxy_url }
        #     type_id = 'playwright'
        #   POST it to the host-plane pods API on request.host_node
        # backend == EC2_NODE:
        #   render user-data via Playwright__User_Data__Builder
        #   (docker run diniscruz/sg-playwright + auto-terminate timer)
        #   call create_node() through EC2__Platform
        # emit 'playwright:stack.created'; return the response

    def list_stacks(self, region)                  -> ...__List
    def get_stack_info(self, region, stack_name)   -> ...__Info | None
    def delete_stack(self, region, stack_name)     -> ...__Delete__Response
    def health(self, region, stack_name)           -> Schema__Health
        # HTTP GET <stack-url>/health/status through Health__HTTP__Probe

  POD-path lifecycle delegates to the host-plane API
  (`/pods`, `/pods/{name}`, `/pods/{name}/stop`, DELETE `/pods/{name}`) —
  it does NOT shell out to docker. Same reasoning as the vault-app v2
  plan: keeps the orchestration reusable on podman / k8s / Fargate.

4.4 `Routes__Playwright__Stack`  (logic-free — pure delegation)

  POST   /playwright/stack            → service.create_stack()
  GET    /playwright/stacks           → service.list_stacks()
  GET    /playwright/stack/{name}     → service.get_stack_info()
  DELETE /playwright/stack/{name}     → service.delete_stack()
  GET    /playwright/stack/{name}/health → service.health()

  Every route returns `.json()` on a Type_Safe schema. No raw dicts.

5. The user experience
═══════════════════════════════════════════════════════════════════════════════

Once `'playwright'` is in `PLUGIN_FOLDERS`, the CLI exposes it the same way
as every other spec — both as HTTP routes (above) and as typer commands:

    sp playwright create  --host-node vault-app-box --image-tag latest
                          → starts a pod, prints the pod URL + api-key header
    sp playwright list                  → Rich table of live Playwright stacks
    sp playwright info  happy-turing     → panel: url, backend, age, health
    sp playwright health happy-turing    → polls /health/status until ready
    sp playwright delete happy-turing    → stops + removes the pod

  EC2-node variant:
    sp playwright create --backend ec2-node --region eu-west-2 --max-hours 2

  With a paired mitmproxy capturing/intercepting proxy (see §9):
    sp playwright create --host-node vault-app-box --with-mitmproxy
    sp playwright create --host-node vault-app-box --with-mitmproxy \
                         --intercept-script vault://scripts/strip-analytics.py

Caller then talks to the returned URL directly — it IS the Playwright
FastAPI (the 25-endpoint surface: /sequence/execute, /browser/*, /screenshot,
…), API-key gated by the value passed at create time. When `--with-mitmproxy`
is set, all browser traffic from that instance flows through the paired
mitmproxy and is captured / intercepted per the addon script.

8. Open questions
═══════════════════════════════════════════════════════════════════════════════

Q1. Port allocation on the POD path — who picks the host port? Architect
    lean: host-plane allocates and returns it (the pods API already owns
    the host's port space); the spec service just asks for "a port".

Q2. How does the caller reach a POD-backend instance from outside the
    host? Via SSM port-forward to the host, or does the host publish a
    port range? Architect lean: SSM port-forward by default (matches the
    vault-app stack's "one external port" posture); a published range is
    an opt-in.

Q3. Does `sp playwright` need its own vault writes (`VAULT_WRITES`
    capability is in the spec manifest)? Architect lean: the running app
    already stages artefacts via the `VAULT` sink; the stack launcher
    itself writes nothing. Keep the launcher vault-free.

Q4. Intercept-script source — vault path (`vault://…`), a local file
    uploaded at create time, or a repo-baked default? Architect lean:
    support all three with precedence (explicit `--intercept-script` >
    image-baked default); the vault path is the primary path for
    user-authored scripts. See §9.4.

9. `--with-mitmproxy` — paired capturing / intercepting proxy
═══════════════════════════════════════════════════════════════════════════════

Bring back the OSBot-Playwright capability: an `sp playwright` instance can
be launched with a paired mitmproxy that captures (and optionally rewrites)
all browser traffic, driven by a user-defined intercept/addon script.

9.1 What the flag does
  `--with-mitmproxy` on `sp playwright create` makes the launcher provision
  TWO containers instead of one, wired together:

    ┌─────────────────────────┐      ┌──────────────────────────┐
    │ sg-playwright            │ ───► │ mitmproxy                 │ ───► internet
    │ diniscruz/sg-playwright  │ HTTP │ sg_compute_specs/mitmproxy │
    │ env:                     │ proxy│ - intercept/addon script   │
    │  SG_PLAYWRIGHT__DEFAULT_  │      │ - NDJSON capture to stdout │
    │   PROXY_URL=http://<mitm> │      │ - CA cert served on /ca    │
    │  SG_PLAYWRIGHT__IGNORE_   │      │   (Routes__CA, existing)   │
    │   HTTPS_ERRORS=1          │      └──────────────────────────┘
    └─────────────────────────┘

  Without the flag: one container, no proxy — unchanged from §2.

9.2 No new runtime code in the playwright service
  The service ALREADY supports this. `Browser__Launcher` reads
  `SG_PLAYWRIGHT__DEFAULT_PROXY_URL` at browser-launch and applies
  `proxy={'server': url}`, and reads `SG_PLAYWRIGHT__IGNORE_HTTPS_ERRORS`
  to pass `ignore_https_errors=True` (so Chromium accepts mitmproxy's
  forged certs). `--with-mitmproxy` is purely a LAUNCHER concern — it sets
  those two env vars on the playwright container. The cert-trust story is
  `IGNORE_HTTPS_ERRORS=1`; if a future caller needs real cert pinning
  instead, the mitmproxy spec's `/ca` route already serves the CA PEM for
  injection into the browser's trust store — left as a follow-up, not
  needed for day one.

9.3 What the launcher does — `Playwright__Mitmproxy__Companion`
  POD backend:
    1. Start the mitmproxy pod first via the host-plane pods API:
       image   = the mitmproxy spec's image, env carries the addon-script
                 reference, NO external port (docker-network only).
    2. Start the sg-playwright pod with
         SG_PLAYWRIGHT__DEFAULT_PROXY_URL = http://<mitmproxy-pod>:8080
         SG_PLAYWRIGHT__IGNORE_HTTPS_ERRORS = 1
    3. Both pods carry the same `stack_name` + `type_id='playwright'` tag
       so `list` / `info` / `delete` treat them as one stack.
  EC2_NODE backend:
    The `Playwright__User_Data__Builder` emits a 2-service docker-compose
    (playwright + mitmproxy on a shared docker network) — the same shape
    `scripts/provision_ec2.py` produces today, lifted into the spec.

  Delete tears down BOTH pods/containers.

9.4 Intercept / addon script
  - `--intercept-script <ref>` points at the mitmproxy addon script.
    Precedence: explicit ref > image-baked default (a no-op passthrough
    that logs NDJSON to stdout, matching the vault-app v2 plan).
  - Primary source is a vault path (`vault://scripts/<name>.py`) so scripts
    are user-authored and versioned, not baked into an image. The mitmproxy
    spec's interceptor registry is the loading mechanism — reuse it; do not
    invent a new one.
  - This is the "like we had in the past" piece: the OSBot-Playwright
    intercept-script workflow, now driven by a CLI flag + vault-resident
    script instead of a hand-edited container.

9.5 Boundaries preserved
  - `Step__Executor` remains the only `page.*` caller — the proxy is set at
    launch by `Browser__Launcher`, not per-step.
  - mitmproxy stays passive-by-default (capture + log); rewriting happens
    only when the addon script does it. No admin API exposed on the
    sidecar (consistent with the vault-app v2-delta plan §8).
  - The proxy is internal to the stack's docker network — never a
    published port.
