# Architect Review — Vault Sync architecture thread

## Trigger

Human asked the Architect to review the 13 May 2026 dev brief on the "Vault Synchronizer Tool" and decide whether to build it as a separate repo or fold it into existing specs. Thread occurred 2026-05-16.

## Decisions reached

| # | Question | Decision | Rationale |
|---|---|---|---|
| D1 | Repo location | New spec under `sg_compute_specs/vault_sync/` (same repo as `sg_compute_specs`); contract spec lives in *this* repo's `library/docs/specs/` | Reuses `Spec__CLI__Builder`; aligns with the third-agent-CLI pattern; EC2-resident workload belongs in SG/Compute's lifecycle |
| D2 | Canonical vault model | None — vault-sync is peer-to-peer between SG/API instances. Its own state (mappings) lives in a dedicated "vault-sync vault" | Per human: "the job of this tool is to sync vaults between multiple instances" |
| D3 | Transport | `sgit` exclusively, as a CLI sub-process | Most mature SG/API client; no new wire protocol invented |
| D4 | SG/Compute integration | CLI sub-process for v1; REST API later | Lower coupling; matches what's stable today |
| D5 | Authorization | Possession of `SG_VAULT_SYNC__KEY` env var. No allowlist layer | Admin-only operation; allowlist over-engineered for current trust model |
| D6 | Conflict policy | Fast-forward-or-fail. No auto-merge. Both vaults untouched on failure | Per human: "fail fast and leave each vault as before" |
| D7 | Webhook receiver | Defer to v2. v1 is CLI-only | Per human: "leave the web hook part for later, lets first do this manually and via the cli commands" |
| D8 | Per-target credential storage | Inside the vault-sync vault for v1 | Per human: "for now we need to store them there"; concentration risk acknowledged, hardening is v2 |
| D9 | Sync vs. async | Default sync. Async only for idempotent fan-out (`push --all`) and future webhook-triggered pulls | Per human: "most actions should be sync; only async ones would be idempotent triggered actions like push and pull" |
| D10 | Controller framework | `Fast_API__Vault__Sync` using `Serverless__Fast_API` + `X-API-Key`, same pattern as Playwright service | Per human: "Fast_API__Vault__Sync will Serverless__Fast_API features and the protection of the AUTH key" |
| D11 | Deploy / viewer scope | **Removed entirely.** vault-sync does not spin up EC2, does not deploy viewers | Per human: "that's legacy and should be removed; this service doesn't really deploy anything (it just syncs vaults, i.e. sgit push and pull)" |

## Brief items dropped from scope

This review records what was cut from the source brief, so the trail is auditable:

- `vault-sync deploy ephemeral <vault-id>` verb — gone
- `vault-sync deploy persistent <vault-id> --domain <name>` verb — gone
- "Scenario 2: vault → fresh deploy → live URL" — gone
- Acceptance criteria #2 (deploy-ephemeral works), #7 (tool runs on EC2 as a deployable spec for itself was kept; the *deploys* are gone), #9 (scenario 2 verified) — replaced
- Hooks system (Phase 4 in brief) — deferred to v2
- Visualisation UI (Phase 3 in brief) — deferred; v1 emits JSON only

## Brief items kept

- CLI-first architecture
- `push` / `pull` / `sync` / `status` verbs
- Vault graph as a JSON-emitting concept
- Long-lived FastAPI controller (the brief hinted at this; we made it explicit)
- Mappings / target registry
- "One repo, multiple specs" structure (chose `sg_compute_specs` as the home)

## Risks acknowledged

| Risk | Severity | Mitigation |
|---|---|---|
| Concentration of trust in `SG_VAULT_SYNC__KEY` | High | Documented as known; v2 hardening tracked. Operators must treat this key as crown-jewel. |
| Per-target credentials in same vault as mappings | Medium | Documented; v2 will isolate per-target credentials into separate vault keys |
| Sgit-only transport — single point of dependency | Low | SGit team is reachable for clarifications; spec defers wire details to them |
| Async fan-out push has no durable retry on controller crash | Low for v1 | Re-firing converges (idempotent); operator can re-trigger manually. Durable queue is v2. |
| No audit log of who triggered what sync | Medium | Tracked as v2 hardening item. |

## Good failure / bad failure

Per CLAUDE.md rule 27, classifying the failures surfaced during the thread:

- **Good failure (caught early):** the brief's `deploy` verb conflated two responsibilities (sync + provisioning). Human caught it during the review thread, not after code was written. Saved a phase of misdirected work.
- **Good failure (caught early):** the architect's first instinct was to put a folder-as-queue inside the mappings vault for async ops. Human's "most actions should be sync" steer killed that prematurely-clever design before it landed.
- **No bad failures** in this thread.
