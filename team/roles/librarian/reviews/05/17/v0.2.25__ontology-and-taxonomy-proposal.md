# Librarian — Ontology & Taxonomy Proposal for the SG Playwright Repo

> **PROPOSAL — does not exist yet.** This document does not change the repo layout. It captures findings, proposes an ontology + taxonomy, and lists concrete, phased moves for human ratification. The Librarian ROLE explicitly forbids reorganising the repo without confirmation (`ROLE.md` → "Never reorganise the repo without confirmation").

---

## 0. Executive Summary

The Playwright repo's knowledge base has grown to **683 markdown files** and **2,169 Python files** across two parallel doc trees (`library/` and `team/`). Three structural pressures now show up consistently:

1. **Reality fragmentation** — the reality domain tree has *three* coexisting layouts (flat `v0.1.X__what-exists-today.md` snapshots, the `v0.1.31/` numbered-parts archive, and the new per-domain tree). Only 1 of 11 domains has fully migrated. `sg-compute/index.md` is already 545 lines.
2. **Brief sprawl** — "briefs" exist in **four** semantically-overlapping homes: `team/comms/briefs/` (170 files), `team/humans/dinis_cruz/briefs/` (~60), `library/briefing/` (9), `sg_compute/brief/` (2). The naming `briefing/` vs `briefs/` is non-obvious.
3. **Monolith leakage** — 48 markdown files exceed 300 lines (5 over 1,000); 16 Python files exceed 500 LOC (4 over 1,000). Critical reference docs (`v0.20.55__*-catalogue-v2.md`, 1,200–1,500 lines each) use Python-style `═══` H1 banners that render as walls of H1s on GitHub — violating `library/guides/v0.2.15__markdown_doc_style.md`.

The two sibling repos (`SGraph-AI__App__Send`, `SGraph-AI__Tools`) have already faced this and converged on a small set of patterns we can adopt directly — most importantly a **fractal domain tree** with a **pointer-log changelog**, a **diff-aware versioned catalogue**, and **stable cross-doc IDs** (`INC-NNN`, `M-NNN`, `B-NNN`).

This proposal:

- Defines an **ontology** — the seven kinds of knowledge that live in this repo, and where each kind belongs.
- Defines a **taxonomy** — placement, naming, and growth rules that flow from the ontology.
- Lists **patterns to steal** from `sgai-send` and `sgai-tools`.
- Provides a **phased migration plan** (M-001 … M-007) — none of which is destructive in phase 1.

## 2. Proposed Ontology — Seven Kinds of Knowledge

An ontology answers: *what KINDS of things live here?* The repo currently mixes them. Proposing seven, with explicit definitions and a single canonical home for each:

| # | Kind | Definition | Canonical home | Authority |
|---:|---|---|---|---|
| 1 | **REALITY** | What demonstrably exists today, verified against committed code. The truth ledger. | `team/roles/librarian/reality/` | Canonical. If not here, it doesn't exist. |
| 2 | **CATALOGUE** | A navigable map of REALITY — fractal, "what is where" indexes. | `library/catalogue/` | Derived from REALITY; refreshed each session. |
| 3 | **SPEC** | A contract for something — schema, route, naming convention. May describe shipped or proposed behaviour. | `library/docs/specs/` | Aspiration until REALITY confirms. |
| 4 | **GUIDE** | A non-negotiable pattern, style rule, or how-to. | `library/guides/` | Authoritative once adopted. |
| 5 | **REFERENCE** | Append-only historical context — architecture briefs, decisions log, debrief index, phase summaries. | `library/reference/` + `team/claude/debriefs/` | Immutable history. |
| 6 | **WORK** | In-progress collaboration — briefs (cross-role), plans (multi-step), reviews (per-role), changelog entries. Time-bucketed. | `team/comms/` + `team/roles/{role}/reviews/` | Ephemeral; deliverables get promoted. |
| 7 | **SCRATCH** | Agent and human working space; not part of the formal graph. | `team/humans/dinis_cruz/{briefs,claude-code-web}/` | Read-only for the Librarian unless promoted. |

The three failure modes the current repo shows are all ontology violations:

- **REALITY ↔ WORK confusion** — `team/comms/briefs/v0.2.6__qa-vault-app/01__what-exists-today.md` borrows REALITY's filename.
- **SPEC ↔ CATALOGUE confusion** — `v0.20.55__schema-catalogue-v2.md` lives in `specs/` despite being a CATALOGUE (1,439 lines proves it).
- **WORK ↔ SCRATCH confusion** — the 1,943-line architect plan is promotion-grade WORK parked in SCRATCH.

### 2.1 Knowledge lifecycle

Every node moves through this pipeline:

```
SCRATCH (human or agent draft)
   ↓ promote
WORK (cross-role brief / plan / review)
   ↓ ratify (Architect / Dev sign-off)
SPEC (contract) ───────────────→ REFERENCE (decision log / phase debrief)
   ↓ implement (Dev)
REALITY (what exists)
   ↓ index
CATALOGUE (where to find it)
```

GUIDE sits orthogonal — it documents *how* nodes are produced (Markdown style, testing patterns, naming).

## 3. Proposed Taxonomy — Placement, Naming, Growth Rules

A taxonomy answers: *given a thing of this KIND, where does it go and how is it named?*

### 3.1 Universal naming rule

Every formal Markdown deliverable carries a version prefix matching the project version file at the time of authorship:

```
v{X.Y.Z}__{kind}__{slug}.md
```

| Token | Source | Example |
|---|---|---|
| `v{X.Y.Z}` | `version` (root, currently `v0.2.25`) | `v0.2.25` |
| `{kind}` | One of: `reality`, `catalogue`, `spec`, `guide`, `reference`, `brief`, `plan`, `review`, `debrief`, `master-index`, `health-scan`, `ontology`, `findings` | `master-index` |
| `{slug}` | Kebab-case noun phrase | `vault-publish-spec` |

Date-bucketed WORK uses `team/{owner}/reviews/MM/DD/v{X.Y.Z}__{kind}__{slug}.md` (matches `sgai-send`).

**One project, one version file.** Today the repo has three: `version` (v0.2.25), `sg_compute/version` (v0.1.162), `sg_compute_specs_playwright/version` (v0.2.12). CLAUDE.md tells agents to read `sgraph_ai_service_playwright/version` — which **does not exist**. This needs Architect/Dev consensus; the Librarian proposes the root `version` (v0.2.25) as the canonical Librarian version, with sub-package versions used only for sub-package internal naming.

### 3.2 Fractal-growth rule (single hard threshold)

Adopt `sgai-send`'s rule unchanged: **when any Markdown file exceeds ~300 lines, split it.** Today `sg-compute/index.md` (545) and 47 other files breach this. Apply the rule retroactively to anything authored under `team/roles/librarian/`; flag (don't auto-split) for GUIDE/SPEC files owned by other roles.

The split strategies are not mutually exclusive — pick by file type:

| File type | Split strategy |
|---|---|
| Reality domain index | Per-area sub-files under same domain dir (e.g. `sg-compute/cli.md`, `sg-compute/services.md`) — fractal, like `sgai-send`. |
| Single large doc with chronological growth (e.g. per-spec rounds) | Per-subject `rounds.md`, headings = rounds — like `sgai-tools/tools/{slug}/rounds.md`. |
| Onboarding sequence | Numbered siblings under same parent, README as cover sheet — already used in `library/briefing/`. |
| Catalogue | Diff-aware versioned shards (`v{ver}__catalogue__{slice}.md`) — like `sgai-tools/library/catalogue/`. |

### 3.3 Stable cross-doc IDs

Adopt the `sgai-send` convention. The Librarian mints IDs and maintains a registry under `team/roles/librarian/ids/`:

| Prefix | Owns | Example |
|---|---|---|
| `M-NNN` | Migration backlog items (replaces today's `B-001`-style entries in `DAILY_RUN.md`) | `M-001__reality-tree-completion` |
| `INC-NNN` | Incidents (post-mortems, hot fixes that triggered process change) | `INC-001__sg-name-prefix-block` |
| `B-NNN` | Generic backlog items in `DAILY_RUN.md` | `B-007__broken-link-scan` |
| `D-NNNN` | Debriefs (chronological — already exists implicitly via date) | optional, low priority |

IDs survive file moves and renames; documents reference each other by ID, not by path.

## 4. Patterns to Steal — from `sgai-send` and `sgai-tools`

Five patterns are directly applicable and low-cost. Listed by impact:

### 4.1 Pointer-log changelog (`sgai-send`)

Already partly implemented in `team/roles/librarian/reality/changelog.md`. Tighten the format to one-line-per-change with a hard pointer to the master-index file that holds the detail:

```
2026-05-17 | sg-compute/index.md           | vault-publish spec landed (P1b..P2d) — see reviews/05/17/v0.2.25__master-index__vault-publish.md
2026-05-16 | sg-compute/index.md, cli/...  | sg aws billing MVP — see ...
```

Already 143 lines today; rewrite is small.

### 4.2 "Verified by" rolling paragraph (`sgai-tools`)

A single `reality/verified-by.md` file with one ever-growing paragraph: "Reality verified through {date} via {method}: {commit-range}." Cheap append-only signal of direction-of-travel for any reader.

### 4.3 `proposed/` sub-folders per domain (`sgai-send`)

Already started in `host-control/proposed/index.md`. Roll out to the other 10 domains as part of the migration (M-001 below). Spin out individual proposals to their own files when they warrant design (`proposed/{topic}.md`).

### 4.4 Diff-aware catalogue shards (`sgai-tools`)

See §3.6 above. The index file in `library/catalogue/` should show "what changed since the last shard version" — instant orientation for someone who last touched the repo a week ago.

### 4.5 Numbered-parts split with cover sheet (`sgai-tools`)

For files that genuinely belong as one document but are too long, use `{base}.md` (cover sheet) + `{base}__1__{section}.md`, `{base}__2__{section}.md`, etc. Lighter than full per-topic spin-out. **Recommended target:** `v0.20.55__schema-catalogue-v2.md` (1,439 lines) → cover sheet + 4 parts.

### 4.6 Things NOT to steal

- **`team/explorer/{role}/` path** (sgai-tools) — we already migrated to `team/roles/{role}/`. Don't reintroduce.
- **15 numbered reality shards in one folder** (sgai-tools `v0.1.31/`) — our 11-domain tree is cleaner; complete that migration instead of going back to numbered shards.
