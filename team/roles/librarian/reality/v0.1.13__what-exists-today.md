**Changes since v0.1.12 (milestone — "100% clean state per call"):**
- Version file bumped `v0.1.12 → v0.1.13`.
- **`Browser__Launcher` refactored to fresh-per-call `sync_playwright` + Chromium** (commits `8922484`, `cdba60c`, `22dd98a`, `6869632`, `906ba05`). No singleton runtime. Each `launch()` starts a new Node subprocess (~420 ms) + Chromium (~130 ms) and returns `Schema__Browser__Launch__Result(browser, playwright, playwright_start_ms, browser_launch_ms)`. `register(session_id, result)` stashes both handles; `stop(session_id)` closes the browser **and** stops the sync_playwright subprocess together. The launcher registry is empty after every stateless request — verified by a 20-sequential-calls reliability test.
- **`Sequence__Runner.execute()` wraps the step loop in try/finally** so `browser_launcher.stop(session_id)` runs even when a step raises. Close-after teardown duration is returned as part of the timings block.
- **Per-phase timing breakdown surfaced on every sequence response** — new `Schema__Sequence__Timings` (`playwright_start_ms`, `browser_launch_ms`, `steps_ms`, `browser_close_ms`, `total_ms`). JSON field `timings` on `Schema__Sequence__Response` and `Schema__Quick__Html__Response`. `/quick/screenshot` returns raw PNG bytes, so the same fields are emitted as HTTP headers: `X-Playwright-Start-Ms`, `X-Browser-Launch-Ms`, `X-Steps-Ms`, `X-Browser-Close-Ms`, `X-Total-Ms`.
- **`/quick/html` + `/quick/screenshot` endpoints live** under `Routes__Quick` (tag=`quick`). Flat request/response schemas (`Schema__Quick__Html__Request/Response`, `Schema__Quick__Screenshot__Request`) — designed so Swagger UI shows a minimal example body (`url` + one or two optional fields) rather than the full Sequence request tree. Each call spins up a throwaway session, runs a tiny internal sequence, and tears the session down.
- **Payload-size primitives lifted past the `Safe_Str__Text__Dangerous` 64 KB default:**
  - `Safe_Str__Page__Content` — 10 MB cap on HTML returned by `GET_CONTENT` / `/quick/html` (real pages range 100 KB – 5 MB).
  - `Safe_Str__Artefact__Inline` — 20 MB cap on inline artefact base64 payloads (typical PNG ≈ 100 KB).
- **New schemas:** `schemas/browser/Schema__Browser__Launch__Result`, `schemas/sequence/Schema__Sequence__Timings`, `schemas/quick/Schema__Quick__Screenshot__Result` (internal-only — `/quick/screenshot` still returns raw `image/png`; the schema pairs `png_bytes` with `timings` so the route can emit headers alongside the body).
- **Error surface hardened:** `quick_html` / `quick_screenshot` wrap the service call in `try/except` that emits `HTTPException(502, last-1800-chars-of-traceback)` because `osbot-fast-api`'s `Type_Safe__Route__Wrapper` squashes every exception to `"{Type}: {msg}"` without a traceback — this was slowing debug iteration to one deploy per guess.
- **Production-verified** behind CloudFront (`https://dev.playwright.sgraph.ai`): three back-to-back `/quick/screenshot` runs came back at 1164 ms / 1039 ms / 800 ms total, all with `X-Playwright-Start-Ms ≈ 420`, `X-Browser-Launch-Ms ≈ 130`, and a consistent ~550 ms fixed clean-state overhead.

### Service classes (12 of 12)
- `service/JS__Expression__Allowlist` — deny-all by default; exact-match gate for the EVALUATE step.
- `service/Request__Validator` — single home for cross-schema validation (spec §9). Methods: `validate_session_create`, `validate_step`, `validate_sink_configs`, `validate_step_ids_unique`, `validate_sequence_timeout`, `reject`. Raises `HTTPException(422)` carrying `Schema__Error__Response`.
- `service/Capability__Detector` — env-driven deployment target detection + per-target `Schema__Service__Capabilities` profiles (lambda, claude_web, laptop, ci, container). Builds `Schema__Service__Info` for `/info` and `Schema__Health__Check` for `/health`.
- `service/Session__Manager` — in-memory session lifecycle (create/get/close, capture config, start time, artefact accumulation, action counters, active-session listing, healthcheck). No Playwright calls; Browser objects are held as opaque refs.
- `service/Sequence__Dispatcher` — parse surface (`parse_single_step`, `parse_steps`) wrapping `dispatcher/step_schema_registry.parse_step`. `execute_step` deferred to Phase 2.5 (needs `Step__Executor`).
- `service/Artefact__Writer` — sink routing for INLINE (base64), VAULT, S3, LOCAL_FILE. Assembles `Schema__Artefact__Ref` with size + 10-hex sha256 identity hash. `write_bytes_to_local` fully implemented; `write_bytes_to_vault` / `write_bytes_to_s3` / `read_from_vault` / `write_to_vault` are subclass-overridable seams (real vault + osbot-aws clients not yet wired). Typed capture wrappers `capture_screenshot(data, sink_config)` / `capture_page_content(...)` / `capture_pdf(...)` (Phase 2.9) pin the artefact type so Step__Executor callers don't repeat it; each returns `None` when the sink_config is disabled.
- `service/Credentials__Loader` — vault ↔ Playwright `BrowserContext` glue. `apply(session_id, session_manager, credentials)` loads cookies / storage state via the writer's vault seams and sets `extra_http_headers`. `save_state_to_vault` persists state back. No direct vault client; no `page.*` calls (context-level only).
- `service/Browser__Launcher` — real Chromium process lifecycle, **fresh-per-call as of v0.1.13**. `launch(browser_config)` starts a brand-new `sync_playwright()` Node subprocess AND launches a fresh Chromium — no singleton runtime, zero cross-request state bleed (proxies, cookies, stuck handles, thread affinity). Returns `Schema__Browser__Launch__Result(browser, playwright, playwright_start_ms, browser_launch_ms)` so the runner can surface per-phase timings. `register(session_id, result)` stashes **both** handles together; `stop(session_id)` closes the Browser and stops the Node subprocess in one call, returning `browser_close_ms` (`Safe_UInt__Milliseconds`) — double-stop is idempotent. `stop_all()` loops `stop` over every session; healthcheck reports `active_browsers={len(self.browsers)}` with no singleton flag. `ENV_VAR__CHROMIUM_EXECUTABLE` escape hatch for pre-installed Chromium builds. Default launch args per spec §5.2: `--no-sandbox --disable-gpu --disable-dev-shm-usage --single-process --use-mock-keychain`. Only LOCAL_SUBPROCESS + CHROMIUM supported today (CDP_CONNECT, BROWSERLESS, Firefox, WebKit raise `NotImplementedError`). The only non-`Step__Executor` class permitted to import `playwright.sync_api`.
- `service/Step__Executor` — **only class (alongside `Browser__Launcher`) permitted to import `playwright.sync_api`**. `execute(page, step, step_index, capture_config)` dispatches on `step.action`. First-pass handlers (Phase 2.9): `execute_navigate` / `execute_click` / `execute_fill` / `execute_screenshot` / `execute_get_content` / `execute_get_url`. Screenshot and GET_CONTENT bytes are handed to `artefact_writer.capture_*()` — Step__Executor never writes to a sink directly. Every handler catches `Exception`, times the step, and returns a uniformly-shaped `Schema__Step__Result__*` (PASSED or FAILED with error_message). Deferred handlers (`execute_press / select / hover / scroll / wait_for / video_start / video_stop / evaluate / dispatch_event / set_viewport`, 10 total) raise `NotImplementedError` with a `Phase 2.11` marker rather than falling through to AttributeError. Helpers: `passed_result`, `failed_result`, `resolve_id` (falls back to `Step_Id(str(step_index))`), `filter_refs` (drops None refs from disabled sinks), `now_ms` (wall-clock seam for tests).
- `service/Playwright__Service` — thin orchestrator. Composes Capability__Detector + Session__Manager + Browser__Launcher + Credentials__Loader + Artefact__Writer + Request__Validator + **Action__Runner** + **Sequence__Runner**. `setup()` primes detection (idempotent) and shares Session__Manager + Capability__Detector + Request__Validator (plus Browser__Launcher + Credentials__Loader + Sequence__Dispatcher + Step__Executor for the sequence runner) into both runner instances so they see live service state instead of default-constructed copies. Health methods: `get_service_info()`, `get_capabilities()`, `get_health()`. Session methods (Phase 2.10 Slice A): `session_create()` (uses `launch_result.browser`; calls `browser_launcher.register(session_id, launch_result)` after launch) / `session_list()` / `session_get()` / `session_save_state()` (None → 404) / `session_close()` (None → 404) / `generate_trace_id()`. Action method (Phase 2.10 Slice B subset): `execute_action(request)` delegates to `Action__Runner.execute()`. Sequence method (Phase 2.10 Slice C): `execute_sequence(request)` delegates to `Sequence__Runner.execute()`. **Quick methods (v0.1.13):** `quick_html(request: Schema__Quick__Html__Request) -> Schema__Quick__Html__Response` (spins up a throwaway session, runs a navigate + optional click + `GET_CONTENT` sequence, tears the session down, returns flat url/final_url/html/duration_ms + `timings`) and `quick_screenshot(request: Schema__Quick__Screenshot__Request) -> Schema__Quick__Screenshot__Result` (same shape, returns `png_bytes` decoded from the INLINE artefact's base64 payload plus `timings`). Both wrap the core service call in `try/except` that emits `HTTPException(502, last-1800-chars-of-traceback)` so callers see the real failure instead of osbot-fast-api's `"{Type}: {msg}"` squash.
- `service/Action__Runner` — Layer-0 single-action execution (Phase 2.10 Slice B). `execute(request: Schema__Action__Request) -> Schema__Action__Response`: looks up session (404 if missing), resolves trace_id (falls back to session's), resolves page via `get_or_create_page(browser)` (lazy context + page creation because a freshly launched browser has neither), resolves capture_config (request override -> session default), parses the step dict via `sequence_dispatcher.parse_single_step`, validates via `request_validator.validate_step` (raises 422 on JS-allowlist or sink-compat reject), executes via `step_executor.execute`, records on `session_manager.record_action` (bumps counters + collects artefact refs), returns `Schema__Action__Response`. Spec §4.3 called `sequence_dispatcher.execute_step(...)` which doesn't exist yet; Action__Runner calls `Step__Executor` directly instead, keeping `Sequence__Dispatcher`'s responsibility narrow (wire → typed schema).
- `service/Sequence__Runner` — Layer-3 multi-step execution (Phase 2.10 Slice C + v0.1.13 clean-state refactor). `execute(request: Schema__Sequence__Request) -> Schema__Sequence__Response`: resolves sequence_id + trace_id (auto-generate when absent); `resolve_session` returns `(session_id, playwright_start_ms, browser_launch_ms)` — named-session lookup (404 when missing + no `browser_config` fallback; 422 when neither set) reports zero boot timings, else ad-hoc launch via `browser_launcher.launch` + `browser_launcher.register(session_id, result)` + `session_manager.create` replaying `validate_session_create` + `credentials_loader.apply`; parses all step dicts via `sequence_dispatcher.parse_steps`; rejects duplicate step ids via `request_validator.validate_step_ids_unique`; resolves the active page via `get_or_create_page`; **wraps the entire step loop in try/finally** so `close_session_after` teardown (`session_manager.close` + `browser_launcher.stop(session_id)`) runs even when a step raises; walks the queue validating + executing each step through `step_executor.execute` and recording every result on `session_manager.record_action`; after a failure with `halt_on_error=True` the remaining steps are emitted as `SKIPPED`; derives `Enum__Sequence__Status.{COMPLETED,FAILED,PARTIAL}` (`COMPLETED` on zero failures, `FAILED` on failure+halted, `PARTIAL` on failure+continued); assembles a `Schema__Sequence__Timings` block (`playwright_start_ms`, `browser_launch_ms`, `steps_ms`, `browser_close_ms`, `total_ms`) and returns `Schema__Sequence__Response` with aggregated counters, cumulative artefact refs drawn from every executed step, post-run `session_info`, and the `timings` block.
