### Service classes (11 of 12)
- `service/JS__Expression__Allowlist` ‚Äî deny-all by default; exact-match gate for the EVALUATE step.
- `service/Request__Validator` ‚Äî single home for cross-schema validation (spec ¬ß9). Methods: `validate_session_create`, `validate_step`, `validate_sink_configs`, `validate_step_ids_unique`, `validate_sequence_timeout`, `reject`. Raises `HTTPException(422)` carrying `Schema__Error__Response`.
- `service/Capability__Detector` ‚Äî env-driven deployment target detection + per-target `Schema__Service__Capabilities` profiles (lambda, claude_web, laptop, ci, container). Builds `Schema__Service__Info` for `/info` and `Schema__Health__Check` for `/health`.
- `service/Session__Manager` ‚Äî in-memory session lifecycle (create/get/close, capture config, start time, artefact accumulation, action counters, active-session listing, healthcheck). No Playwright calls; Browser objects are held as opaque refs.
- `service/Sequence__Dispatcher` ‚Äî parse surface (`parse_single_step`, `parse_steps`) wrapping `dispatcher/step_schema_registry.parse_step`. `execute_step` deferred to Phase 2.5 (needs `Step__Executor`).
- `service/Artefact__Writer` ‚Äî sink routing for INLINE (base64), VAULT, S3, LOCAL_FILE. Assembles `Schema__Artefact__Ref` with size + 10-hex sha256 identity hash. `write_bytes_to_local` fully implemented; `write_bytes_to_vault` / `write_bytes_to_s3` / `read_from_vault` / `write_to_vault` are subclass-overridable seams (real vault + osbot-aws clients not yet wired). Typed capture wrappers `capture_screenshot(data, sink_config)` / `capture_page_content(...)` / `capture_pdf(...)` (Phase 2.9) pin the artefact type so Step__Executor callers don't repeat it; each returns `None` when the sink_config is disabled.
- `service/Credentials__Loader` ‚Äî vault ‚Üî Playwright `BrowserContext` glue. `apply(session_id, session_manager, credentials)` loads cookies / storage state via the writer's vault seams and sets `extra_http_headers`. `save_state_to_vault` persists state back. No direct vault client; no `page.*` calls (context-level only).
- `service/Browser__Launcher` ‚Äî real Chromium process lifecycle. `launch(browser_config)` spawns via `sync_playwright().chromium.launch(...)`; lazy runtime start; per-session `browsers` registry; `stop(session_id)` / `stop_all()`; healthcheck. `ENV_VAR__CHROMIUM_EXECUTABLE` escape hatch for pre-installed Chromium builds. Only LOCAL_SUBPROCESS + CHROMIUM supported today (CDP_CONNECT, BROWSERLESS, Firefox, WebKit raise `NotImplementedError`). The only non-`Step__Executor` class permitted to import `playwright.sync_api`.
- `service/Step__Executor` ‚Äî **only class (alongside `Browser__Launcher`) permitted to import `playwright.sync_api`**. `execute(page, step, step_index, capture_config)` dispatches on `step.action`. First-pass handlers (Phase 2.9): `execute_navigate` / `execute_click` / `execute_fill` / `execute_screenshot` / `execute_get_content` / `execute_get_url`. Screenshot and GET_CONTENT bytes are handed to `artefact_writer.capture_*()` ‚Äî Step__Executor never writes to a sink directly. Every handler catches `Exception`, times the step, and returns a uniformly-shaped `Schema__Step__Result__*` (PASSED or FAILED with error_message). Deferred handlers (`execute_press / select / hover / scroll / wait_for / video_start / video_stop / evaluate / dispatch_event / set_viewport`, 10 total) raise `NotImplementedError` with a `Phase 2.11` marker rather than falling through to AttributeError. Helpers: `passed_result`, `failed_result`, `resolve_id` (falls back to `Step_Id(str(step_index))`), `filter_refs` (drops None refs from disabled sinks), `now_ms` (wall-clock seam for tests).
- `service/Playwright__Service` ‚Äî thin orchestrator. Composes Capability__Detector + Session__Manager + Browser__Launcher + Credentials__Loader + Artefact__Writer + Request__Validator + **Action__Runner**. `setup()` primes detection (idempotent) and shares Session__Manager + Capability__Detector + Request__Validator into the `Action__Runner` instance so the runner sees live service state instead of default-constructed copies. Health methods: `get_service_info()`, `get_capabilities()`, `get_health()`. Session methods (Phase 2.10 Slice A): `session_create()` / `session_list()` / `session_get()` / `session_save_state()` (None ‚Üí 404) / `session_close()` (None ‚Üí 404) / `generate_trace_id()`. Action method (Phase 2.10 Slice B subset): `execute_action(request)` delegates to `Action__Runner.execute()`. Sequence method still outstanding (Slice C).
- `service/Action__Runner` ‚Äî Layer-0 single-action execution (Phase 2.10 Slice B). `execute(request: Schema__Action__Request) -> Schema__Action__Response`: looks up session (404 if missing), resolves trace_id (falls back to session's), resolves page via `get_or_create_page(browser)` (lazy context + page creation because a freshly launched browser has neither), resolves capture_config (request override -> session default), parses the step dict via `sequence_dispatcher.parse_single_step`, validates via `request_validator.validate_step` (raises 422 on JS-allowlist or sink-compat reject), executes via `step_executor.execute`, records on `session_manager.record_action` (bumps counters + collects artefact refs), returns `Schema__Action__Response`. Spec ¬ß4.3 called `sequence_dispatcher.execute_step(...)` which doesn't exist yet; Action__Runner calls `Step__Executor` directly instead, keeping `Sequence__Dispatcher`'s responsibility narrow (wire ‚Üí typed schema).

### Docker infrastructure (Phase 2.8)
- `docker/Docker__SGraph_AI__Service__Playwright__Base` ‚Äî shared init for all four Docker classes. `setup()` instantiates `Create_Image_ECR` against `docker/images/sgraph_ai_service_playwright/` and `Deploy_Lambda` against `fast_api.lambda_handler.run`. Helpers: `api_docker()`, `image_uri()` (`:latest`), `repository()`, `path_docker()`, `path_dockerfile()`, `lambda_function()`. Module constants: `IMAGE_NAME='sgraph_ai_service_playwright'`, `LOCAL_PORT=8888`, `CONTAINER_PORT=8000`. (Spec uses hyphenated `SGraph-AI`; Python normalises to `SGraph_AI` per CLAUDE.md ¬ß16.)
- `docker/Build__Docker__SGraph_AI__Service__Playwright` ‚Äî `build_docker_image()` stages the dockerfile + requirements.txt + whole package source in a tempdir, then calls the docker SDK `client.images.build()` **directly** (bypassing osbot_docker's `Docker_Image.build()` because it's `@catch`-decorated and swallows `BuildError`/`APIError` into a `{'status': 'error'}` dict that was silently slipping past the `assert result is not None` check). Passes `dockerfile='dockerfile'` explicitly because the daemon defaults to `Dockerfile` (capital D, case-sensitive on Linux); returns `{'status': 'ok', 'image_id', 'tags'}` which the test verifies contain the ECR URI. Also exposes `create_container()` / `start_container()` / `created_containers()` (container lifecycle), `dockerfile()` (file content), `image_architecture()`.
- `docker/ECR__Docker__SGraph_AI__Service__Playwright` ‚Äî `ecr_setup()` (idempotent repo create), `publish_docker_image()` (runs `check_for_docker_config_json()` first, then `push_image()`). `check_for_docker_config_json()` carries forward the Docker Desktop credsStore workaround (`{'credsStore': 'desktop'}` breaks the Python Docker API ECR login ‚Üí delete the file).
- `docker/Lambda__Docker__SGraph_AI__Service__Playwright` ‚Äî `create_lambda(wait_for_active)` is an **upsert** that builds Lambda at `memory_size=5120 MB` (osbot_aws reads `memory_size`, **not** `memory` ‚Äî wrong attr silently falls back to 512 MB and Playwright OOMs at cold start with `Runtime.ExitError`), `architecture=x86_64`, `timeout=300s`, wires env vars; if the function already exists it calls `update_lambda_image_uri()` + `update_lambda_configuration()` in place, otherwise `create()`. **Never deletes the function** ‚Äî CloudFront (`dev.playwright.sgraph.ai` ‚Üí `*.lambda-url.<region>.on.aws`) is wired to the URL host UUID, and deleting the function (or its URL config) regenerates that UUID and breaks the origin. `set_lambda_env_vars()` propagates seven env vars from the process env to the Lambda env (five `SG_PLAYWRIGHT__*` + two `FAST_API__AUTH__API_KEY__*` for the middleware) and pins `SG_PLAYWRIGHT__DEPLOYMENT_TARGET='lambda'`. `create_lambda_function_url()` preserves any existing URL config (so CloudFront's origin stays stable) and only calls `function_url_create_with_public_access()` the first time (statement 1: `lambda:InvokeFunctionUrl` + `StringEquals lambda:FunctionUrlAuthType=NONE`); it then always enforces statement 2 via boto3 `add_permission(InvokedViaFunctionUrl=True)` (statement 2: `lambda:InvokeFunction` + `Bool lambda:InvokedViaFunctionUrl=true`) ‚Äî AWS requires **both** statements for public Function URLs created after October 2025 (osbot_aws only emits the first; without the second AWS returns 403 at the URL). `update_lambda_function()` + `function_url()` for post-deploy ops. Tuning constants: `LAMBDA_MEMORY_MB=5120`, `LAMBDA_ARCHITECTURE='x86_64'`, `LAMBDA_TIMEOUT_SECS=300`.
- `docker/Local__Docker__SGraph_AI__Service__Playwright` ‚Äî label-based container reuse (`source=local__docker__sgraph_ai_service_playwright`). `create_or_reuse_container()` / `delete_container()`, `GET()` / `POST()` helpers via `requests`, `local_url()` prefixes `http://localhost:8888`, `uvicorn_server_running()` + `wait_for_uvicorn_server_running()` poll container logs for `Uvicorn running on `. Module constant: `LABEL_SOURCE`.
